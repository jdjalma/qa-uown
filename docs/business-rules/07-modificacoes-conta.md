Rewind/Replay, settlement, invoice modification, frequency change, due date move, approval change, additional lease and FPD adjustment.

## 17. Rewind/Replay (Payment Correction Engine)

### What It Is

Mechanism to correct or restructure payment allocations. It works like "undo all allocations, then redo from scratch in the correct order".

### What It's For

When something changes after payments have been allocated (reversal, frequency change, schedule regeneration), the allocations may no longer be correct. Instead of fixing them surgically, the system undoes everything and redoes it.

### How It Works

| Phase | Action |
|------|------|
| **Rewind** | Reverses allocations of all PAID payments. If the account was paid off (PAID_OUT, etc.), it returns to ACTIVE |
| **Replay** | Reapplies all payments in chronological order, re-evaluating EPO, allocation, fees |

### When It's Used

- Payment reversal (returned check, NSF)
- Frequency change
- Schedule regeneration
- EPO payoff (reallocation to the EPO receivable)

## 35. Settlement (Payoff Agreement)

### What It Is

Settlement is the **debt negotiation** process where UOwn accepts an amount lower than the total owed to close the account. Typically used for severely delinquent accounts.

### What It's For

Recovers part of the value from accounts that would likely be charged-off (total loss). It's better to receive part than nothing.

### How It Works

1. **An agent or bot (Skit.ai) calls the delinquent customer**
2. **Offers a payoff agreement** with a reduced amount (e.g. owes $1,500, offer of $800)
3. **The customer accepts** and provides credit card details
4. **The transaction is processed** as a CC SALE with the flag `isSettlementPayment = true`
5. **If approved:** The account changes to `SETTLED_IN_FULL`, an email is sent
6. **If the payment is reversed:** The account returns to ACTIVE

### How to Enable/Trigger

- **Via human agent:** TMS endpoint `makeCreditCardPayment` with the settlement flag
- **Via Skit.ai bot:** Automatically through an offer file generated by the `createSkitDelinquentOfferFileSweep` sweep
- **Via API:** `POST /uown/los/settleApplication` with `ApplicationSettleRequest`

### Pre-Conditions for Settlement via `settleApplication`

| Condition | Rule |
|----------|-------|
| All items delivered | `numberOfItemsDelivered == numberOfItems` |
| No item cancelled | Items with status `CANCELLED` block the settlement |
| No item already paid | Items with status `PAID` indicate the account is already paid off |
| Partial settlement | The flag `partialSettlement: boolean` allows settlement of a subset of the items |

### Concurrency Control

The system uses the Hazelcast Settlement Request Map to prevent duplicate simultaneous offers for the same lead.

### What to Verify

- Account with status `SETTLED_IN_FULL` and `settledInFullDateTime` populated
- "Settled in Full" email sent
- CC transaction with `isSettlementPayment = true`

## 36. Invoice Modification

### What It Is

Service that allows modifying the terms of a lease **after signing** by creating a new lease reference.

### What It's For

When a settlement agreement is accepted or the lease terms need to change, the system creates a new lead with the modified terms linked to the original.

### How It Works

1. Validates that the lead is SIGNED or beyond
2. Cancels the associated account (if it exists)
3. If FUNDED: Creates a refund request to the merchant
4. Creates a new application from the original data
5. Copies payment methods (CC/ACH) to the new lead
6. Links the new lead to the settlement offer

### How to Trigger

```
POST /uown/los/modifyInvoiceForLead/{leadPk}
```
**Response:** `ModifyInvoiceResponse` with `newLeadPk`

### What to Verify

- New lead created with the correct status
- Previous account cancelled
- Payment methods copied

## 41. Payment Frequency Modification

### What It Is

Service that allows **changing the payment frequency** of a lease (e.g. weekly to bi-weekly) and records an audit trail. Reachable both via the administrative interface AND as a **customer self-service flow** on the Website portal (`/payment-frequency`, sidebar Payments → Payment Flexibility → "Change your payment schedule" → VIEW MY OPTIONS).

### What It's For

Customers may need to change the frequency to align with their paycheck cycle.

### Recorded Data

| Field | Description |
|-------|-----------|
| `fromFrequency` | Previous frequency (WEEKLY, BI_WEEKLY, etc.) |
| `toFrequency` | New frequency |
| `modDate` | Modification date |
| `agentUsername` | Who made the change — for a **customer self-service Save on the Website portal, this is the literal string `"customer portal"`** (NOT `SYSTEM`, NOT the customer's name, NOT a UI username) |

### How to Trigger

Via the administrative interface or the frequency change endpoint. The `ChangeFrequencyService` automatically calls the audit record. **Customer self-service:** `POST /uown/svc/changePaymentFrequency`, payload `{accountPK, newFrequency, firstDueDay, secondDueDay}` (Semi-Monthly destination) or `{accountPK, newFrequency:"BI_WEEKLY", nextPayDate}` (Bi-Weekly destination — no day-of-month fields). Response is the full updated `SchedSummaryInfo` row.

### Website self-service UI rules

- The "Payment Frequency" dropdown **excludes the current frequency** and **never lists Monthly** as an option, regardless of the account's current frequency.
- Selecting **Semi-Monthly** as the destination reveals two extra pickers: **First Payment Day** (exactly days 1–17) and **Second Payment Day**, whose option set is constrained to a **14–20 day gap** from the selected First Payment Day. Both pickers validate purely by option-set truncation (invalid days are simply absent, not offered-and-rejected).
- Selecting **Bi-Weekly** as the destination reveals a **"When is your next payday?"** date field instead of the day pickers; only dates from tomorrow through 15 days out are selectable.
- **Known defect (non-blocking, tracked for PO/dev triage):** for First Payment Day ∈ {14,15,16,17}, the Second Payment Day option set is truncated at calendar day 31 (`first+14`) instead of offering the full `first+14..first+20` range — day 31 does not exist in April/June/September/November/most Februaries, yet SAVE FREQUENCY remains enabled with it selected. Confirmed live for First=17 across 7 independent reproductions (sandbox); the full First=14/15/16 matrix remains unconfirmed.
- Repeated frequency changes on the same account (verified live up to 6 consecutive Saves / 3 full Bi-Weekly↔Semi-Monthly cycles) do **not** hit any enforced limit — the backend tracks a `frequencyChanges` counter but no gate rejects the Save because of it. There is no product-side "maximum number of changes" as of this writing (2026-07-01); the historical production drift bug (schedule derived to 12/30/2027, ACCT 545697) was traced to a recalculation defect, not a missing counter limit, and was fixed via MR !286 (schedule recalculation logic) rather than a change-count cap.

### Impact

- The receivables schedule is **regenerated** (Rewind/Replay executed)
- Installment amounts change (more smaller installments or fewer larger ones)
- Change history recorded in `uown_frequency_mods`
- `uown_sv_activity_log` gets a `log_type='FREQUENCY_CHANGE'` row (`created_by='customer portal'` for self-service, `creation_source='USER_ACTION'`, notes e.g. `"Payment frequency changed from BI_WEEKLY to SEMI_MONTHLY"`), plus an `INFORMATION` row from the Rewind/Replay EPO receivable regen side effect

## 42. Due Date Move

### What It Is

Allows **postponing or bringing forward** receivable due dates by N days.

### What It's For

- **Skip payment:** Skip an installment in case of hardship
- **Payment arrangement:** Reorganize dates to align with paychecks
- **Alignment:** Adjust all future installments

### How to Trigger

```
POST /uown/svc/moveDueDatesByDays/{accountPk}?moveNumberOfDays=7&fromDueDate=2026-02-28
```

Or via TMS: `POST /uown/tms/moveDueDatesByDays`

### Recorded Data

| Field | Description |
|-------|-----------|
| `movedFromDueDate` | Original date |
| `movedByDays` | Number of days (positive = postpone) |
| `adjustmentType` | Type (SCHEDULE_SHIFT, etc.) |
| `agentUsername` | Who did it |

### Restrictions

- Can only adjust receivables with a past date
- All future installments are shifted by the same number of days

### What to Verify

- Receivables with updated dates
- Due date moves report (sweep `generateDueDateMovesReport`)

## 59. Approval Amount Override

### What It Is

Allows administrators to change a lead's credit approval amount after underwriting.

### What It's For

When the amount approved by UW is not appropriate (too high or too low), an admin can do a manual override with a justification.

### Prerequisites

| Condition | Required |
|----------|-------------|
| Non-null approval amount | Yes |
| Amount <= maximum limit per client type | Yes (default: $5,000) |
| UW data existing on the lead | Yes |
| Lead status `UW_APPROVED` or `CONTRACT_CREATED` | Yes |

### Maximum Limit per Client Type

Configurable via `max.approval.amount.{ClientType}` (e.g. `max.approval.amount.RETAIL`). General default: $5,000.

### What Happens on the Change

1. **UW Response updated:** `creditLimit` changes to the new value, `decision = "ACCEPT"`, adverse reasons cleared
2. **Old data preserved:** Previous UW data saved in the lead's notes as "Old UW data [JSON]"
3. **maxApprovalAmount:** Updated only if the new value > current value
4. **Alert created** with old/new values and the admin's comment
5. **LeadModification recorded** with type `APPROVAL_AMOUNT_CHANGE`

### How to Trigger

```
POST /uown/los/overrideApprovalAmount
Body: LeadApprovalAmountOverrideRequest { leadPk, approvalAmount, comment }
```

## 60. Additional Lease (Add Lease)

### What It Is

Allows a customer to add a new lease using the remaining credit from a previously funded lease.

### What It's For

Customers with remaining credit (open-to-buy) can finance additional products without needing a full new application.

### Prerequisites

| Condition | Required |
|----------|-------------|
| Original lead with status `FUNDED` (configurable) | Yes |
| Remaining credit available (`openToBuy > 0`) | Yes |
| ApplicationRequest retrievable from the original lead | Yes |

**Config:** `com.uownleasing.svc.service.AddLeaseService.valid.statuses.for.add.lease` (default: `"FUNDED"`)

### How It Works

1. **Validates the status** of the original lead
2. **Retrieves the cached ApplicationRequest** from the original lead
3. **Clears** previous invoice and item data from the request
4. **Configures vendor skips** (to avoid redundant checks):
   - `skip.sentilink`: false (default -- runs Sentilink again)
   - `skip.neustar`: true (default -- skips Neustar)
   - `skip.seon`: true (default -- skips SEON)
   - `skip.lexisnexis`: true (default -- skips LexisNexis)
5. **Creates a new application** via `SendApplicationService`
6. **If approved:**
   - The new lead gets a `refLeadPk` pointing to the original lead
   - The original lead is marked with `addedSecondLease = TRUE`
   - Activity logs created on both leads

### How to Trigger

```
POST /uown/los/addLease
```

## 68. First Payment Due Date Adjustment

### What It Is

Logic that ensures the first payment due date (FPD) is always a minimum distance in the future, automatically adjusting when necessary.

### What It's For

Prevents receivables from being created with due dates that have already passed or are too close, which would cause immediate delinquency.

### Adjustment Rule

1. The FPD must be at least **N days in the future** (config: `difference.in.days.between.first.receivable.next.pay.date`, default: 3)
2. If the FPD does not meet the minimum, it is pushed to the **next payment date** based on the frequency (from employment or from the payment frequency)
3. The adjustment is **recursive** -- it repeats until it finds a valid date in the future
4. SVC accounts have a separate config for the minimum interval

### Impact

- Affects receivable creation in both LOS and SVC
- Ensures the first payment always falls on a valid future date

## 70. Payment Reallocation

### What It Is

A Servicing portal feature that allows moving the partially paid amount of a receivable to another eligible receivable on the same account.

### What It's For

Correct incorrect allocations without needing to reverse and re-post payments. Useful when a payment was allocated to the wrong receivable.

### Validation Rules

| Rule | Detail |
|-------|---------|
| Maximum amount | `amount <= partialPaymentAmount` of the source receivable |
| Same type | The target receivable must have the same `receivableType` as the source receivable |
| Cannot be PAID_IN_FULL | Receivables with `allocationStatus = PAID_IN_FULL` cannot be a target |
| Cannot be the same | The source receivable cannot be the same as the target |
| Amount > 0 | The reallocated amount must be greater than zero |
| Comment required | Requires an explanation (max 500 characters) |

### Permission

Requires `reallocate_receivable` in the Servicing permissions model.

## 71. Reset Default Schedule (KORNERSTONE)

### What It Is

A feature exclusive to Kornerstone accounts (`company = 'KORNERSTONE'`) that allows resetting the payment schedule to the configured default.

### What It's For

Restores the default schedule parameters when previous modifications (frequency change, due date moves) have made the schedule irregular.

### Availability

Appears in the Servicing portal only when `company === 'KORNERSTONE'` — completely hidden for UOWN accounts.

## 72. Agent Attribution in the Modification Report (`agent_username`)

### What It Is

Every lead status change (`LEAD_STATUS_CHANGE`) is recorded in `uown_lead_modifications`. The `agent_username` column (UI "Agent Name" in `/modificationReport`, endpoint `POST /uown/los/getModifiedLeads`) must identify the human agent who triggered the action — or `SYSTEM` **exclusively** when the action was triggered by backend automation with no human actor.

### Rule

| Case | Expected `agent_username` |
|------|---------------------------|
| Human agent clicks an action in the portal (e.g. "Set to Expired", "Change to Signed") | the agent's real username (e.g. `jmendes.gow`) |
| GowSign/SignWell webhook callback (`CONTRACT_CREATED → SIGNED`, `SIGNED → SIGNED` re-sign) | `SYSTEM` (correct — no user session in the HTTP context) |
| Sweep/scheduler | `SYSTEM` (correct) |

- The agent's identity comes from an **HTTP header `username`** (NOT from JWT), read by `RequestFilter.java`. `ChangeLeadStatusService` is the **only** creator of `LEAD_STATUS_CHANGE` records.
- `ThreadAttributes.getUsername()` returns `"SYSTEM"` when the header is blank. A direct API call to `changeLeadStatus` **without** the `username` header writes `SYSTEM` — this is a test-simulation gap, NOT a backend bug.

### Implication for tests

A human status action MUST be exercised via the browser (inviolable rule #14): the portal SPA sends the `username` header. Assertion: `agent_username === '<agent>'` AND `!== 'SYSTEM'`, with a negative guard (no SYSTEM row for the fresh lead).

### Bug history (#1315, fixed R1.53.0)

Pre-fix, an outbound webhook corrupted the `ThreadLocal` BEFORE `agent_username` was read → blank username → `SYSTEM` written for real human actions. Fix (MR svc!1464 captures `agentName` at the start of the method; svc!1470 does save/restore of the `ThreadAttributes` around the webhook). Confirmed in qa2 + source code 2026-06-18.
