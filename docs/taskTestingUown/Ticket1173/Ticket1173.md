UOWN | Origination & Servicing | Visually differentiate Activity Log types using colors



Synopsis
Both Origination and Servicing contain Activity Log tables that display records generated automatically by the system, triggered by user actions, or manually added as notes. This ticket proposes implementing visual color differentiation for easier categorization.


Business Objective
Enhance clarity and speed in reviewing Activity Logs, allowing business and support teams to instantly recognize the type of log without reading the full text. This reduces interpretation errors, improves triage workflows, and strengthens operational efficiency.


Feature Request | Business Requirements
1. Apply distinct colors based on log type
Activity Logs must follow this visual color scheme:
A. Manual logs (typed by user/agent) – COLOR: RED (#dc3545)
      Includes notes, comments, and any text manually added.
      Indicates intentional documentation by the user/agent.
B. Logs generated by user actions – COLOR: GREEN (#28a745)
      Logs triggered by user operations, such as:
      Adding a new credit card
      Editing Merchant information
      Any user-driven action that produces an automatic log
      Indicates that a user action directly caused the log.
C. All other logs – COLOR: BLACK
      Includes system-generated logs not tied to manual input or direct user-triggered actions.
      This remains the standard color already used.
2. Consistent behavior across modules
      The color logic must be identical in both Origination and Servicing.
      All pages displaying Activity Logs should apply this pattern.
3. Maintain existing log behavior
      No changes to how logs are generated.
      Only the visual display should be updated based on the categories above.



Testing Steps

Overview
Activity logs are color-coded based on their creationSource to help distinguish between manual entries, user actions, system-generated logs, and API calls.
Color Mapping
RED - MANUAL_ENTRY
Logs manually created by users through the "Add New Log" modal.
Examples:
User clicks "Add New Log" and creates a log with type DEFAULT, INFORMATION, INTERNAL, etc.
Where to test:
Servicing: Account detail page → Activity Logs section → Add New Log button
Origination: Lead detail page → Activity Logs section → Add New Log button
GREEN - USER_ACTION
Logs automatically generated as a result of user actions in the UI.
Examples:
REVIEW logs
DATA_CHANGE logs triggered by user edits (e.g., updating account information, changing lead status)
CORRESPONDENCE logs (when user sends correspondence)
STATUS_CHANGE logs (when user changes account/lead status)
MERCHANT_DATA_CHANGE logs (when user updates merchant information)
PROGRAM_DATA_CHANGE logs (when user updates program information)
Any log created by an authenticated user (email address as username)
Where to test:
Edit account/lead information → Check for green DATA_CHANGE logs
BLACK - SYSTEM_GENERATED or API_CALL
Logs generated automatically by the system or external API calls.
Examples:
IMPORT logs (data imports, migrations)
Scheduled task logs (usernames like SWEEP, TASK, SCHEDULED_TASK)
System-generated logs (usernames like SYSTEM, AUTOMATED, BATCH)
API call logs (usernames ending with  API or starting with TMS-)
Logs with blank or SYSTEM username

Notes
Old logs without creationSource will default to BLACK
The creationSource is automatically detected based on the username pattern when not explicitly set

UOWN | Origination & Servicing | Diferenciar visualmente os tipos de Activity Log usando cores

**Sinopse**
Tanto Origination quanto Servicing contêm tabelas de Activity Log que exibem registros gerados automaticamente pelo sistema, acionados por ações do usuário ou adicionados manualmente como notas. Este ticket propõe implementar diferenciação visual por cores para facilitar a categorização.

**Objetivo de Negócio**
Aumentar a clareza e a rapidez na revisão dos Activity Logs, permitindo que as equipes de negócios e suporte reconheçam instantaneamente o tipo de log sem precisar ler o texto completo. Isso reduz erros de interpretação, melhora fluxos de triagem e fortalece a eficiência operacional.

---

## **Requisição de Funcionalidade | Requisitos de Negócio**

### **1. Aplicar cores distintas com base no tipo de log**

Os Activity Logs devem seguir o seguinte esquema visual de cores:

**A. Logs manuais (digitados pelo usuário/agente) – COR: VERMELHO (#dc3545)**

- Inclui notas, comentários e qualquer texto adicionado manualmente.
- Indica documentação intencional feita pelo usuário/agente.

**B. Logs gerados por ações do usuário – COR: VERDE (#28a745)**

- Logs acionados por operações feitas pelo usuário, como:

  - Adicionar um novo cartão de crédito
  - Editar informações do Merchant
  - Qualquer ação do usuário que produza um log automático

- Indica que uma ação do usuário causou diretamente o log.

**C. Todos os demais logs – COR: PRETO**

- Inclui logs gerados pelo sistema que não estão ligados a entrada manual ou ações diretas do usuário.
- Permanece a cor padrão já usada.

---

### **2. Comportamento consistente em todos os módulos**

- A lógica de cores deve ser idêntica em Origination e Servicing.
- Todas as páginas que exibem Activity Logs devem aplicar este padrão.

---

### **3. Manter o comportamento atual dos logs**

- Nenhuma alteração em como os logs são gerados.
- Apenas a exibição visual deve ser atualizada conforme as categorias acima.

---

## **Passos de Teste**

### **Visão Geral**

Os Activity Logs são coloridos com base em seu `creationSource` para ajudar a diferenciar entradas manuais, ações do usuário, logs gerados pelo sistema e chamadas de API.

---

### **Mapeamento de Cores**

### **VERMELHO – MANUAL_ENTRY**

Logs criados manualmente pelos usuários através do modal "Add New Log".

**Exemplos:**

- Usuário clica em "Add New Log" e cria um log com tipo DEFAULT, INFORMATION, INTERNAL etc.

**Onde testar:**

- Servicing: Página de detalhes da conta → seção Activity Logs → botão Add New Log
- Origination: Página de detalhes do lead → seção Activity Logs → botão Add New Log

---

### **VERDE – USER_ACTION**

Logs gerados automaticamente como resultado de ações do usuário na interface.

**Exemplos:**

- Logs de REVIEW
- Logs DATA_CHANGE gerados quando usuário edita algo (ex.: atualizar informações da conta, mudar status do lead)
- Logs de CORRESPONDENCE (quando o usuário envia correspondências)
- Logs de STATUS_CHANGE (quando o status da conta/lead é alterado)
- MERCHANT_DATA_CHANGE (quando atualiza merchant)
- PROGRAM_DATA_CHANGE (quando atualiza programa)
- Qualquer log criado por um usuário autenticado (username é um e-mail)

**Onde testar:**

- Editar informações de conta/lead → verificar logs DATA_CHANGE em verde

---

### **PRETO – SYSTEM_GENERATED ou API_CALL**

Logs gerados automaticamente pelo sistema ou por chamadas de API externas.

**Exemplos:**

- Logs de IMPORT (importações, migrações)
- Logs de tarefas agendadas (usernames como SWEEP, TASK, SCHEDULED_TASK)
- Logs do sistema (usernames como SYSTEM, AUTOMATED, BATCH)
- Logs de chamadas de API (usernames terminando com API ou começando com TMS-)
- Logs com username vazio ou SYSTEM

---

### **Notas**

- Logs antigos sem `creationSource` serão exibidos como PRETO.
- O `creationSource` é detectado automaticamente com base no padrão do username quando não definido explicitamente.

    public static LogCreationSource detectFromUsername(String username) {
        if (isBlankOrSystem(username)) {
            return LogCreationSource.SYSTEM_GENERATED;
        }

        String upperUsername = username.toUpperCase();

        if (isApiCall(username)) {
            return LogCreationSource.API_CALL;
        }

        if (isSystemGenerated(username, upperUsername)) {
            return LogCreationSource.SYSTEM_GENERATED;

        // Default to USER_ACTION for authenticated users (email addresses, regular usernames)
