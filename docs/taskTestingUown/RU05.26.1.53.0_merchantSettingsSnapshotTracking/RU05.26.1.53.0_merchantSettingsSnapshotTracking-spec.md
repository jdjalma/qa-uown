## Demand summary (one line)

On UW approval the system writes an immutable **lead snapshot** of the merchant's UW config (EPO5, EPO10, UW Pipeline, Fraud Threshold); on funding/account-creation it copies that into an immutable **account snapshot**; later merchant-config edits must never mutate either snapshot.

## Scope

**IN:**

- **Lead snapshot creation on UW approval** (AC-01 / AC-04) — the primary trigger; one row in `uown_los_lead_merchant_settings_snapshot` with all four UW values + timestamp. Why: core of the feature.
- **No lead snapshot on UW denial** (AC-01 negative) — denial branch must NOT write a snapshot. Why: confirms the write is gated on the `UW_APPROVED` engine path, not on any lead reaching the pipeline.
- **Account snapshot creation on funding** (AC-01 / AC-04) — copy of the four values into `uown_sv_account_merchant_settings_snapshot` linked to account+lead+merchant. Why: second half of the feature and currently **unproven in qa2 (0 rows)** — highest empirical risk.
- **Lead-snapshot immutability when merchant settings change after approval** (AC-02 / AC-03) — editing each of the four settings does not mutate the existing lead row. Why: the whole compliance value proposition is immutability.
- **Account snapshot derives from lead snapshot, not live merchant** (AC-02 / AC-04) — when merchant config changes between approval and funding, the account snapshot must equal the lead snapshot, not the new live value. Why: this is the subtle correctness invariant (TC-05) that a naive implementation would get wrong by re-reading the merchant.
- **No account snapshot when no lead snapshot exists** (TC-06) — account still created; account snapshot silently skipped. Why: explicit AC; also the negative baseline for the account-snapshot write path.
- **Merchant `terraceFinance` (pk 26 / `OL90202-0001`)** as the single fixture merchant, with the four values re-asserted at runtime (`epo5=true, epo10=true, uw_pipeline='Test', fraud_threshold=1`). Why: confirmed live; the all-null clones (pk 3/54/82) would make "captures all four values" and the `uw_pipeline`/`fraud_threshold` immutability outline meaningless.

**OUT:**

- **UI rendering of the snapshot** — there is no portal screen that displays the snapshot tables (audit/backend artifact). Why: no UI affordance exists to render → Rule #14 exception applies (see Test Strategy §1). If a snapshot-viewer screen is later added, this becomes IN.
- **Brand parity (UOWN + Kornerstone) for the snapshot itself** — see Test Strategy §3. Why: the snapshot captures merchant UW config; the write hook is brand-orthogonal and `terraceFinance` is a `TERRACE_FINANCE` client type (a UOWN-pool merchant). Adding a KS merchant tests the same hook against a different config row, not a different code path. Documented exception to `ssn-test-modalities` §5, justified below.
- **16m / Second Look / multi-modality coverage** (`ssn-test-modalities` modalities B/C) — the snapshot fires on **every** approval regardless of term; `program_pk` is captured but term selection is not part of the snapshot logic under test. Why: modality is orthogonal to the snapshot write; covering 13m approval is sufficient to exercise the hook. Documented per `ssn-test-modalities` §4 "Quando omitir modalidade".
- **CC-payment / post-funding servicing flows** — not required to reach the account snapshot; `driveLeadToFunding` settles without CC. Why: qa2 `makeCreditCardPayments` returns 500 (caveat); avoid.
- **State / locale matrix** — snapshot has no state-dependent template behavior (unlike GoSign). Why: no evidence of per-state branching in the snapshot hook.
- **`program_pk` value assertion against a specific program** — captured as present/non-contradictory, not asserted to an exact program. Why: `terraceFinance` program mapping is not part of the AC; over-asserting would couple the test to merchant program drift.

**AMBIGUOUS / Questions for PO (see Open Questions for full list):**

- Is there a **non-DELETE** path to produce a fundable lead with no lead snapshot (import source KORNERSTONE/UOWN_V1, `addLease`, skip-UW)? Discovery says skip-UW still writes the snapshot, leaving DB DELETE as the only known route — which needs authorization.
- Does snapshot creation emit an **activity log / note** (Rule #13)? Not yet observed; must be confirmed empirically (see §2).
- Are there **other columns** in scope beyond the four (e.g., `program_pk` semantics, `agent`/`web_user_id` population)?

## Test Strategy

- **Overall approach:** **Hybrid — API setup + DB validation; UI for the merchant-setting CHANGE only.**
- **Environments:** qa2 (primary) + stg (DoD re-run). qa1 avoided per discovery (DV360/sendApplication instability history).
- **Suites to activate:** none of the dual-brand / signing-regression suites apply (see §3). This is a standalone task suite under `docs/taskTestingUown/`.

### §1 — Strategy per scenario + Rule #14 justification

The **snapshot itself is a backend audit artifact with no UI affordance** — no Origination/Servicing/Website/AMS screen renders `*_merchant_settings_snapshot`. Per Rule #14 exception (a) "admin/ops with no UI exposed" + (c) "cross-cutting DB validations", **API+DB is the legitimate, justified strategy for asserting snapshot rows.** This is NOT the Daniel's-Jewelers anti-pattern: there is no customer-visible render to miss; the oracle genuinely lives in the DB.

**BUT the merchant-setting CHANGE in AC-02/AC-03 (TC-03a–d) has a real UI** — the Origination "Merchant Setting" page (bulk-update, ref `.claude/oracles/1309-...`). Decision, weighing Rule #14 (UI-first) and Rule #18 (UI→API→DB discovery order):

- **TC-03a–d (immutability): drive the merchant-setting change via the Origination UI**, then assert the lead snapshot is unchanged in DB. Justification: the change is a real agent action with a UI affordance; Rule #14 says exercise it via browser; the bug class ("editing the merchant mutated the snapshot") is exactly the kind of side effect that a UI edit (which may run different update paths than a raw API PATCH) could trigger. The merchant edit is the *exercise*; the snapshot is the *oracle* (DB).
  - **qa2 caveat handling:** the merchant-setting page edits config rows; do NOT route this through `MerchantConfigurator.configureByName` (fails silently in qa2). The UI edit is a direct agent action on the page, not the configurator helper.
- **TC-05 (account snapshot after merchant change): also change Fraud Threshold via the Origination UI** between approval and funding, then fund via API. Same justification.
- All **setup** (create approved lead, drive to funding) is API — Rule #14 exception (b): setup that accelerates the test. The feature under test is the snapshot write, not the new-application flow, so `createPreQualifiedApplication` + `driveLeadToFunding` for setup is correct (not the `feedback_setup_via_ui_new_application` anti-pattern — that applies when the new-application flow *is* the feature).

| TC | Setup | Exercise | Oracle | UI? |
|----|-------|----------|--------|-----|
| TC-01 lead snapshot on approval | API: pre-qualified + approve (ending-in-non-9) | API approval | DB lead snapshot | No (justified §1) |
| TC-02 no snapshot on denial | API: `generateTestSSN(false)` | API denial | DB lead-snapshot absent | No (justified §1) |
| TC-03a–d immutability ×4 | API: approved lead + snapshot | **UI: edit setting on Origination Merchant Setting page** | DB lead snapshot unchanged | **Yes** |
| TC-06-baseline positive account snapshot | API: approved lead | API: `driveLeadToFunding` | DB account snapshot present + 4 values | No (justified §1) |
| TC-05 account derives from lead (merchant changed) | API: approved lead | **UI: edit Fraud Threshold** → API: `driveLeadToFunding` | DB account snapshot == lead snapshot, ≠ new live value | **Yes** (edit) |
| TC-07 no account snapshot w/o lead snapshot | API: approved lead → **DELETE lead row (AUTH-REQUIRED)** | API: `driveLeadToFunding` | DB account snapshot absent + account exists | No — `test.skip` until auth |
| TC-08 activity-log probe | piggyback on TC-01/TC-06-baseline | (none) | DB `uown_los_lead_notes` probe | No |

### §2 — Activity-log coverage (Rule #13)

Rule #13 requires every business action to have a corresponding log; snapshot creation is a business/audit action. **Whether the snapshot write emits a note is currently UNKNOWN** — the discovery did not observe one, and `uown_*_merchant_settings_snapshot` are themselves audit tables (they may BE the log).

Decision: **TC-08 — empirical probe, not an assumption.** During TC-01 and TC-06-baseline, after the snapshot row appears, query `uown_los_lead_notes` for a note correlated to the approval/funding window referencing the snapshot (e.g. `notes ILIKE '%snapshot%'` or a `[*Snapshot*Service]` prefix).

- If a note exists → record its exact prefix/content and make it a required assertion in TC-01/TC-06-baseline going forward (feed `activity-log-validation`).
- If NO note exists → state explicitly, with evidence (query + zero rows), classified as **`[OBSERVAÇÃO]` possible observability gap** (per `bug-classification` — single-env observation, not a confirmed bug). The snapshot tables themselves (`row_created_timestamp`, `agent`, `web_user_id`) serve as the audit trail; absence of a `lead_notes` entry is reported to PO/dev as a question, not failed as a blocker. This satisfies Rule #13's "spec without log coverage is incomplete" by covering the log *question* explicitly rather than silently skipping it.

This is the honest application of Rule #13 to a feature where the log location is itself undetermined: we assert what exists and surface what does not.

### §3 — Brand coverage (`ssn-test-modalities` §5) decision

`ssn-test-modalities` §5 makes UOWN+Kornerstone parity inviolable **for features that create applications**. This task's feature is **not** the application-creation flow — it is a backend snapshot of merchant UW config triggered by approval/funding. The relevant axiom (§2) is that **merchant capability is config-driven and brand-orthogonal.** The snapshot write hook reads `uown_merchant_program` / merchant config columns identically regardless of `Company`. A Kornerstone merchant would exercise the *same* hook against a *different config row* — testing data, not a distinct code path.

**Decision: brand coverage is OUT for this task, with documented justification (not a silent skip).** Rationale logged here per §4's "toda omissão DEVE ser explícita":
- The snapshot columns (`epo5/epo10/uw_pipeline/fraud_threshold/program_pk/merchant_pk`) have no brand-conditional logic.
- The discovery confirmed snapshots fire identically for diverse merchant types (PAY_TOMORROW all-null, Tire Agent, terraceFinance) — brand/client-type is not a branch.
- `terraceFinance` (TERRACE_FINANCE) is the AC-named fixture with the four known values pre-set.

**Open question to PO (informative):** if the implementation special-cases Kornerstone merchants anywhere in the snapshot write, add a KS-merchant variant of TC-01/TC-06-baseline. Until evidence of brand branching exists, single-merchant coverage is sufficient and brand is treated as orthogonal.

### §4 — Smoke vs full / parallelization

- **Smoke (blocker subset):** TC-01 (lead snapshot) + TC-06-baseline (account snapshot) — the two positive writes. ≤ generous-timeout budget given qa2 slowness.
- **Full:** all TCs including immutability ×4, TC-05, TC-08; TC-07 in `test.skip`.
- **Parallelization:** each TC creates its own fresh lead (fresh SSN) → independent and parallelizable. **Exception:** all TCs share `terraceFinance` config. TC-03a–d and TC-05 **mutate** merchant settings via UI → these **must run serial** relative to each other and to TC-01/TC-06-baseline (which assert the snapshot captured the *pre-edit* live values). Mark the merchant-mutating TCs `test.describe.serial`; positive-write TCs that only read merchant config can run before any mutation. Pass `skipMerchantPreflight: true` everywhere (qa2 `configureByName` silent-fail caveat) and re-assert the four merchant values via DB at runtime instead of relying on preflight auto-heal.

## Scenarios (prioritized)

> Test design techniques applied: **State Transition** (lead lifecycle → snapshot trigger points), **Decision Table** (which of the 4 settings is edited × snapshot unchanged), **Equivalence Partitioning** (approved vs denied UW outcome), **Error Guessing** (account snapshot re-reads live merchant — the TC-05 trap). SSN selection per `ssn-test-modalities`.

### TC-06-baseline (P0, HIGHEST RISK) — Positive account snapshot on funding

- **Technique:** State transition (UW_APPROVED → SIGNED → FUNDING → account created) + consequence oracle.
- **Persona:** UOWN support agent / compliance officer (backend audit; no customer-facing surface).
- **Run order:** FIRST. Gates TC-05 and TC-07.
- **Setup:** re-assert `terraceFinance` (pk 26) has `epo5=true, epo10=true, uw_pipeline='Test', fraud_threshold=1` (DB SELECT, fail-fast if drifted — do NOT auto-heal/mutate). Refresh Kount + GDS sweeps. `createPreQualifiedApplication({ merchant: terraceFinance, ssn: generateTestSSN(true), skipMerchantPreflight: true })` → poll approval (generous timeout, qa2 slow). Capture `leadPk`, `leadUuid`.
- **Exercise:** `driveLeadToFunding(api, merchant, ctx)` (SIGNED → settle → FUNDING). Then resolve `accountPk` via `SELECT pk FROM uown_sv_account WHERE lead_pk = $1`.
- **Validations:**
  - DB lead snapshot exists (precondition for the copy) — Q1 below.
  - DB account snapshot: exactly one row for `account_pk`, with `epo5=true, epo10=true, uw_pipeline='Test', fraud_threshold=1`, `lead_pk` = originating lead, `merchant_pk = 26`, `row_created_timestamp` present — Q2.
  - Account snapshot values **equal** the lead snapshot values (copy integrity).
  - Account created successfully (`uown_sv_account` row exists, no error).
  - Activity log: TC-08 probe (see §2).
- **Edge/pitfalls:** account snapshot has never fired in qa2 — if 0 rows after generous wait, this is a **`[HIPÓTESE]` candidate finding**, NOT auto-pass; reproduce once more in fresh data before any classification (Rule #10). `driveLeadToFunding` uses `settleApplication` (no CC) — avoids qa2 CC 500.

### TC-01 (P0) — Lead snapshot on UW approval, all four values

- **Technique:** Equivalence partitioning (valid/approved class) + consequence oracle.
- **Persona:** compliance officer (audit).
- **Setup:** re-assert terraceFinance four values (DB SELECT, fail-fast). Refresh Kount/GDS sweeps. `createPreQualifiedApplication({ merchant: terraceFinance, ssn: generateTestSSN(true), skipMerchantPreflight: true })`.
- **Exercise:** approval via the standard pre-qualified path (API). Capture approval timestamp `t0`.
- **Validations:**
  - DB: exactly one row in `uown_los_lead_merchant_settings_snapshot` for `lead_pk`, with `epo5=true, epo10=true, uw_pipeline='Test', fraud_threshold=1`, `merchant_pk=26`, `row_created_timestamp >= t0`.
  - "Exactly one" — `COUNT(*) = 1` (idempotency oracle; guards against double-write).
  - Activity log: TC-08 probe.
- **Edge cases covered:** all four columns asserted individually (AC-04); count assertion (idempotency).
- **Pitfalls:** merchant values were NULL until set by user — MUST re-assert at runtime, never assume persistence. Use `generateTestSSN(true)` not a fixed SSN (avoid spurious denial from stale sweeps; non-9 ending).

### TC-02 (P1) — No lead snapshot on UW denial

- **Technique:** Equivalence partitioning (invalid/denied class) + negative consequence oracle.
- **Persona:** compliance officer.
- **Setup:** re-assert terraceFinance values. `createPreQualifiedApplication({ merchant: terraceFinance, ssn: generateTestSSN(false), skipMerchantPreflight: true })` → poll for `UW_DENIED` / `appApprovalStatus=DECLINED` (deterministic, vendor-independent; **denied path can take >180s in qa2 → generous timeout**).
- **Validations:**
  - DB: **no** row in `uown_los_lead_merchant_settings_snapshot` for `lead_pk` (use `waitForRecordAbsent` with a settle window, then confirm `COUNT(*) = 0`).
  - Lead status is the denial variant (`UW_DENIED`), confirming engine-decline (not a pre-UW blacklist/state deny, which `ssn-test-modalities` distinguishes).
  - Activity log: not required to assert presence here (no snapshot action occurred); the denial's own logging is out of scope.
- **Pitfalls:** ending-in-9 is the only deterministic denial; do NOT use catalog-external SSNs (stale GDS/Kount sweep → spurious denial that would falsely "pass" this negative test for the wrong reason). Distinguish `UW_DENIED` from `DENIED`/`BLACKLIST_DENIED`.

### TC-03a–d (P1) — Lead-snapshot immutability under merchant-setting edits

- **Technique:** Decision table — row = which setting edited (`fraud_threshold`, `uw_pipeline`, `epo5`, `epo10`); expected = that column unchanged in snapshot AND all other columns unchanged.
- **Persona:** UOWN admin/agent on Origination "Merchant Setting" page (desktop 1440×900, internal portal, Rule #15).
- **Run order:** serial; AFTER TC-01/TC-06-baseline captured pre-edit values. Each sub-case ends by **restoring** the merchant value (or each uses a freshly approved lead + edits, then the suite restores terraceFinance to the contract values in teardown).
- **Setup (per sub-case):** approved lead via TC-01 setup → snapshot recorded with original value. Capture snapshot row.
- **Exercise:** **via Origination Merchant Setting UI**, change the one setting for terraceFinance to a different value (e.g. `fraud_threshold 1→5`, `uw_pipeline 'Test'→'Prod'`, `epo5 true→false`, `epo10 true→false`). Confirm the UI persisted the change (reload the Merchant Setting page → new value shown — consequence oracle, `check-points`).
- **Validations:**
  - DB: the lead snapshot row for `lead_pk` still shows the **original** value for the edited column.
  - DB: no other column in that snapshot row reflects the edit (assert all four against captured originals).
  - Cross-check: the live merchant config DID change (proves the edit was real, not a no-op) — distinguishes "immutable snapshot" from "edit silently failed".
- **Edge cases covered:** each of the four columns independently (a partial-update bug touching one column would be caught).
- **Pitfalls:** edits via UI, not `MerchantConfigurator.configureByName` (qa2 silent-fail). Teardown MUST restore terraceFinance to `epo5=true, epo10=true, uw_pipeline='Test', fraud_threshold=1` so later/parallel runs aren't poisoned (the edit is a real config mutation; leaving it drifted breaks TC-01/TC-06-baseline reruns). The merchant-config write here is part of the feature exercise (allowed), NOT a forbidden raw DB mutation.

### TC-05 (P0, HIGH RISK) — Account snapshot derives from lead snapshot, not live merchant

- **Technique:** Error guessing (the "re-read live merchant at funding" trap) + state transition.
- **Persona:** compliance officer.
- **Run order:** serial, after TC-06-baseline proves the account snapshot fires.
- **Setup:** approved lead (TC-01 setup) → lead snapshot with `fraud_threshold=1`. Capture lead snapshot.
- **Exercise:** **via Origination Merchant Setting UI**, change terraceFinance `fraud_threshold 1→5` (now live ≠ snapshot). Then `driveLeadToFunding` → resolve `accountPk`.
- **Validations:**
  - DB: account snapshot `fraud_threshold = 1` (the approval-time/lead-snapshot value), **NOT 5** (the live value).
  - DB: account snapshot `fraud_threshold == lead snapshot fraud_threshold`.
  - DB: neither lead nor account snapshot reflects `5`.
- **Edge cases covered:** the precise compliance invariant that a naive impl gets wrong.
- **Pitfalls:** restore terraceFinance to `fraud_threshold=1` in teardown. This TC is meaningless if TC-06-baseline hasn't proven the account snapshot writes at all → strict ordering.

### TC-07 (P0, AUTH-REQUIRED — implement as `test.skip`) — No account snapshot without a lead snapshot

- **Technique:** Negative consequence oracle + state transition (artificial no-snapshot state).
- **Persona:** compliance officer / dev (admin-override path the ticket describes).
- **Setup:** approved lead (TC-01 setup) → lead snapshot written. Resolve the lead snapshot row.
- **AUTH-REQUIRED step:** `DELETE FROM uown_los_lead_merchant_settings_snapshot WHERE lead_pk = $1`. **This is a raw DB DELETE → forbidden without explicit user authorization (CLAUDE.md Exception 2 / Rule #9). Implement the TC fully but guard it with `test.skip` and an inline comment citing the authorization requirement.** Do not run until the user grants per-operation authorization.
- **Exercise:** `driveLeadToFunding` → resolve `accountPk`.
- **Validations:**
  - DB: **no** row in `uown_sv_account_merchant_settings_snapshot` for `account_pk` (`waitForRecordAbsent` + `COUNT(*) = 0`).
  - DB: `uown_sv_account` row for `lead_pk` exists (account created successfully despite skipped snapshot).
  - Activity log / error: account creation produced no error.
- **Pitfalls:** asserting "absence" is only valid once TC-06-baseline proved the positive write — otherwise a permanently-broken write path would make this falsely green. Document that dependency. The DELETE is on a snapshot row created by THIS test's fresh lead (not a shared/pre-existing record) — minimizes blast radius once authorized.
- **Open question (carried):** confirm with dev whether a non-DELETE route exists (import source KORNERSTONE/UOWN_V1, `addLease`, a skip-UW config that genuinely bypasses the hook, or a pre-deploy lead). If yes, replace the DELETE with that route and remove the `test.skip`.

### TC-08 (P2, observability probe) — Activity log on snapshot creation

- **Technique:** Exploratory probe + conservative classification.
- Piggybacks on TC-01 and TC-06-baseline (no separate setup).
- **Validations:** query `uown_los_lead_notes WHERE lead_pk = $1 AND row_created_timestamp >= t0` for any note referencing the snapshot/UW-config write.
  - Note present → capture exact prefix/content; promote to required assertion in TC-01/TC-06-baseline; feed `activity-log-validation`.
  - Note absent → report `[OBSERVAÇÃO] possible observability gap` with the zero-row query as evidence; raise as PO/dev question; do NOT fail the pipeline (snapshot tables' own timestamp/agent columns are the audit trail). See §2.

## Exact DB validation queries

> All read-only (SELECT) — allowed without authorization. The single DELETE (TC-07) is AUTH-REQUIRED and isolated below. Use `db.waitForRecord` / `db.waitForRecordAbsent` (signature: `(table, whereClause, params, timeoutMs?)`) for async waits; `db.queryOne` / `db.query` for value assertions. Re-confirm column names against `information_schema` once at suite start (db-polling-pattern pitfall #7) — names below are from the confirmed discovery probe.

```sql
-- [terraceFinance config re-assert — run before each positive TC; fail-fast, do NOT mutate]
SELECT epo5, epo10, uw_pipeline, fraud_threshold
  FROM uown_merchant            -- merchant pk 26 / ref_merchant_code 'OL90202-0001'
 WHERE pk = 26;
-- expect: epo5=true, epo10=true, uw_pipeline='Test', fraud_threshold=1
-- (verify actual merchant table/column names at suite start; discovery referenced merchant config cols)

-- [TC-01 / TC-06-baseline / TC-03 / TC-05 — lead snapshot read]
SELECT pk, epo5, epo10, uw_pipeline, fraud_threshold,
       merchant_pk, program_pk, row_created_timestamp
  FROM uown_los_lead_merchant_settings_snapshot
 WHERE lead_pk = $1
 ORDER BY pk DESC;

-- [TC-01 idempotency — exactly one]
SELECT COUNT(*) AS n
  FROM uown_los_lead_merchant_settings_snapshot
 WHERE lead_pk = $1;
-- expect n = 1

-- [TC-02 — lead snapshot ABSENT after denial]
SELECT COUNT(*) AS n
  FROM uown_los_lead_merchant_settings_snapshot
 WHERE lead_pk = $1;

SELECT pk FROM uown_sv_account WHERE lead_pk = $1;

-- [TC-06-baseline / TC-05 — account snapshot read]

-- [TC-06-baseline — exactly one account snapshot]

  FROM uown_sv_account_merchant_settings_snapshot

-- expect n = 0

-- [TC-07 — account still created]

-- expect 1 row, no error
```

What changed is **the governing rule, not the technical analysis**. [[ssn-test-modalities]] §5 makes UOWN+Kornerstone parity **inviolable for any feature that creates applications** — and this suite *does* create applications (`createPreQualifiedApplication` → approval → funding) to reach the snapshot triggers. The §5 checklist item "Cada CT Kornerstone valida `uown_sv_account.company='KORNERSTONE'`?" is itself a brand-cross-cutting oracle the UOWN-only suite cannot provide. The user invoked §5 explicitly; "brand-orthogonal hook" is an argument about the *write*, not a waiver of the *parity rule* that governs application-creating suites. Therefore: **brand coverage is now IN**, scoped to re-running the two positive-write TCs (lead snapshot on approval, account snapshot on funding) against a KS merchant, plus the mandatory `company='KORNERSTONE'` cross-check.

### Decision: KS immutability variant — OMIT (justified)

Considered: a `TC-KS-03` that edits a KS1011 setting via the Origination Merchant Setting UI and asserts the KS lead snapshot is unchanged (the KS analogue of UOWN TC-03a–d).

**Omit, because:**
1. **Immutability is brand-orthogonal.** TC-03a–d already proves, per-column via decision table, that a merchant-setting edit does not mutate an existing snapshot. The immutability invariant lives in the snapshot table's write/no-update logic, which has **no `Company` branch** (the original §3 analysis holds for this sub-claim). Spending that on a brand-orthogonal invariant is poor risk allocation ([[risk-based-prioritization]] 80/20).
