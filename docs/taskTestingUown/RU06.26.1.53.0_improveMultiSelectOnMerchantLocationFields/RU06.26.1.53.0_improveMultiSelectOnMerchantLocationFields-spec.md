## PO Extension Contract (for qa-implementer) — loaded [[page-object-pattern]]

**Reuse, do not recreate (rule #2).** Extend `MerchantLocationFilterPO` (`src/pages/origination/merchant-location-filter.po.ts`). The existing #1292 methods assume **close-on-pick** and force-close via Escape — they are correct for the #1292 spec and MUST NOT be modified or broken. Add NEW methods for the stay-open world; the existing `tests/e2e/origination/multi-select-filters.spec.ts` must still pass.

### Why existing methods can't be reused for stay-open
- `selectOption()` (lines ~259-298): explicitly waits for the portal to become **hidden** after a pick (`waitFor({state:'hidden'})`) and force-closes via `closeDropdown()` (Escape). Under the new behavior the menu stays open → that hidden-wait would either no-op or mask the new behavior. It also reopens the dropdown before each pick (`openDropdown` at top of `selectOptions` loop), which is the very thing this task makes unnecessary.
- `getCheckedOptionNames()` / `listAvailableOptions()` (lines ~395-463): they open AND **close** the dropdown as a side effect → cannot be used to read order/checked-state WHILE the menu must remain open mid-pick.
- There is no method to assert open-state or to read rendered option order.

### New methods to add (signatures + what each asserts)

```ts
/** AC1/AC2 — true if the menu portal is currently visible. No side effect (does NOT open/close). */
async isDropdownOpen(): Promise<boolean>
//   impl: SELECTORS.filterMenuPortal .isVisible({ timeout }) — no Escape, no reopen.

/**
 * AC1/AC2/AC4 — pick options KEEPING the dropdown open between picks (the stay-open flow).
 * Opens the dropdown ONCE, then for each option: type-to-narrow (roster is 6885),
 * click the narrowed top option, assert the menu is STILL open, assert checkbox :checked.
 * Does NOT reopen, does NOT Escape. Leaves the menu open for order assertions.
 */
async selectOptionsKeepingOpen(label: string, options: string[]): Promise<void>
//   asserts internally: after each click isDropdownOpen()===true; the option's
//   input[type=checkbox] is :checked. Throws if menu closed (that = stay-open regression).

/**
 * AC3 — ordered list of currently-rendered option texts WITHOUT closing the menu.
 * Returns the leading window (e.g. first 20) so the test can assert selected ⊆ top.
 */
async getRenderedOptionOrder(label: string, limit = 20): Promise<string[]>
//   reads SELECTORS.filterOption rows in DOM order; NO openDropdown/closeDropdown.

/** AC4 — checked-state of a specific option by exact text, read from native checkbox. */
async isOptionChecked(label: string, optionText: string): Promise<boolean>
//   oracle: option row -> input[type="checkbox"]:checked  (NOT aria-selected — it is null in stg).

/**
 * Helper — type into the combobox to narrow the 6885-roster to a target, return whether
 * exactly one (or the exact match) is now at the top. Needed because the roster is too
 * large to scroll. Reuses the existing controlByLabel anchor + an input lookup.
 */
async narrowTo(label: string, term: string): Promise<string[]>  // returns visible option texts after typing
```

Notes for the implementer:
- Keep selectors in `SELECTORS` (rule); the needed ones already exist (`filterMenuPortal`, `filterOption`, `multiSelectFilterButton`, `filterClearIndicator`). Add only a combobox-input selector if one isn't reusable from `controlByLabel`.
- CSS-Module hash (`index-module_customOptionStyles__CSG9m`) is build-drift-prone (pitfall #47/#50): rely on `SELECTORS.filterOption`'s prefix match + visible text, never the exact hash; validate live before declaring stable (run ≥2×, per [[dom-investigation]] pitfall).
- `aria-selected` is `null` in stg — do NOT use it as the checked oracle; use the native checkbox.

## Scenarios (prioritized) — loaded [[test-design-techniques]]

Techniques: AC1/AC2/AC3 are **state-transition** on the dropdown (closed→open→open-after-pick→open-after-2nd-pick) + **sequence/order** assertion. AC4 = **decision** (checked vs unchecked). AC5 = **filter-results equivalence** (rendered rows ⊆ selected set).

### CT-01 — Leads: dropdown stays open, multi-pick without reopen, selected-on-top (P0, @smoke)
- **Technique:** state transition (open-state across 2 sequential picks) + order assertion + checkbox decision.
- **Persona:** UOWN/merchant **agent** on Origination desktop (1440×900) filtering the Leads list — wants to tick several merchants fast without the dropdown snapping shut each time ([[user-journey-perspective]]: the pain point is exactly "reopening between every pick").
- **Page:** Leads (`navigateAndWaitForTable`).
- **Setup:** navigate; open Filters panel; open Merchant dropdown. No data created (read-only).
- **Steps:**
  1. Open Merchant dropdown; assert `isDropdownOpen()===true`.
  2. `narrowTo('Merchant','101 Mobility')` → pick it (click). Assert `isDropdownOpen()===true` AFTER the pick (AC1).
  3. `isOptionChecked('Merchant','101 Mobility')===true` (AC4).
  4. WITHOUT reopening, `narrowTo` a second distinct merchant → pick it. Assert menu still open (AC2) and 2nd option checked (AC4).
  5. `getRenderedOptionOrder('Merchant')` → assert BOTH selected merchants occupy the leading indices (top of list), ahead of any unselected option (AC3).
- **Validations:**
  - UI: `isDropdownOpen()` true after pick #1 and pick #2; selected count text "2 items selected"; both checkboxes `:checked`; selected texts at indices 0..1 of rendered order.
  - DB: none (read-only filter — see rule exceptions).
  - Activity log: N/A (navigation/read action, not a business action — rule #13 exception, reaffirmed below).
- **Edge cases covered:** multi-pick without reopen (the headline); reorder after the 2nd pick (most-recent-first observed in gate).
- **Pitfalls considered:** CSS-Module hash drift (use prefix+text); 6885-roster (type-to-narrow, never scroll); do NOT use `aria-selected` (null in stg); do NOT let the existing close-on-pick `selectOption` leak in (use the new keep-open method).

### CT-02 — Open To Buy: stay-open + reorder on the REACTIVE-apply variant (P0, @regression)
- **Technique:** state transition on the second apply-mode branch of the shared component.
- **Persona:** agent reviewing Open-To-Buy by merchant — same multi-pick need, but OTB applies reactively (no Search button).
- **Page:** Open To Buy (`navigateToOpenToBuy`).
- **Setup:** navigate; open Filters panel; open Merchant dropdown. Read-only.
- **Steps:**
  1. Open Merchant dropdown; assert open.
  2. Pick 2 distinct merchants via `selectOptionsKeepingOpen` (no reopen between picks). Assert menu stayed open across both (AC1/AC2).
  3. `getRenderedOptionOrder` → selected on top (AC3).
  4. Confirm no server error after the reactive apply settles (OTB has no Search; the menu-close/settle path is where #1292 OTB broke twice).
- **Validations:**
  - UI: menu open after each pick; selected-on-top; no "Internal Server Error" banner; OTB table re-renders.
  - DB: none. Activity log: N/A.
- **Edge cases covered:** reactive-apply branch interacting with stay-open (different close timing than Leads' explicit Search).
- **Pitfalls considered:** OTB roster for `manager` may not contain a specific favorite merchant → pick first-N narrowed available (mirror #1292 `pickFirstNAvailable`); OTB has no Search button → don't wait on one.

### CT-03 — Leads: applied multi-filter returns matching rows + checkbox toggle off (P1, @regression)
- **Technique:** filter-results equivalence (AC5) + checkbox decision toggle (AC4 deselect).
- **Persona:** agent who ticked 2 merchants, applies, and expects the grid to show only those merchants' leads.
- **Page:** Leads.
- **Setup:** navigate; open filter; open dropdown.
- **Steps:**
  1. Pick 2 distinct merchants keeping open.

- **Validations:**
  - UI: deselected checkbox unchecked + menu still open; applied grid rows' Merchant column ⊆ selected set (or explicit empty-state).

- **Edge cases covered:** deselect-while-open; results-match after a partial deselection; empty-result tolerance.
