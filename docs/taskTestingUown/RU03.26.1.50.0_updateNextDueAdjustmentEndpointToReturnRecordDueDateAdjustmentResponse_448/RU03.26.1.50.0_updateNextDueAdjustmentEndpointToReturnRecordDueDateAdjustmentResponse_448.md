## Descrição

Atualizar o endpoint TMS/IVR de ajuste de due date (`POST /uown/tms/v1/accounts/{accountPk}/next-due-date/adjustments`) para retornar um `DueDateAdjustmentResponse` com `accountPk`, `originalDueDate` e `newDueDate`. O endpoint também deve gravar um registro na tabela `uown_due_date_moves` para histórico.

Validações:
- `offset` entre 0-7 dias (bean validation `@Max(7)`)
- WEEKLY: offset máximo 3 dias
- Demais frequências: offset máximo 7 dias
- `dueDate` null: auto-resolve via `getNextReceivable`
- `dueDate` explícita: deve corresponder à próxima receivable da conta

## Cenários

### Cenário: Cenário 1 (CT-01)

Move due date via modal no Servicing portal (E2E), verifica toast de sucesso, navega para History page e valida DB.

Exemplos:
| AccountPk | NextDueDate | Offset | History | DB Record |
|-----------|-------------|--------|---------|-----------|
| 4461 | 03/20/2026 | 3 | 3 registros (screenshot) | days=3, type=SCHEDULE_SHIFT, agent=manager |

#### Como verificar manualmente

1. Login no Servicing (qa1) com manager
2. Pesquisar conta 4461
3. Confirmar modal "Customer Information Confirmation"
4. Ir para "Due Amounts" → "Move Due Date"
5. Selecionar scheduled date, preencher nova data (+3 dias), SAVE
6. Verificar toast de sucesso
7. Verificar DB: `SELECT * FROM uown_due_date_moves WHERE account_pk = 4461 ORDER BY row_created_timestamp DESC LIMIT 1`

**PASSOU**
