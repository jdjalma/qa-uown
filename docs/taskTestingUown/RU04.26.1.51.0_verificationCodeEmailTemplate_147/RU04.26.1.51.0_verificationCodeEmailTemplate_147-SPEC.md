## Triple Validation Applicability

Triple validation (API + DB + UI) does NOT apply in the traditional sense for this task because:
- There is no new data-display page or history table
- The task validates **email HTML template content** (not a UI table rendering backend data)

However, each CT implements a **dual-layer validation**:
- **API layer**: `sendVerificationCode` endpoint returns success (200/204)
- **Email layer**: IMAP retrieval confirms email delivery with correct HTML body content
- **DB layer**: `uown_login_attempt` table confirms code storage, expiration time, and account association
- **UI layer**: Customer Portal login succeeds with the code extracted from email

## Interaction Strategy

**UI-first with API/DB support.**

- The login flow (navigate to portal, enter email, receive OTP, enter code) is executed via UI (WebsiteBasePage)
- Email content validation is done via IMAP (EmailHelpers.getEmailContent) -- no UI for this
- DB queries are used as:
  - Fallback for OTP retrieval when IMAP is slow
  - Validation that `uown_login_attempt` records are created correctly
  - Pre-test verification that customer accounts exist and are ACTIVE

## Edge Cases

| # | Edge Case | How the test handles it |
|---|-----------|------------------------|
| 1 | IMAP auth failure or slow delivery | DB fallback: query `uown_login_attempt` for code (same pattern as task #146). Poll for up to 30s with 2s interval |
| 2 | Email body contains IMAP headers before HTML | Use `rawEmail.indexOf('<!DOCTYPE')` to extract clean HTML before content assertions (same pattern as task #486 screenshot step) |
| 3 | Multiple emails matching "Verification" subject | `getEmailContent` returns the most recent by date. Additional guard: verify the code in the email matches what was stored in `uown_login_attempt` |
| 4 | Quoted-printable encoding in email body | `EmailHelpers.decodeQuotedPrintable()` handles `=XX` sequences and soft line breaks. Assertions should use decoded body |
| 5 | Kornerstone account not available in qa2 | Test should skip KS group with clear error message. Pre-flight DB query checks account existence before test execution |
| 6 | Login attempt rate limiting | Each brand test uses a different email. CT-06 (resend) uses the same email but with enough delay between attempts |
| 7 | Code expiration (5 minutes default) | Verification code expires after 5 minutes. Tests must enter the code within this window. Total timeout for email fetch is 150s, leaving ~3 minutes for code entry |
| 8 | Template not yet deployed to qa2 | If template assertions fail (e.g., old template without lock icon), log the actual HTML content and fail with descriptive message indicating deployment may be pending |
| 9 | Same lock_icon.png used by both brands | Both templates reference `lock_icon.png` from GCS -- the icon is shared; only the background color differs (blue #1dbae4 vs green #8fc31f) |

## Implementation Notes for subagent-impl-e2e

1. **Follow task #486 pattern** for email HTML validation: `BrandExpectations` interface, `extractImageUrls()`, `extractLinks()`, `stripHtmlTags()` helpers, screenshot via `chromium.launch()`.

2. **Follow task #146 pattern** for website login: `WebsiteBasePage`, `getOtpFromDb()` fallback helper, `uown_login_attempt` table queries.

3. **test.describe.configure({ mode: 'serial' })** -- CT-01 and CT-02 share the same login flow (CT-01 validates email content mid-flow, CT-02 completes login). Same for CT-03/CT-04.

4. **Shared state**: Use `emailByBrand` record to store `EmailContent` objects across serial tests within the same describe block, same as task #486.

5. **CT-05 (code format)** can run after both brand groups, using the stored email bodies from ctx. It validates structural HTML elements, not brand-specific content.

6. **CT-06 (resend)** is a separate test that starts a fresh login flow. It does NOT depend on CT-01/CT-02 shared state.

7. **KS account discovery**: Before test execution, include a precondition step that queries the DB for a Kornerstone ACTIVE account with a Gmail-routable email. If none found, skip KS tests with `test.skip('No KS account available in qa2')`.

8. **Image URL verification**: Extract all `src` attributes from `<img>` tags, verify they point to `storage.googleapis.com/uown/`, and GET each URL to confirm HTTP 200.

9. **Hidden code div**: Parse the HTML for `user-select:all` div content. In the Thymeleaf-rendered email, this div contains the full 6-digit code as plain text (not individual digits).

10. **Template name in CorrespondenceService**: UOWN uses `VerificationCode`, Kornerstone uses `KORNERSTONE_VerificationCode`. The template files are at `verification-code-email.html` (UOWN) and `kornerstone/verification-code-email.html` (KS). The mapping between template name and file path is in `TemplateInfo.xlsx` / `KS_Templates.xlsx`.
