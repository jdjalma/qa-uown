## 0. Synopsis

Brick-and-mortar merchants reported that `CUSTOMER FIRST NAME` and `CUSTOMER LAST NAME` sit too far right in Origination tables, forcing horizontal scroll. The fix moves customer name columns to the left (positions previously held by `SALES REP` + `MERCHANT SUPPORT`), demoting those two columns to later positions.

Three pages in scope: **Overview**, **Leads**, **Funding**. Two pages have ordering exceptions:
- **Leads**: `CUSTOMER NAME` must appear immediately **BEFORE** `INVOICE NUMBER`.
- **Funding**: `CUSTOMER NAME` must appear immediately **AFTER** `FUNDING QUEUE STATUS`.

## 2. Acceptance Criteria — restated from ticket + derived

### AC explicit (from ticket)

- **AC1** — Customer-name column(s) appear in the **higher-priority positions** previously occupied by `Sales Rep` and `Merchant Support`. **Per page** (post-MCP § 0.5):
  - Overview: `First Name` + `Last Name` (2 cols) before `Sales Person` / `Sales Rep Code` / `Merchant Support`.
  - Leads: `Customer Name` before `Invoice Number` (Leads has no Sales Rep / Merchant Support columns — AC1 reduces to AC3).
  - Funding: `Customer Name` before `Sales Rep Code` (Funding has no Merchant Support column — AC1 reduces to AC4 + "before Sales Rep Code").
- **AC2** — `Sales Rep` / `Merchant Support` move to later (less-prominent) positions but remain present where they exist (only Overview has both; Funding only has Sales Rep Code; Leads has neither).
- **AC3** — On **Leads**, `Customer Name` appears immediately **BEFORE** `Invoice Number`.
- **AC4** — On **Funding**, `Customer Name` appears immediately **AFTER** `Funding Queue Status`.
- **AC5** — At default agent viewport (1440×900), the customer-name column(s) must be **fully visible within the initial viewport without horizontal scroll**. NOTE: full table scroll is **inevitable** (Overview=27, Leads=17, Funding=36 columns); the AC is about the customer-name `<th>`+cells specifically, not the whole table.
- **AC6** — Sorting / filter / pagination / export functionality remains intact on the affected columns.

### AC derived (implicit / from product reflexes)

- **AC7 (implicit)** — Header `<th>` accessible name (visible text) matches business label (the reorder must not silently rename the column).
- **AC8 (implicit)** — Row cells under each `<th>` continue to render the correct value (i.e. reorder is by column INDEX, not by column key — a regression class: re-indexed headers but cells unchanged means data appears in wrong column).
- **AC9 (implicit, regression)** — Click-to-sort on customer name column actually sorts by customer name field (not by whatever index used to be there).
- **AC10 (implicit, regression)** — CSV export header row and data rows are consistent with each other (matching column at index N has matching cell at index N). The order CSV ⇄ UI may differ (see Section 1 ambiguous), but internal CSV consistency is asserted.

### Floor rules applied

- **Data-cell mis-alignment** is the highest-impact failure mode: when a `<th>` reorder is done by visual index but cells are populated by stable key, the resulting table shows correct values under each header. When done wrong (header reordered, cells indexed by position), agents see customer first name under `SALES REP` header, etc. This is a **trust-of-data** class bug — promoted to P1 floor.

### Final priority decision

- **P0**: none — non-revenue, non-customer-facing.
- **P1**: CT-09 (data-cell alignment per row).
- **P2**: CT-01, CT-02, CT-03 (column order per page), CT-04 (scroll), CT-05 (sort), CT-06 (filter), CT-08 (CSV).

- **Type**: E2E (per-page, parameterized over Overview/Leads/Funding)
