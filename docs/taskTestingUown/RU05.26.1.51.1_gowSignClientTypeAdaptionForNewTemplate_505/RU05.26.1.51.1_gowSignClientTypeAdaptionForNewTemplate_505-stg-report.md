## Análise das falhas remanescentes

### Regressão stg — `submitApplication` retorna `embeddedSigningUrl=null` para customer non-CA (12 fails)

Cenários: CT-R02..R04, CT-R06..R08, CT-R10..R12, CT-R14..R16. O setup falha em `expect(result.embeddedSigningUrl).toBeTruthy()` porque o backend retorna:

```json
{
  "embeddedSigningUrl": null,
  "esignClient": null,
  "error": "There was a problem generating lease contract. Please contact merchant for next steps"
}
```

O caminho CA (Daniel's, Paramount, TireAgent) continua funcionando — só o caminho SIGNWELL fallback (non-CA) está degradado em stg.

**Bug classification (CLAUDE.md regra #11):** **[OBSERVAÇÃO] regressão de ambiente stg, não regressão do hotfix #505.** O hotfix toca seleção de template e normalização de `clientType`; o caminho que está quebrado (geração de contrato no fallback non-CA) não passa pela lógica de template GowSign. Critérios:
- (a) reprodução em dados fresh ✓ (3 runs consecutivos)
- (b) issue não é referenciada no escopo da #505
- (c) não é artefato de teste — vendor/upstream backend retornando erro

Recomenda-se reportar a regressão fora do canal da #505.

### CT-R13 (5thAve × CA → expected SIGNWELL, observed GOWSIGN)

Expectativa errada da matriz, não defeito do produto. A matriz assume override INSTORE (merchant.state=NY → effective state NY → SIGNWELL). Em stg, todos os merchants seguem `customer.state` (sem override INSTORE para 5thAve), então 5thAve × cust=CA roteia para GOWSIGN (CA template, standard). Mantido como FAIL pra surfacear o gap — o próprio comentário do código (linha 169-170) admite que a classificação INSTORE/ONLINE é "best-guess for stg".

### CT-07 (account row null — auto-funding)

Daniel's-OL90205-0079 em stg não tem `isSignedToFunding=true`, então não cria `uown_sv_account` automaticamente após STORED. CT-07 espera `account` para abrir o portal Servicing e baixar o documento. Config divergente entre qa2 e stg — não é bug do hotfix #505 nem do teste.
