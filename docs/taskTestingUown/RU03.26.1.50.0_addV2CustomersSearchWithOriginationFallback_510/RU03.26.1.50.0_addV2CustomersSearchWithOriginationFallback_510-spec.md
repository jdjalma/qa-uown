## Program × Brand Coverage

Feature é endpoint de consulta agnóstico de programa/brand. Matriz abaixo é preenchida por completude; todas as variações modalidade × brand seriam redundantes para este escopo.

| | UOWN | Kornerstone |
|-|:----:|:-----------:|
| Modalidade A — 13m apenas | CT-01..CT-15 (fixture D/B) | N/A |
| Modalidade B — 13m+16m | N/A | N/A |
| Modalidade C — 16m Second Look | N/A | N/A |
| Modalidade D — Denied | N/A | N/A |

**Justificativas de N/A:**
- **Kornerstone:** endpoint retorna profile/lead/account independentemente de brand; `customerAccountDomain` não varia com brand. Cobrir Kornerstone seria redundante. Se o response incluir `company` ou algum campo brand-specific, promover um CT suplementar na Fase 5; para o planejamento atual, fora de escopo.
- **Modalidade B/C/D:** programa do lease não altera a busca — endpoint consulta por phone/ssn/dob, não por plano. Modalidade A cobre o happy path mínimo para fundar em qa2.
- **Modalidade D (Denied):** um denied lead ainda existe em `uown_los_lead` — seria idêntico ao Slot B do ponto de vista do endpoint. Omitido como redundante.

## Edge Cases
- **qa2 env-gap (Slot A não reproduzível):** scenarios.md justifica N/A; report deve registrar explicitamente.
- **Not-found semântica inconsistente entre v1 e v2:** se um retornar `404` e o outro `200-empty`, isso é bug de contrato — CT-05 captura.
- **`dob` ambíguo** (matcha múltiplos customers): CT-07 passo 3 — backend pode retornar 400, lista, ou primeiro match. Documentar, não falhar por expectativa.
- **Race: lead promovido a account durante o teste:** Slot B pode virar Slot D se job de funding rodar entre `createPreQualifiedApplication` e a chamada do endpoint. Mitigação: DB guard imediatamente antes da chamada API (passo 2-3 de CT-02/CT-15).
- **Swagger endpoint ausente em qa2:** CT-13 pode degradar pra `[OBSERVAÇÃO]` — não falhar.
- **TMS API key rotacionada:** CT-09 9b usa `"invalid-key-xyz"` literal — se backend aceitar qualquer string não-vazia como válida, é bug de auth.
- **Paralelismo (workers múltiplos):** fixtures Slot D/B criam lease fresh por worker via `runId` + `generateUniqueEmailAlias` — sem colisão de phone/ssn entre workers.
- **Classificação conservadora de bug** (regra 11): observação em dado pré-existente NÃO escala pra `[CONFIRMADO]` sem reprodução em fresh. Para este teste, cada slot é fresh por design, mas divergências de shape/path precisam de confirmação com o usuário antes de reportar.
