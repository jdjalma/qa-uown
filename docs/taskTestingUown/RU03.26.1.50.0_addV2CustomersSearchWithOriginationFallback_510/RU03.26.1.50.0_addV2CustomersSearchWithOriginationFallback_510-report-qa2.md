## Descrição

Criar novo endpoint `POST /uown/tms/v2/customers/search` para consulta unificada de cliente a partir do IVR (Five9), substituindo o legado `POST /uown/tms/v1/accounts/search`. O backend tenta primeiro Servicing (`uown_sv_account`); se não encontrar, faz fallback automático para Origination (`uown_los_lead`). A resposta usa o mesmo envelope paginado do V1 (`{ content: CustomerProfile[], size: number }`) e adiciona, **dentro de cada profile**, os campos novos `leadPk`, `leadStatus` e `customerAccountDomain` (enum `SERVICING | ORIGINATION`) para o atendente saber a fonte do dado.

Critérios de aceite principais:
- `/v2/customers/search` responde `200` com profile quando cliente está em Servicing (`customerAccountDomain=SERVICING`).
- Fallback automático para Origination quando cliente não está em Servicing (`customerAccountDomain=ORIGINATION`).
- Not-found consistente entre `/v1` e `/v2` (ambos `200` com `content=[]`, `size=0`).
- `customerAccountDomain` reflete corretamente a origem do dado.
- Quando cliente existe em ambos, Servicing vence a prioridade.
- Body vazio `{}` → `400` com mensagem documentada.
- `/v1/accounts/search` permanece funcional mas marcado `@Deprecated` (swagger).
- Nenhum campo sensível (SSN completo, senha, token) vaza no response.

### CT-08 — Tipos/formatos inválidos: delta entre spec e comportamento real do Spring/Jackson

**Cenário de negócio:** Documentar o que acontece quando o cliente da API manda payload malformado. O spec oficial implica que entradas inválidas devem dar 400/422. A realidade é que a validação server-side é leniente (herdada do V1), então alguns inputs "errados" passam silenciosamente em vez de rejeitar. Esse teste **não falha o build** por causa dessa divergência — documenta como `[OBSERVAÇÃO]` para o time BE decidir se vale enrijecer.

**O que o teste faz (6 sub-casos documentando o comportamento real):**
| # | Payload | Status real | Classificação |
|---|---------|------------|--------------|
| 8a | `{ phone: 1234567890 }` (JSON number, não string) | **200** | `[OBSERVAÇÃO]` Jackson coage `Number→String` porque o request record declara `String phone` sem `@Pattern`. V1 tem comportamento idêntico — não é regressão da task. |
| 8b | `{ last4SSN: "12345" }` (5 dígitos em vez de 4) | **200 (empty)** | `[OBSERVAÇÃO]` Sem `@Size(4)`. String vai direto pra SQL; não match → envelope vazio. |
| 8c | `{ last4SSN: "abcd" }` (não-numérico) | **200 (empty)** | `[OBSERVAÇÃO]` Sem `@Pattern("\\d{4}")`. Mesmo padrão do 8b. |
| 8d | `{ dob: "01/15/1990" }` (MM/DD/YYYY em vez de ISO) | **400** | `LocalDate` tem parse estrito YYYY-MM-DD. Jackson deserializer rejeita. |
| 8e | `{ phone: "" }` (string vazia) | **400** | `StringUtils.hasText("")` é false → cai na validação "at least one identifier". |
| 8f | JSON malformado `{phone:` | **400** | Parser Jackson rejeita na deserialização. |

Para 8a/b/c, o teste aceita `[200, 400, 422]` e documenta como `[OBSERVAÇÃO]` em comentário no código.

**PASSOU** (1.2s)

> 8a/b/c alimentam a observação #1 do final do report — sugestão de hardening para o time BE adicionar `@Pattern`/`@Size` no `FindCustomerRequest`. Aplicaria a V1 também.

### CT-16 — Divergência SVC/LOS: quando atendente edita dados em Servicing, cada domínio responde pelo "seu" phone

**Cenário de negócio (o mais sutil do conjunto):**

No mundo real, depois que um cliente vira funded:
- O registro em Origination (`uown_los_lead` + `uown_los_phone`) é **congelado no estado que estava no momento do funding** — isso é o histórico da aplicação, ninguém mexe mais.
- O registro em Servicing (`uown_sv_account` + `uown_sv_phone`) é **mutável** — atendente/cliente podem editar phone, email, endereço ao longo da vida do lease.

Consequência: é perfeitamente possível (e comum) que um mesmo cliente tenha **phones diferentes** entre os dois domínios depois de alguma edição pós-funding. Essa divergência é um caso real do negócio e o `/v2` deve se comportar corretamente em ambos os lados:
- Buscar pelo phone **atual** (o que está em Servicing agora) → encontra em Servicing → retorna `SERVICING`.
- Buscar pelo phone **antigo** (o do momento do funding, ainda em Origination) → não encontra em Servicing → fallback para Origination → retorna `ORIGINATION`, mas com o mesmo `leadPk`.

Esse é o proxy válido para o Scenario A da matriz oficial ("Servicing ✅ / Origination ❌"). O cenário literal não existe na plataforma (todo account SV nasce de um lead LOS), mas a divergência de dados captura o **mesmo efeito observável** — Servicing tem dados que Origination não tem.

**Setup:** Reusa slot D (funded, phone inicialmente sincronizado entre SVC e LOS). Colocado no final da suite para não contaminar CTs anteriores que dependem do phone original.

**O que o teste faz:**

1. **Capturar PKs do customer em Servicing** — `GET /uown/svc/getPrimaryCustomerContactInfo/{accountPk}` retorna `phoneList[0].pk`, `phoneInfo.phonePK`, `phoneInfo.customerPK`. Esses PKs são necessários pro update (modo "update existing" vs "create new").

2. **Gerar phone novo determinístico** — `415-2XX-XXXX` (NANP válido, não toll-free — tentei 800 antes e o backend rejeitou com `SvcException: Invalid Phone Number`). Suffix derivado do `leadPk` pra evitar colisão com leads seed.

3. **Editar phone em Servicing via API oficial** — `POST /uown/svc/createOrUpdatePrimaryCustomerContactInfo` com `phoneList: [{ pk, phoneInfo: { phonePK, customerPK, areaCode='415', phoneNumber=<novo>, phoneType }}]`. Nada de UPDATE direto no DB — tudo via endpoint público que o Servicing expõe.

4. **DB guard confirma a divergência**:
   - `SELECT count(*) FROM uown_sv_phone WHERE account_pk=? AND CONCAT(area_code, phone_number)=<NEW_PHONE>` → ≥1 (Servicing tem o novo).
   - `SELECT count(*) FROM uown_los_phone WHERE lead_pk=? AND CONCAT(area_code, phone_number)=<ORIGINAL_PHONE>` → ≥1 (Origination preserva o antigo).
   
   > **Correção importante pós-debug:** a primeira versão da query olhava `uown_los_customer.cell_phone_number`, que volta **null** pós-funding nesta arquitetura. A fonte correta do phone histórico do lead é `uown_los_phone` (tabela dedicada com `area_code + phone_number`, estrutura espelhada ao `uown_sv_phone`). Essa descoberta só veio quando o usuário olhou `uown_los_phone` direto e viu que o phone original ESTAVA lá.

5. **Busca por phone NOVO** → `POST /v2/customers/search { phone: NEW }` deve achar em Servicing:
   - `200` + `content[]` contém o slot D pelo `leadPk`.
   - `customerAccountDomain === 'SERVICING'` (o dado atual vem do Servicing).

6. **Busca por phone ORIGINAL** → `POST /v2/customers/search { phone: ORIGINAL }` **não** acha em Servicing (phone foi trocado lá), mas **acha** em Origination:
   - `200` + `content[]` contém o mesmo slot D pelo mesmo `leadPk`.
   - `customerAccountDomain === 'ORIGINATION'` (fallback disparou — dado vem do histórico LOS).
   - **Mesmo `leadPk`**: prova que o backend resolveu o mesmo cliente por dois paths diferentes.

## Observações para o time backend (não bloqueiam)

1. **Validação leniente em `FindCustomerRequest`**: sem `@Pattern`/`@Size`/`@NotBlank` nos campos `phone` e `last4SSN`, Jackson coage silenciosamente (`Number→String`) e strings de qualquer comprimento/formato passam para a SQL. Se o critério real é "phone = string de 10 dígitos", recomendo adicionar `@Pattern(regexp="\\d{10}")` e `@Size(min=4,max=4)` em `last4SSN`. Impacto: mesma regra vale para V1 (herança).
2. **Header `Deprecation` ausente em `/v1`**: swagger já reporta `deprecated=true`, mas não há header HTTP `Deprecation` (RFC 8594) nas responses. Clientes machine-readable que não consultam swagger (ex: proxy do Five9) não teriam como detectar o status de deprecation em runtime. Melhoria de baixa prioridade.
3. **Nomenclatura do campo `last4SSN` (request) × `last4Ssn` (response)**: inconsistência camelCase entre body e content. Não afeta funcionamento (é só o JSON property name), mas pode gerar atrito no consumidor. Herdado do V1.
4. **Persistência do phone em LOS pós-funding** (achado do CT-16): o phone histórico do lead persiste em `uown_los_phone` (tabela dedicada `area_code + phone_number`, espelhada a `uown_sv_phone`) mesmo após o funding. `uown_los_customer.cell_phone_number` aparece como `null` na maioria dos leads, mas a tabela canônica dessa informação é `uown_los_phone`. Vale documentar explicitamente no schema doc que `uown_los_phone` é a fonte única de phones históricos do lead — o campo `cell_phone_number` em `uown_los_customer` é legado/redundante e não deve ser usado como single-source-of-truth. Não bloqueia a task; é sugestão de clareza de schema.
