## Dados de teste — Matriz Servicing × Origination (qa2)

A estratégia é criar **fixtures fresh** quando possível, seguindo Test Data Hierarchy (`.claude/rules/testing.md`):

| Slot | Estado esperado | Como gerar |
|------|-----------------|------------|
| **S-ONLY** (A) | Existe em Servicing, **sem** lead em Origination | *Raro* — em qa2 todo account tem lead de origem. **Fallback:** usar account migrado (se houver) OU classificar cenário A como N/A com justificativa (todos os accounts também existem em Origination). |
| **O-ONLY** (B) | Existe em Origination, **não** em Servicing | Criar application via `sendApplication` e **parar antes de `driveLeadToFunding`** → lead em `uown_los_lead` sem `uown_sv_account` correspondente. |
| **NEITHER** (C) | Não existe em nenhum | Phone `5550000000` + last4SSN `0000` + dob `1900-01-01` — garantir via DB query que não existe. |
| **BOTH** (D) | Funded → account em servicing + lead em origination | `createPreQualifiedApplication` + `driveLeadToFunding` (pipeline padrão). Este é o caso **comum** na prática. |

**Decisão de desenho:** assumimos que slot D cobre o caso "exists in both"; se qa2 tiver algum account synthetic sem lead origin (slot A real), usamos. Caso contrário, cenário A vira `[N/A — slot não reproduzível em qa2]` e argumentamos que o fallback lógico (slot D) já cobre "Servicing priority".

Worker-scoped IDs: todo lead/application criado usa `generateUniqueEmailAlias()` + `RUN_ID` pra não colidir com paralelismo.

## User Stories

### US-IVR-01: Consulta unificada de cliente (IVR/TMS)

**Como** sistema IVR (Five9) integrado ao TMS,
**Quero** consultar um cliente por phone/last4SSN/dob e receber dados equivalentes esteja ele em Servicing ou Origination,
**Para** que o atendente de callcenter não precise saber em qual domínio buscar e nenhum cliente vazado seja "invisível".

**Critérios de aceite:**
- [ ] `/v2/customers/search` responde `200` com profile quando cliente está em Servicing
- [ ] `/v2/customers/search` faz fallback automático para Origination se não achar em Servicing
- [ ] `/v2/customers/search` responde not-found (404 ou 200-empty) quando não há match em nenhum domínio
- [ ] `customerAccountDomain` no response reflete corretamente a origem do dado
- [ ] Quando cliente existe em **ambos**, Servicing vence (Origination não sobrescreve)
- [ ] Body vazio `{}` → `400` com mensagem documentada
- [ ] `/v1/accounts/search` permanece funcional mas está marcado `@Deprecated`
- [ ] Nenhum campo sensível (SSN completo, senhas, tokens) vaza no response

## Cenários de Teste

> Cada CT é 1 caso API. Tags: `@api @critical @tms` (todos regression).

### CT-01: [Happy — Slot D] /v2 retorna Servicing quando customer existe em ambos

**Tipo:** API
**Portal:** TMS/IVR
**Slot de dados:** D (funded account, lead origem preservado)

**Pré-condição:**
- Criar lease fresh via `createPreQualifiedApplication` + `driveLeadToFunding` → captura `{ accountPk, leadPk, customer.phone, customer.last4SSN, customer.dob }`
- DB guard: `SELECT 1 FROM uown_sv_account WHERE pk = $accountPk` e `SELECT 1 FROM uown_los_lead WHERE pk = $leadPk`

**Passos:**
1. `POST /uown/tms/v2/customers/search` com `{ phone }`
2. Repetir com `{ last4SSN, dob }` (multi-field search)

**Resultado esperado:**
- Status `200`
- Body valida schema `FindCustomerResponse + CustomerProfile`
- `customerAccountDomain === "SERVICING"`
- `leadPk` presente e numérico, bate com o criado
- `leadStatus` presente e string não-vazia
- Campos core (nome, telefone, identificadores) batem com a fixture criada
- `[reflex]` → nenhum campo com SSN completo (`ssn` / `socialSecurityNumber`) nem `password` no response

### CT-02: [Happy — Slot B] /v2 faz fallback para Origination

**Slot de dados:** B (lead criado, **sem** funding)

**Pré-condição:**
- `createPreQualifiedApplication` **sem** `driveLeadToFunding` → `leadPk` em `uown_los_lead`, sem `uown_sv_account`
- DB guard: `SELECT count(*) FROM uown_sv_account WHERE primary_phone = $phone` → `0`

**Passos:**
1. `POST /uown/tms/v2/customers/search` com `{ phone }` do lead

**Resultado esperado:**
- Status `200`
- `customerAccountDomain === "ORIGINATION"`
- `leadPk` bate com o `uown_los_lead.pk`
- `leadStatus` reflete o estado atual do lead (ex: `UW_APPROVED` ou equivalente, a confirmar)
- Profile preenchido com dados de origination

### CT-08: [Validation/Erros] Tipos e formatos inválidos (v2)

**Sub-cenários:**

| Sub | Payload | Esperado |
|-----|---------|----------|
| 8a | `{ phone: 1234567890 }` (número, não string) | `400` ou `422` — documentar |
| 8b | `{ last4SSN: "12345" }` (5 dígitos) | `400` — "last4SSN must be 4 digits" ou similar |
| 8c | `{ last4SSN: "abcd" }` (não-numérico) | `400` |
| 8d | `{ dob: "01/15/1990" }` (formato MM/DD/YYYY) | `400` se backend aceita só ISO, senão documentar |
| 8e | `{ phone: "" }` (string vazia — conta como ausente?) | `400` se backend faz `blankToNull` e cai em "at least one required" |
| 8f | Payload malformado (JSON inválido) | `400` |

**Importante:** documentar o comportamento **real**, não falhar o teste por divergência de mensagem — o objetivo é confrontar expectativa com contrato.

### CT-11: [Contract] Shape do response /v2 bate com FindCustomerResponse + CustomerProfile

**Passos:** reusa resposta do CT-01 (slot D) e CT-02 (slot B)

**Asserções:**

- Campos obrigatórios **não** são `null`: `leadPk`, `leadStatus`, `customerAccountDomain`
- Tipos: `leadPk: number`, `leadStatus: string`, `customerAccountDomain: string`
- Nenhuma chave sensível: `ssn`, `socialSecurityNumber`, `password`, `apiKey`, `token` no payload

### CT-14: [Consistência] Core fields batem entre /v1 e /v2 para o mesmo customer

**Pré-condição:** slot D

**Passos:**
1. `r1 = /v1/accounts/search({ phone })`
2. `r2 = /v2/customers/search({ phone })`

**Asserções:**
- `r2.name === r1.name` (ou equivalente em CustomerProfile)
- `r2.phone === r1.phone`
- Identifiers centrais (accountPk/customerPk) batem
- /v2 **não** altera nenhum valor do /v1 — só adiciona

### CT-15: [Search order] /v2 tenta Servicing primeiro (slot B forçado)

**Setup:** criar lead slot B + garantir via DB que **não há** account em servicing para esse phone

**Passos:**
1. `/v2/customers/search({ phone })`

**Asserções:**
- Resposta: `customerAccountDomain === "ORIGINATION"`
- Interpretação: se tivesse tentado Origination primeiro, não daríamos pra distinguir — o único sinal de "ordem" é o `customerAccountDomain` reflete a origem correta. A **ordem** em si não é observável por API (seria via logs/tracing). Asserção prática: existe lead em orig, não existe account em svc, domain vem ORIGINATION → ordem está consistente com specs.
