## Bugs de Aplicação Encontrados

### BUG-01 — Mobile Number não é persistido após salvar o merchant

**Status:** OPEN
**Severidade:** High

**Descrição:** Ao preencher o campo Mobile Number (tanto Primary Contact quanto Alternate Contact) na página de edição do merchant e clicar em SAVE, o valor não é persistido. Ao recarregar a página, o campo volta vazio. A consulta ao banco de dados confirma que as colunas `primary_contact_mobile` e `alt_contact_mobile` permanecem null. O toast de sucesso do save também não aparece (string vazia), o que pode indicar que o save falhou silenciosamente ou que os novos campos não estão sendo incluídos no payload enviado ao backend.

**Como Reproduzir:**
1. Acessar `https://origination-qa2.uownleasing.com` e fazer login como admin
2. Navegar para `/merchantEdit/OL90202-0001` (merchant TerraceFinance)
3. Na seção Primary Contact, preencher o campo Mobile Number com `(555) 123-4567`
4. Clicar no botão "SAVE"
5. Observar que nenhum toast de sucesso aparece
6. Recarregar a página — o campo Mobile Number está vazio
7. Consultar o banco:
   ```sql
   SELECT primary_contact_mobile, alt_contact_mobile
   FROM uown_merchant
   WHERE ref_merchant_code = 'OL90202-0001';
   ```
   Resultado: ambos os campos são `null`

**Evidência:** CT-05 registrou `Persisted contactMobile value: ""` e CT-06 registrou `Persisted alternateContactMobile value: ""`. CT-07 confirmou via DB que ambas as colunas são null.

**Cenário que detectou:** CT-05, CT-06 (confirmado por CT-07)

**Causa provável:** O frontend provavelmente não está incluindo os campos `primaryContactMobile` e `altContactMobile` no payload do POST/PUT enviado à API `createOrUpdateMerchant`, ou o proxy do Origination não está encaminhando esses novos campos para o backend SVC. A migration de banco criou as colunas corretamente (CT-10), então o problema está na camada de aplicação (frontend ou API proxy).
