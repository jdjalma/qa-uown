# Cenários de Teste — Redirecionamento de Links de Aplicação Inválidos

Verificação de que o cliente que acessa um link de aplicação que não funciona — por qualquer motivo — é levado automaticamente para a página de busca de lojas parceiras, em vez de cair em uma tela quebrada. Cobertura em ambas as marcas (UOWN e Kornerstone).

## Resumo geral

O objetivo é eliminar becos sem saída na jornada do cliente. Hoje, links antigos ou inválidos compartilhados por clientes, agentes ou impressos em material físico podem apontar para merchants desativados, terminados ou inexistentes — e quando o cliente clica, fica preso. Com a correção, sempre que o link não puder ser honrado, o cliente é redirecionado para a página oficial de busca de lojas, onde pode encontrar uma loja ativa próxima e seguir a aplicação por um caminho válido.

## Cenários

### CT-01 — Cliente acessa link de merchant UOWN desativado e é levado para a busca de lojas

**O que é testado:** Cliente clica em um link de aplicação de um merchant da marca UOWN que está desativado no sistema.

**Resultado esperado:** O sistema reconhece que o merchant não pode atender e redireciona o cliente automaticamente para a página de busca de lojas parceiras da UOWN. O cliente nunca enxerga uma tela quebrada.

---

### CT-02 — Cliente acessa link de merchant Kornerstone desativado e é levado para a busca de lojas

**O que é testado:** Mesmo cenário do CT-01, agora para a marca parceira Kornerstone.

**Resultado esperado:** O sistema redireciona o cliente para a página de busca de lojas. O comportamento é consistente entre as marcas — nenhum cliente, independentemente da origem, fica travado.

---

### CT-03 — Cliente acessa link de merchant UOWN cuja relação com a UOWN foi encerrada

**O que é testado:** Cliente clica em um link de um merchant UOWN que teve a parceria comercial encerrada com a UOWN (terminated).

**Resultado esperado:** O sistema redireciona o cliente para a página de busca de lojas, da mesma forma que faz para merchants desativados.

---

### CT-04 — Cliente acessa link de merchant Kornerstone cuja relação foi encerrada

**O que é testado:** Mesmo cenário do CT-03, agora para a marca Kornerstone.

**Resultado esperado:** Cliente é redirecionado para a página de busca de lojas.

---

### CT-05 — Cliente acessa link com código de merchant que não existe

**O que é testado:** Cliente clica em um link cujo código de merchant nunca existiu — pode ser um typo, um link inventado, ou um código gerado em ataque de varredura.

**Resultado esperado:** O sistema redireciona para a página de busca de lojas. Várias variações de código inválido foram testadas (códigos com letras aleatórias, formatos malformados, códigos só com números) — todas levam o cliente ao destino correto.

---

### CT-06 — Cliente acessa o link de aplicação sem informar nenhum merchant

**O que é testado:** Cliente acessa apenas a URL base de aplicação, sem código de merchant nenhum.

**Resultado esperado:** Como não há merchant para abrir, o sistema redireciona o cliente diretamente para a busca de lojas, oferecendo a oportunidade de descobrir uma loja parceira próxima.

---

### CT-07 — Cliente acessa o link de aplicação com uma barra extra no final

**O que é testado:** Variação comum de URL — cliente acessa a URL com uma barra a mais no final (`/getApplication/` em vez de `/getApplication`).

**Resultado esperado:** O sistema trata a variação como equivalente e redireciona o cliente para a busca de lojas.

---

### CT-08 — Cliente acessa link de merchant UOWN ativo e o formulário de aplicação abre normalmente

**O que é testado:** Cenário do caminho feliz. Cliente acessa o link de um merchant UOWN ativo e válido — não deve haver redirecionamento.

**Resultado esperado:** O formulário de aplicação carrega normalmente para o cliente seguir o fluxo de crédito. A correção que redireciona links inválidos **não afeta** os links bons.

---

### CT-09 — Cliente acessa link de merchant Kornerstone ativo e o formulário de aplicação abre normalmente

**O que é testado:** Mesmo cenário do CT-08, agora para a marca Kornerstone. Cliente acessa link de merchant Kornerstone ativo e válido.

**Resultado esperado:** O formulário de aplicação da Kornerstone carrega normalmente. Caminho feliz preservado nas duas marcas.

---

### CT-10 — Link manipulado para tentar enviar o cliente para um site externo malicioso

**O que é testado:** Cenário de segurança. Atacante manipula a URL com parâmetros maliciosos tentando usar o redirect do UOWN para mandar o cliente para um site falso (phishing).

**Resultado esperado:** O sistema ignora completamente os parâmetros maliciosos e redireciona o cliente sempre para a página oficial de busca de lojas. A URL não pode ser usada como vetor de phishing.

---

### CT-12 — Link com caracteres incomuns ou tamanho extremo

**O que é testado:** Cliente (ou ferramenta automatizada) acessa o link com caracteres especiais, espaços codificados, caractere nulo, ou um código gigantesco (mais de 300 caracteres).

**Resultado esperado:** O sistema não quebra em nenhuma das variações. Em casos extremos (caractere nulo), o sistema responde de forma defensiva sem expor erro interno. Em todos os outros casos, o cliente é redirecionado normalmente.

---

### CT-13 — Cliente acessa o link na versão não-segura (sem HTTPS)

**O que é testado:** Cliente acessa o link via `http://` em vez de `https://` — pode acontecer com links antigos ou copiados sem protocolo.

**Resultado esperado:** O sistema migra automaticamente para a versão segura (HTTPS) e depois redireciona o cliente para a busca de lojas. Em nenhum momento dados trafegam em canal não-seguro.

---
