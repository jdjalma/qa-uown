## Descrição

Adiciona suporte ao tipo de arranjo `SETTLEMENT` nos endpoints de payment arrangement:

- Campo `arrangementType` (SETTLEMENT / NORMAL) introduzido nos endpoints de criação de arranjo (CC e ACH).
- **SETTLEMENT**: quando todos os parcelamentos são processados, o arranjo passa para `SUCCESS` e a conta transita para `SETTLED_IN_FULL`.
- **NORMAL**: o arranjo completa com `SUCCESS` mas a conta permanece com o status original (ex.: `ACTIVE`).
- Nova tabela `uown_sv_payment_arrangement` com colunas: `status`, `arrangement_type`, `is_active`, `previous_rating`, `current_rating`, `payment_type`, `username`, `start_date`, `end_date`, `amount`, `notes`.
- FKs adicionadas nas colunas `arrangement_pk` de `uown_sv_credit_card_transaction`, `uown_sv_achpayment`, e `uown_los_credit_card_transaction`.

### BUG-02 — Profituity ACH inativo em STG

| Campo | Valor |
|-------|-------|
| **Severidade** | Low (ambiente) |
| **Componente** | svc / Profituity integration / ACH sweeps |
| **Ambiente** | STG |
| **Cenários afetados** | CT-06, CT-08, CT-SM-02 (fluxos ACH end-to-end) |

Os sweeps ACH retornam 200 OK (HTTP estrutural OK) mas os pagamentos permanecem PENDING. Mesmo após ativar sweep tasks (`is_active=true`) e configurar `append.simulated.response.column: "false"`, o endpoint `sendAchPaymentsSweep` não retorna em <60s e os pagamentos não transitam de PENDING. Possíveis causas: Profituity sem credenciais/certificados válidos em stg, endpoint inalcançável, ou serviço não reiniciado após mudança de config.

**Diferença comportamental stg vs qa1:**

| Comportamento | QA1 | STG |
|--------------|-----|-----|
| CC payments | Síncrono (SUCCESS imediato) | Síncrono (SUCCESS imediato) |
| CC sweep | Inativo (disabled em qa1) | Ativo mas lento (processa todo o ambiente) |
| ACH sweep send | Inativo (Profituity off) | Ativo (tasks `is_active=true`), sweep dispara mas não retorna <60s |
| ACH pagamentos | Ficam PENDING | Ficam PENDING (Profituity não processa) |
| CC decline rollback | Arranjo pode ser criado parcialmente | Rollback completo (arranjo=null) |
| Toast UI | Aparece e persiste | Aparece e some rapidamente (backend mais lento) |
