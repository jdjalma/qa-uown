## Reprodução do null nextPayDate

### Evidência de qa1 (investigação 2026-05-23)

Query rodada em qa1 (`SELECT` read-only — Exception 3 OK):

```sql
SELECT COUNT(*) FROM uown_los_sched_summary s
  JOIN uown_los_lead l ON l.pk = s.lead_pk
WHERE s.next_payment_due_date IS NULL AND l.status = 'SIGNED';
-- result: 20 rows
```

```sql
SELECT COUNT(*) FROM uown_los_sched_summary s
  JOIN uown_los_lead l ON l.pk = s.lead_pk
WHERE s.next_payment_due_date IS NOT NULL AND l.status = 'FUNDED'
  AND l.row_created_timestamp >= NOW() - INTERVAL '14 days';
-- result: 89 rows
```

**Interpretação:** O null em `uown_los_sched_summary.next_payment_due_date` para lead SIGNED é estado natural e reprodutível pelo caminho atual do sistema em qa1. Os 20 leads SIGNED null são vítimas do bug pré-fix — não conseguiram avançar para SETTLED. Os 89 leads FUNDED são casos onde o caminho deu certo (com NPD populado em algum momento). (source: DB qa1 read-only query, 2026-05-23)

### Método de reprodução escolhido

**Método primário (default):** Caminho fidedigno via API.

1. `createPreQualifiedApplication` (ou pipeline manual `sendApplication` + `sendInvoice` + `getMissingFields` + `submitApplication` + `changeLeadStatus SIGNED`) com o body padrão de `buildSendApplicationBody`.
2. Imediatamente após `changeLeadStatus SIGNED`, query `SELECT next_payment_due_date FROM uown_los_sched_summary WHERE lead_pk=$leadPk`.
3. **Se NULL** → cenário do bug reproduzido naturalmente. Prosseguir para Settle.
4. **Se NOT NULL** → fix do submit (`fixPlanIdConditionWhenNextPayDateIsNotProvided`) já preencheu o campo no caminho normal. **Fallback ativado** (ver abaixo).

**Justificativa:** memory `feedback_setup_via_ui_new_application` e regra #14 (UI-first) priorizam caminho real do customer/agent. O bug existe no sistema atual — 20 leads SIGNED null em qa1 comprovam. Forçar override no body só faz sentido se o caminho natural não produzir null, e a evidência sugere que produz em pelo menos uma fração dos casos.

**Método fallback (se NPD vier populado naturalmente):** Override explícito no body do `sendApplication`:

```ts
const body = buildSendApplicationBody(merchant, applicant, order);
body.mainNextPayDate = undefined;
body.mainLastPayDate = undefined;
body.mainPayFrequency = undefined;
// mainEmployerName/employment data: deixar populado (não reduzir AC a "no employment")
await api.application.sendApplication(body);
```

(source: `src/api/bodies/application.body.ts:197` confirma que builder seta `mainNextPayDate=now+7d` sempre; bypass exige override explícito antes do POST. Overload `sendApplication(body)` existe em `src/api/clients/los-partner-application.client.ts`.)

**Decisão de implementação:** o `qa-implementer` deve fazer **smoke do método primário em CT-01 primeiro** — rodar uma vez, checar via DB se NPD veio null. Se sim, todos os 12 CTs seguem método primário. Se não, ativar fallback (override) em todos. Documentar o resultado no report.

### Dataset de regressão real disponível em qa1

20 leads SIGNED com `next_payment_due_date=NULL` em qa1 podem ser usados como **dataset secundário de regressão**: aplicar Settle Application em um subset (ex: 3 leads) via UI Origination e validar fix. Isso NÃO substitui CTs com lead fresh (regra #9 Test Data Hierarchy — fresh > reuso), mas é evidência adicional valiosa que `qa-validator` pode rodar como sanity após CTs verdes. (Para extrair lista de leadPks, query: `SELECT s.lead_pk FROM uown_los_sched_summary s JOIN uown_los_lead l ON l.pk=s.lead_pk WHERE s.next_payment_due_date IS NULL AND l.status='SIGNED' LIMIT 20;`)
