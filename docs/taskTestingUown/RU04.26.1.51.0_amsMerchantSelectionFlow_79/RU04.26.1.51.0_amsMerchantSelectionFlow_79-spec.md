## Interaction Strategy

**UI-first approach.** All 12 CTs drive the AMS portal browser. API calls (`AmsClient`) are used for:
- **Preconditions**: `beforeAll` to discover usernames and capture original `merchantCodes` for teardown
- **Post-verification**: After UI save operations, `GET /user/{username}` confirms persistence
- **CT-11 only**: Pure API round-trip verification (no UI)
- **Teardown**: `afterAll` restores `merchantCodes` to original values via `PUT /user/{username}`

## Triple Validation Assessment

This task is a **UI component replacement** on the AMS user details page. It does NOT introduce:
- New API endpoints (uses existing `GET/PUT /user/{username}`)
- New DB tables or columns (`merchantCodes` field on `user` table already exists)
- A data-display page (no table/list fetched from a dedicated GET endpoint)

**Triple validation is partially applicable.** CTs that modify and save merchant assignments (CT-05, CT-06, CT-08, CT-09) include:
1. **UI action**: modify merchants via React Select, click SAVE
2. **UI verification**: verify badges in read-only mode after save
3. **API verification**: `GET /user/{username}` confirms `merchantCodes` field matches

DB verification is **not included** because the AMS backend uses a separate PostgreSQL instance (not the main `uown` DB accessible via `db` fixture). The `AmsClient` API verification serves as the persistence check.

## Edge Cases

1. **Empty merchant list after clear all**: CT-10 clicks "clear all" then cancels. If the user accidentally saves with zero merchants, `merchantCodes` becomes empty string or null. The `afterAll` teardown restores original state regardless.

2. **Select All mutual exclusivity**: When "Select All" is chosen, individual tags must be replaced. If the component does NOT enforce this (bug), CT-08 will catch it — individual tags would still be visible alongside "Select All".

3. **Dropdown portal overlay (React Select)**: The React Select dropdown renders in a portal. Options must be clicked with care. The existing `selectMerchantInEdit()` method uses keyboard navigation (ArrowDown + Enter) which bypasses portal overlay issues. Extend this pattern for "Select All" selection.

4. **Login session expiration**: AMS portal login may expire between CTs in serial mode. If a navigation to `/users/[username]` redirects to login, the test should re-login. The `loginToPortalWithOptions` helper handles this, but the test should check `isAmsLoginPage()` before each CT and re-login if needed.

5. **Race condition on save**: After clicking SAVE, the component transitions from edit mode to read-only. The `saveMerchantsEdit()` method waits for the pencil icon to reappear (indicating edit mode closed). If the save request is slow (>10s), this timeout may need increasing.

6. **CT-08 to CT-09 ordering dependency**: CT-08 sets `test.tester` to `*`, and CT-09 expects `*` as the starting state. These CTs MUST run in serial order. The `test.describe.configure({ mode: 'serial' })` ensures this.

7. **Searchbox disabled detection**: When `merchantCodes = "*"`, the "Search for Current Merchants" searchbox should be disabled. The disabled state could be an HTML `disabled` attribute or a CSS class (`--is-disabled`). The test should check both `isDisabled()` and visibility.

8. **Tag text format difference between edit and read-only modes**: In read-only mode, badges show "Name - Code" in green. In edit mode, React Select multi-value tags show "Name - Code x" (with remove button). The `getMerchantTags()` method extracts text differently depending on mode — ensure the page object method handles both.

9. **CT-05 merchant selection**: To add a merchant that is NOT already assigned, the test needs to know which merchants are currently assigned. Strategy: use `getMerchantTags()` to capture current badges, then pick a search term for a merchant not in the list. Alternatively, query the API for available merchants. The existing `selectMerchantInEdit(searchTerm)` searches and selects the first non-"Select All" option — the search term should target a known merchant code that exists in qa2 but is rarely pre-assigned.

10. **AMS does NOT use the main `uown` DB**: The AMS backend has its own PostgreSQL database. The `db` fixture connects to the main `uown` DB, NOT the AMS DB. Therefore, DB-level validation (`db.getSingleRow(...)`) is NOT possible for AMS `user` table queries. API verification via `AmsClient.getUser()` is the persistence validation layer.
