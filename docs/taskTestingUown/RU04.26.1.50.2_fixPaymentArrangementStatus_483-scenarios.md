## Contexto do Bug

Três comportamentos incorretos identificados em produção relacionados à lógica de Payment Arrangement:

1. **CC DENIED sem atualização de rating**: Quando um CC arrangement era negado (DENIED), a validação de `postingDate == startDate` e `startDate <= today` não era realizada, impedindo a chamada a `updateRatingForAccount`. Com o fix, essas condições são verificadas e o rating é corretamente atualizado para `P`.

2. **Transição prematura para IN_PROGRESS**: Quando um arrangement tinha `startDate` no futuro e pagamentos em PENDING, o listener transitava o status para `IN_PROGRESS` prematuramente. O fix adiciona a verificação de `startDate` antes de qualquer processamento.

3. **CC APPROVED sem atualização de rating**: Quando um CC arrangement era aprovado com sucesso, as condições de data (`postingDate == startDate`, `startDate <= today`) também não eram verificadas, impedindo a chamada correta a `updateRatingForAccount`. O fix garante que o rating seja atualizado para `P` ao processar pagamentos aprovados dentro da janela de data esperada.

## User Stories

**Como** agente de cobrança,
**Quero** que o rating letter `P` seja atualizado corretamente quando um acordo de pagamento em CC é negado com as condições de data satisfeitas,
**Para** que a conta reflita o status correto de *promise to pay* e não seja processada indevidamente por sweeps automáticos.

#### Critérios de Aceite
- [ ] Arrangement criado com 1 CC payment, `postingDate == startDate`, `startDate <= today`
- [ ] CC payment com status `DENIED`
- [ ] Sistema chama `updateRatingForAccount(...)` — condições de data satisfeitas
- [ ] Rating da conta atualizado para `P` na tabela `uown_sv_account`
- [ ] Activity log registra a atualização de rating

**Como** agente de cobrança,

### US-SVC-02: Arrangement com data futura não transiciona para IN_PROGRESS antes do início

**Como** agente de cobrança,
**Quero** que um acordo de pagamento com `startDate` no futuro permaneça em `NOT_STARTED` até chegar a data de início,
**Para** que os status dos pagamentos e do arrangement estejam sempre consistentes com a realidade operacional.

#### Critérios de Aceite
- [ ] Arrangement criado com 3 parcelas, `startDate` no futuro
- [ ] Todos os pagamentos em status `PENDING`
- [ ] Status do arrangement permanece `NOT_STARTED` (não transiciona para `IN_PROGRESS`)
- [ ] Log gerado indicando que o arrangement ainda não começou

## SQL Auxiliar para Busca de Contas Elegíveis (qa2)

```sql
-- Buscar contas ACTIVE em FifthAveFurnitureNY (KS3015) em qa2
-- Sem arrangement ativo (para isolamento de teste)
SELECT a.pk, a.account_status, a.rating, m.merchant_name
FROM uown_sv_account a
JOIN uown_sv_merchant m ON a.merchant_pk = m.pk
WHERE m.merchant_number = 'KS3015'
  AND a.account_status = 'ACTIVE'
  AND a.rating IS NULL
  AND NOT EXISTS (
    SELECT 1 FROM uown_sv_payment_arrangement p
    WHERE p.account_pk = a.pk AND p.is_active = true
  )
ORDER BY a.pk DESC
LIMIT 10;
```
