## Detailed CT Design Notes

### React Select Interaction Pattern

All filter dropdowns use `@uownleasing/common-ui` `InputField` with `type="select"` or `type="multi-select"`, which renders React Select under the hood. The proven interaction pattern from `BasePage.selectFilterOption()` and `BasePage.selectReactOption()` should be used:

**Multi-select (Overview, Open to Buy)**:
- Merchant/Location fields render with CSS class `filter__control` or `react-select__control`
- Opening: click the control element or the dropdown indicator
- Options appear in `filter__menu-portal` or `filter__menu`
- Each option has class `filter__option` or `[role="option"]`
- "Select All" option has `value: 'all'`, `label: 'Select All'`
- Selected values appear as multi-value chips with `filter__multi-value` class
- Clear indicator: `filter__clear-indicator` (X button to remove all)
- Individual chip remove: `filter__multi-value__remove`

**Single-select (Leads, New Application, Sales Rep)**:
- Same React Select but `isMulti=false`
- Selected value appears as `filter__single-value`
- Clear: click `filter__clear-indicator`
- Opening: click control or dropdown indicator

### Scoping Dropdowns

CRITICAL: Multiple React Select dropdowns exist on the same page. The page object must scope each dropdown by its label or form field name:
- Overview: scope by `label` text ("Merchant", "Location") within the filter panel
- Leads: scope by filter label within `FilterTable` component
- Sales Rep: scope within the `CollapsableEditLayout` with title "Merchant Info"
- Use pattern: `page.locator('label:has-text("Merchant")').locator('..').locator('.filter__control')` or similar

### Select All Mutual Exclusivity Logic (from source code)

```
Given: Select All is currently selected in Merchant
When:  User selects a specific merchant "Acme Corp"
Then:  Select All is REMOVED, only "Acme Corp" remains

Given: Merchant field has specific merchants selected (e.g., "Acme Corp", "Beta Inc")
When:  User selects "Select All"
Then:  All specific merchants are REMOVED, only "Select All" remains
```

This is implemented in `handleMerchantChange` (Open to Buy) and `getOverviewFilterProps` onChange (Overview).

### Location --> Merchant Auto-Add (Multi-select)

From `openToBuy/index.tsx` lines 168-176:
- When locations are selected (non-Select All), `allMerchantData` is filtered by `merchantLocation`
- Matching `merchantCode`s are set in `selectedMerchantCodes`
- The Merchant field itself is NOT auto-populated in Open to Buy (only `merchantCodes` are tracked)

BUT from `overview-table-config/index.tsx`, the Overview Location onChange does NOT auto-add merchants -- it only filters.

The task description says "selecting location(s) without merchant (multi) -> merchants correspondentes are auto-added to Merchant". This means the NEW `MerchantLocationFilters` component will implement this behavior. The test should verify the NEW behavior as described.

### Location --> Merchant Auto-Populate (Single-select)

From `get-leads-by-criteria-table-config/index.tsx` lines 255-278:
- When a location is selected, `merchantObject` is searched for merchants matching that location
- First match (`respectiveMerchants[0]`) is auto-set as Merchant value

### Clear Merchant --> Location Reset (Single-select)

From `get-leads-by-criteria-table-config/index.tsx` lines 233-236:
- When merchant is cleared (`!merchantName`), `setLocationNames([])` and `formik.setFieldValue('location', '')` are called

## Edge Cases

| # | Edge Case | How Test Handles It |
|---|-----------|-------------------|
| 1 | **Only 1 merchant available** | CT-UI-01/04 assumes 2+ merchants. If only 1, Select All logic still works but exclusivity test is trivial. Pre-check: count merchant options on page load. Skip mutual exclusivity sub-step if < 2 merchants. |
| 2 | **No locations for selected merchant** | Location dropdown should show no options. CT-UI-04 should handle empty location list gracefully (assert empty options). |
| 3 | **React Select dropdown portal renders outside filter panel** | Dropdowns may use `menuPortal` that renders at body level. Use `page.locator(SELECTORS.filterMenuPortal)` or `page.locator('[role="option"]')` not scoped to filter panel for option selection. |
| 4 | **Stale filter state from previous navigation** | Each CT reloads the page or navigates fresh to ensure clean state. |
| 5 | **Sales Rep panel not in write mode** | CT-UI-11/12 must click the edit icon first. If panel is not editable (permissions), the test should detect and skip/fail-fast. |
| 6 | **No lead in UW_APPROVED on qa2** | CT-UI-11/12 setup step queries DB. If no lead found, create one via `sendApplication` API with approved SSN. Fallback: skip CT-UI-11/12 with `test.skip()` and log reason. |
| 7 | **Sort indicators in Leads table headers** | `LeadsPage.getCleanHeaders()` already strips sort indicator characters. No additional handling needed. |
| 8 | **Open to Buy requires form validation (Location is required)** | Yup validation requires at least 1 location. CT-UI-06 must select locations before submitting. |
| 9 | **Network latency on location fetch** | After selecting a merchant, `getLocationNamesByMerchant` API call happens. Wait for dropdown options to update before asserting. Use `waitForResponse` or poll dropdown option count. |
