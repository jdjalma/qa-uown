## Descrição

Adicionar a página "Frequency Changes" no menu History do portal Servicing. A página deve exibir o histórico de alterações de frequência de pagamento de uma conta.

**Requisitos identificados:**

- Item de menu "Frequency Changes" acessível pelo dropdown History na barra lateral do Servicing
- Página renderiza tabela com colunas: Date, User, From, To, Old Payment, New Payment, First Due Date, Second Due Date
- Endpoint: `GET /uown/svc/accounts/{accountPk}/frequency-changes`
- Permissão necessária: `frequency_history [view]`
- Registros armazenados em `uown_frequency_mods`
- Ciclo de frequências suportadas: `WEEKLY`, `BI_WEEKLY`, `MONTHLY`, `SEMI_MONTHLY`
- `SEMI_MONTHLY` é a única frequência que exige `second_due_date`

## Bugs de Aplicação Encontrados

### BUG-01 — Tabela Frequency Changes vazia após SPA navigation (cache)

**Status:** OPEN
**Severidade:** Medium

**Descrição:**
A navegação SPA via menu History > Frequency Changes (router.push) não carrega os dados imediatamente na tabela. Um full page load (`page.goto`) carrega os dados corretamente após expiração do cache.

**Comportamento por ambiente:**
- **qa1:** `page.reload()` carrega dados imediatamente (10 rows visíveis)
- **stg:** SPA navigation e primeiro reload não carregam dados (cache mais longo); `page.goto` na tentativa 2 carrega dados corretamente (10 rows visíveis)

**Causa provável:** `frequencyChangesHistory` em `domain/stores/customer.tsx` falta `@observable`, impedindo MobX de notificar React na navegação client-side. Em stg, cache do frontend/API aumenta o tempo de expiração.

**Workaround no teste:** Retry via `page.goto` (até 3 tentativas) — dados carregam consistentemente na tentativa 2.
