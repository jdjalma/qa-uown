## Descrição

No fluxo CompleteApplication (tela CC/ACH), merchants com NeuroID habilitado estavam bloqueando clientes legítimos com a mensagem "Failed to Verification Identification". A causa raiz é que o serviço externo NeuroID, em várias execuções, retorna o status `PROFILE_NOT_FOUND` — que NÃO é um erro real, apenas significa que aquela identidade ainda não tem histórico no NeuroID. O comportamento anterior tratava esse status como falha e barrava a aplicação.

**Fix implementado em `NeuroIdVerificationService.java` (MR !1372):**

Adiciona um early-exit condicional no serviço:

```
if !skip.neuroid.verification:
    if neuroIdVerification != null
       AND status == PROFILE_NOT_FOUND
       AND approve.on.profile.not.found == true:
          updateLeadStatus → NEURO_ID_APPROVED
          neuroIdVerification.status = APPROVE
          persist → return "APPROVE"          ← EARLY EXIT (bypass)

    if (null | success==null | success==false | status != SUCCESS):
          updateLeadStatus → NEURO_ID_ERROR
          com reason="NeuroId verification failed" + LogType.INTERNAL  ← outside-scope
          neuroIdVerification.status = DECLINE
```

O fix tem dois efeitos principais:
1. **In-scope:** quando a config `approve.on.profile.not.found=true` está ativa e o NeuroID retorna `PROFILE_NOT_FOUND`, o lead é aprovado imediatamente (`NEURO_ID_APPROVED` + `FraudStatus.APPROVE`) e o fluxo prossegue para Terms & Conditions + e-sign normalmente, sem interromper o cliente.
2. **Outside-scope (audit improvement):** o caminho de falha genuína passa a chamar `updateLeadStatus` com `reason="NeuroId verification failed"` + `LogType.INTERNAL` (ambos não-nulos), melhorando a rastreabilidade regulatória.

**Ambiente qa2 (confirmado):**
- `approve.on.profile.not.found=true` (bypass ativo) — 1ª execução
- `approve.on.profile.not.found=false` (bypass desativado) — 2ª execução, após toggle pelo usuário para validar a regressão do caminho feliz sob a flag OFF
- Siteids NeuroID: `items340` (SEND_APP) / `depth355` (SUBMIT_APP)
- Merchants habilitados sob teste: Saslow's Jewelers (`IL90206-0003`) e Bodega Furniture (`KS1011`), ambos com `use_neuro_id_check=true`

**Requisição adicional do usuário:** capturar todo o tráfego de rede relacionado ao NeuroID durante a execução da tela de contrato, como evidência objetiva do comportamento do serviço externo.

### CT-07

**Objetivo:** Verificar que, após o bypass (ou sucesso genuíno) do NeuroID, o cliente consegue avançar naturalmente por Terms & Conditions + e-sign até um estado pós-contrato, comprovando que a aplicação não é mais interrompida.

**O que é verificado:** Este CT prova o objetivo de negócio da feature: "cliente completa CompleteApplication sem interrupção". O fluxo UI é navegado até o fim (T&C + e-sign), e o `lead_status` final é polled no DB até alcançar `CONTRACT_CREATED` ou além.

- Saslow (lead 15350): bypass acionado → Terms & Conditions completados → e-sign (SignWell embedded iframe) completado → `lead_status=SIGNED` (avançou além de `CONTRACT_CREATED`).
- Bodega (lead 15351): sucesso genuíno do NeuroID → Terms & Conditions completados → o iframe SignWell não ficou visível em 30s (ver bloco "Observações outside-scope"), mas o DB já mostrava `lead_status=CONTRACT_CREATED` — prova de que o NeuroID não interrompeu o fluxo. O incidente de UI pós-NeuroID não afeta a validação da feature.

Exemplos:
| Lead | Estado final DB | Bloqueio "Failed to Verification Identification" em tela? |
|------|-----------------|-----------------------------------------------------------|
| 15350 (Saslow) | SIGNED | NÃO |
| 15351 (Bodega) | CONTRACT_CREATED | NÃO |

#### Como verificar manualmente

1. No browser, continuar do CT-04 (já submetido o CC/ACH).

## Observações outside-scope (não bloqueantes, não são bugs da feature)

> Aplicado o Bug Triage Protocol — os itens abaixo foram confirmados como **fora do escopo** da feature testada ou como **limitações de ambiente**, e NÃO figuram em `## Bugs de Aplicação Encontrados`.

1. **Saslow's Jewelers Henderson (merchant_pk=8) — programas ausentes no início do run:** a primeira tentativa de `sendApplication` devolveu `transactionStatus=E4 / "An unexpected error has occurred"` com `paymentDetailsList=[]` por falta de programs configurados no merchant em qa2. O usuário corrigiu mid-run inserindo programas via DB (autorização explícita). Após a correção, o fluxo executou normalmente. → Config de ambiente, não é bug da feature.

2. **SignWell e-sign iframe timeout em Bodega qa2 (lead 15351):** o `#SignWell-Embedded-Iframe` não ficou visível em 30s para o Bodega. Contudo, o DB já mostrava `lead_status=CONTRACT_CREATED` (pós-NeuroID), o que valida o objetivo da feature (não houve bloqueio pelo NeuroID). → Incidente de UI pós-NeuroID, não relacionado ao fix de `NeuroIdVerificationService`. Possível investigação separada sobre performance/flakiness do SignWell em qa2.

3. **`uown_neuro_id_verification.site_id` null em ambos os registros (pks 17 e 18):** apesar dos siteids `items340`/`depth355` estarem configurados corretamente na config qa2 e dos endpoints terem sido invocados (`caller=SUBMIT_APP` preenchido), a coluna `site_id` ficou `null` nas duas linhas. → Potencial gap de audit data — recomenda-se confirmar no código Java se `siteId` é persistido no caminho atual do `NeuroIdVerificationService` (fora do diff do MR !1372). Não bloqueia a feature alvo.

4. **MR !1376 (closed, não merged):** rename do config key para camelCase (`approveOnProfileNotFound`). Fechado sem merge — o prod e qa2 continuam usando a versão dotted `approve.on.profile.not.found`. Mantido aqui apenas como referência caso o rename seja feito no futuro. Sem impacto no teste atual.

5. **`classifyNeuroIdOutcome` retornou `UNKNOWN` no dump de rede:** como explicado no CT-04, o SDK frontend do NeuroID não expõe o status final no body das responses observáveis pelo browser. O status real está em `uown_neuro_id_verification` e é validado pelo CT-05. → Limitação conhecida do SDK, não é bug.

6. **`PROFILE_NOT_FOUND` é não-determinístico em qa2:** das 4 execuções de fluxo completo (2 merchants × 2 runs), apenas a 1ª (lead 15350, Saslow, flag=ON) acionou `PROFILE_NOT_FOUND` naturalmente — todas as subsequentes retornaram `SUCCESS`. O comportamento do serviço externo NeuroID depende da completude dos eventos do SDK frontend na sessão (heurística do provedor) e da presença de perfil comportamental correlato. Isso **não impede a cobertura do bypass** (já comprovado no run 1) mas **impede cobrir o caminho NEURO_ID_ERROR de forma determinística** sem mock. → Limitação do ambiente; o caminho de falha enriquecida (outside-scope da tarefa) fica registrado como follow-up.
