## Contexto

**Sintoma reportado em PROD:** merchants com NeuroID habilitado recebiam "Failed to Verification Identification" na tela CC/ACH mesmo com dados corretos. A causa raiz é NeuroID retornando `PROFILE_NOT_FOUND` (não um erro literal).

**Fix em `NeuroIdVerificationService.java`:**

```
if !skip.neuroid.verification:
   if neuroIdVerification != null
      AND status == PROFILE_NOT_FOUND
      AND approve.on.profile.not.found == true:
         updateLeadStatus → NEURO_ID_APPROVED
         neuroIdVerification.status = APPROVE
         persist → return "APPROVE"            ← EARLY EXIT (bypass)

   if (null | success==null | success==false | status != SUCCESS):
         updateLeadStatus → NEURO_ID_ERROR
         com reason="NeuroId verification failed" + LogType.INTERNAL   ← outside-scope
         neuroIdVerification.status = DECLINE
```

**Merchants sob teste (qa2):**

| Ref | Display name | Username | Senha (qa2) |
|-----|--------------|----------|-------------|
| `IL90206-0003` | Saslow's Jewelers | `saslowJewelers` | `U0wn_saslowJewelers_fGoj3p` |
| `ks1011` | Bodega Furniture | `kornerstone` | `U0wn_Kornerstone_012c` |

> **Nota:** Saslow's Jewelers não está no catálogo `src/data/merchants.ts` — adicionar na Fase 5.

**Config qa2 confirmada (NeuroIdVerificationService):**
```
approve.on.profile.not.found: "true"       # bypass ATIVO em qa2
# approveOnProfileNotFound: "true"         # (commented) — MR !1376 rename, não ativo
neuro:
  id:
    siteid:
      SEND_APP: "items340"                 # siteid usado na chamada a partir de sendApplication
      SUBMIT_APP: "depth355"               # siteid usado na chamada a partir de submitApplication
```

> **Implicação:** o happy-path do bypass (CT-06) é ativo no ambiente — se NeuroID retornar `PROFILE_NOT_FOUND`, lead → `NEURO_ID_APPROVED` automaticamente.

**Detecção por rede (user request):** capturar TODAS as requisições e respostas relacionadas a NeuroID via `page.on('request')` / `page.on('response')`. Escopos:

- Frontend SDK: `*.neuroid.com`, `*.neuro-id.com`, `*.nid.com`, chamadas ao bundle JS do SDK
- Siteids para filtro/dedup: `items340` (SEND_APP), `depth355` (SUBMIT_APP) — presentes em query string ou payload
- Backend svc: POST `/uown/los/sendApplication`, POST `/uown/los/submitApplication` (cuja response indica resultado da verificação NeuroID no backend)
- Qualquer response body / header indicando `neuroIdStatus`, `fraudStatus`, etc.

## User Stories

### US-NEURO-01: Cliente em merchant com NeuroID habilitado conclui aplicação mesmo quando NeuroID não encontra perfil

**Como** cliente final submetendo CC/ACH em merchant com NeuroID habilitado,
**Quero** não ser bloqueado quando o serviço NeuroID retorna `PROFILE_NOT_FOUND`,
**Para** concluir minha aplicação sem erro externo bloqueante.

**Critérios de aceite:**
- [ ] Quando `approve.on.profile.not.found=true` + `neuroIdStatus=PROFILE_NOT_FOUND` → lead transiciona para `NEURO_ID_APPROVED`.
- [ ] `uown_neuro_id_verification.status` = `APPROVE` (registro persistido).
- [ ] Fluxo continua até `CONTRACT_CREATED` ou além (não é interrompido).
- [ ] Ao reabrir a URL pós-submit, apenas o botão submit aparece — não há campos visíveis.
- [ ] Log interno registrado: `"[NeuroIdVerificationService] Profile not found — skipping per config"`.

### US-NEURO-02: Auditoria registra audit log completo em falhas de NeuroID (impacto outside-scope)

**Como** analista de compliance,
**Quero** que toda falha de NeuroID deixe registro auditável com `reason` + `logType`,
**Para** rastrear regulatoriamente a decisão de negar a aplicação.

**Critérios de aceite:**
- [ ] Em falha (não-PROFILE_NOT_FOUND OU bypass desativado), `updateLeadStatus` é chamado com `reason="NeuroId verification failed"` e `LogType.INTERNAL` — ambos não-nulos.
- [ ] `uown_neuro_id_verification.notes` começa com `"Neuro Id verification Failed"` + `errorMessage`.
- [ ] Lead transiciona para `NEURO_ID_ERROR` + FraudStatus `DECLINE`.

### US-NEURO-03: QA detecta e inspeciona tráfego de rede do NeuroID durante a aplicação

**Como** QA validando o fix,
**Quero** que o teste capture e reporte todas as requisições/respostas NeuroID (SDK + backend),
**Para** ter prova objetiva do status retornado pelo serviço externo e da decisão do backend.

**Critérios de aceite:**
- [ ] Teste escreve artifact (JSON ou anexo Playwright) com `{url, method, status, requestBody?, responseBody?}` de cada request NeuroID interceptado.
- [ ] Presença de pelo menos uma request ao domínio NeuroID ou ao endpoint backend que consome NeuroID.
- [ ] Status observado (`SUCCESS`, `PROFILE_NOT_FOUND`, `FAILED`, etc.) é persistido no relatório do teste.

### US-NEURO-04: Ambos os merchants configurados no qa2 têm suporte mapeado no automation

**Como** automation engineer,
**Quero** que Saslow's Jewelers e Bodega Furniture estejam registrados e resolvíveis em `buildTestData`,
**Para** executar o teste sem hard-coded credentials.

**Critérios de aceite:**
- [ ] `getMerchant('SaslowsJewelers', 'qa2')` retorna credenciais corretas.
- [ ] `getMerchant('BodegaFurniture', 'qa2')` retorna credenciais corretas.
- [ ] Senhas qa2 configuradas sem vazar em logs.

### CT-06: Lead transiciona para NEURO_ID_APPROVED quando bypass é acionado (PROFILE_NOT_FOUND + flag ON)

**Tipo:** DB + API validation
**Pré-condição:** CT-05 evidenciou `neuro_id_status = PROFILE_NOT_FOUND` E config `approve.on.profile.not.found = true` no qa2 para o merchant.

**Passos:**
1. Poll `uown_los_lead.lead_status` para o `leadPk` — aguardar transição.
2. Verificar `uown_los_lead_log` (ou equivalente) por entrada com reason `"[NeuroIdVerificationService] Profile not found — skipping per config"`.

**Resultado esperado:**
- `lead_status` transitou para `NEURO_ID_APPROVED` (ou seguinte estado no fluxo, ex: `CONTRACT_CREATED`).
- Entrada de log interno presente com reason esperado.
- `uown_neuro_id_verification.status = APPROVE`.

**Tags:** `@regression @qa2 @critical @happy-path`

**Condicional:** se `neuro_id_status != PROFILE_NOT_FOUND` na execução, este CT é reportado como `N/A — condição de bypass não acionada` e CT-07 assume a validação. Bypass flag CONFIRMADA ativa em qa2 (`approve.on.profile.not.found=true`), então o único cenário em que CT-06 não é acionado é NeuroID retornar `SUCCESS` genuíno (também positivo).
