## Descrição

Implementação de autenticação para parceiros MERCHANT na API LOS. O `RequestFilter` no SVC valida o header `Authorization` para paths sob `/uown/los/merchant/applications/*`, exigindo formato `Bearer <key>` para `apiUserType = MERCHANT`. O Istio/Envoy intercepta qualquer header `Authorization: Bearer *` e valida como JWT antes do request chegar ao SVC, criando um conflito com o fluxo hex key do `RequestFilter`. Os testes validam o comportamento do `RequestFilter` enviando a raw hex key (sem Bearer prefix) para bypassar o Istio.

## Bloqueio Documentado: Conflito Istio/Envoy vs RequestFilter

O step B (happy path Bearer) não pode ser validado end-to-end porque o Istio intercepta `Authorization: Bearer *` e exige JWT. O CT-09 documenta e prova esse comportamento. O `RequestFilter` do SVC está correto (aceita `Bearer <hex_key>` para MERCHANT), mas nunca é alcançado quando o Bearer prefix está presente.

```
Bearer <hex_key> → Istio → 401 "Jwt not in Header.Payload.Signature format"
<hex_key> (raw)  → Istio ignora → SVC RequestFilter → 401 "Invalid API KEY" (MERCHANT exige Bearer)
```

**Solução proposta:** `ApiKeyService.java:111` — `MERCHANT -> true` (aceitar ambos os formatos).
