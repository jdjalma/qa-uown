  - 🟢 executável em sandbox hoje sem setup adicional
  - 🟡 precisa setup (UPDATE/UI/criação de dado) — autorizado pelo usuário

- **Convenções de severidade do gap**:
  - 🔴 ALTA — risco regulatório, financeiro ou de imagem em produção
  - 🟡 MÉDIA — comportamento errado mas contornável

### Grupo 2 — Filtros de elegibilidade (regressão dos 27 filtros do SQL)

Cada cláusula do `StickyRecoverSweep.SQL` é um CT regressão. Os principais já documentados no SPEC inicial:

- `cct.status='DENIED'` (positivo) + outros status (negativos: APPROVED, ERROR, PENDING)
- `cct.cc_vendor='CHANNEL_PAYMENTS_CC'` (positivo) + outros vendors (USAEPAY, OMNIFUND)
- `cct.cc_transaction_type='SCHEDULED'` (positivo) + outros tipos (RERUN, DAILY_DELINQUENCY_RUN, REQUEST, _100_DAY_DELINQUENCY_RUN, OTHER)
- `cct.cc_action='SALE'` (positivo) + CREDIT, AUTHENTICATION, TOKENIZATION, CAPTURE (negativos)
- `cct.posting_date = today-7` exato — D-6 ignorado, D-8 ignorado
- `account.account_status='ACTIVE'` + outros (CANCELLED, PAID_OUT, CLOSED, etc — negativos)
- `cc.auto_pay=true` + autopay desligado
- `rating NOT IN (P,C,D,B)` + cada rating excluído
- `delinquency_as_of_date <= today` + conta sem delinquência
- Lista de errors excluídos (8 itens) — cada um como CT negativo
- `comment NOT LIKE 'Idempotent transaction was run. %'`
- `agent_username NOT IN ('SpecialProcess#5014')`
- Dedupe `NOT EXISTS uown_sticky` quando sticky_transaction_id já existe

Status: ✅ documentados no SPEC; ❌ ainda não automatizados.

## Parte B — Questionamentos abertos (perguntas de negócio para Yuri/Gustavo)

Lista exaustiva das dúvidas levantadas durante a investigação que **dependem de alinhamento com produto/dev** antes de virar CT formal.

### B1 — Regras de negócio do recovery

1. **Qual a expectativa quando o cliente quita o lease durante recovery ativa?** Cancel `RESOLVED`? Aguardar Sticky tentar e capturar o duplicado? Cancelar manualmente via agente?
2. **Cliente em payment arrangement (rating=P) criado APÓS submit à Sticky** — recovery deve ser cancelada? Em qual momento (rating muda primeiro, ou um sweep guarda)?
3. **Bankruptcy declarado (rating=C/D/B) APÓS submit** — qual a obrigação legal de pausar (automatic stay)? Quanto tempo?
4. **Conta vendida (SOLD/rating=S) APÓS submit** — UOWN tem obrigação contratual de cancelar com Sticky? Multa por cobrar dívida vendida?
5. **Cliente desliga autopay durante recovery** — Sticky continua usando token guardado conta como falta de consentimento MIT?
6. **Cliente troca cartão durante recovery** — Sticky deveria retentar com token novo? Cancel + new submit? Ou aceitar perda dessa recovery?

### B2 — Fees e tipos de receivable

7. **NSF_FEE negada deve virar candidata a Sticky?** Custo da operação (~$0.X por chamada Sticky) vs valor da fee ($25–$30) — vale a pena?
8. **PROCESSING_FEE / MISC_FEE / MANUAL_FEE — mesmo questionamento.**
9. **PROTECTION_PLAN_FEE** — é parte da parcela ou cobrança separada? Se separada, faz sentido recover só essa?
10. **REINSTATEMENT_FEE** — fee aplicada quando conta é reativada. Negada gera recovery? Provavelmente sem sentido (conta acabou de voltar do CANCELLED, recovery 7 dias depois cobra fee de reativação que já foi resolvida).

### B3 — Comunicação ao cliente

11. **Cliente deveria ser notificado antes da 1ª tentativa Sticky?** Requisito regulatório em algum estado US?
12. **Cliente deveria ser notificado de cada tentativa?** Frequência de comunicação aceitável.
13. **`recovery.successful` aciona "Payment Received" email padrão ou template específico?** Diferenciar para cliente saber que foi via Sticky?
14. **`recovery.unsuccessful` aciona algum email/escalação?** Cliente deve saber que tentamos e falhamos?

### B4 — Lead note / activity log

15. **Padrão de texto do lead_note Sticky** — formato esperado? `[StickyRecover][submit]`, `[StickyRecover][webhook][recovery.started]`, etc?
16. **Granularidade**: cada attempt_failed gera lead_note ou só o start/end?
17. **Cancel via API gera lead_note diferente do cancel automático (guardian sweep)?** Auditoria precisa distinguir?

### B5 — Allocation strategy quando recovery successful

18. **Onde o pagamento da recovery successful aterrissa?**
    - Receivable original (que já está com `delinquency_as_of_date` no passado)?
    - Ou na primeira parcela ativa em aberto (que pode ser uma diferente, dias depois)?
19. **Valor cobrado pela Sticky pode ser diferente da parcela atual** (ex: parcela teve juros adicionados nos 30 dias). Sticky retentou valor antigo. Como reconciliar?
20. **Late fee acumulado durante o período Sticky** — entra junto na cobrança ou continua em aberto?

### B6 — Operação e admin

21. **Yuri/ops conseguem cancelar manualmente uma recovery sem entrar no banco?** Botão no Servicing? Endpoint admin?
22. **Yuri/ops conseguem desligar o sweep em emergência?** Hoje é UPDATE manual em `uown_scheduled_task.is_active=false`.
23. **Dashboard/relatório de taxa de recovery (sucesso vs unsuccessful)** existe? Onde?
24. **Como ops sabe que uma recovery ficou travada** (RECOVERY_STARTED por X dias sem webhook)?

### B7 — Sticky-side (perguntar ao suporte deles)

25. **`dunningProfileId` enviado no body do `/recover` é honrado ou sobreposto pelo gateway profile default?** Validamos empiricamente: enviamos 223, webhook reporta 225. Pergunta direta.
26. **Qual a janela máxima de retentativa de cada dunning profile (223 weekly, 224 biweekly, 225 monthly)?** Dias totais? Número de tentativas?
27. **Sticky aceita resubmit do mesmo `originalTransaction.id` após sessão `unsuccessful`?** Doc diz que sim, mas com algum cooldown?
28. **Quando o gateway responde "Insufficient funds", Sticky considera retryable por default?** Como interpreta `Kount Denied`, `Do not Honor`, decline customizados?
29. **Bulk APIs (`bulk-recovery`)** — UOWN poderia migrar para reduzir overhead? Vantagens vs 1-a-1?

### B8 — Ambientes e config

30. **`paymentProfileId=153`** é só sandbox? Qual o ID em prod? Já está pronto?
31. **Hardcoded sandbox fallback no `StickyRecoverConfig.java`** — quando promove pra prod, qual a salvaguarda de não cair no fallback?
