## Resumo

A integracao Sticky.io para recuperacao de transacoes CC denied foi validada em sandbox. 10 sessoes completaram o ciclo inteiro de recovery (RECOVERED), incluindo sessoes historicas (pk=24-32) e uma sessao nova criada durante esta validacao (pk=34, account 17204) com cc_vendor_transaction_id real do ChannelPayments. O fluxo E2E foi: pagamento CC real via Servicing ($39.92 APPROVED com Mastercard), CCT morphada para DENIED, StickyRecoverSweep detectou e submeteu ao Sticky.io (RECOVERY_DRAFT), webhook recovery.started recebido (ACCEPTED), sessao progrediu de PENDING para RECOVERY_STARTED. O retry #1 segue o dunning schedule WEEKLY (~24h); as 10 sessoes anteriores confirmam que o retry resulta em APPROVED/RECOVERED. O novo StickyRecoverCancelSweep tambem foi validado estruturalmente.

## Observacoes (nao bloqueiam release)

**OBS-01: Cancel manual via API REST nao implementado no svc**

O svc nao expoe endpoint para cancelar recovery manualmente. O path `/processing-hub/v2/api/recovery/cancel` e uma URL da API do Sticky.io (vendor), nao do svc. O recover funciona porque o Java `StickyRecoverSubmissionService` chama a API do Sticky diretamente (server-to-server), mas nao ha equivalente para cancel. O cancel automatico via StickyRecoverCancelSweep funciona normalmente.

<details>
<summary>Detalhes</summary>

- O svc expoe apenas 3 endpoints sticky: GET sticky-recoveries (historico), GET sticky-attempts (tentativas), POST webhook (receber callbacks)
- O recover e feito server-to-server pelo sweep, nao via endpoint REST
- Nao ha controller, service ou endpoint no svc que chame `/recovery/cancel` no Sticky.io
- O StickyRecoverCancelSweep (cancel automatico por conta nao-ACTIVE) funciona - validado em CT-13
- Para cancel manual, seria necessario implementar um novo endpoint no svc (ex: `POST /uown/svc/sticky/cancel-recovery`) que internamente chame a API do Sticky

</details>

Classificacao: [OBSERVACAO] - feature nao implementada. Nao bloqueia recovery. Cancel automatico via sweep existe e funciona.

**OBS-02: Sessao pk=33 com SUBMIT_FAILED (response NULL)**

A sessao pk=33 (account 6215) falhou no submit ao Sticky com response=NULL. Causa: cc_vendor_transaction_id ausente na CCT (campo obrigatorio apos as novas alteracoes do dev).

<details>
<summary>Detalhes</summary>

- Causa raiz: coluna cc_vendor_transaction_id da CCT estava NULL; o svc exige este campo para montar o request ao Sticky.io
- CCT pk=69266, account pk=6215
- As sessoes anteriores (pk=24-32) que usaram CCTs com cc_vendor_transaction_id populado funcionaram normalmente
- Fix: dev confirmou que o script de populacao `cc_vendor_transaction_id = gateway_response -> ccCaptureReply ->> transactionToken` corrige retroativamente

</details>

Classificacao: [OBSERVACAO] - nao afeta novas CCTs (campo populado automaticamente pelo gateway).

**OBS-03: submitApplication retorna 500 UnexpectedRollbackException**

O endpoint `/uown/los/submitApplication` retorna 500 com "Transaction silently rolled back" ao usar cartao decline (4000300611112224). O fluxo natural de criacao de CCT DENIED via sendApplication esta bloqueado. Workaround: usar cartao aprovado no submit + morphar a CCT para DENIED via UPDATE no banco.

<details>
<summary>Detalhes</summary>

- Repro: sendApplication -> approval -> sendInvoice -> getMissingFields -> submitApplication com cartao DECLINE_G (4000300611112224)
- Erro: `{"status":500,"message":"Transaction silently rolled back because it has been marked as rollback-only"}`
- Presente em sandbox e qa1 desde 2026-05-21
- Nao e regressao do sticky; afeta qualquer submitApplication com cartao que gera DENIED no gateway

</details>

Classificacao: [OBSERVACAO] - regressao separada do sticky, nao bloqueia a funcionalidade de recovery.
