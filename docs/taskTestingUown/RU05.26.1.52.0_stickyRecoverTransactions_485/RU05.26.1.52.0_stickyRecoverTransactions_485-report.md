#### Diagnóstico das falhas iniciais (RESOLVIDOS)

**CT-01 — Quartz scheduler delay (test bug):**

- Setup fast-path estava correto: cct 82932 / account 17178 passou todas as 14 cláusulas SQL do sweep (verificado clause-by-clause via `src/scripts/diagnose-sticky-485-v2.ts`)
- Sweep `pk=6344540` rodou às `01:29:22 UTC` com `processed=1`, sem erros → outbound `pk=6` com `RECOVERY_DRAFT` + `stickyTransactionId=9442241e…`
- **Causa raiz:** `triggerScheduledTask` enfileira no Quartz mas a execução é assíncrona; observada janela de até ~37 min até o scheduler pegar a task. O timeout de 90s do `waitForStickySession` era insuficiente.
- **Fix:** novo helper `waitForStickyRecoverSweepRun` aguarda entrada nova em `uown_sweep_logs` antes de pollar `uown_sticky`; timeout aumentado para 6 min. CT-01 e CT-08 atualizados.

**CT-11 — Race condition no UI test (test bug):**

- API endpoint `/uown/svc/accounts/17176/sticky-recoveries?page=0&size=1000` retorna `Page<>` corretamente: `content[0].creditCardTransactionPk=82912`, `recoveryStatus="RECOVERY_STARTED"`, `stickyTransactionId="80587c4…"` (validado via `src/scripts/probe-sticky-endpoint.ts`)
- FE `domain/stores/payment.tsx::getStickyRecoveries` (commit `9c2e651` MR !691) já trata ambos os shapes (`Array` E `Page<>.content[]`)
- **Causa raiz:** `loadCreditCardHistory(accountPk)` chama `getCCTransactions` + `getStickyRecoveries` em `Promise.all`. Rows renderizam quando getCCTransactions resolve (rápido) — antes do getStickyRecoveries propagar via MobX. Cells exibem `—` (fallback) até o sticky data chegar e MobX re-renderizar.
- **Fix:** page object ganhou `waitForStickyRecoveriesResponse()` (escuta `waitForResponse`) + `waitForStickyCellPopulated(txPk)` (poll até cell deixar de ser `—`). UI spec arma `waitForResponse` ANTES da navegação para não perder o response do bootstrap.

> O doc anterior `…-fe-bug-servicing-page-mapper.md` foi **RETRATADO** — não há bug de produto. Detalhes lá.

#### 1. Rating `M` (MR Money Owed) não excluído do sweep — semântica TBD

**Severidade:** 🟡 Média · **Categoria:** Produto · **Evidência:** conta 17176 → sticky_pk=7, txn=`80587c4...`

**O que é o rating `M`** (fonte: `common/src/main/java/com/uownleasing/common/enumeration/RatingLetter.java`)
Letra do enum `RatingLetter` com a descrição `"MR Money Owed"`. **Semântica operacional não-óbvia.** A doc histórica do projeto afirmava que `M` representava "Military/SCRA" e desligava auto-pay, mas:

- O enum oficial define apenas `"MR Money Owed"` — sem menção a SCRA;
- **Nenhum** filtro SQL no svc inteiro exclui `M`: nem `ScheduledACHPayments`, nem `ScheduledCreditCardPayments`, nem `RerunACH/CC`, nem `CCDailyScheduledDeniedRerun`, nem `CustomerPortalReminder`, nem `CCVintageRun`, nem `DelinquencyRerunCC`;
- Ou seja, do ponto de vista do sistema, `M` é **um rótulo sem efeito sistêmico via SQL** — possivelmente usado por agentes de cobrança como anotação semântica, ou via toggle separado em `cc.auto_pay`, ou ainda como item TODO de produto.

**Observação empírica**
Em sandbox, criamos uma conta com `rating='M'` (conta 17176) — o `StickyRecoverSweep` submeteu normalmente à Sticky (sticky_pk=7), exatamente o mesmo comportamento que `rating IS NULL`.

**Por que isso é uma Melhoria — não um bug confirmado**
Sticky seguir o mesmo padrão dos outros sweeps (não excluir `M`) é, hoje, **coerente** com o resto do sistema. Para classificar como bug é necessário confirmar com Gustavo/Yuri **qual é a intenção operacional do rating `M`**:

- (a) Se `M` é apenas anotação visual sem efeito sistêmico → Sticky está correto em não excluir;
- (b) Se `M` deveria desligar auto-pay (como a doc histórica afirmava) → há bug em todos os sweeps de pagamento do svc, e o gap precisa ser corrigido em escala, não só no Sticky;
- (c) Se `M` foi pensado para uso futuro / WIP → escopo deve ser definido por produto.

**Sugestão técnica**

1. Confirmar com Gustavo Martins / Yuri Araujo qual é a intenção operacional de `rating='M'`.
2. Se a resposta for (b), abrir tarefa separada para alinhar **todos** os sweeps de pagamento (Sticky + Scheduled + Rerun + etc) a excluir `M`.
3. Se a resposta for (a) ou (c), **fechar este item como esperado** e ajustar a doc `06-conta-ciclo-vida.md` removendo a afirmação de que `M` desliga auto-pay.

#### 2. Ratings `F` (Fraud) e `S` (Sold Accounts) não excluídos do `StickyRecoverSweep`

**Severidade:** 🔴 Alta · **Categoria:** Regulatório · **Evidência:** conta 17177 (`F`) → sticky_pk=6, txn=`930fdf6...`; conta `S` não reproduzida empiricamente (sandbox tem 1 conta `S=pk 16616` sem cct DENIED) — inferida do filtro `B,C,D,P` do `StickyRecoverSweep` que comprovadamente não contém `F` nem `S`.

**Precedente no código**
Os sweeps de recovery/rerun mais conservadores do svc **já excluem** `F` e `S`:

- `DelinquencyRerunCCPaymentsSweepService` (linha 269, 286) exclui `B,C,P,S,D,E,F,G,L,U`;
- `CCVintageRunHelper` (linha 94) exclui `B,C,P,S,D,E,F,G,L,U`;
- `CustomerPortalReminder` (BootstrapService:858) exclui `B,C,D,F,G,S,U`.

O `StickyRecoverSweep`, por outro lado, exclui apenas `B,C,D,P` — **não alinhado** com os outros sweeps de mesma natureza.

**Cenário real (F — Fraud)**
Maria abriu disputa com seu cartão alegando que compras passadas foram não-autorizadas. Investigação interna confirmou a fraude e a conta foi marcada `rating='F'` (enum: `"Fraud"`). Uma CCT antiga ainda está em D-7. `DelinquencyRerunCC` correto excluiria — mas o `StickyRecoverSweep` ignora a fraude confirmada e submete à Sticky, que retenta o mesmo cartão fraudado por 30 dias. Cada tentativa vira novo chargeback que volta contra UOWN.

**Cenário real (S — Sold Accounts)**
Conta de Carlos foi vendida a um debt buyer (`rating='S'`, enum: `"Sold Accounts"`); contratualmente UOWN não tem mais direito de cobrar. Sticky continua tentando, debitando uma conta sobre a qual UOWN não tem mais nenhum interesse jurídico.

**Impacto percebido**

- **(F)** chargebacks acumulados pioram o BIN reputation no card network e custam $15–25 por dispute;
- **(S)** cobrança em dívida vendida configura double-collection sob UCC §9 e potencialmente FDCPA — risco de processo do próprio debt buyer;
- **Inconsistência interna** — `DelinquencyRerunCC` e `Sticky` cobrem o mesmo terreno (retentar CCTs negadas), mas com filtros diferentes — fácil para um cliente cair em um e não no outro.

**Sugestão técnica**
Estender a cláusula `a.rating NOT IN (...)` do `StickyRecoverSweep` para incluir pelo menos `F` e `S`, alinhando com `DelinquencyRerunCC`. Idealmente, padronizar a lista entre os sweeps de recovery (`StickyRecoverSweep`, `DelinquencyRerunCC`, `CCDailyScheduledDeniedRerun`) com revisão de produto.

#### 3. SQL bug: `comment NOT LIKE 'Idempotent…'` descarta silenciosamente CCTs com `comment IS NULL`

**Severidade:** 🔴 Alta · **Categoria:** Produto · **Evidência:** setup com 3 CCTs retornou `processed=0` até forçar `comment='Sticky setup'`

**Cenário real**
Uma CCT recém-criada via `CreateScheduledCreditCardPaymentsSweep` tem `comment=NULL` (default da tabela). Cai em D-7 com decline genuíno. O sweep aplica `cct.comment NOT LIKE 'Idempotent transaction was run. %'` — em ANSI SQL `NULL NOT LIKE '…'` resulta em `NULL` (não `TRUE`), então a CCT é **silenciosamente excluída do conjunto**. Em produção, a vasta maioria das CCTs naturais tem `comment=NULL`.

**Impacto percebido**

- Feature foi mergeada operando "parcialmente" — boa parte do volume real de CCTs negadas (provavelmente a maior fatia) nunca chega à Sticky;
- Receita perdida silenciosa, sem alarme;
- Só descobrimos porque o setup retornou zero linhas processadas.

**Sugestão técnica**
Reescrever como `(cct.comment IS NULL OR cct.comment NOT LIKE 'Idempotent transaction was run. %')` — mesmo ajuste no constant `StickyRecoverSweepService.SQL` e no SQL persistido em `uown_scheduled_task`.

#### 4. Campos "Highly Recommended" da Sticky não estão sendo enviados

**Severidade:** 🔴 Alta · **Categoria:** Integração · **Evidência:** payload `uown_sticky_outbound_log` pk=4 (account_pk=17176)

**Cenário real**
Uma CCT foi negada por Kount (fraude provável) — decline definitivo. O submit à Sticky vai sem `originalTransaction.retryable=false`, sem `networkAdviceCode`, sem `bin`, sem `processorDeclineCode`, e sem `billingAddress`. A Sticky, sem o sinal `retryable=false`, **retenta 7+ vezes em 30 dias** — todas negadas pelo issuer porque o flag de fraude já está colado ao PAN. Sem `billingAddress`, o AVS no issuer falha e mais tentativas viram decline. Sem `networkAdviceCode`, a Mastercard "Do Not Retry" advice é ignorada.

**Impacto percebido**

- Taxa de sucesso Sticky cai drasticamente para o volume Kount-denied (quase certo: 0%);
- Custo por chamada Sticky paga sem retorno;
- Tentativas excessivas em PANs marcados podem ativar bloqueio reputacional do BIN no card network — UOWN paga caro por tentar o que nunca daria certo.

**Sugestão técnica**
Estender o constructor `StickyRecoveryRequest` (na lib `com.uownleasing:sticky:1.52.0`) para aceitar esses campos, e setar `retryable=false` em `StickyRecoverSubmissionService` quando o `error` casar com uma lista de erros não-recuperáveis (Kount Denied, fraude genérica, AVS hard failures). Popular `billingAddress` a partir do `CCInfo` (`addr1/city/state/zip` já disponíveis no service).

#### 5. `dunningProfileId` pode estar sendo ignorado pela Sticky

#### 6. Janela rígida `posting_date = today - 7 days` sem catch-up

**Severidade:** 🟡 Média · **Categoria:** Produto · **Evidência:** SQL do sweep em `uown_scheduled_task` (pk=80)

**Cenário real**
Quarta-feira o pod do scheduler crashou por 6 horas. Todas as CCTs cujo `posting_date = wednesday - 7d` jamais serão processadas — o filtro do sweep usa `=`, não `>=`. Próximo feriado prolongado, próximo deploy malsucedido, próxima outage: mesma perda. A Sticky não impõe limite máximo de idade da transação, então a restrição é puramente UOWN-side.

**Impacto percebido**

- Receita perdida silenciosamente (sem alerta, sem métrica de "CCT skipped");

- Cliente que poderia ter sido recuperado simplesmente nunca foi tentado.

**Sugestão técnica**
Mudar para `posting_date BETWEEN CURRENT_DATE - INTERVAL '14 days' AND CURRENT_DATE - INTERVAL '7 days'`, ou usar uma coluna dedicada `submitted_to_sticky boolean` na `cc_transaction`. Adiciona catch-up sem retentar transações recentes em duplicidade.

#### 7. Dedupe local mais restritivo que o da Sticky

**Severidade:** 🟡 Média · **Categoria:** Produto · **Evidência:** sub-query `NOT EXISTS (... WHERE sticky_transaction_id IS NOT NULL)` no SQL do sweep

#### 11. Exceptions da aplicação virando `declineReason`

**Severidade:** 🟡 Média · **Categoria:** Integração · **Evidência:** CCTs sandbox pk=68330, pk=68983 com `error='org.springframework.orm.jpa.JpaSystemException: ...'`

**Cenário real**
Uma CCT teve uma transação tentada, mas durante o flow uma `JpaSystemException` foi lançada no svc (lock de transação, deadlock, foreign key violation transitória). A coluna `error` da `cc_transaction` ficou com o stack trace bruto: `org.springframework.orm.jpa.JpaSystemException: could not extract ResultSet...`. Sete dias depois, o sweep pega essa CCT (ela passa pelo filtro) e submete à Sticky com `declineReason="org.springframework.orm.jpa.JpaSystemException..."`. A heurística de retry da Sticky não consegue mapear esse texto a nenhum bucket conhecido, retenta cegamente, queima chamadas.

**Impacto percebido**

- Custo Sticky em CCTs que nem chegaram ao gateway (erro foi interno UOWN, não decline real);
- Logs Sticky com lixo de stack trace;
- Possível leak de informação interna (nomes de classes Java, namespaces) para a Sticky.

**Sugestão técnica**
Estender o SQL do sweep com `AND cct.error NOT LIKE '%Exception%' AND cct.error NOT LIKE 'org.springframework.%' AND cct.error NOT LIKE 'java.lang.%'`, e adicionar um fallback no `StickyRecoverDeclineReasonMapper` para pular submissão (não enviar à Sticky) em vez de mandar stack trace.

#### 13. Nenhum mecanismo cancela a recovery quando a conta muda de estado durante o ciclo Sticky

**Severidade:** 🔴 Alta · **Categoria:** Regulatório · **Evidência:** contas 17176 (PAID_OUT) e 17177 (CANCELLED) → `recovery_status` continuou RECOVERY_STARTED após 3 sweeps + 30s

**Escopo do gap (importante)**
O entry filter `a.account_status = 'ACTIVE'` (igualdade estrita) **impede que sessões novas sejam criadas** para contas em `CLOSED/PAID_OUT/PAID_OUT_EARLY/CANCELLED/CHARGED_OFF/SETTLED_IN_FULL` (confirmado empiricamente em 2026-05-21 com cct 82934 / account 17169 — sweep_log pk=6345191, `processed=0`). O gap aplica-se **exclusivamente** a sessões criadas quando a conta era `ACTIVE` e que **sobrevivem à transição posterior** — ninguém re-avalia `account_status` depois da criação. Confirmado para `PAID_OUT` (17176) e `CANCELLED` (17177); `CLOSED` listado abaixo é inferido por simetria (mesma classe de transição pós-criação) — não reproduzido empiricamente porque a conta de teste foi setada em CLOSED antes do submit.

**Cenário real**
Cliente teve parcela negada, sweep submeteu à Sticky (`recovery_status='RECOVERY_STARTED'`). Três dias depois o cliente quita o contrato inteiro via early payoff — `account_status` vira `PAID_OUT_EARLY`. Sticky, alheia, continua retentando o cartão pelos próximos 27 dias seguindo o dunning profile. Confirmado empiricamente: atualizamos conta 17176 para `PAID_OUT` e conta 17177 para `CANCELLED` durante recoveries ativas, disparamos `paidOutAccountsSweep`, `reverseAchPaymentsSweep` e `StickyRecoverSweep`, esperamos 30s — `recovery_status` continuou `RECOVERY_STARTED`, zero chamadas a `/cancel-recovery`, zero entradas em activity log. Em zero ocorrências históricas em sandbox (`uown_sticky_outbound_log` filtrado por `CANCELED/RESOLVED/DEFERRED`) o produto sequer chamou o endpoint cancel.

**Impacto percebido — riscos concretos em prod**


| Cenário do cliente                                                                                               | Risco                                                                                                                                            |
| ---------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `PAID_OUT`, `PAID_OUT_EARLY`, `SETTLED_IN_FULL`                                                                  | Cliente recebe charge pelo valor já pago → dispute garantida                                                                                     |
| `CANCELLED`, `CLOSED`                                                                                            | Cobrança contratualmente nula                                                                                                                    |
| `rating=S/SOLD`                                                                                                  | Double collection sob UCC §9                                                                                                                     |
| `rating=C/D/B` atribuídos depois do submit (cliente declarou falência — Confirmed/Pending/Discharged Bankruptcy) | **Violação do automatic stay federal** — cobrança contra cliente em bankruptcy é proibida por 11 USC §362; penalidade direta sob bankruptcy code |
| Autopay desligado durante o ciclo                                                                                | Cobrança sem consentimento ativo                                                                                                                 |
| `rating=P` (payment arrangement)                                                                                 | Cobrança após acordo firmado                                                                                                                     |


**Sugestão técnica**
Implementar sweep diário `StickyRecoveryGuardianSweep` (ou listener event-driven) que:

1. Itere `SELECT * FROM uown_sticky WHERE recovery_status IN ('RECOVERY_STARTED','SUBMITTED')`;
2. Valide para cada sessão que `account_status`, `rating`, `cc.auto_pay` e `cc.is_valid_card` ainda atendem aos filtros originais;
3. Para cada discrepância chame `POST /processing-hub/v2/api/recovery/cancel` com o motivo apropriado:
  - `RESOLVED` para `PAID_OUT/PAID_OUT_EARLY/SETTLED_IN_FULL`;
  - `CANCELED` para `CANCELLED/CLOSED/SOLD/CHARGED_OFF` e ratings `C/D/B`;
  - `DEFERRED` para autopay temporariamente desligado ou cartão inválido.

Cada cancel deve gerar `lead_note` (Melhoria #14).

#### 16. Cobranças de fees (NSF_FEE, PROCESSING_FEE, etc.) negadas podem ser submetidas à Sticky indistintamente

**Severidade:** 🟡 Média · **Categoria:** Produto · **Evidência:** filtros do `StickyRecoverSweep` não restringem por `receivable_type`

**Cenário real**
Cliente teve parcela regular de $188 negada (cartão sem saldo). UOWN automaticamente gera uma NSF_FEE de $25 e cria uma CCT separada para cobrá-la. Ambas caem em D-7 e ambas são candidatas no sweep. **Os filtros do `StickyRecoverSweep` não discriminam `receivable_type*`* — ambas vão para Sticky em paralelo. Pior cenário: cliente paga os $188 fora-de-banda no dia 5 (ligou no CS, fez payment arrangement), mas a NSF de $25 continua na Sticky retentando por mais 25 dias até no dia 30 conseguir capturar. Cliente vê no extrato um charge de $25 "do nada" 30 dias depois da regularização e abre dispute.

**Volumetria sandbox (receivable_types elegíveis hoje)**


| Tipo                  | Rows sandbox |
| --------------------- | ------------ |
| `REGULAR_PAYMENT`     | 2.000.000+   |
| `PROTECTION_PLAN_FEE` | 25.950       |
| `PROCESSING_FEE`      | 16.337       |
| `NSF_FEE`             | 11.968       |
| `REINSTATEMENT_FEE`   | 41           |
| `MISC_FEE`            | 13           |
| `MANUAL_FEE`          | 9            |


**Impacto percebido**

- Custo operacional Sticky frequentemente maior que o valor da própria fee ($25 NSF retentado 7 vezes pode custar mais que o valor recuperado);
- Cobrança "duplicada" percebida pelo cliente quando a parcela já foi paga por outra via;
- Volume de chargebacks por fees pequenas;
- Volume de calls no CS desproporcional ao valor.

**Sugestão técnica — opções a alinhar com produto (Yuri)**


| Opção | Filtro SQL adicional                                                                    | Trade-off                                                       |
| ----- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------- |

| (c)   | Coluna `amount_threshold` na config + `AND cct.amount >= $config.stickyMinAmount` ($50) | Configurável por env, mais flexível                             |


Decisão é de negócio — alinhar quais tipos devem entrar antes de codificar.
