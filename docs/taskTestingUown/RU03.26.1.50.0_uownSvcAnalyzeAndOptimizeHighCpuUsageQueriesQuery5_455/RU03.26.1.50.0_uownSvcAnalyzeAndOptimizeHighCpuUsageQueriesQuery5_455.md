## Descrição

A query `SmsOptInConfirmationResponse.sql` foi identificada como causadora de alto uso de CPU e consumo excessivo de recursos do banco de dados. A otimização reestruturou a query como CTE e adicionou 2 índices parciais em `uown_sms_queue`, removendo o índice combinado antigo que não era mais utilizado pelo novo plano de execução.

**Comportamento de negócio:** Se `smsCount = 0` → sistema envia opt-in (primeira vez). Para contas KORNERSTONE, `smsCount` é forçado para `≥ 1` via lógica GREATEST → opt-in nunca é reenviado.

### Cenário: Cenário 2 — CT-02

**O que é feito:** Consulta `pg_indexes.indexdef` para verificar que ambos os novos índices são índices parciais com cláusula `WHERE ... IS NOT NULL`.

**O que acontece:** Ambos os indexdef contêm `WHERE (lead_pk IS NOT NULL)` e `WHERE (account_pk IS NOT NULL)` respectivamente.

**O que é verificado:**
- `idx_uown_sms_queue_lead_pk` indexdef: `CREATE INDEX idx_uown_sms_queue_lead_pk ON public.uown_sms_queue USING btree (lead_pk) WHERE (lead_pk IS NOT NULL)`
- `idx_uown_sms_queue_account_pk` indexdef: `CREATE INDEX idx_uown_sms_queue_account_pk ON public.uown_sms_queue USING btree (account_pk) WHERE (account_pk IS NOT NULL)`
- Ambos contêm `where` e `is not null`

| Índice | indexdef contém WHERE | indexdef contém IS NOT NULL | qa1 | stg |
|--------|:--------------------:|:--------------------------:|:---:|:---:|
| `idx_uown_sms_queue_lead_pk` | SIM | SIM | PASSOU | PASSOU |
| `idx_uown_sms_queue_account_pk` | SIM | SIM | PASSOU | PASSOU |

#### Como verificar manualmente

1. Executar: `SELECT indexname, indexdef FROM pg_indexes WHERE indexname IN ('idx_uown_sms_queue_lead_pk', 'idx_uown_sms_queue_account_pk');`

### Cenário: Cenário 7 — CT-07

**O que é feito:** Busca no banco o lead mais recente (`lead_pk > 100`, ordenado por `lead_pk DESC`) que possui ≥2 registros em `uown_sms_queue`. Calcula o `COUNT(*)` direto como baseline. Executa a CTE `SmsOptInConfirmationResponse` com esse `leadPk` e `accountPk=null`.

**O que acontece:** A CTE retorna `smsCount` igual à contagem direta de SMS do lead. Como `smsCount≥1`, `checkFirstSms=false` — o opt-in NÃO é reenviado.

**O que é verificado:**
- Lead recente encontrado com SMS existente (filtro `lead_pk > 100 ORDER BY lead_pk DESC` seleciona o mais recente)
- `CTE smsCount` = `COUNT(*) direto` — cross-validação confirma que CTE não alterou contagem
- Regressão: CTE refatorada NÃO alterou a contagem funcional

| Ambiente | Lead PK | COUNT(*) direto | CTE smsCount | Match? | checkFirstSms |
|----------|:-------:|:---------------:|:------------:|:------:|:-------------:|
| qa1 | 9999 | 2 | 2 | SIM | false (opt-in NÃO reenviado) |
| stg | 999990 | 18 | 18 | SIM | false (opt-in NÃO reenviado) |

#### Como verificar manualmente

1. Encontrar lead recente com SMS: `SELECT lead_pk, COUNT(*) FROM uown_sms_queue WHERE lead_pk > 100 GROUP BY lead_pk HAVING COUNT(*) >= 2 ORDER BY lead_pk DESC LIMIT 1;`
2. Executar CTE com o leadPk encontrado e accountPk=null
3. Resultado esperado: `smsCount` = COUNT direto

**PASSOU** (qa1 + stg)

**O que acontece:** A CTE calcula `is_kornerstone_account=0` (conta não é KORNERSTONE). O `GREATEST(count, 0) = count` — a lógica GREATEST não interfere. O `smsCount` é exatamente igual à contagem real de SMS.

**O que é verificado:**
