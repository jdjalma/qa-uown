## Cenarios validados

### Requisitos da tarefa

#### 1. Proxy `/iojs` strip upstream funcional (UOWN)

> ✅ **PASSOU** · UOWN apply em `https://apply-qa1.uownleasing.com/{token}/start` via redirector `getApplication/OW90218-0001` (TireAgent)

Na pagina de start UOWN o iovation loader inicializa corretamente, expondo os campos canonicos `uri_hook=/iojs/`, `subkey=IOVATION_KEY`, `version=general5`, `loaderVer=5.1.0` e `tp_host=https://mpsnare.iesnare.com`. Nenhuma flag `fp_static_wdp_load_failure` ou `fp_dyn_wdp_load_failure` aparece, indicando que ambos os scripts (static e dyn) carregaram sem erro.

**Evidencia:** leadPk=11968 (TireAgent aprovado em $1,292).

#### 2. Proxy `/iojs` strip upstream funcional (Kornerstone)

> ✅ **PASSOU** · Kornerstone apply em `https://apply-qa1.kornerstoneliving.com/{token}/start` via redirector `getApplication/KS3015` (FifthAveFurnitureNY)

Mesmo comportamento do UOWN: o `window.IGLOO.loader` em Kornerstone expoe os mesmos campos canonicos com os mesmos valores. Shape identico ao UOWN, prova de que a config nginx do proxy `/iojs/` foi aplicada em ambos os hosts (`apply-qa1.uownleasing.com` e `apply-qa1.kornerstoneliving.com`).

**Evidencia:** leadPk=11969 (KS3015).

#### 3. Endpoints do proxy retornam 200 em runtime real (UOWN + Kornerstone)

> ✅ **PASSOU** · cross-brand

Os 4 endpoints (UOWN static, UOWN dyn, Kornerstone static, Kornerstone dyn) respondem HTTP 200 durante o fluxo real de cliente.

**Detalhes das responses:**

**UOWN static:** `https://apply-qa1.uownleasing.com/iojs/general5/IOVATION_KEY/static_wdp.js?loaderVer=5.1.0&compat=false&tp=true&tp_split=false&fp_static=true&fp_dyn=true&flash=false` → HTTP 200

**UOWN dyn:** `https://apply-qa1.uownleasing.com/iojs/5.12.0/IOVATION_KEY/dyn_wdp.js?loaderVer=5.1.0&...` → HTTP 200
(note: dyn URL contem `5.12.0` no segmento de versao, nao `general5`; loader troca para versao especifica apos carregar o static)

**Kornerstone static:** `https://apply-qa1.kornerstoneliving.com/iojs/general5/IOVATION_KEY/static_wdp.js?...` → HTTP 200

**Kornerstone dyn:** `https://apply-qa1.kornerstoneliving.com/iojs/5.12.0/IOVATION_KEY/dyn_wdp.js?...` → HTTP 200

Nenhum 4xx ou 5xx observado. Cookies de tracking emitidos pelo backend iovation real (`Set-Cookie: fp_token_`*).

**Evidencia:** responses observadas durante CT-01 e CT-02.

#### 4. Placeholder `IOVATION_KEY` nao vaza em response

> ✅ **PASSOU** · cross-brand

A URL servida ao browser contem o literal `IOVATION_KEY` no path (o frontend bundle nao recebe a chave real; substituicao acontece server-side, no nginx, antes do upstream call). Essa caracteristica de design nao gera leak: as URLs com placeholder respondem 200 normalmente, e nao ha resposta de erro que expusesse a chave real ou outros dados sensiveis. Comportamento esperado e seguro.

**Evidencia:** inspecao do bundle frontend + verificacao das 4 responses do proxy.

#### 5. Persistencia do fingerprint em `uown_los_lead.iovation_fingerprint_text` (UOWN)

> ✅ **PASSOU** · UOWN

Apos submit completo do wizard de aplicacao UOWN (entry via `getApplication/OW90218-0001`, preenchimento das 3 paginas, approval automatico em $1,292 da TireAgent), `uown_los_lead WHERE pk = 11968` retorna `iovation_fingerprint_text` nao-nulo com **6433 caracteres**. Valor comeca com `0400r3EiY+dmiIGVebKatfMjIHo8F1LTs5GHo7cW...` (blackbox real emitido pelo iovation script). Nao corresponde a nenhuma string placeholder (`iovation finger print not available`, `iovation finger print not captured`, `fingerprint placeholder`).

**Evidencia:** leadPk=11968, fingerprint length=6433, prefix `0400r3EiY+dmiIGVebKatfMjIHo8F1LTs5GHo7cW...`
