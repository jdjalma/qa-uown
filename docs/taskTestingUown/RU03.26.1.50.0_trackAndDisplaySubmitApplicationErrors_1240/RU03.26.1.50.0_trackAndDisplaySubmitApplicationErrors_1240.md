**O que é verificado:**

- `appApprovalStatus` = `APPROVED`

- Cada item possui `planId` não-nulo que bate com regex `/^(WK|BW|BWK|SM|MN)(13|16)$/`

### Cenário 3 — CT-03

**O que é feito:** Envia `POST /uown/los/sendApplication` para **PayPossible** (PP00001-0001) com o campo `mainNextPayDate` **completamente omitido** do body JSON.

**O que acontece:** Antes do fix (commit `128bb5fd8`), o backend lançava `IllegalArgumentException("Please provide next pay date")` não tratada, resultando em HTTP 500. Após o fix, o backend detecta que `nextPayDate` é null, consulta a config `allow.default.nextPayDate.for.merchant.paypossible = true` e aplica a data padrão `LocalDate.now().plusWeeks(2)`. A aplicação é processada normalmente e aprovada.

**O que é verificado:**
- HTTP status ≠ 500 (crash evitado — core do Fix 1)
- HTTP 200 com `appApprovalStatus = APPROVED` (config de default date habilitada para PayPossible)

| mainNextPayDate | HTTP | appApprovalStatus | Nota |
|-----------------|------|-------------------|------|
| (campo omitido) | 200 | APPROVED | Fix 1: data padrão `now().plusWeeks(2)` aplicada |

> **Nota:** Com TerraceFinance (config `allow.default.nextPayDate = false`), este cenário retorna HTTP 400 / DECLINED. Com PayPossible (config `= true`), o backend aplica a data padrão. Ambos os comportamentos confirmam o Fix 1 funcionando.

#### Como verificar manualmente

1. Enviar `POST https://stg-svc.uownleasing.com/uown/los/sendApplication` com body igual ao CT-01, porém **sem** o campo `mainNextPayDate`
2. Verificar que a resposta NÃO é HTTP 500 (era o crash antes do fix)
3. Confirmar HTTP 200 com `appApprovalStatus = "APPROVED"` — a data padrão foi aplicada automaticamente
4. Para verificar o comportamento inverso: repetir com merchant TerraceFinance (config `= false`) e confirmar HTTP 400

**PASSOU**
