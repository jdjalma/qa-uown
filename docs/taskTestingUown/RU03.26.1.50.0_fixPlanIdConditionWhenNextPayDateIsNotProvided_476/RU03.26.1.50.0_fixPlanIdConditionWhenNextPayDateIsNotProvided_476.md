## Descrição

O backend SVC tinha dois problemas no fluxo de `planId`:

**Fix 1 — `UownClient.getPaymentOptions()`**

Anteriormente lançava exceção `"Please provide next pay date"` mesmo quando payment options já existiam no banco de dados. O fix implementa a seguinte lógica:

1. Primeiro verifica se já existem payment options no banco → retorna se encontradas
2. Se não existem e `nextPayDate` é null → verifica config `allow.default.nextPayDate.for.merchant.{code}`
3. Se a config permite → usa data padrão `LocalDate.now().plusWeeks(2)`

**Fix 2 — `MissingRequiredFieldsService.resolveAndSetMerchantProgramFromPlanId()`**

Anteriormente lançava `NullPointerException` quando o `merchantProgram` não podia ser resolvido a partir do `planId`. O fix adicionou guard condicional.

**Formato do planId:** `{frequencyAbbr}{termMonths}` — exemplos: WK13, BW13, BWK16, SM13, MN16

A aplicação é processada normalmente e aprovada.

**O que é verificado:**
- HTTP status ≠ 500 (crash evitado — core do fix)

|-----------------|------|-------------------|------------|
| (campo omitido) | 200 | APPROVED | Fix 1: default date `now().plusWeeks(2)` aplicada |
