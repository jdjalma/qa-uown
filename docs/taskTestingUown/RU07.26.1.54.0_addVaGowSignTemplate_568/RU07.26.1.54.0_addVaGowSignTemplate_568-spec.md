## Test Strategy

- **Approach:** Hybrid, **UI-first** (rule #14). Setup via API (`sendApplication` → `sendInvoice` → `submitApplication`/CC pre-auth) to reach CONTRACT_CREATED; **exercise + assert via the rendered GowSign iframe/PDF** (content cannot be read from a backend log — this is the Daniel's Jewelers CA / AL BR-06 lesson); DB assertions as cross-cutting guards (template selection, activity log, dispatch variables).
- **Why not API-only:** the entire ticket is visual PDF content with MUST/MUST-NOT string assertions and blank-token detection — API-only reading logs is exactly what missed BUG-01. API-only is confined to setup + the `assertSelectedTemplateForLead` / `uown_los_lead_notes` / dispatch-variable DB guards.
- **Two-track split forced by the NeuroID blocker:**
  - **Track A (automatable, CI):** VA **13m** via `TerraceFinance` (VA `validMerchant`, ONLINE, UOWN gateway `secure-<env>.uownleasing.com`, no NeuroID). Full render + sign with the existing helper chain (`createPreQualifiedApplication` → `sendInvoice` → `MissingDataFormPage` → `TermsOfAgreementPage.acceptAndProceedWithProtectionPlan(false)` → `AlternativeContractModalPage` → `signGowSignInFrame`). Covers CT-01 negative control (13m → `VA_2025_SAC`), CT-05/CT-07 sanity reference, CT-08 signing on the 13m. **Only if a VA 13m template exists** (discovery).
  - **Track B (MCP-manual, not CI):** VA **16m** via a Kornerstone ONLINE merchant serving VA. Drive `/complete` CC pre-auth → reload → "Sign Contract" (NeuroID bypass, AL BR-07) → Terms → Proceed → read the GowSign iframe. This is the ONLY way to capture the 16m-specific content (CT-02/CT-03/CT-04/CT-06) and to freeze the oracle checkpoints from `[EXPECTED]` to `[CONFIRMED]`. Manual, driven by MCP Playwright.
- **Environments:** primary **qa2** (sibling states verified there); re-list VA templates first — if VA is not deployed in qa2, fall back to **stg** (where LA/IL 16m were proven live). **stg is required for DoD.** Avoid qa1 for routing if the DV360 outage recurs; qa1 is still valid for a SignWell VA baseline (no VA GowSign template there → fallback).
- **Suites to activate:** `gowsign-contract-content*` + `gowsign-signature-fields` for the template change (per gowsign-knowledge "template change → run all content specs"). SignWell visual diff only if a blank-token finding appears.
- **Merchant preflight (rule #12):** required for the Track A fresh 13m lead (`ensureMerchantReady` via `createPreQualifiedApplication`). The Kornerstone Track B merchant config (16m program active, VA in `valid_states`) is a precondition to verify, not to auto-heal blindly — auto-heal must run BEFORE UW approval so the merchant snapshot is correct (snapshot immutability pitfall).
- **Test data (rule #9):** fresh lead per scenario, fresh SSN. No reuse. Kornerstone requires `bankData` (`TEST_BANK` routing `123456780` / account `160781900000`) on `sendApplication`, else 400. CC pre-auth cardholder last name MUST equal applicant last name (AL BR-02) and use `VISA_APPROVED` (AL BR-03).
- **Money assertions:** `toBeCloseTo` (float tolerance), never `toEqual` (float-repr rule).

## Scenarios (prioritized)

Priority buckets from risk-based scoring (Novelty+Integration+Boundary+History × Customer+Function+Audit). All content scenarios inherit the P0 "template change → visual" floor rule.

### Scenario 1 — Routing & term→template selection (P0)
- **Oracle:** CT-01. **Technique:** decision table (state VA × term {13,16} → {`VA_2025_SAC`, `VA_2025_SAC_16_MONTHS`}, both GOWSIGN) + state-transition (UW_APPROVED → CONTRACT_CREATED where the esign doc is born).
- **Persona:** agent/system (routing is internal); the customer only sees the resulting contract.
- **Setup:** pre-run `getGowSignTemplatesForState(db,'VA')` to confirm the inventory and term coverage. Fresh VA lead, 16m via Kornerstone (Track B) and 13m via TerraceFinance (Track A).
- **Steps:** 1) create + approve the lead; 2) `sendInvoice`, pick WK16 (and WK13 for the control); 3) drive to CONTRACT_CREATED; 4) `assertSelectedTemplateForLead(db, leadPk, 'VA_2025_SAC_16_MONTHS')` and, for the control, `'VA_2025_SAC'`.
- **Validations:** DB: `uown_esign_document.client='GOWSIGN'` + template name/id match. If VA is 16m-only → the 13m control is skipped (documented, not silently dropped).
- **Edge cases:** VA still SIGNWELL (template not deployed) → S1 fails fast, blocks the suite. INSTORE merchant would route by store state — use ONLINE so customer state VA drives the lookup.
- **Pitfalls considered:** stale `state-merchant-matrix.ts` VA row; esign doc only exists at CONTRACT_CREATED, not UW_APPROVED.

### Scenario 2 — Item 4 Promotional-Payoff Option content (P0)
- **Oracle:** CT-02. **Technique:** decision table (MUST-present phrases vs MUST-NOT phrases) + error-guessing (standard-VA-SAC leakage).
- **Persona:** customer reading the contract before signing.
- **Setup:** rendered `VA_2025_SAC_16_MONTHS` body (Track B, MCP-manual).
- **Steps:** 1) capture iframe `body.innerText()`, normalize whitespace; 2) assert MUST list; 3) assert MUST-NOT list.
- **Validations (UI render):** MUST — exact heading "Promotional-Payoff Option:", "at any time", "$ [costPriceWithFeeNoTax] plus tax", "daily lease fees ... through the ... exercise", "late payments will void". MUST NOT — /During the first .* days/, "This option expires on", "Cash Price of the item(s)", "You have other purchase options described below", `{{epoDays}}`/`{{epoExpiryDate}}` (populated or blank).
- **Edge cases covered:** blank day-window (empty vs absent), exercise-date vs current-date accrual wording.
- **Pitfalls considered:** VA uses exercise-date accrual (like FL/TN), NOT current-date (CA/TX) — do not assert TX/CA phrasing.

### Scenario 3 — Item 4a Lease-Purchase Ownership / EPO tail (P0, highest defect probability)
- **Oracle:** CT-03. **Technique:** decision table + BVA on monetary tokens (non-empty check).
- **Persona:** customer verifying the total they will pay to own.
- **Setup:** same rendered 16m body + `uown_esign_document.request.document.variables`.
- **Steps:** 1) assert the VA tail regex; 2) assert `nextPaymentDueAmount` non-empty (regression watch); 3) assert MUST-NOT list.
- **Validations (UI render + DB):** MUST — "must be current to elect EPO", "EPO = daily lease charges through exercise date + fees/taxes − on-time payments", tail `[totalNumberOfPayments] payments of $ [nextPaymentDueAmount] (plus tax)`, `$ [contractAmount] (including tax) (including the application fee)`, "Total Cost,", "and you will own the Property". `nextPaymentDueAmount` renders non-empty. MUST NOT — /\d+% discount/, TN-style "plus $ [salesTax] in sales tax" + separate "processing fee". DB guard: no `[DocumentDispatchService][GowSign] … variables map missing …` note for this lead.
- **Edge cases covered:** blank `nextPaymentDueAmount` (the AL/OH regression); TX tail (firstPaymentDueAmount) vs VA tail (nextPaymentDueAmount) confusion.
- **Pitfalls considered:** VA tail is TX-structured but uses `nextPaymentDueAmount` — a genuine new baseline; float tolerance on the money cross-check.

### Scenario 4 — Token render smoke (P0)
- **Oracle:** CT-04. **Technique:** equivalence partitioning over the token set + error-guessing (raw `{{ }}`).
- **Persona:** customer (a blank/`{{token}}` is customer-visible and legally material).
- **Setup:** rendered 16m body + dispatch variables map.
- **Steps:** 1) for each monetary token, assert the dispatch value appears in the body (commas stripped, `toBeCloseTo`); 2) `expect(body).not.toMatch(/\{\{|\}\}/)`; 3) assert the brand logo `<img>` at top; 4) assert absence of the "variables map missing" dispatch note.
- **Validations (UI + DB):** costPriceWithFeeNoTax, nextPaymentDueAmount, contractAmount, totalNumberOfPayments, companyInfoBrandPhone all non-empty and matching the dispatch variables.
- **Edge cases covered:** phone/brand token, contract "Total Cost" token.
- **Pitfalls considered:** the header state token (e.g. `AGREEMENT-VA`) is the reliable state anchor — the literal word "Virginia" may not appear in the flatten (OH "Ohio" false-negative precedent); anchor on `AGREEMENT-VA`, not "Virginia".

### Scenario 5 — Lease-purchase-plan footnote (P1)
- **Oracle:** CT-05. **Technique:** decision table (MUST vs MUST-NOT).
- **Persona:** customer reading the plan summary/footnotes.
- **Setup:** rendered 16m body.
- **Validations (UI render):** MUST — "Traditional [N] Month Lease Purchase Plan", "with a Promotional Payoff option", "Early Purchase Option Available Off Unpaid Balance", populated `[leasePurchasePlan]` grid. MUST NOT — EPO payoff chart / `[earlyPurchaseOption]` table, numbered promotional/EPO option list.
- **Edge cases covered:** empty grid, leaked CA-style chart.
- **Pitfalls considered:** VA is NOT a CA/NY/PA chart state — the `ca_chart`/`[earlyPurchaseOption]` block must be absent.

### Scenario 6 — Singular EARLY PURCHASE OPTION appendix + page order (P1)
- **Oracle:** CT-06. **Technique:** decision table + sequence/state check (page order).
- **Persona:** customer reading the appendix.
- **Setup:** rendered 16m body.
- **Validations (UI render):** MUST — singular "EARLY PURCHASE OPTION", "This Agreement has a 16-month term for ownership.", single intro "For a 16-month lease, the EPO price is calculated as:" (mind the `16-\nmonth` line break), the same 5 bullets as TN, "call us at [phone]". MUST NOT — plural "EARLY PURCHASE OPTIONS", Option 1/2 structure, "Purchase the Property by [epoExpiryDate]", sub-points a.–e., customer-initials line. Page order: the section immediately after is Item 14 (ACH Payments); the EPO page occurs exactly once (no duplication).
- **Edge cases covered:** duplicated EPO page, plural-vs-singular heading.
- **Pitfalls considered:** the `16-\nmonth` line break defeats an exact `"16-month"` match (OH precedent) — use a `16-\n?month` tolerant match.

### Scenario 7 — Sanity §1–§14 vs standard VA SAC (P1, VA-differentiator)
- **Oracle:** CT-07. **Technique:** diff/comparison against the standard VA SAC baseline.
- **Persona:** compliance/agent verifying state-law clauses survived the 16m adaptation.
- **Setup:** rendered body (13m Track A as the standard-VA reference where feasible; 16m Track B for the target).
- **Validations (UI render):** §3 term starts on the **delivery date** (not paycheck); §7 agreement expires on a missed payment, reinstatement **5 / 21 / 45 days at 2/3 of Total Cost**; return language "no further obligation, except for any past due payments due". No 16m-exclusive clause leaks the standard-VA promo/%-discount/plural-Options-with-initials.
- **Edge cases covered:** reinstatement-tier confusion.
- **Pitfalls considered:** **do NOT** assert IL's 16/60-day, IN's 120-day, or TN's 90/180-day reinstatement here — VA is uniquely 5/21/45 at 2/3 Total Cost. This is the single most likely cross-state copy error.

### Scenario 8 — Signing completion + activity log (P1)
- **Oracle:** CT-08. **Technique:** use-case (end-to-end ceremony) + activity-log reflex (rule #13).
- **Persona:** customer completing the signature; agent sees SIGNED.
- **Setup:** a rendered VA GowSign contract (13m Track A end-to-end automatable; 16m via Track B MCP-manual).
- **Steps:** complete the ceremony (Start → adopt signature/initials → Sign All → **Finish-to-finalize dialog** → redirect appComplete).
- **Validations:** DB — `uown_los_lead.lead_status='SIGNED'`; `uown_esign_document` progresses SENT_TO_CUSTOMER → SIGNED with `doc_signed_time_stamp` populated. Activity log (rule #13) — `uown_los_lead_notes` contains `[ContractService][isLeaseOrLeaseModSigned]…SIGNED`, `[EsignRedirectService][updateSignStatus]…to SIGNED`, `[ESIGNSERVICE][parseCCPeekConsent]`.
- **Edge cases covered:** lease stuck at CONTRACT_CREATED if the "Finish to finalize" dialog is not clicked (AL framework gap).
- **Pitfalls considered:** assert on `lead_status='SIGNED'` (authoritative), not on the esign status enum; Buddy widget swallows the Proceed click (re-click / raw DOM click); `signGowSignInFrame` may not click the final Finish dialog — click `frame.getByRole('button',{name:/^Finish$/i}).last()`.
