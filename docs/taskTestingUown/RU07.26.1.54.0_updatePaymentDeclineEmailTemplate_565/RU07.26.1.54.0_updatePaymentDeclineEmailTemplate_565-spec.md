Legacy Payment Decline HTML templates replaced with the new HubSpot-style design for **both brands**:
- UOwn: `payment-decline-email.html` (template_name `PaymentDeclineEmail`)

## Scope

**IN:**
- **UOwn decline email content + rendering** ‚Äî subject, greeting/first-name, portal CTA, phone, closing, HubSpot layout (AC1, AC3, AC4). It is customer-facing correspondence; rule #14 (rendering-first) applies by analogy ‚Äî content bugs (empty placeholders, wrong brand, off-allow-list image) are only visible in the rendered email, not in backend logs.
- **Kornerstone decline email content + rendering** ‚Äî KS branding, KS support phone `(888) 521-5111`, KS portal URL, KS closing, `KORNERSTONE_` template (AC2). Dual-brand parity is mandatory (the task explicitly ships both).
- **First-name fallback** ‚Äî `"Customer"` when the first name is empty (AC3, ticket Scope ¬ß1). This is the classic placeholder-leak failure class (BUG-01 Daniel's Jewelers CA).
- **Subject line exactness** ‚Äî `"Action Needed: Issue Processing Your Recent Payment"` (AC4).
- **Preview/preheader text** ‚Äî `"Ensure your account stays active by resolving payment issues."` (AC5).
- **Multi-client rendering spot-check** ‚Äî Gmail + Outlook, desktop + mobile (AC6, TP5). Visual, cannot be inferred from the DB body.
- **Trigger integrity** ‚Äî the payment-failure flow still enqueues the decline email with no trigger-logic regression (AC7, TP1/TP2).
- **DB template reload** ‚Äî new HTML renders (not legacy) for both companies after `TemplateManagementService` reload (AC8).
- **AC9 settlement conditional copy** ‚Äî declines tied to a `SETTLEMENT` arrangement swap two copy lines; everything else invariant. **Includes the NORMAL-arrangement negative** (regression corollary).
- **Activity log for the decline event** (rule #13) ‚Äî the payment-failure business action must leave an audit trail.

**OUT:**
- **`PaymentReceiptService` trigger logic changes** ‚Äî ticket Scope ¬ß4 says none expected; we validate the email still fires (AC7) but do not re-test the whole payment-decline detection engine.
- **SMS template `payment-decline-sms.txt`** ‚Äî ticket Scope ¬ß4 explicitly leaves it unchanged unless requested separately.
- **The full servicing payment-decline mechanics** (CC gateway decline codes, ACH RETURNED sweep chain) ‚Äî reused as *setup*, not under test here. Their correctness is covered by payment-flows and the arrangement oracles.
- **Other correspondence templates** (Welcome, Approval, Settled-in-Full, reminders) ‚Äî not touched by this diff; no shared-template regression risk (each template is its own file).
- **Non-Gmail/Outlook clients** (Apple Mail, Yahoo, etc.) ‚Äî AC6 scopes "major clients"; the ticket names Gmail + Outlook. Broader client matrix is a nice-to-have (P3).

## Test Strategy

- **Approach:** **Hybrid** ‚Äî API/UI setup drives a fresh FUNDED account and triggers a genuine payment decline (rule #9 fresh data); **assertion is content-first on the rendered email** (`uown_email_queue.email_body` + IMAP-delivered HTML) plus a visual render spot-check (rule #14). Never DB-log-only.
- **Justification of level:** the feature's whole value is what the customer *sees* in the email. Per rule #14 (rendering-can't-be-inferred-from-logs, BUG-01 origin), the email body must be read/rendered, not merely confirmed "a row exists". API-only reading of `template_name` would miss an empty placeholder, wrong-brand logo, or off-allow-list image.
- **Setup / Exercise / Assertion split** (per scenario):
  - *Setup* (fast, not under test): `driveLeadToFunding` / `createPreQualifiedApplication` ‚Üí ACTIVE account (payment-flows canonical sequence). For KS, pick a merchant whose `ref_merchant_code` starts with `KS` (e.g. `KS3015` / `KS3023`) ‚Äî brand discriminator is the ref code prefix, NOT `merchant_name`. For AC9, additionally create a `SETTLEMENT` payment arrangement via the existing Make Payment modal / arrangement API (see payment-arrangement oracle ‚Äî `ArrangementType` is an explicit UI React Select, NOT backend-derived).
  - *Exercise*: trigger a real payment decline ‚Äî declined card (`ALL_TEST_CARDS.VISA_DECLINED` / `DECLINE_*`) for CC, or RETURNED ACH. **Exact trigger path is Open Question Q1** ‚Äî resolve via discovery (rule #18) before locking the setup.
  - *Assertion*: subject/copy/branding/placeholder/portal-link/phone against `email_body` (CT checkpoints) + image allow-list regex + IMAP-rendered visual check for AC6 + activity log (rule #13).
- **Email inspection helpers:** `src/helpers/email.helpers.ts` (IMAP: `snapshotInboxUid` ‚Üí `getEmailContent`/`getEmailLink`, UID-watermark freshness) and `src/helpers/settled-in-full.helpers.ts` (`getEmailQueueRecord`, `waitForEmailQueueRecord`, `waitForEmailQueueDispatched`, `countEmailQueueRows`). Use plus-addressing per `runId` (`uniqueEmail(...)`) to isolate the inbox. Do NOT create new email helpers (rule #2).
- **Environment:** primary run where the `emailSweep` + a payment processor are available ‚Äî `dev1`/`qa1` have the sweep confirmed (email-templates-catalog ¬ß4). Prefer the env used by existing payment-arrangement / decline setup; **stg required for DoD** (`project_qa_task_structure`). Flag if the chosen env lacks a real CC/ACH processor (affects Q1/Q3). Avoid qa1 if the DV360 outage is active ‚Äî fall back to qa2.
- **Suites to activate:** dual-brand (mandatory ‚Äî both templates ship). No signing-regression (no signing touched). No lease-edit / submitApplication regression (handler untouched).
- **Smoke vs full:** smoke = S1 (UOwn happy content) + S4 (AC9 settlement) ‚Äî the two P0s. Full = all seven.
- **Parallelization:** scenarios use independent fresh accounts ‚Üí parallelizable, BUT they share the IMAP inbox ‚Üí use per-`runId` plus-addressing (safe to parallelize). KS and UOwn use different merchants ‚Üí no config-drift collision.

## Scenarios (prioritized)

### S1 ‚Äî [P0] UOwn non-settlement payment decline renders the new design
- **Technique:** Use-case + equivalence (valid class: UOwn, no arrangement) + error-guessing (placeholder leak).
- **Persona:** Customer (recipient); Agent/system triggers the decline. Customer reads the email on mobile ‚Äî the CTA button + phone must be tappable/legible.
- **Setup [reflex]:** fresh FUNDED UOwn account via `driveLeadToFunding`; unique customer first name + `uniqueEmail`; no SETTLEMENT arrangement. Merchant preflight auto (rule #12).
- **Exercise:** trigger a servicing payment that declines (see Q1).
- **Validations (Oracle CT-01):**
  - Rendered email: subject exact `"Action Needed: Issue Processing Your Recent Payment"`; greeting `"<FirstName>, we had trouble processing your payment."` with the real name; copy `"...your most recent payment."` + `"...your account in good standing..."`; NO `"settlement"` strings.
  - Portal CTA `href` starts `https://portal.uownleasing.com` + `customerPortalParameters`.
  - Support phone `(877) 357-5474`; closing `"Thank you for choosing Uown."`.
  - Image allow-list: zero `<img src>`/CSS-bg URLs outside `https://storage.googleapis.com/uown/` (negative regex, email-templates-catalog ¬ß3).
  - DB: `uown_email_queue` row `template_name='PaymentDeclineEmail'`, `status IN ('SENT','STORED')`, `email_body_type='HTML'`, `to_email_addresses`==customer email.
  - **Activity log [reflex, rule #13]:** decline event note present (table per Q4).
- **Edge cases covered:** original (non-settlement) copy = the AC9 regression baseline.
- **Pitfalls considered:** placeholder leak (`${...}`, `[FIRST NAME]`), legacy-template-still-cached (AC8), off-allow-list image.

### S2 ‚Äî [P0] Kornerstone non-settlement decline renders KS branding
- **Technique:** brand-parity equivalence (KS class).
- **Persona:** KS Customer.
- **Setup [reflex]:** fresh FUNDED account on a KS merchant (`ref_merchant_code` starts `KS`, e.g. `KS3015`); no SETTLEMENT arrangement.
- **Exercise:** trigger decline.
- **Validations (Oracle CT-02):** `template_name='KORNERSTONE_PaymentDeclineEmail'`; subject exact; KS phone `(888) 521-5111` (NOT UOwn's); portal CTA uses KS `customerPortalUrl` + params (not `portal.uownleasing.com`); KS closing; KS logo/social/contact assets on the GCS allow-list with NO UOwn logo leak; `from_email_address` = KS sender. Activity log (rule #13).
- **Edge cases covered:** wrong-brand-asset leak, UOwn phone/URL bleeding into KS.
- **Pitfalls considered:** brand discriminator via ref-code prefix not merchant name; KS uses `customerPortalUrl` (relative) vs UOwn's hardcoded host.

### S3 ‚Äî [P1] First-name fallback renders "Customer"
- **Technique:** BVA on the name field (empty boundary) + error-guessing.
- **Persona:** Customer with empty first name.
- **Setup:** fresh account whose customer first name is empty/blank.
- **Exercise:** trigger decline.
- **Validations (Oracle CT-03):** greeting `"Customer, we had trouble processing your payment."`; body contains NO raw `${`, NO `[FIRST NAME]`, NO empty `", we had trouble"`.
- **Edge cases covered:** the empty-placeholder failure class (rule #14 origin).
- **Pitfalls considered:** Thymeleaf fallback wiring identical to `decline-email.html`.

### S4 ‚Äî [P0] Settlement-arrangement decline swaps to settlement copy (AC9)
- **Technique:** decision-table row (`arrangement_type=SETTLEMENT`).
- **Persona:** Customer on a settlement plan (delinquent, negotiated payoff) ‚Äî sensitive copy: the email must reference the *settlement*, not a generic payment.
- **Setup [reflex]:** fresh FUNDED account; create an active `SETTLEMENT` payment arrangement via the Make Payment modal / arrangement API (payment-arrangement oracle ‚Äî explicit `SETTLEMENT` React Select). The declined payment must be linked (`payment_arrangement_pk`) to that arrangement.
- **Exercise:** the payment tied to the SETTLEMENT arrangement declines (see Q3 ‚Äî may need authorized RETURNED, Exception 3).
- **Validations (Oracle CT-04):** copy A `"Unfortunately, we were unable to process your settlement payment."`; copy B `"To keep your settlement offer in good standing, please make a payment as soon as possible."`; original copy strings ABSENT; subject/greeting/CTA/phone/closing IDENTICAL to the brand baseline (only the two lines differ); `arrangement_type='SETTLEMENT'` confirmed in DB and linked to the declined txn. Activity log (rule #13).
- **Edge cases covered:** conditional copy swap; "all other messaging unchanged" invariant.
- **Pitfalls considered:** over-trigger (any arrangement ‚Üí settlement copy) is caught by S5; setup may require DB mutation (flag, don't assume).

### S5 ‚Äî [P1] NORMAL-arrangement decline keeps original copy (regression guard on the AC9 condition)
- **Technique:** decision-table negative row (`arrangement_type=NORMAL`) ‚Äî the exact boundary of `getArrangementType()==SETTLEMENT`.
- **Persona:** Customer with a normal installment arrangement.
- **Setup [reflex]:** fresh FUNDED account with an active `NORMAL` payment arrangement; declined payment linked to it.
- **Exercise:** trigger decline on the NORMAL-arrangement payment.
- **Validations (Oracle CT-05):** original non-settlement copy present; body contains NEITHER `"settlement payment"` NOR `"settlement offer"`; linked `arrangement_type='NORMAL'` confirmed.
- **Edge cases covered:** distinguishes "arrangement exists" from "arrangement is SETTLEMENT" ‚Äî the only scenario that catches an over-broad condition.
- **Pitfalls considered:** why distinct from S1 ‚Äî S1 has NO arrangement, S5 has a NORMAL arrangement; only S5 exercises the false-positive path.

### S6 ‚Äî [P1] Subject + preview text + multi-client rendering (AC5, AC6)
- **Technique:** use-case + visual.
- **Persona:** Customer scanning inbox (subject/preheader) then opening on mobile + desktop.
- **Setup:** reuse a rendered email from S1/S2 (either brand) via IMAP.
- **Exercise:** fetch delivered HTML via IMAP; render at 1440 (desktop) and 375 (mobile).
- **Validations (Oracle CT-06):** subject exact; preheader `"Ensure your account stays active by resolving payment issues."` present in body; visual: logo, CTA button, footer visible, not broken/overlapping at both viewports (rule #14 ‚Äî cannot infer from DB body).
- **Edge cases covered:** preheader hidden-text, mobile layout break.
- **Pitfalls considered:** Gmail/Outlook CSS mangling; mobile-only regression (rule #15 Website viewport principle applied to email).

### S7 ‚Äî [P1/P2] Trigger integrity + DB reload + activity log (AC7, AC8, rule #13)
- **Technique:** state-transition (payment ‚Üí declined) + audit reflex.
- **Persona:** system/agent; audit reviewer.
- **Setup:** any fresh FUNDED account.
- **Exercise:** trigger decline.
- **Validations (Oracle CT-07):** exactly one new decline-email row per decline (no dup, no missing); brand-correct `template_name`; new HTML (not legacy) renders ‚Üí confirms `TemplateManagementService` reload for BOTH companies (AC8); **activity log** for the decline present + content (rule #13, table per Q4).
- **Edge cases covered:** silent no-email (worst case), duplicate email on retry (idempotency reflex).
- **Pitfalls considered:** email-sweep availability per env (email-templates-catalog ¬ß4); activity-log table lead_pk vs account_pk (post-funding ‚Üí `uown_sv_activity_log`).
