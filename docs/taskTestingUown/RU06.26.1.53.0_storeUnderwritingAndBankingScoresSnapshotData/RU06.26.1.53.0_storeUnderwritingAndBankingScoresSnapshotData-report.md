## Descrição

No underwriting via **GDS**, o backend persiste dois scores INTEGER de snapshot histórico em `uown_los_uwdata`
(lado lead) e `uown_sv_uwdata` (lado account):

- `npm_segment` — lido de `$.data.out.npm_segment` da resposta GDS.
- `tam_score` — lido de `$.data.out.tam_score` da resposta GDS.

**Correção factual (root cause confirmada 2026-06-21/22 — ver §Causa-raiz):** os DOIS campos são lidos pelo MESMO
método `GdsResponseParser.setCurrentUWConfigData`, chamado **incondicionalmente** em todo parse de resposta GDS
bem-sucedida — **não há gate de "branch 16m" nem de "TireAgent" no backend** `[svc-source:GdsResponseParser.java:41,58-62]`.
O caveat "`tam_score` só TireAgent / só 16m" é propriedade do **modelo do vendor DataView360** (qual campo o vendor
EMITE no JSON), não do código UOWN. Campo ausente no JSON vira `null` via JsonPath (`DEFAULT_PATH_LEAF_TO_NULL +
SUPPRESS_EXCEPTIONS`). A descrição anterior ("gravado no branch 16m") estava incorreta e foi substituída.

Não há superfície de UI própria (escrita silenciosa no backend, como `lambda_segment`) → o comportamento observável
é **persistência em DB**. A regra #14 (UI-first) é satisfeita exercitando o **gatilho real** (application → bank data
→ decisão de underwriting GDS); as duas colunas novas são validação **cross-cutting de DB** (exceção c da #14). A
decisão UW é uma transição de status → seu activity log é asserido (regra #13).
