## Scope

### IN

- **Enabled Download CSV below the limit, Overview** (`all-filtered-leads.csv`, 27 cols) — core regression surface (AC-01); the fix must not break existing small exports. **Decisive UI check: open the file and compare rows/columns to the portal.**
- **Enabled Download CSV below the limit, Leads** (`leads-results.csv`, 17 cols incl. SSN) — same util now drives Leads; brand-new export path for this screen (AC-01/AC-05).
- **Empty-table state, both screens** — both Download CSV and Email CSV disabled; **no** directing tooltip in the empty case (AC-06; negative control for AC-03).
- **Permission gate** — Download CSV renders only with `hasDownloadPermission && headers.length>0`; Email CSV renders regardless (AC-07).
- **Email CSV availability + modal** — Email CSV always rendered (disabled only when table empty); opens modal "Which email should we send this CSV file to?"; Send disabled until address entered (AC-04, partial AC-07).
- **Filter re-evaluation** — narrowing/widening filters re-evaluates availability (AC-08), exercised on the reproducible (enabled↔empty) axis on QA2; the enabled↔disabled axis is source-confirmed only (see test-data).
- **Disabled Download CSV + directing tooltip/toast (the point of the fix)** — IN scope as **objective**, but **execution-blocked on QA2**; covered at source level + UI on a high-volume/seeded env when available (AC-02, AC-03). This is explicitly carried as the highest-risk, lowest-reproducibility item.

### OUT (with justification)

- **Reproducing the pod crash / `ERR_STRING_TOO_LONG`** — the fix's observable is the disabled button + directing message, not the crash. Rule #14: validate at the UI affordance, not by re-triggering an outage. (Scenarios file §test-data agrees.)
- **Email CSV success-toast wording + actual email delivery** — sending a real CSV email is a side effect; the success-toast copy is an unconfirmed `[gap]` (KB G2). We assert the request is *accepted* (modal closes / no error), not the delivered email content. Listed as Open Question.
- **The 27-column Overview content correctness / Config Columns interaction** — already covered by task #1295 / `overview-csv-export-merchant-support`. We assert filename + column **count/header set** + row count on the happy path, not per-column business correctness (no regression to that prior coverage intended).
- **Kornerstone brand parity** — CSV export is a portal-level Origination listing feature, brand-agnostic (same component/util); no brand-specific code in the diff. Not promoted to dual-brand. (Re-evaluate only if PO says the Leads/Overview export differs per tenant.)
- **Mobile/responsive viewport** — Origination is an internal agent portal (rule #15: `1440×900` único, Bootstrap `d-lg-block`). No mobile axis.
- **Backend `getLeadsInDateRange` / BFF proxy `Buffer.toString()` unit behaviour** — the mitigation is **client-side** (the guard prevents the request from being issued). The proxy buffering is the root cause, not the fix surface. OUT unless PO asks for a server-side cap test.

## Test Strategy

- **Approach:** **Hybrid, UI-first.** Decisive checks are all UI affordances (button enabled/disabled state, tooltip/toast copy, modal, and — for the happy path — **opening the downloaded file**). Per rule #14 these cannot be replaced by reading backend logs. API/DB are used only for **setup** (reaching a large/empty/within-limit filtered set) and **cross-cutting assertion** (row-count reconciliation against the listing endpoint).
- **Why not API-only:** the entire fix is a **client-side render gate** (button disabled, tooltip text, error toast). There is no server endpoint that "returns the disabled state" — the behaviour exists only in the browser. API-only would test nothing the fix changed. (Decision tree in `test-strategy-decision`: feature has UI affordance ⇒ UI obligatory.)
- **Setup / Exercise / Assertion split** (per scenario): Setup = navigate + apply filters to produce the target result set (mostly UI filter actions, since the filters *are* how a user reaches each state; date-range widening is the lever for volume). Exercise = view/hover/click the export controls. Assertion = button state + tooltip/toast copy + modal + (happy path) opened file content + DB row-count reconciliation.
- **Environments:** **QA2** primary (per task). **DoD requires Staging** too — flag that the reproducible CTs must also run on `stg` before the task closes. Disabled-path UI render needs a **high-volume env or seeded wide date range** (Q2) — not QA2 as-is.
- **Smoke vs full:** Smoke = CT-01 (enabled Overview, file diff) + CT-06 (Email CSV modal) — fast, proves the two everyday paths. Full = all CTs + the source-confirmed disabled-path note.
- **Suites to activate:** none beyond this task's file — no signing/payment/dual-brand surface touched. (regression-suites-map not triggered.)
- **Parallelization:** CTs are independent and read-only on data (no mutation) → parallelizable. **Caveat:** they share the QA2 listing data; an empty-result CT and an enabled CT must use **disjoint filters** so neither depends on the other's state. No merchant mutation → `merchant-preflight` is **NOT** run (rule #12: tests operating on existing listing data must not mutate merchant config — `skipMerchantPreflight` / skip the helper).

## Test-Data Strategy (the central problem — disabled path)

Applying `test-data-hierarchy` (fresh-via-automation is default; reuse is the documented exception):

**The conflict:** the highest-risk behaviour (disabled guard) needs **one filtered set whose estimated CSV size > 48 MiB** = ≈66,000 Overview rows (~762 bytes/row, 27 cols) or proportionally more narrow Leads rows (17 cols). QA2 accrues ~44 leads/day. Creating ~66k fresh leads via `sendApplication` purely to render a disabled button is **disproportionate** (cost ≫ risk it covers, and the fix is a pure client-side float comparison with no per-lead business logic). Fresh-data-default does not mean "fabricate 66k rows for a render check."

**Decision (per axis):**

| Path | Data approach | Level | Justification |
|------|---------------|-------|---------------|
| **Enabled (below limit)** | **Fresh-or-existing wide date range that returns a non-empty set well under 48 MiB** (trivially true on QA2 — ~44/day ⇒ ~0.03 MB). Read-only over listing data. | **E2E (UI exercise + open file)** | Reproducible today; this is the broad-blast-radius regression check. Reuse of existing listing rows is acceptable here because the assertion is *export-mechanics* (filename/columns/row-count vs portal), not per-record business state — no mutation, no bug-classification risk. |
| **Empty** | **Narrow filter guaranteed to return zero** (e.g. impossible LeadPk / future-only date / non-existent invoice). Fresh, deterministic. | **E2E** | Fully reproducible on QA2; deterministic empty set. |
| **Permission** | Requires a **role without CSV download permission** on QA2 (Q-role). | **E2E (conditional)** | If no such role/account exists, CT-09 is source-confirmed only — surfaced, not assumed. |
| **Disabled (>48 MiB)** | **NOT reproduced via fresh fabrication on QA2.** Two sanctioned options, in preference order: (a) **high-volume env / seeded wide date range** with ≈66k+ leads (Q2) — preferred if it exists; (b) **source-level confirmation** (deployed bundle constants `r=50331648`, comparison `≤`, exact tooltip string, error-toast path) — already done in KB, primary truth for this frontend behaviour. **No DB INSERT/UPDATE to fake the count** (rule #9 / Exception 3 — forbidden without explicit authorization, and would be a non-fresh artifact anyway). | **Source-confirmed; UI-rendered only on high-volume env** | The disabled guard is a deterministic float comparison decoded at source; a 66k-row fabrication adds cost without adding confidence in the comparison logic. We do **not** silently mark AC-02 "passed via UI" on QA2 — the report must state "source-confirmed; UI-disabled render pending high-volume env." |

**Anti-pattern explicitly avoided:** no `UPDATE`/`INSERT` to manufacture `totalRows`; no reuse of a mutated record to "force" the disabled state. The honest classification is `[CONFIRMADO at source]` + `[BLOQUEADO no UI por volume de dados]`, never a false green.

## Per-AC Technique Mapping (`test-design-techniques`)

- **Boundary Value Analysis (BVA)** — the size guard is a single numeric boundary at `50331648` bytes (48 MiB), **inclusive**. Logical boundary cases: `< 48 MiB` ⇒ enabled (CT-01/02, reproducible), `== 48 MiB` ⇒ enabled (source-only, Q4), `> 48 MiB` ⇒ disabled (CT-08, source-confirmed). Plus the degenerate boundary `totalRows == 0` ⇒ estimate 0 ⇒ but disabled-by-empty (CT-03/04). This is the primary technique for AC-01/02/06.
- **Decision Table** — the render/enabled matrix is a clean AND/OR table over four conditions; each distinct outcome column is a CT:

  | Condition | C1 | C2 | C3 | C4 | C5 |
  |-----------|----|----|----|----|----|
  | `hasDownloadPermission` | Y | Y | Y | Y | N |
  | `headers.length > 0` | Y | Y | Y | Y | (n/a) |
  | table empty | N | N | Y | N | N |
  | size > 48 MiB | N | Y | N | Y(empty=N) | N |
  | → Download CSV | **enabled** | **disabled+tooltip** | **disabled, no tooltip** | **disabled+tooltip** | **not rendered** |
  | → Email CSV | enabled | enabled | **disabled** | enabled | enabled |

  C1=CT-01/02, C2/C4=CT-08, C3=CT-03/04, C5=CT-09. (C2 and C4 collapse to the same outcome — the size-exceeded branch — so one disabled-render CT covers both.)
- **State transition (lightweight)** — AC-08 filter re-evaluation: `enabled → (widen to empty) → disabled-no-tooltip → (narrow to non-empty within limit) → enabled`. The enabled↔disabled-by-size transition is the source-only segment. CT-07 covers the reproducible segments.
- **Equivalence partitioning** — Email CSV address input (CT-06): empty ⇒ Send disabled; any non-empty ⇒ Send enabled. (Email-format validation depth is OUT — KB doesn't confirm client-side format checking; assert only the empty/non-empty gate.)
- **Error guessing / known smells** — SSN column present in `leads-results.csv` (PII in a file on disk → cleanup the downloaded artifact); size estimate sensitivity to row *width* (same row count, wider content can flip — note in CT-08 reasoning, not a separate CT since unreproducible).

## Activity-log note (rule #13 / `activity-log-validation`)

Rule #13 mandates a log assertion for every **business action**. This feature is a **client-side read-only export gate** — viewing/filtering a listing and building a CSV in the browser are **not** business state mutations and produce **no** `uown_los_lead_notes` row. Therefore:

- For **Download CSV / disabled-guard / empty / permission** CTs: **no activity log expected**, and absence is **correct behaviour**, not an observability gap. The SPEC records this explicitly so the validator does not flag a missing log as a failure.
- For **Email CSV** (CT-06): the async email *send* is the only candidate side effect. Per `activity-log-validation` §"Gaps known", email dispatch typically lands in `uown_email_queue` / `uown_correspondence_logs`, **not** `uown_los_lead_notes`. Since we deliberately do **not** trigger a real send (side effect OUT, Q3), there is **no log to assert** in scope. If PO requires asserting the send (Q3), add: after confirming a real address, poll `uown_email_queue` for a fresh row with the CSV-export template (template name uncaptured — needs discovery). **This is the only place a log could matter; flagged, not silently dropped.**

Verdict: log coverage is **complete by being explicitly N/A** for the read-only paths, with a conditional hook for the Email CSV send pending Q3.
