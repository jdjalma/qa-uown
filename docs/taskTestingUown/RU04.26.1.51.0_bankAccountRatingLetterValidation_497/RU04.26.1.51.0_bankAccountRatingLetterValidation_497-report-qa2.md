## Descrição

Em produção, as contas **566171** e **528451** tiveram pagamentos ACH criados por um agente interno (`airral.capili`) usando `achProcessType=REQUEST` com `postingDate` futuro. O fluxo `createOrUpdateAchPayment` deveria auto-setar o rating letter para `P` (Payment Arrangement) quando todas as condições fossem atendidas, porém o check `username == "customer portal"` no backend barrou a atualização para usuários internos. Como consequência, o rating letter precisou ser ajustado manualmente em produção, desligando o auto-pay.

**Escopo de QA solicitado:**
1. Adição de bank account com `Set as default payment? = Yes`
2. Adição de bank account com `Set as default payment? = No`
3. Deleção de bank account
4. Validação do rating letter em UI (Servicing Information + Activity Log), API (`getAccountByPk`) e DB (`uown_sv_account.rating`) em todos os cenários
5. Replicação do cenário exato do bug #497 (ACH `REQUEST` + posting futuro por usuário interno) via UI e via API

## Cenários

### CT-01

**Objetivo:** Validar que adicionar uma bank account com `Set as default payment? = Yes` em uma conta com rating neutro (`null`) preserva o rating letter e habilita ACH no autopay.

**O que é verificado:** O sistema aceita o cadastro da conta bancária como método padrão, adiciona `ACH` ao conjunto `autoPayTypes` da conta (conforme business rule 58) e mantém o campo `rating` em `uown_sv_account` inalterado. A UI (Servicing Information), a API (`getAccountByPk.rating`) e o DB retornam exatamente o mesmo valor de rating que existia antes da ação. O activity log registra o evento `BANK_ACCOUNT` com mensagem `ADDED : BankAccount[..., autoPay = TRUE]` e não há registro de "Rating letter changed".

Exemplos:
| Campo | Valor Observado |
|-------|-----------------|
| Rating antes | `null` |
| Rating depois (UI/API/DB) | `null` |
| Bank account `auto_pay` | `true` |
| ACH em `autoPayTypes` | SIM |
| accountPk | 11376 |
| Duração | 34.8s |

#### Como verificar manualmente

1. Criar conta ACTIVE via fluxo de originação (merchant `TerraceFinance` em `qa2`): originar lead, enviar contrato, coletar assinatura, settlement e funding — resultado é um `accountPk` com `rating = null`.
2. Logar no Servicing em `https://servicing.qa2.uown.com` como `automation.gow`.
3. Navegar para `/customer/{accountPk}` e confirmar no card "Servicing Information" que o campo **Rating Letter** está vazio ou `-`.

   Esperado: `rating` = `NULL`, `auto_pay_types` contém `ACH`, `auto_pay = true`, `is_deleted = false`.
7. API: `GET /account/getAccountById/{accountPk}` — resposta deve ter `rating: null` e `autoPayTypes` contendo `ACH`.
