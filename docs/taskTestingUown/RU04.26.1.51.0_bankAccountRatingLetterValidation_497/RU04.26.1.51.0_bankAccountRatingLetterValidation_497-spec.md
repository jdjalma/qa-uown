## Edge Cases

- **Fix ainda não deployado (CT-06):** usar `expect.soft()` em todas as assertions de rating pós-ação. Teste deve coletar estado UI+API+DB+log completo e o relatório deve listar o gap como "bug de aplicação pendente" (não falhar o pipeline).
- **Rating `C` / `M` (não apenas `P`):** CT-05 cobre `P`; variações com `C` (bankruptcy) e `M` (military) seguem a mesma regra de precedência (§58). Considerar parametrizar CT-05 se o orquestrador pedir; por padrão, apenas `P` conforme scenarios.
- **Routing number de teste:** `121042882` é routing público (Wells Fargo) — válido no sandbox mas garantir que não dispare validação externa real. Se sandbox rejeitar, fallback para routing específico do env (`SANDBOX_TEST_ROUTING`).
- **Múltiplas bank accounts (CT-04):** ao deletar BA2, validar por **último 4 dígitos do account number** (não pelo PK), pois a UI expõe apenas o mascarado. Garantir que BA1 e BA2 tenham últimos 4 diferentes no setup.
- **Activity log race condition:** logs podem ser assíncronos. Usar `db.waitForRecord` com timeout 10s para aguardar entry `BANK_ACCOUNT ADDED` / `Deleted Bank Account` antes de assertar.
- **Autopay banner "ACH autopay turned off":** aparece **apenas** quando havia autopay ativo (CT-03). Em CT-04 (delete não-default), banner NÃO deve aparecer — asserção negativa.
- **CT-05 setup de rating=P:** prefer criar via payment arrangement (caminho natural). Se não for determinístico, solicitar autorização explícita ao usuário antes de rodar UPDATE direto no DB (conforme `security.md`: INSERT/UPDATE/DELETE precisam de autorização).
- **Autenticação Servicing:** `tests/auth.setup.ts` tem bloco Servicing comentado (memory). Antes de CT-01..CT-06 rodarem, habilitar e persistir `servicing-internal.json` — tarefa do `subagent-impl-e2e`.
- **Posting date = today+7 em fim de semana/feriado:** regra de negócio pode deslocar para dia útil. CT-06 deve derivar `postingDate` do primeiro dia útil ≥ today+7 para evitar rejeição.
