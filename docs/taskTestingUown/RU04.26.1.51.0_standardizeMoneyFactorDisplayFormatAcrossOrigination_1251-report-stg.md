## Descricao

O Money Factor e a taxa de lease do programa, armazenada no banco como decimal (ex: `0.1846`). Para o usuario, esse valor e exibido multiplicado por 100 com o icone `%` — ou seja, `18.46%`. O fix da #1251 padronizou a coluna "Money Factor" da tabela de Programs para usar a mesma funcao de conversao (`toMoneyFactorDisplayValue`) ja utilizada no formulario de detalhes:

```
toMoneyFactorDisplayValue(0.1846)
= trimTrailingZeros((0.1846 * 100).toFixed(4))
= trimTrailingZeros("18.4600")
= "18.46"
```

Esta execucao em stg valida que o fix da coluna da tabela esta promovido para o ambiente seguinte. A diferenca observada em CT-02 entre tabela (`"18.46"`) e formulario (`"18.459999999999997"`) **nao e bug funcional** — e artefato de representacao IEEE 754 (mesmo numero, formatacao distinta). Detalhes na secao Observacoes.

## Observacoes (nao sao bugs)

### OBS-01 — Tabela e formulario formatam o mesmo numero de formas diferentes (arredondamento, nao bug)

| Campo | Valor |
|-------|-------|
| **Ambiente** | stg (mesmo comportamento em qa2) |
| **Observado em** | CT-02 |
| **Classificacao** | Inconsistencia de **formatacao**, nao bug funcional |
| **Severidade** | Cosmetica (UX) — valor numerico esta correto, apenas a renderizacao em string difere |

**Descricao:** Ao abrir `/programs/1`, o campo "Money Factor" exibe `18.459999999999997` enquanto a coluna da tabela em `/programs` exibe `18.46`. Sao **representacoes diferentes do mesmo numero** — JavaScript nao consegue representar `0.1846 * 100` exatamente como `18.46` em IEEE 754 e expoe a aproximacao binaria quando o resultado e usado sem arredondamento. Equivalencia matematica:

```js
0.1846 * 100                          // 18.459999999999997
(0.1846 * 100).toFixed(4)             // "18.4600"
Number("18.459999999999997")          // 18.459999999999997
Number("18.46")                       // 18.46
0.1846 * 100 === 18.46                // false (precisao binaria)
Math.abs(0.1846 * 100 - 18.46) < 1e-10 // true (igualdade pratica)
```

Por que **nao e bug**:
- O valor armazenado no DB (`numeric(9,6) = 0.1846`) nao foi alterado
- A coluna da tabela exibe `18.46` (formatado corretamente para o usuario final)
- O campo do formulario carrega a representacao binaria do mesmo numero — nenhum dado e perdido nem corrompido
- Re-salvar sem editar produz `18.459999999999997 / 100 = 0.18459999999999997`, que arredondado para `numeric(9,6)` no DB volta a ser `0.1846`

**Sugestao de melhoria (UX, nao critica):** `toMoneyFactorFormValue` poderia aplicar `.toFixed(4)` para alinhar a formatacao do input com a da tabela, evitando que o usuario veja a string longa. Mas isso e polimento visual, nao correcao de bug.

```typescript
// atual
export const toMoneyFactorFormValue = (value: number) => value * 100;

// sugestao (puramente cosmetica)
export const toMoneyFactorFormValue = (value: number) =>
  parseFloat((value * 100).toFixed(4));
```

**Comparativo entre ambientes:**

| Ambiente | Tabela | Formulario | Mesma observacao? |
|----------|--------|------------|-------------------|
| qa2 | `18.46` | `18.459999999999997` | sim |
| **stg** | `18.46` | `18.459999999999997` | sim (re-confirmado nesta execucao) |

**Ajuste sugerido para o teste CT-02:** comparar valores numericos com tolerancia em vez de strings literais — ex.:

```typescript
expect(Number(tableValue)).toBeCloseTo(Number(formValue), 4);
```

Isso reflete a intencao real (valor exibido equivalente) sem falhar por artefato de representacao.
