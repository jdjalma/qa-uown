## Scope

**IN:**
- **Content of the rendered `IN_2025_SAC_16_MONTHS` GowSign contract** (iframe/PDF): Item 4 (Promotional-Payoff Option), Item 4a (Lease-Purchase Ownership / EPO), the singular EARLY PURCHASE OPTION appendix, the lease-plan footnote, and the Indiana state-specific body (§1/§6(a)/§7/§14 + pre-auth brackets). — This is the deliverable of #579; a wrong clause is customer-facing and regulated.
- **Provider + template routing** for an IN 16-month lease (→ GOWSIGN / `IN_2025_SAC_16_MONTHS`) with a 13-month negative control. — The stale `state-merchant-matrix.ts` IN row (`SIGNWELL`) must not mask a real GowSign selection.
- **Dynamic token resolution** — every `{{token}}` renders non-empty, zero literal `{{ }}`, values cross-checked against `uown_esign_document.request` (appendix-H H.6). — The AL/OH `{{nextPaymentDueAmount}}` blank regression is the canonical migration bug.
- **Negative assertions (AC6)** — the wrong-cluster wordings (day-count promo, discount %, dual-option appendix, hardcoded $15 NSF, 60-day reinstatement, wrong-state header) must be ABSENT. — Migration regressions manifest as leftover donor-template text.
- **Signing completion + activity log** (rule #13) for the setup path that reaches the render. — No log = nothing happened.

**OUT:**
- **SignWell regression / visual diff** — IN is being migrated *to* GowSign; there is no SignWell IN 16m baseline to diff against, and IN routes to GowSign once the template exists. (Contrast: the general GoSign-rollout rule mandates SignWell diff only where a SignWell contract still renders for that state/term. Flag revisited in Open Questions if IN 13m still routes SignWell.)
- **EPO *calculation* correctness** (the numeric payoff cascade) — #579 is a *contract-content* task; the EPO math is owned by `04-calculos-financeiros.md` and its own tests. We assert the value renders and is non-empty/consistent with the dispatch variables, not that the payoff formula is arithmetically correct.
- **Other 16m states** (CA/TX/FL/AL/OH/NY/IL/MA…) — covered by their own tickets/oracles. IN sanity uses them only as a reference cluster.
- **Dual-brand parity** — routing is by effective state + template availability, not brand; the merchant that reaches IN 16m is dictated by ABB eligibility (see Ambiguous). Not a submitApplication-handler change, so the dual-brand floor rule does not apply.
- **Post-signing lifecycle** (funding, activation, payments) — out of the template-content scope.

## Test Strategy

- **Approach:** **Hybrid, UI-first (rule #14).** API/discovery setup to reach the rendered contract → UI render inspection of the GowSign iframe/PDF (the deliverable is *what the customer sees*, not a log) → DB assertions for routing selection, token cross-check, and the signing activity log.
- **Justification:** the entire ticket is visual/PDF content verification. Per rule #14 and appendix-H H.6, rendered placeholders / clause text / red labels CANNOT be validated by reading backend logs (Daniel's Jewelers CA origin). API-only is restricted to: (a) lead setup to CONTRACT_CREATED, (b) the CT-01 routing DB read, (c) the CT-04 dispatch-variable cross-check, (d) the CT-08 log assertion.
- **Data (rule #9):** fresh IN customer + fresh SSN per run via the realistic factory; merchant via `ensureMerchantReady` preflight (rule #12) — the 16m program must be active on the chosen merchant. NO reuse of an existing IN lease; NO direct DB UPDATE (Exception 3).
- **Environments:** primary render capture wherever an IN 16m lease is reachable (IL used **stg**; local/sandbox tunnels were down for IL). Confirm IN template availability per env with `getGowSignTemplatesForState` before asserting. DoD requires **stg** validation.
- **CI-automatable split (inherited from IL):**
  - **Automatable now:** CT-01 routing + the 13m negative control render, IF IN 13m routes via a non-NeuroID gateway (e.g. TerraceFinance) — reuse existing helpers (`createPreQualifiedApplication` → `sendInvoice` → `MissingDataFormPage`/`ContractCompletePage` → `TermsOfAgreementPage` → `AlternativeContractModalPage` → `signGowSignInFrame`).
  - **MCP-manual only (if NeuroID-gated):** CT-02..CT-06 content + CT-08 signing for the 16m contract. This is the expected outcome given the IL precedent; treat CI automation of 16m as blocked until a stg NeuroID test-merchant bypass exists.
- **Suites to activate:** signing content family — `gowsign-contract-content*`, `gowsign-signature-fields`; NO SignWell visual diff (see OUT). `signing-regression/multi-state-signing.spec.ts` gains IN as a covered state once frozen.
- **Existing artifacts to reuse (rule #2):** `getGowSignTemplatesForState`, `assertSelectedTemplateForLead`, `getEsignDocumentByLeadAndClient` (`src/helpers/gowsign-template-db.helpers.ts`); `waitForLeadStatus`, `waitForEsignDocumentStatus`, `findLeadNoteContaining` (`src/helpers/esign-db.helpers.ts`); `signGowSignInFrame` (`src/helpers/gowsign-signing.helper.ts`); `AlternativeContractModalPage.getGowSignFrame`. Do NOT recreate.

## Scenarios (prioritized)

### S1 — Routing: IN 16m selects `IN_2025_SAC_16_MONTHS` (GowSign) + 13m negative control  [P0]
- **Oracle:** CT-01. **Technique:** decision table (state IN × term {13,16} → template/client), state-transition (…→ CONTRACT_CREATED where the esign doc is born).
- **Persona:** agent/system (routing is invisible to the customer; validated via DB).
- **Setup:** discovery pass (rule #18) — `getGowSignTemplatesForState(db,'IN')` to confirm inventory + find the merchant route that yields WK16 (and WK13 for the control). Fresh IN customer, merchant preflight.
- **Steps:** drive an IN 16m lead to CONTRACT_CREATED → assert selection; repeat for 13m as control.
- **Validations:**
  - DB: `assertSelectedTemplateForLead(db, leadPk, 'IN_2025_SAC_16_MONTHS')`, `client='GOWSIGN'`; 13m → IN 13m document_name (or documented SIGNWELL fallback if no IN 13m template), `client` ≠ unexpected.
  - Activity log (rule #13): CONTRACT_CREATED transition note present in `uown_los_lead_notes`.
- **Pitfalls:** stale `state-merchant-matrix.ts` IN=SIGNWELL (ignore matrix, trust live DB, rule #16); esign doc only exists at CONTRACT_CREATED, not UW_APPROVED.

### S2 — Item 4 + Item 4a + token smoke render correctly  [P0]
- **Oracle:** CT-02, CT-03, CT-04. **Technique:** decision table (MUST vs MUST NOT clause variants), error-guessing (blank token, leftover donor text), BVA-adjacent (float cent tolerance on money).
- **Persona:** customer (Website, reads the contract in the GowSign iframe — potentially mobile).
- **Setup:** rendered IN 16m contract (CI if non-NeuroID; else MCP-manual capture).
- **Steps:** capture `frame.innerText()` (iframe by class `alternative-contract-vendor_iframe__nSb3A` / substring `gowsign`, NOT `gowsign.com`); assert Item 4 + Item 4a MUST/MUST NOT; assert token smoke.
- **Validations:**
  - UI: Item 4 heading exactly "Promotional-Payoff Option:"; daily-accrual-through-exercise-date phrase; "void this option". Item 4a tail "{{totalNumberOfPayments}} payments of $ {{nextPaymentDueAmount}} (plus tax)" + "(including the application fee)" + "Total Cost" + "and you will own the Property".
  - UI (negative, AC6): no day-count promo / expiry / lesser-of-Cash-Price / "other purchase options"; no `%` discount / `firstPaymentDueAmount` / separate `salesTax` line / current-date accrual.
  - UI + DB: `nextPaymentDueAmount` non-empty; every money token cross-checks `uown_esign_document.request` variables via `toBeCloseTo` (float repr — never `toEqual`); zero literal `{{ }}`; logo present; no "variables map missing" dispatch note.
- **Edge cases:** `16-\nmonth` line break (OH lesson); comma-stripped money comparison.
- **Pitfalls:** Item 4a is a partial new-baseline — do NOT reuse an IL/AL assertion verbatim; ground on AC2.

### S3 — Singular EARLY PURCHASE OPTION appendix + plan footnote  [P1]

- **Pitfalls:** page duplication (assert single occurrence, IL CT-06 lesson).

### S4 — Indiana state-specific body (§1/§6(a)/§7/§14 + pre-auth brackets)  [P1]
- **Oracle:** CT-06. **Technique:** decision table (Indiana values vs wrong-state/stale-value), equivalence partitioning (nsfFee dynamic vs hardcoded).
- **Persona:** customer + (compliance/legal as the ultimate stakeholder — regulated lease content).
- **Setup:** rendered IN 16m contract.
- **Validations:**
  - UI: header "CONSUMER LEASE-PURCHASE AGREEMENT-IN" (anchor on `AGREEMENT-IN`, not the word "Indiana"); "Lessee should read the contract in its entirety…"; §1 "(*) Total delivery fee"; §6(a) shows the dynamic `{{nsfFee}}` value (non-empty numeric); §7 "120 days" to reinstate; pre-auth forgo-debit `[late/delinquency/reinstatement]` + `[late/delinquency/fees may accrue…]` brackets; consent "[Click Here]" brackets outside the link text; §14 ACH labels CHANGE OF INFORMATION / RETURNS / CANCELLATION in **red** (assert via computed style/class, not text alone — color is not conveyed by text).
  - UI (negative, AC6): no hardcoded "$15.00" declined charge; no "60 days" reinstatement; no "*(*)" delivery footnote; no EPO payoff schedule table; no wrong-state header.
- **Pitfalls:** `{{nsfFee}}` is dynamic — assert the resolved value is present AND that the literal `$15.00` is absent (cross-check the configured NSF fee via nsf-fee rule / `05-pagamentos.md`); red-label assertion must read style, per rule #14 (color is a visual affordance).

### S5 — Signing completes + activity log (rule #13)  [P2 / blocked-if-NeuroID]

- **Setup:** rendered IN 16m contract (MCP-manual if NeuroID-gated).
- **Steps:** `signGowSignInFrame(page, frame, { preauthChoice:'yes', waitForCompleted:true })`.
- **Validations:**
  - UI: signing ceremony completes → redirect/appComplete.
  - DB: `waitForLeadStatus(db, leadPk, 'SIGNED')`; `uown_esign_document.status` reaches SIGNED then STORED, `doc_signed_time_stamp` non-null (post-signature: query by lead+client — `request` becomes the string `getDocumentStatus`).
  - Activity log (rule #13): `[ContractService][isLeaseOrLeaseModSigned]…SIGNED`, `[EsignRedirectService][updateSignStatus]…to SIGNED`, `[ESIGNSERVICE][parseCCPeekConsent]`.
- **Pitfalls:** NeuroID reload → "Sign Contract" trick (MCP-manual only); Buddy widget swallows "Proceed" click.
