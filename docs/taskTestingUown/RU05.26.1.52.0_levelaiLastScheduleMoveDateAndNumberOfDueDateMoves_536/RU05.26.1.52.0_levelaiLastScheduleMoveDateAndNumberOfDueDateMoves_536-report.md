## Improvements (recommended, non-blocking — for qa-doc-keeper)

1. **Encode the WEEKLY 3-day offset cap in catalog (per regra #11)** — F-004 surfaced an undocumented backend constraint now baked into the spec via adaptive offset. Per regra #11, this should be lifted into a domain skill (e.g., new section in [[application-lifecycle]] or a move-due-date skill). Primary source: `svc MoveDueDatesService:119`.

Adding this schema to a domain skill (e.g., a new "DueDateMoves table contract" section) prevents the same drift on future specs.
3. **Encode the timezone-naive pitfall (per regra #11)** — F-006 surfaced a general framework pitfall: `since`-style helpers using `since.toISOString()` against `timestamp without time zone` columns misbehave on any host not in UTC. Worth adding to [[db-polling-pattern]] / [[helpers-catalog]] as a documented anti-pattern with remediation (prefer fresh-account hygiene + no time bound; or use server `NOW()`; or `AT TIME ZONE 'UTC'` casts). Reference fix shipped here: drop the `since` parameter (matches `waitForDueDateMovesRowCount` / `getTopDueDateMoveRow` pattern).
4. **Cosmetic `"from dueDate"` (F-002)** — variable-name leak in activity log notes; not blocking #536. Owner = move-due-date feature team, separate ticket.
