## Descrição

O `CalculatorStep` agora verifica o flag de configuração `allowMultipleOptionsFor` (default: `"TERRACE_FINANCE"`). Quando o `clientType` de um merchant integrado via API está na lista permitida, `sendApplication` retorna TODAS as opções de pagamento (planos de 13 e 16 meses). Para merchants API fora da lista, apenas o termo máximo (16m) é retornado.

O filtro é aplicado APENAS quando TODAS as condições são verdadeiras:
1. `eligibleTerms` não está vazio (GDS aprovou múltiplos termos)
2. Merchant NÃO está na lista `allowMultipleOptionsFor`
3. `IntegrationType == API`
4. `termMonths.size() > 1`

**Commit:** `25f71ae1f` — `uown/backend/svc#488 Return multiple payment option programs for Terrace. Add config`

**Dados de teste:** SSN `888888888` → GDS retorna `lambdaSegment=5` ou `6` → `eligible_terms="13,16"`. Perfil fixo (nome, endereço) necessário para passar o `DataMismatchStep`.

## Cenários

### CT-01

**Objetivo:** Validar que TerraceFinance (clientType `TERRACE_FINANCE`, presente na lista `allowMultipleOptionsFor`) recebe opções de pagamento para AMBOS os termos de 13 e 16 meses.

**O que é verificado:** Ao enviar uma aplicação via `sendApplication` para TerraceFinance com SSN 888888888 (que gera `eligible_terms="13,16"` no GDS), o sistema retorna `paymentDetailsList` com 5 entradas — planos de 13m e 16m. O `CalculatorStep` detecta que TERRACE_FINANCE está na lista `allowMultipleOptionsFor` e NÃO aplica o filtro de max term, retornando todas as opções ao merchant.

Exemplos:
| Campo | Valor |
|-------|-------|
| Merchant | TerraceFinance (`OL90202-0001`) |
| clientType | `TERRACE_FINANCE` |
| SSN | `888888888` |
| lambdaScore | 6 |
| eligible_terms | `"13,16"` |
| appApprovalStatus | `APPROVED` |
| paymentDetailsList count | 5 |
| termInMonths | `[13, 16]` |

#### Como verificar manualmente

1. Via Postman: `POST https://svc-stg.uownleasing.com/uown/los/sendApplication` com body JSON contendo:
   - `userName: "terraceFinance"`, `setupPassword: "U0wn_terraceFinance_xJ9z4p"`, `merchantNumber: "OL90202-0001"`
   - `mainSSN: "888888888"`, `mainFirstName: "debra"`, `mainLastName: "swarey"`
   - `mainAddress1: "121 Madison Ave 5"`, `mainCity: "New York"`, `mainStateOrProvince: "NY"`, `mainPostalCode: "10016"`
   - Line items + order total (obrigatório para calculator rodar)
2. Verificar `appApprovalStatus = "APPROVED"`
3. Inspecionar `paymentDetailsList`: deve conter entradas com `termInMonths: 13` E `termInMonths: 16`
4. Verificar que os `planId` nos `redirectUrl` incluem WK13, BW13, WK16, BW16

**PASSOU**

### CT-02

**Objetivo:** Validar que PayPossible (clientType `PAY_POSSIBLE`, NÃO presente na lista `allowMultipleOptionsFor`) recebe APENAS opções do termo máximo (16 meses).

**O que é verificado:** Ao enviar uma aplicação para PayPossible com o mesmo SSN 888888888 (que também gera `eligible_terms="13,16"`), o `CalculatorStep` detecta que PAY_POSSIBLE NÃO está na lista `allowMultipleOptionsFor`, que é merchant API, e que há múltiplos termos elegíveis — aplica o filtro de max term. A response contém APENAS planos de 16m (WK16, BW16), sem planos de 13m.

Exemplos:

   - `userName: "payPossible"`, `setupPassword: "U0wn_payPossible_eSOZdN"`, `merchantNumber: "PP00001-0001"`

3. Inspecionar `paymentDetailsList`: deve conter APENAS entradas com `termInMonths: 16`

### CT-03

**Objetivo:** Validar via banco de dados que ambos os leads (CT-01 e CT-02) possuem `eligible_terms = "13,16"` na tabela `uown_los_uwdata`, confirmando que a diferença de resultado é exclusivamente do filtro no `CalculatorStep`.

**O que é verificado:** A consulta SQL confirma que AMBOS os leads receberam `eligible_terms = "13,16"` do GDS (mesmo SSN, mesmo lambda score). A diferença de payment options na response API (5 vs 3) é causada exclusivamente pelo filtro configurável no `CalculatorStep`, não por diferença na elegibilidade do UW.

Exemplos:
| Lead | eligible_terms | lambda_segment | uw_status |
|------|---------------|----------------|-----------|
| 6559056 (TerraceFinance) | `"13,16"` | 6 | APPROVED |
| 6559057 (PayPossible) | `"13,16"` | 6 | APPROVED |

#### Como verificar manualmente

1. Executar query no DB de stg:
   ```sql
   SELECT lead_pk, eligible_terms, lambda_segment, uw_status
   FROM uown_los_uwdata
   WHERE lead_pk IN (6559056, 6559057)
   ORDER BY lead_pk;
   ```
2. Verificar que ambos têm `eligible_terms = '13,16'` e `uw_status = 'APPROVED'`

**PASSOU**

### CT-04

**Objetivo:** Validar que quando `eligible_terms` contém apenas 1 termo (ex: `"13"`), o filtro do `CalculatorStep` NÃO é aplicado — o merchant API recebe todas as opções para aquele termo.

**O que é verificado:** Ao enviar uma aplicação para PayPossible (merchant API, NÃO na lista `allowMultipleOptionsFor`) com um SSN aleatório que gera `eligible_terms = "13"` (apenas 1 termo), a condição `termMonths.size() > 1` do filtro é falsa. O `CalculatorStep` retorna TODAS as opções de 13m sem filtrar, confirmando que o filtro de max term é aplicado APENAS quando há múltiplos termos elegíveis.

Exemplos:
| Campo | Valor |
|-------|-------|
| Merchant | PayPossible (`PP00001-0001`) |
| clientType | `PAY_POSSIBLE` |
| SSN | Aleatório (não 888888888) |
| lambdaScore | 3 |
| eligible_terms | `"13"` |
| appApprovalStatus | `APPROVED` |
| paymentDetailsList count | 2 |
| termInMonths | `[13]` (único termo, sem filtro) |

#### Como verificar manualmente

1. Via Postman: `POST /uown/los/sendApplication` com PayPossible e um SSN qualquer (não terminando em 9)
2. Verificar `appApprovalStatus = "APPROVED"`
3. Inspecionar `paymentDetailsList`: deve conter opções de 13m (WK13, BW13)
4. Confirmar que o filtro NÃO foi aplicado — todas as opções para o termo elegível são retornadas

**PASSOU**
