## Setup base (executa 1x antes de cada cenário)

Cada cenário precisa de um **lead novo** (regra de Test Data Hierarchy: dados frescos por CT).

### Passo 1 — Criar aplicação

```http
POST https://svc-stg.uownleasing.com/uown/los/sendApplication
Content-Type: application/json
X-API-Key: <UOWN_API_KEY>

{
  "merchantNumber": "OW90218-0001",
  "merchantName": "TerraceFinance",
  "firstName": "Manual",
  "lastName": "Repro<TIMESTAMP>",
  "ssn": "666112233",
  "dob": "1985-06-15",
  "email": "manual-repro-<TIMESTAMP>@example.com",
  "phone": "5551234567",
  "address1": "123 Test St",
  "city": "Los Angeles",
  "state": "CA",
  "zip": "90001",
  "monthlyIncome": "5000",
  "orderTotal": "1000",
  "orderDescription": "Manual repro #507"
}
```

> Substitua `<TIMESTAMP>` (e o sufixo do `lastName`/`email`) por um valor único — `date +%s` ou similar. SSN não pode terminar em 9 (causa UW_DENIED).

**Capture da resposta:**
- `authorizationNumber` → use como `<leadPk>`
- `accountNumber` → use como `<leadUuid>`
- `paymentDetailsList[0].redirectUrl` → use como `<contractUrl>`. Extraia daí:
  - `<shortCode>` = primeiro segmento do path (ex: `gO9USNot` em `/gO9USNot/complete`)
  - `<planId>` = query param `planId` (ex: `WK13`)

### Passo 2 — Bind merchant program

> **OBRIGATÓRIO** — sem isso, o `authorizeCreditCard` retorna 500 com `"Merchant program is required to determine fee to be charged for lead {pk}"`. Pitfall conhecido.

```http
GET https://svc-stg.uownleasing.com/uown/los/missing-fields/<shortCode>?planId=<planId>
X-API-Key: <UOWN_API_KEY>
```

**Esperado:** HTTP 200 com lista de campos faltantes (vazio em fresh leads). Se 500 → o redirectUrl foi invalidado (ver pitfall #21 em `application-lifecycle-protocol.md`).

## CT-01 — Approved CC: shape allow-list (4 campos)

**Objetivo:** confirmar que o response retorna **exatamente** os 4 campos do slim DTO.

### Ação

```http
POST https://svc-stg.uownleasing.com/uown/los/authorizeCreditCard
Content-Type: application/json
X-API-Key: <UOWN_API_KEY>

{
  "leadPk": "<leadPk>",
  "ccNumber": "5146515000000019",
  "ccExp": "12/2030",
  "cvc": "123",
  "ccFirstName": "Manual",
  "ccLastName": "Repro<TIMESTAMP>"
}
```

### Esperado

- HTTP **200**
- Body **exatamente**:
  ```json
  {
    "status": "APPROVED",
    "error": null,
    "errorCode": null,
    "preAuthStatus": "NOT_RUN"
  }
  ```

### Como validar

1. Conte as chaves: deve dar **4** (não 3, não 5).
2. Confirme que **NÃO existe**: `gatewayRequest`, `gatewayResponse`, `token`, `ccToken`, `ccNumber`, `creditCardTransactionPk`, `authorizationCode`, `transaction`, `cardData`, `rawResponse`, `rawRequest`.
3. `preAuthStatus` deve ser um destes valores: `SESSION_UNAVAILABLE`, `SUCCESS`, `DENIED`, `ERROR`, `NOT_RUN`. Em **stg** é `NOT_RUN` (Kount preauth desabilitado).

### Falha indicaria

Mais de 4 campos no body → regressão da fix #507 (slim DTO quebrado).

## CT-02 — Deny-list deep traversal

**Objetivo:** confirmar **profundamente** que nenhum campo sensível existe em qualquer nível do JSON.

### Ação

Igual ao CT-01.

### Como validar

1. Cole o body em https://jsonpath.com/ ou similar.
2. Para cada chave proibida abaixo, busque com JSONPath `$..<chave>` — deve retornar **vazio**:

| Categoria | Chaves proibidas |
|-----------|------------------|
| Raw card data | `ccNumber`, `creditCardNumber`, `cardNumber`, `cvc`, `cvv`, `ccExp`, `expiry`, `expirationDate` |
| Gateway internals | `gatewayRequest`, `gatewayResponse`, `gatewayCorrelationId`, `gatewayTransactionId`, `transactionToken`, `requestID`, `idempotencyKey` |
| Tokens / PKs | `token`, `ccToken`, `creditCardTransactionPk`, `creditCardPk`, `authorizationCode` |
| Raw transaction | `transaction`, `creditCardTransaction`, `ccTransaction`, `ccCaptureReply`, `purchaseTotals`, `amount`, `capturedAmount` |
| Outros blobs | `cardData`, `rawResponse`, `rawRequest` |

3. Verifique o tamanho do body:
   ```bash
   echo -n '<body>' | wc -c
   ```
   Deve ser **~77 bytes** (sanity guard: <500 bytes).

### Falha indicaria

Qualquer chave da lista presente em qualquer profundidade → regressão de segurança.

## CT-03 — Decline real via lastName mismatch

**Objetivo:** validar que o slim DTO se mantém quando o backend recusa de verdade.

### Ação

Crie um lead novo (passos 1 e 2). Então:

```http
POST https://svc-stg.uownleasing.com/uown/los/authorizeCreditCard
Content-Type: application/json
X-API-Key: <UOWN_API_KEY>

{
  "leadPk": "<leadPk>",
  "ccNumber": "5146515000000019",
  "ccExp": "12/2030",
  "cvc": "123",
  "ccFirstName": "Manual",
  "ccLastName": "MISMATCHEDLAST"
}
```

> **Nota:** `ccLastName` propositalmente diferente do `lastName` que foi usado no `sendApplication`. Isso força o path `uownClient.declineCreditCard` no backend (`svc/ApplicationService.java:172-173`).

### Esperado

- HTTP **200**
- Body:
  ```json
  {
    "status": "ERROR",
    "error": "Credit Card is invalid.",
    "errorCode": null,
    "preAuthStatus": "NOT_RUN"
  }
  ```

### Como validar

- 4 chaves exatas (mesmo do CT-01).
- `status === "ERROR"` (ou `"DENIED"`).
- `error` não-nulo (mensagem do backend).
- Deny-list ainda zerada.

### Falha indicaria

Body com mais de 4 campos OU vazamento de detalhes do gateway na mensagem de erro.

## CT-04 — ccNumber incompleto ⚠️ (com ressalva)

**Objetivo:** validar comportamento do backend com CC malformado.

### Ação

Crie um lead novo (passos 1 e 2). Então:

```http
POST https://svc-stg.uownleasing.com/uown/los/authorizeCreditCard
Content-Type: application/json
X-API-Key: <UOWN_API_KEY>

{
  "leadPk": "<leadPk>",
  "ccNumber": "4111",
  "ccExp": "12/2030",
  "cvc": "123",
  "ccFirstName": "Manual",
  "ccLastName": "Repro<TIMESTAMP>"
}
```

### ⚠️ Comportamento observado em stg (NÃO bate com o Gherkin literal)

- HTTP **500**
- Body: **Spring exception envelope**, não slim DTO:
  ```json
  {
    "message": "Invalid card. Please try again",
    "status": 500
  }
  ```

> **Atenção:** o Gherkin original pedia *"apenas os 4 campos do DTO mínimo"* mesmo nesse caso. O que o backend entrega é um shape diferente (2 campos, `status` é o HTTP code numérico, não o enum `CCTransactionStatus`).

### Como validar

1. Confirme que **NÃO há leak**: deny-list (lista do CT-02) deve estar zerada.
2. Body deve ser pequeno (<300 bytes).
3. **Decisão de pass/fail**:
   - Se você considera que `#507` cobre só o controller principal (interpretação atual da automação): **PASS** — sem leak, scope ok.
   - Se você considera que o Gherkin é o contrato literal: **FAIL** — backend deveria retornar slim DTO mesmo no caminho de erro pré-controller. **Levante na issue do GitLab.**

### Falha indicaria

Vazamento de gateway info no `message` (ex: stack trace, headers de gateway).

## CT-07 — Body adversarial + headers extras

**Objetivo:** garantir que campos/headers extras NÃO forçam um response verboso.

### Ação

Crie um lead novo. Então:

```http

{
  "leadPk": "<leadPk>",
  "ccNumber": "5146515000000019",
  "ccExp": "12/2030",
  "cvc": "123",
  "ccFirstName": "Manual",
  "ccLastName": "Repro<TIMESTAMP>",

  "debug": true,

```

> **Não** envie `Accept: application/json+full` — causa 406 Not Acceptable e não é o ponto do teste.

### Esperado

- HTTP **200**

### Como validar

1. Confirmar que `requestFullResponse`, `debug`, `includeGatewayData` **não aparecem** no response.
2. Slim DTO holds (mesmo do CT-01).

### Falha indicaria

Backend respeitando algum dos hints adversariais e expondo dado extra.

## CT-E2E-01 — Slim DTO no browser real (DevTools)

**Objetivo:** equivalente direto ao TC-02 do QA original — validar via DevTools que o frontend recebe o slim DTO.

### Setup

1. Crie um lead novo via `sendApplication` (passo 1 do setup base).
2. Faça `getMissingFields` (passo 2 do setup base).

### Ação

1. Abra Chrome/Firefox.
2. Abra **DevTools → Network** (Ctrl+Shift+I → tab "Network").
3. Marque "Preserve log".
4. Filter: `authorizeCreditCard`.
5. Cole o `<contractUrl>` capturado no setup na barra de endereço e navegue.
6. Preencha o formulário com o cartão `5146515000000019`, exp `12/2030`, cvc `123`, nome do applicant.
7. Submeta.

### Como validar (DevTools)

1. Encontre a chamada `POST /uown/los/authorizeCreditCard` no Network.
2. Tab **Preview** (ou **Response**) deve mostrar:
   ```json
   {"status":"APPROVED","error":null,"errorCode":null,"preAuthStatus":"NOT_RUN"}
   ```
3. Tab **Headers** → **Response Headers** → `content-length` deve ser ~77 bytes.
4. Procure visualmente: nenhum campo `gatewayRequest`, `gatewayResponse`, `token`, etc.

### Falha indicaria

Qualquer campo extra visível no Preview/Response → regressão da fix.

> **Nota stg:** após submeter o pagamento, o passo de e-sign (SignWell) pode não carregar em stg (flake conhecido). **Isso não invalida a validação da #507** — o que importa já foi capturado na chamada de `/authorizeCreditCard`.

## CT-E2E-02 — Mensagem de erro renderizada (UI)

**Objetivo:** confirmar que UI mostra erro apropriado **sem expor detalhes técnicos do gateway**.

### Setup

Crie um lead novo (passos 1 e 2).

### Ação

1. DevTools aberto, filter `authorizeCreditCard`, "Preserve log".
2. Navegue ao `<contractUrl>`.
3. Preencha o formulário com um cartão **decline** OU dados inválidos:
   - Use cartão `4000000000000002` (Stripe decline) — em stg pode aprovar mesmo assim
   - Ou use `lastName` diferente do applicant (forces decline real)
4. Submeta.

### Como validar

1. **Network:** chamada `/authorizeCreditCard` retorna body slim (mesmo shape do CT-03).
2. **UI:** mensagem de erro visível para o usuário (pode ser toast, banner, alert):
   - Procure por elementos com `role="alert"`, `.error`, `.toast`, ou texto vermelho
   - Mensagem deve ser **friendly** (ex: "Credit Card is invalid", "Please try a different card")
3. **Não deve haver:** stack trace, JSON dump, detalhes do gateway visíveis ao usuário.

### Falha indicaria

- UI mostrando JSON bruto / stack trace / gateway response → regressão UX.
- Mensagem de erro contendo `gatewayResponse`, `transactionId`, etc → regressão de segurança.
