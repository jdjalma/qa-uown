## Pipeline Justification — Why API-only (CLAUDE.md Rule #15)

CLAUDE.md Rule #15 makes UI-first the default and API-only the exception. This task qualifies as the **(c) cross-cutting wire-payload validation** exception:

- The bug being fixed is **the JSON serialized by the backend** for `POST /uown/los/authorizeCreditCard`. Sensitive fields (`gatewayRequest`, `gatewayResponse`, full `ccNumber`, `token`, raw transaction object) used to leak in the response body.
- A UI test cannot reliably observe what the wire payload contains — the frontend silently discards unknown fields. By the time the data reaches a rendered component, the leak is already invisible to the test. Manual QA confirmed this required Postman + DevTools network inspection.
- There is **no UI affordance** that exposes the difference between "frontend got the slim DTO" vs "frontend got the fat DTO and ignored the extra keys" — both render identically. The defect is structurally invisible from the browser.
- A deep-traversal allow-list / deny-list assertion against the raw JSON is the only mechanism that catches a future regression where someone re-adds `gatewayResponse` to the controller response.

> **Frontend smoke is already covered downstream.** The existing TireAgent / contract-page E2E suites continue to exercise the authorize call through the browser — they prove the new minimal DTO does not break the flow. This SPEC focuses on the wire contract.

## Allow-list & Deny-list (Single Source of Truth)

**Allow-list (top-level keys allowed in the slim DTO).** Backend ground-truth is the Java record `AuthorizeCreditCardResponse(CCTransactionStatus status, String error, String errorCode, PreAuthStatus preAuthStatus)` — **EXACTLY** these 4 fields are serialized. The previous SPEC erroneously included `message`; this has been removed (Gap 7).

```
ALLOWED_TOP_LEVEL_KEYS = { 'status', 'error', 'errorCode', 'preAuthStatus' }
```

Each CT now asserts `Object.keys(respBody).length === 4` in addition to the subset check, so a regression that adds a 5th key fails LOUDLY (not silently as a deny-list miss).

**`preAuthStatus` enum (Gap 1).** Backend `PreAuthStatus` enum: `{ SESSION_UNAVAILABLE, SUCCESS, DENIED, ERROR, NOT_RUN }`. On `stg`, Kount preauth is disabled by env config and the value observed is `NOT_RUN`. The slim-DTO assertion validates **membership** in the enum set (env-independent) — never asserts a specific value (env-dependent). All CTs perform this validation through the shared `assertSlimDto` helper.

> Source: backend Java record (svc `AuthorizeCreditCardResponse.java`), `CCTransactionStatus` enum, `PreAuthStatus` enum. The `AuthorizeCreditCardResponseBody` TS interface in `src/api/responses/credit-card.response.ts` currently still declares `authorizationCode` and `creditCardTransactionPk` as optional — **the deny-list check is what catches them at runtime**, the typed interface alone is insufficient because TypeScript's structural typing tolerates extra fields on `BaseResponseBody`.
>
> **Follow-up (not part of this SPEC's pass criteria, but flagged for impl-api):** the typed interface should be slimmed to `{ status: string; error?: string; errorCode?: string; preAuthStatus?: string; }` (no `extends BaseResponseBody` — the backend record has no `message`) and the now-removed fields deleted. impl-api owns this edit.

**Deny-list (forbidden keys at ANY nesting depth).** Deep-traversal of the parsed JSON. Match by exact key name, case-sensitive. If any of these appear at any depth, CT fails with the offending JSON path.

```
FORBIDDEN_KEYS = {
  // raw card data
  'ccNumber', 'creditCardNumber', 'cardNumber',
  'cvc', 'cvv',
  'ccExp', 'expiry', 'expirationDate',
  // gateway internals
  'gatewayRequest', 'gatewayResponse',
  'gatewayCorrelationId', 'gatewayTransactionId',
  'transactionToken', 'requestID', 'idempotencyKey',
  // tokens / persisted PKs
  'token', 'ccToken',
  'creditCardTransactionPk', 'creditCardPk',
  'authorizationCode',
  // raw transaction shape
  'transaction', 'creditCardTransaction', 'ccTransaction',
  'ccCaptureReply', 'purchaseTotals', 'amount', 'capturedAmount',
}
```

> The `transactionToken`, `requestID`, `idempotencyKey`, `ccCaptureReply`, `purchaseTotals` entries are derived directly from real `gatewayResponse` blobs already in our test corpus (see `docs/taskTestingUown/Ticket1154/Ticket1154.md` lines 2109, 2616, 3338) — the exact shape that the bug used to leak. Including them by name catches a regression that re-introduces a flat copy of those fields without the `gatewayResponse` envelope.

> `amount` and `capturedAmount` are listed because they appear inside `ccCaptureReply` payloads. If a future regression flattens those into the top-level DTO, the deny-list catches them.

**False-positive guard.** `status`, `message`, `error`, `errorCode`, `preAuthStatus` are **never** in `FORBIDDEN_KEYS`. The deep-traversal must skip values that are non-objects (so a string value of `"transaction failed"` under `message` does not match a `transaction` key).

## Edge Cases

- **Endpoint returns HTTP 200 with `status='DECLINED'` instead of HTTP 4xx.** CT-03 step 2 must NOT assert `resp.ok === false`; assert on body fields instead. Manual QA confirmed the success-path returns `{status:"APPROVED", error:null, errorCode:null, preAuthStatus:"SUCCESS"}`, implying the failure path uses the same envelope with different values rather than HTTP error codes.
- **`ccToken` field name variants.** Backend Java code may serialize as `ccToken`, `cc_token`, or `creditCardToken` depending on Jackson settings. The deny-list catches all three (`ccToken`, `token`). If a CT fails with an unexpected variant, add it to `FORBIDDEN_KEYS` in the same PR — that is a finding, not a flake.
- **Empty `error` field.** Manual QA observed `error: null, errorCode: null` on the success path. `null` values for allow-listed keys are OK. The allow-list check is on KEY presence, not value type.
- **Network retry / 5xx.** If `resp.ok === false` with HTTP 5xx, that is a separate bug (endpoint flakiness) and the test should fail with a clear message — do NOT swallow and re-attempt the assertions on undefined body. Wrap the call in a single retry only if the failure is `ECONNRESET`-class transport noise.
- **Masked PCI logging.** Step 4 of CT-02 must mask any leaked card-number value before printing it into the test failure message. Never echo a real (or test) PAN into a Playwright report. Hash or print only `<key> at <path> (value masked)`.
- **Run isolation.** Each CT runs in its own `test()` (NOT shared `beforeAll`) so a CT-01 failure does not poison CT-02/CT-03. Acceptable cost: 3× the lead creation. Each setup is ~60s; full suite ~5 minutes.
- **Backend not yet deployed in target env.** If `stg` has not received the hotfix yet, CT-01 step 4 (allow-list) will fail with a list of the old fields. That is expected pre-deploy and is a **valid signal**, not a bug in the test. Tag the run as `pre-deploy regression check` in the report. Re-run after the hotfix lands.
