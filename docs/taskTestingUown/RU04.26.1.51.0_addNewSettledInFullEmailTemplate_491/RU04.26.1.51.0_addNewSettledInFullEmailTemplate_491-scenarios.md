## Contexto da Feature

Adicionar novo email template **"Settled in Full"** disparado pelo sweep `settledInFullAccountEmailSweep` quando uma conta atinge `account_status = SETTLED_IN_FULL`. Template deve existir para **UOWN** e **Kornerstone**, com imagens hospedadas no GCS bucket (sem URLs externas).

### Arquitetura relevante

| Componente | Valor |
|------------|-------|
| Sweep task name | `settledInFullAccountEmailSweep` |
| Cron | `0 0 2 ? * MON-FRI` |
| Trigger manual | `POST /uown/svc/triggerScheduledTask/settledInFullAccountEmailSweep` |
| Inspeção SQL | `GET /uown/svc/getScheduledTaskByName/settledInFullAccountEmailSweep` |
| Template UOWN | `SettledInFullEmail` |
| Template Kornerstone | `KORNERSTONE_SettledInFullEmail` |
| From UOWN | `CustomerService@uownleasing.com` |
| From Kornerstone | `CS@kornerstoneliving.com` |
| Tabela enfileiramento | `uown_email_queue` |
| Tabela log | `correspondence_logs` |
| Condição de elegibilidade | `account_status='SETTLED_IN_FULL'` AND `settled_in_full_date_time IS NOT NULL` AND `rating NOT IN ('E','F','U')` AND dia da semana Mon–Fri AND janela de datas dependente do dia (Mon/Tue=4 dias atrás, Wed=2–4 dias, Thu/Fri=2 dias) |
| Resolução do email | `uown_sv_email` onde `email_type='PRIMARY'` AND `do_not_email=false` |

### Estratégia de dados de teste — EXCEÇÃO documentada à Test Data Hierarchy

> ⚠️ Este spec **reusa fixtures pré-existentes** (accountPk 11385–11413) em vez de criar contas fresh via automação por CT. Ver `.claude/rules/testing.md § Test Data Hierarchy` — este é um caso da exceção "fixture impossível de criar via automação dentro do tempo do teste".

**Justificativa (inviolável — documentar sempre):**

O sweep `settledInFullAccountEmailSweep` tem um predicate SQL DOW-dependente que exige `settled_in_full_date_time` em uma data exata do passado (4 dias para Mon/Tue, 2–4 dias para Wed, 2 dias para Thu/Fri). Três bloqueios para fluxo fresh completo:

1. **sif_date:** criar uma conta fresh via happy path resultaria em `settled_in_full_date_time = HOJE`, que não cai na janela do sweep até 4 dias úteis depois. Teste nunca triggerava.
2. **rating=E/F/U (CT-04):** rating é computado pelo sistema baseado em histórico de pagamentos — não pode ser forçado via automação em runtime.
3. **company='KORNERSTONE' (CT-02):** campo herda da lógica de routing backend (merchant + BIN + banking data) — happy path smoke não produz Kornerstone accounts diretamente.

**Estratégia efetiva (approved by user):**

1. User executa Postman happy path para criar shell accounts (UOWN + Kornerstone)
2. User executa UPDATEs autorizados para setar `account_status='SETTLED_IN_FULL'`, `settled_in_full_date_time` na janela do dia, `company='KORNERSTONE'` quando aplicável, `rating='E'` para CT-04
3. Teste roda: `findEligibleSettledInFullAccount` / `findIneligibleSettledInFullAccount` selecionam via SELECT
4. Teste atualiza primary email via `POST /uown/svc/createOrUpdateEmail` para `fintechgroup777@gmail.com`
5. Teste dispara `settledInFullAccountEmailSweep` → enfileira → dispara `emailSweep` → envia → IMAP verifica

**Fixtures em qa2:**
- 11385–11387: UOWN SETTLED_IN_FULL sif=2026-04-17 (CT-01 + CT-05)
- 11388–11390: UOWN SETTLED_IN_FULL rating=E sif=2026-04-17 (CT-04)
- 11391–11395: UOWN ACTIVE (CT-06)
- 11403–11405: KORNERSTONE SETTLED_IN_FULL sif=2026-04-17 (CT-02)

### Re-validação (quando BUG-01 for corrigido pelo dev)

O `settled_in_full_date_time` das fixtures está cravado em `2026-04-17` — só entra na janela do sweep em **Terça 2026-04-21**. Para re-rodar em outro dia:

1. Calcular data alvo: `DATE(today) - N` onde N é 4 (Mon/Tue), 2–4 (Wed) ou 2 (Thu/Fri) — ajustar conforme o DOW do dia do run
2. User autoriza `UPDATE uown_sv_account SET settled_in_full_date_time = 'YYYY-MM-DD 10:00:00' WHERE pk IN (11385, 11386, 11387, 11388, 11389, 11390, 11403, 11404, 11405)`
3. Rodar o teste
4. Se muitos dias passaram, considerar refazer os UPDATEs completos ou criar novos fixtures via smoke

## User Stories

### US-01: Email de settled in full para cliente UOWN

**Como** cliente UOWN com conta `SETTLED_IN_FULL`,
**Quero** receber um email branded UOWN com os dados da quitação,
**Para** ter prova formal da quitação.

**Critérios de Aceite**
- [ ] Email chega no endereço primary do cliente
- [ ] Remetente = `CustomerService@uownleasing.com`
- [ ] Assunto contém identificação do settlement/UOWN
- [ ] Conteúdo segue HTML aprovado (nome do cliente, settlement date, último pagamento)
- [ ] Todas as imagens carregam de URLs `https://storage.googleapis.com/...`

### US-02: Email de settled in full para cliente Kornerstone

**Como** cliente Kornerstone com conta `SETTLED_IN_FULL`,
**Quero** receber o email com branding Kornerstone,
**Para** que a comunicação seja consistente com a marca que contratei.

**Critérios de Aceite**
- [ ] Template usado = `KORNERSTONE_SettledInFullEmail`
- [ ] Remetente = `CS@kornerstoneliving.com`
- [ ] Todas as imagens em GCS (não externas)

### US-03: Regras de elegibilidade do sweep

**Como** operador UOWN,
**Quero** que o sweep ignore contas inelegíveis (rating ruim, opt-out de email, status diferente),
**Para** não enviar emails indevidos.

**Critérios de Aceite**
- [ ] Conta com rating `E`/`F`/`U` não enfileira
- [ ] Conta com primary email `do_not_email=true` não enfileira
- [ ] Conta sem status `SETTLED_IN_FULL` não enfileira

### US-04: Imagens hospedadas no GCS

**Como** responsável pelos templates,
**Quero** que todas as imagens venham do GCS bucket,
**Para** garantir estabilidade visual independente do host original.

**Critérios de Aceite**
- [ ] `<img src="..."` sempre aponta para `storage.googleapis.com`
- [ ] Nenhuma URL externa (ex: `mailchimp`, `sendgrid`, CDN de terceiros)
