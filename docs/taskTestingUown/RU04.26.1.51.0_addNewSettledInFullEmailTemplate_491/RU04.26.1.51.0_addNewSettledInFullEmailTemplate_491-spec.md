### Eligibility SQL (canonical reference ŌĆö used by helpers)

```sql
-- Given: :company IN ('UOWN','KORNERSTONE')
SELECT a.pk              AS account_pk,
       a.account_status,
       a.rating,
       a.settled_in_full_date_time,
       a.company,
       c.pk              AS customer_pk,
       c.first_name,
       c.last_name
  FROM uown_sv_account a
  JOIN uown_sv_customer c ON c.pk = a.customer_pk
 WHERE a.account_status = 'SETTLED_IN_FULL'
   AND a.settled_in_full_date_time IS NOT NULL
   AND (a.rating NOT IN ('E','F','U') OR a.rating IS NULL)
   AND a.company = $1
   AND (
         (extract(DOW FROM CURRENT_DATE) IN (1,2) AND a.settled_in_full_date_time::date >= CURRENT_DATE - INTERVAL '4 days')
      OR (extract(DOW FROM CURRENT_DATE) = 3     AND a.settled_in_full_date_time::date BETWEEN CURRENT_DATE - INTERVAL '4 days' AND CURRENT_DATE - INTERVAL '2 days')
      OR (extract(DOW FROM CURRENT_DATE) IN (4,5) AND a.settled_in_full_date_time::date >= CURRENT_DATE - INTERVAL '2 days')
       )
 ORDER BY a.settled_in_full_date_time DESC
 LIMIT 1;
```

## Test Data Acquisition

Each CT uses one of three strategies (declared in its Preconditions):

- **STRATEGY-A (reuse eligible)**: `findEligibleSettledInFullAccount(company)` returns an existing account. Used by CT-01, CT-02, CT-08, CT-09.
- **STRATEGY-B (mutate to make ineligible)**: Clone strategy-A seed, then update a single field (rating, do_not_email, account_status) to force ineligibility. Used by CT-04, CT-05, CT-06.
- **STRATEGY-C (static metadata)**: No account needed ŌĆö just hit the inspection endpoint. Used by CT-07.

> **Note (Phase 5):** mutation strategies (CT-04/05/06) require `UPDATE` on `uown_sv_account` and `uown_sv_email`. Per security rules, the test MUST request explicit user authorization before executing these UPDATEs, or prefer finding/creating separate fixture accounts. The recommended approach is: pick a **different** eligible account per negative CT and mutate that isolated fixture. Document the UPDATE intent in the test report; fail fast if consent is not given.

## Edge Cases

- **Weekend execution:** sweep cron excludes Sat/Sun. If test runs Sat/Sun against prod DB, trigger still executes but `findEligibleSettledInFullAccount` returns null because the date-window predicate includes `extract(DOW FROM CURRENT_DATE) BETWEEN 1 AND 5`. CTs should `test.skip()` gracefully when no eligible account ŌĆö document in report.
- **IMAP race / stale emails:** `getEmailContent` must filter by `date > triggerTimestamp`. Fase 5 may need to pass `sinceTimestamp` into `getEmailContent` or rely on ImapFlow's built-in `since` filter (already present for subject searches with 10-min window) ŌĆö if flakiness appears, widen or use strict `msgDate > triggerTimestamp` check in the processor.
- **Parallel runs with same inbox:** serialize CT-01/02/03/08/09 via `describe.serial` to ensure the latest email in the inbox is the one produced by the test.
- **Sweep processes all eligible accounts at once:** CT-01 and CT-02 can share one trigger; design Phase 5 implementation to call sweep ONCE then assert per-brand.
- **Mutation authorization:** per security rule, `UPDATE` on `uown_sv_account` / `uown_sv_email` requires explicit consent. Prefer selecting pre-existing fixture accounts via differentiated SELECTs; fall back to `createOrUpdateEmail` (API, allowed) for DNE toggling ŌĆö NOT direct SQL UPDATEs on `uown_sv_account.rating`.
- **Rating fixture scarcity:** if no rating-E/F/U SETTLED_IN_FULL account exists in qa2, CT-04 must `test.skip('no rating-{E,F,U} fixture available in qa2')` and flag in report.
- **Kornerstone scarcity:** if no eligible Kornerstone SIF account exists, CT-02 must `test.skip` and the report should note that the Kornerstone path was not validated in this run.
