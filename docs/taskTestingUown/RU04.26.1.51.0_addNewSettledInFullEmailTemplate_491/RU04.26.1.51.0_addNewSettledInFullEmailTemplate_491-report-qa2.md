## Descrição

A tarefa adiciona o template de e-mail "Settled in Full" para ambas as marcas (UOWN e Kornerstone) e ativa o sweep agendado `settledInFullAccountEmailSweep`, que percorre contas elegíveis e dispara o e-mail de acordo com regras de negócio.

Critérios de aceite observados:

- Template "Settled in Full" existente para a marca UOWN, com imagens hospedadas no GCS (sem URLs externas).
- Template "Settled in Full" existente para a marca Kornerstone, com as mesmas regras.
- Sweep `settledInFullAccountEmailSweep` agendado via Quartz (`cronTrigger = 0 0 2 ? * MON-FRI`), ativo, usando SQL que filtra por `status = SETTLED_IN_FULL`, exclui `rating IN (E, F, U)` (ou `rating IS NULL`), respeita `do_not_email` e aplica janela temporal dependente do DOW.
- E-mail disparado apenas para contas com status `SETTLED_IN_FULL`, rating elegível (≠ E/F/U) e `do_not_email = false`.

## Infra/Test Issues

### INFRA-01 — Quartz não executa manual triggers do sweep em qa2 hoje

**Severidade:** Alta — bloqueia 100% dos CTs positivos do teste.

**Sintoma:**
- `POST /uown/svc/triggerScheduledTask/settledInFullAccountEmailSweep` responde 200.
- `uown_scheduled_task.is_active=true`.
- Nenhuma linha é escrita em `uown_email_queue` ou `uown_correspondence_logs` para as contas alvo nos 60s seguintes.
- `uown_scheduled_task.last_trigger_time` permanece em `2026-04-22T20:41:09.687Z` (carimbo do último run bem-sucedido em 2026-04-22) mesmo após múltiplos triggers em 2026-04-23.

**Evidência de que a config+sql estão corretas** (não é problema do feature):
- Rodando a SQL do sweep (`sqlToPickAccounts`) diretamente contra o DB retornou `[{pk:11476}, {pk:11477}]` — as contas criadas hoje SÃO elegíveis.
- `is_active=true` confirmado.
- CT-07 mostra o contrato do endpoint OK.
- `uown_email_queue` continua recebendo outros templates normalmente hoje (`Welcome`, `PaymentReceiptEmail`, `KORNERSTONE_PaymentReceiptEmail` para as mesmas contas 11476/11477). A pipeline de e-mail como um todo está viva; só a task `settledInFullAccountEmailSweep` não está firing.
- Em 2026-04-22 o mesmo spec funcionou (`last_trigger_time=20:41:09Z`); a quebra é entre 2026-04-22 e 2026-04-23.

**Hipóteses prováveis (escopo: time do svc / infra qa2):**
1. Quartz scheduler em estado `paused`/`misfire` para esta task específica — `resumeScheduledTask` pode re-armar.
2. Lock table Quartz não foi liberada de uma fire anterior (olhar `qrtz_fired_triggers` / `qrtz_triggers`).
3. Container svc em qa2 reiniciou e o Quartz ainda não re-scheduled esta task ativa.
4. Clock drift no container svc (observado em runs anteriores: `sent_time` ~1h atrás do UTC real) causando o cron `0 0 2 ? * MON-FRI` a fire em momento inesperado e bloquear re-fires via trigger manual.

**Próximos passos sugeridos:**
- Time do svc: checar logs do Quartz em qa2 nos últimos 24h por `Job settledInFullAccountEmailSweep`, misfire warnings, lock timeouts.
- Se for lock-related: reiniciar svc qa2 ou liberar locks Quartz manualmente.
- Re-rodar o spec completo depois do unblocking para confirmar que CT-01/CT-02/CT-03/CT-08/CT-09 voltam a passar.

### INFRA-02 — Falta de fixture rating E/F/U em qa2 dentro da janela DOW atual

**Contexto:** CT-04 deu skip porque `findIneligibleSettledInFullAccount(db, 'E'|'F'|'U')` não localizou conta com `settled_in_full_date_time` dentro da janela DOW atual.

**Impacto:** Cobertura do filtro negativo de rating fica via contrato SQL (CT-07 assert contém `rating NOT IN ('E','F','U')`).

**Fix sugerido:** criar fixture longeva em qa2 com rating E/F/U e `settled_in_full_date_time` rolling, OU derivar conta E/F/U via drive + mutação autorizada.
