> âś… **PASSOU** Â· GriffinsFurniture (`KS5936`) em CA com programa `GOW 16 month program`, lead criado via `sendApplication` e levado a ACTIVE via API.

Lead 11929 (account 4878) ficou ACTIVE em qa1 com term=16, state=CA. As tres superficies do cliente leram o mesmo valor de EPO: $3,813.43. O calculo bate exatamente com a formula CA aplicada sobre o lease real (`cashPrice = $3,474.65`, `taxRate = 9.75%`, `processingFee = $0`, `buyoutFee = $0`):

#### 6. Boundary day inclusivo (today == earlyPayoffDateExpiry mantem CA)

> âś… **PASSOU** Â· KS5936 CA 16m apos `setEarlyPayoffDateExpiry(today)`.

Apos shift do `_90DayExpirationDate` para a data de hoje via `POST /uown/svc/createOrUpdateServicingInfo`, o lease permanece elegivel ao EPO sob a formula CA. O `epoBreakdown` do endpoint `getServicingInfo` continua expondo o header `Account 90 Day Payoff Eligible = TRUE` + formula `90 Day Payoff Amount - Total Paid Amount + Fees`. A flag `Eligible for 90-day Pay Off` no Servicing UI tambem se mantem em `Yes`. Confirma `EpoEligibleService` usa comparacao `!isBefore(today)` (estrito-em-igualdade), conforme SPEC.

**Evidencia:** leadPk=12013, accountPk=4942 (KS5936 CA 16m). Apos shift de expiry para today via `createOrUpdateServicingInfo`, `detectEpoRegime(epoBreakdown) = 'ca'`, eligibility flag UI = true.

#### 7. Apos expiry (yesterday) flipa para legacy 16m anytime buyout

> âś… **PASSOU** Â· KS5936 CA 16m apos `setEarlyPayoffDateExpiry(yesterday)`.

Apos shift do expiry para a data de ontem, o `epoBreakdown` muda de **regime estrutural**:

- **Pre-shift (CA regime):** `row[0] = ['Account 90 Day Payoff Eligible', 'State', '90 Day Payoff Amount', ...]`, `row[1] = ['TRUE', 'CA', '3813.43', ..., '90 Day Payoff Amount - Total Paid Amount + Fees', '3813.43']`.
- **Post-shift (legacy 16m):** `row[0] = ['ProgramType', 'MoneyFactor', 'Cost', ..., 'TotalLeaseAmount (cost*moneyFactor - cost)', 'DailyLeaseAmount', 'DaysUsed', 'Anytime buy out amount', ..., 'Formula', 'Balance']`, `row[1] = ['16 Month Anytime Buyout', '2.72', '3474.65', ..., '((DailyLeaseAmount*DaysUsed) + allFees + Tax) - TotalPaymentAmount', '3813.43']`.

A flag `Eligible for 90-day Pay Off` no Servicing UI tambem flipa para `No`.

**Nota numerica importante:** os valores numericos das duas formulas COINCIDEM ($3,813.43) para um lease recem-ativado porque `DaysUsed = 0` faz `DailyLeaseAmount * DaysUsed = 0`, e os fees do programa KS5936 sao zero. Assim ambas as formulas reduzem a `cashPrice + tax`. A asserĂ§ĂŁo e portanto feita sobre o **regime estrutural do breakdown** (cabecalho + label), nao sobre divergencia numerica. Esse comportamento bate com a OBS-4 da SPEC (PDF T0 vs runtime apos 90d) e foi documentado por Priyanka na issue: a divergencia entre 13m e 16m apos 90d e observavel quando o lease ja teve dias usados.

**Evidencia:** leadPk=12014, accountPk=4943 (KS5936 CA 16m). `detectEpoRegime(preBreakdown) = 'ca'`, `detectEpoRegime(postBreakdown) = 'legacy16m'`, activity log `DATA_CHANGE` capturado com mensagem `Changed 90DayExpirationDate from <future> to <yesterday>`.
