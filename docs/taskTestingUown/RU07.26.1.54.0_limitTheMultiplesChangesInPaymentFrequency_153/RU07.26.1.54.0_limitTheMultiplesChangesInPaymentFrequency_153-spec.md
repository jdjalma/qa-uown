**AC review findings (testability rubric):**
- AC1 scores 5/6 — "far into the future" is under-specified. **[REFINED BY QA]** operational oracle: after any single change, the next due date must fall within one frequency interval of today (Bi-Weekly ≤ ~15 days; Semi-Monthly ≤ next 1st/15th-class day) — NOT months/years out. - **Implicit AC (proposed):** a rejected (out-of-range day) Save must NOT mutate frequency, NOT write `uown_frequency_mods`, and NOT regenerate the schedule — the schedule stays on the prior frequency. Covered by B1/B2 negative rows.

## Scenarios (prioritized)

> Every scenario's validation block includes the rule #13 audit assertion and the schedule-regeneration check. `[reflex]` tags mark validations pulled from `qa-domain-reflexes`.

### B1 — Semi-Monthly First Payment Day boundary validation (P0)
- **Technique:** BVA + decision table (valid/invalid partitions of First Payment Day, range 1–17).
- **Persona:** returning customer, Website, mobile-first.
- **Setup:** fresh ACTIVE account on Bi-Weekly.
- **Boundary rows:** First Payment Day = **0** (below min → reject), **1** (min valid → accept), **17** (max valid → accept), **18** (above max → reject). Second Payment Day held at a mid-valid gap (e.g. +15).
- **Steps:** open Change payment schedule → select Semi-Monthly → set First Payment Day = {row} + a valid Second Payment Day → Save.
- **Validations:**
  - Valid rows — UI: success toast + "Current Frequency" = Semi-Monthly; due dates render on the selected day. DB: `uown_frequency_mods` row `fromFrequency=BI_WEEKLY, toFrequency=SEMI_MONTHLY` `[reflex]`; `uown_sv_receivable` regenerated (Rewind/Replay), installment count = `numOfPayments.{term}.SEMI_MONTHLY`.
  - Invalid rows (0, 18) — UI: inline validation error, Save blocked; "Current Frequency" **unchanged** (stays Bi-Weekly). DB: **no** new `uown_frequency_mods` row, schedule **not** regenerated (implicit-AC negative). [check-points: confirm no side effect on reject.]
- **Edge cases covered:** the fix's "update options for first payment day selection".
- **Pitfalls considered:** date pickers may ignore `fill`/`type` (native-setter pattern, payment-flows pitfall #19); React-Select day dropdowns need click-not-selectOption.

### B2 — Semi-Monthly Second Payment Day gap boundary validation (P0)
- **Technique:** BVA on MIN_DAY_GAP=14 / MAX_DAY_GAP=20 (the constants the MR introduced).
- **Setup:** fresh ACTIVE account on Bi-Weekly; First Payment Day fixed at a mid-valid value (e.g. day 3).
- **Boundary rows:** gap = **13** (below min → reject), **14** (min valid → accept), **20** (max valid → accept), **21** (above max → reject).
- **Steps:** select Semi-Monthly → set First Payment Day + Second Payment Day = first + {gap} → Save.
- **Validations:** same pattern as B1. Valid → Save succeeds, `uown_frequency_mods` + regen; the two semi-monthly due dates are spaced by the selected gap and do NOT wrap into a far-future month. Invalid → blocked, no mutation, frequency unchanged.
- **Edge cases covered:** the exact recalculation guardrail that failed in production (drift). Assert the **first** post-change due date is bounded near today, not months out.
- **Pitfalls considered:** gap that pushes the 2nd day past month-end (first=17, gap=20) — confirm the fix's intended behavior in discovery before asserting.

### S3 — Repeated Biweekly↔Semi-monthly toggle: no date drift (P0 — production bug repro)
- **Technique:** state transition (frequency state machine) + error guessing (the reported drift).
- **Persona:** customer who keeps adjusting to match their paycheck.
- **Setup:** fresh ACTIVE account on Bi-Weekly; **record baseline** next due date + frequency.
- **Steps:** Bi-Weekly→Semi-Monthly (valid days)→Save → Semi-Monthly→Bi-Weekly (valid next payday)→Save → repeat ≥3 full cycles.
- **Validations (consequence oracle — check-points, after EACH save):**
  - UI: toast + "Current Frequency" = the just-selected frequency (each cycle).
  - **Anti-drift oracle [REFINED AC1]:** the next due date after each save falls within one frequency interval of today — Bi-Weekly ≤ ~15 days out, Semi-Monthly on the next selected day — and does NOT march progressively further into the future across cycles (the 12/30/2027 symptom).
  - DB: one `uown_frequency_mods` row per successful save (`fromFrequency`/`toFrequency` correct each time) `[reflex]`; after each Rewind/Replay exactly **one** active receivable set (no accumulation of overlapping schedules).
- **Edge cases covered:** AC1, AC4. This is THE bug — a fresh-data reproduction (rule #10) is what turns the production observation into a confirmed pass/fail.
- **Pitfalls considered:** Rewind/Replay leaves stale deactivated receivables — assert on `status=ACTIVE` receivables only (§34); async schedule regen → `waitForRecord` with backoff, not a single query.

### S6 — 3–5 mixed updates: no unintended schedule changes after last save (P0)
- **Technique:** use-case + check-points (repeatable-action-without-unintended-effect).
- **Setup:** fresh ACTIVE account; **record baseline** frequency, next due date, full schedule.
- **Steps:** perform 3–5 self-service changes mixing valid Weekly/Bi-Weekly/Semi-Monthly options → after the **last** save, refresh and re-open the schedule + payment views.
- **Validations:** schedule reflects **only** the last saved frequency; installment count = `numOfPayments.{term}.{lastFREQ}`; **no duplicate/overlapping receivables**, no duplicate charges, no schedule change without a corresponding successful `uown_frequency_mods` row (count of audit rows == count of successful saves) `[reflex]`. Account did NOT silently become Monthly (AC3).
- **Edge cases covered:** AC3, AC4, idempotency of repeated saves.
