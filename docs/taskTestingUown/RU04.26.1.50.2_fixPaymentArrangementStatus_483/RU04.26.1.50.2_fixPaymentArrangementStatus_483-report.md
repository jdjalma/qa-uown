## Cenários

### CT-01

**Objetivo:** Verificar que um pagamento CC com cartão aprovado cria arrangement com status SUCCESS e atualiza rating para P.

**O que é verificado:** Ao submeter um payment arrangement via portal Servicing com cartão CC aprovado (Mastercard 5146...0055) e datas de hoje, o sistema processa o pagamento de forma síncrona, transiciona o arrangement para SUCCESS, registra a transação CC como APPROVED e tenta atualizar o rating da conta para P.

| Dado | Valor |
|------|-------|
| accountPk | 589025 |
| Customer | Keith Hatcher |
| arrangementPk | 107 |
| arrangement status | SUCCESS |
| CC txn status | APPROVED (1 transação) |
| Rating após arrangement | null (bug conhecido #446) |
| Activity log rating | 4 entradas — "Account rating updated to P due to new Payment Arrangement. arrangementPk=107" |

#### Como verificar manualmente

1. Acessar `https://svc-website-stg.uownleasing.com/` e logar como manager
2. Navegar para `Customer Information` da conta 589025: `/customer-information/589025`
3. Criar Payment Arrangement: marcar checkbox `#paymentArrangement`, preencher startDate e endDate com data de hoje, frequency BiWeekly
4. Preencher dados do cartão CC: 5146315000000055, CVC 998, exp 12/28, nome Keith Hatcher
5. Submeter e verificar toast "Payment Arrangement scheduled successfully."
6. Verificar no banco:
```sql
SELECT pk, status, arrangement_type, is_active, payment_type
FROM uown_sv_payment_arrangement WHERE account_pk = 589025 ORDER BY pk DESC LIMIT 1;
-- Esperado: status=SUCCESS, arrangement_type=NORMAL, payment_type=CC

SELECT pk, status FROM uown_sv_cc_transaction WHERE arrangement_pk = 107;
-- Esperado: status=APPROVED

SELECT rating FROM uown_sv_account WHERE pk = 589025;
-- Esperado: P (atualmente null — bug #446)
```

**PASSOU**

### CT-02

No ambiente STG, o gateway de pagamentos não rejeitou o cartão — todas as transações foram APPROVED e o arrangement ficou SUCCESS.

1. Acessar `https://svc-website-stg.uownleasing.com/` e logar como manager
2. Navegar para conta 588998: `/customer-information/588998`
3. Criar Payment Arrangement com cartão 4000300611112224, CVC 123, exp 09/28, nome Brandon Kelly
4. Verificar no banco:

### CT-03

**Objetivo:** Verificar que um arrangement com datas futuras e 3 parcelas BiWeekly permanece NOT_STARTED com todas as transações CC em PENDING.

**O que é verificado:** Ao criar um payment arrangement com data de início futura (5 dias) e data de fim 33 dias à frente, o sistema deveria calcular 3 parcelas BiWeekly e manter o arrangement como NOT_STARTED até que a primeira parcela seja processada na data agendada. As transações CC deveriam estar PENDING.

| Dado | Valor |
|------|-------|
| accountPk | 589025 |
| Customer | Keith Hatcher |
| arrangementPk | 109 |
| startDate | 04/12/2026 |
| endDate | 05/10/2026 |
| arrangement status | IN_PROGRESS (esperado: NOT_STARTED) |
| is_active | true |

#### Como verificar manualmente

1. Acessar `https://svc-website-stg.uownleasing.com/` e logar como manager
2. Navegar para conta 589025: `/customer-information/589025`
3. Criar Payment Arrangement: startDate=04/12/2026, endDate=05/10/2026, frequency=BiWeekly, cartão 5146315000000055
4. Verificar no banco:
```sql
SELECT pk, status, is_active, start_date, end_date
FROM uown_sv_payment_arrangement WHERE pk = 109;
-- Obtido: status=IN_PROGRESS, is_active=true
-- Esperado (Task #483): status=NOT_STARTED

SELECT pk, status, scheduled_date FROM uown_sv_cc_transaction WHERE arrangement_pk = 109 ORDER BY scheduled_date;
-- Esperado: 3 transações com status=PENDING
```

> Falha: `expect(received).toBe(expected) // Object.is equality — Expected: "NOT_STARTED", Received: "IN_PROGRESS"` — arrangement future-dated recebeu IN_PROGRESS ao invés de NOT_STARTED. Fix da Task #483 possivelmente não deployed no STG.

**FALHOU**

## Bugs de Aplicação Encontrados

### BUG-01: Arrangement future-dated recebe IN_PROGRESS ao invés de NOT_STARTED (Task #483)

| Campo | Valor |
|-------|-------|
| **Severidade** | Alta |
| **Ambiente afetado** | stg |
| **Reprodutível** | Sim (100%) |
| **arrangementPk** | 109 |
| **accountPk** | 589025 |

**Descrição:** Um payment arrangement com datas futuras (startDate=04/12/2026, endDate=05/10/2026) recebe status `IN_PROGRESS` imediatamente após criação, quando deveria ser `NOT_STARTED` até a primeira parcela ser processada. Este é exatamente o bug que a Task #483 visa corrigir.

**Evidência:**
```sql
SELECT pk, status, is_active, start_date, end_date
FROM uown_sv_payment_arrangement WHERE pk = 109;
-- status=IN_PROGRESS, is_active=true (esperado: NOT_STARTED)
```
