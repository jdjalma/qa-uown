| **Milestone** | RU04.26.1.51.0 |
| **Labels** | — |
| **Pipeline** | `new-flow` |

## Descrição

Ao navegar para **Account Settings > Update Contact Info** no Customer Portal (`website-qa2.uownleasing.com`), o frontend chamava `POST /uown/svc/sendVerificationCode/{newPhone}` ANTES de salvar o telefone via `createOrUpdatePrimaryCustomerContactInfo`. Como o novo número ainda não estava vinculado a nenhuma conta ativa no banco, o backend retornava HTTP 500: _"We could not find an active account for customer with phone number X in our system."_

**Fix (frontend):** O fluxo foi corrigido para salvar o telefone primeiro via `createOrUpdatePrimaryCustomerContactInfo` e somente depois chamar `sendVerificationCode`.

### CT-03

**Objetivo:** Verificar que `sendVerificationCode` retorna HTTP 200 (e não 500) quando chamado com um telefone que acabou de ser salvo — confirmando que o fix do bug está funcionando no nível de API.

**O que é verificado:** Após salvar um novo telefone via `createOrUpdatePrimaryCustomerContactInfo`, o endpoint `sendVerificationCode` consegue localizar a conta associada ao novo número e retorna HTTP 200. Este cenário reproduz exatamente a sequência corrigida pelo fix: salvar primeiro, verificar depois. Antes do fix, o `sendVerificationCode` era chamado antes do save, resultando em HTTP 500 ("could not find an active account").

|:-----:|-----------------|-----------|
| 1 | `GET /uown/svc/getPrimaryCustomerContactInfo/10855` | HTTP 200 — telefone atual obtido |
| 2 | `POST /uown/svc/createOrUpdatePrimaryCustomerContactInfo` (novo telefone) | HTTP 200 — areaCode=407, phoneNumber=555XXXX |
| 3 | `POST /uown/svc/sendVerificationCode/407555XXXX?company=UOWN` | **HTTP 200** (fix confirmado — não 500) |
| 4 | `POST createOrUpdatePrimaryCustomerContactInfo` (restauração) | HTTP 200 — telefone original restaurado |

#### Como verificar manualmente

1. Salvar um novo telefone para `accountPk=10855` via `POST {svcApiUrl}/uown/svc/createOrUpdatePrimaryCustomerContactInfo` (mesmo body do CT-02, Step 2)
2. Confirmar HTTP 200 na response
3. Chamar `POST {svcApiUrl}/uown/svc/sendVerificationCode/4075558403?company=UOWN` com header `Authorization: {SVC_API_KEY}`
4. Verificar que o status HTTP é **200** (ou 204) — não 500
5. Se retornar 500 com mensagem "could not find an active account for customer with phone number", o bug não foi corrigido
6. Restaurar telefone original via POST com valores originais
