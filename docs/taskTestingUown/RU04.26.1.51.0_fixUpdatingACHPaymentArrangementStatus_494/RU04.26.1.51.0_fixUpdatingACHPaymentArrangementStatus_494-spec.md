## Interaction Strategy

This task primarily validates backend state machine logic (PaymentArrangementACHListener). The interaction strategy follows "API for preconditions, DB for simulation, UI for verification":

1. **API steps**: Create ACH arrangements via `api.paymentArrangement.createOrUpdateAchPayments(body)` — this is the precondition setup
2. **DB steps**: UPDATE ACH payment statuses + call `recalculateAchArrangementStatus` — this simulates the listener behavior since Profituity is not available in qa2 for controlled status transitions
3. **UI steps**: Navigate to Servicing Portal Payment Arrangement page and Customer Information page to verify rendered status values match DB state

**Why not pure UI?** The UI cannot control individual ACH payment statuses. The Servicing Portal only displays arrangement results. The listener is triggered by Profituity sweep responses which cannot be controlled in qa2. DB simulation is the only viable approach for exhaustive status testing.

## New Helper Specification: `recalculateAchArrangementStatus`

The new helper must mirror the refactored Java listener logic from MR !1368:

```typescript
/**
 * Recalculates ACH payment arrangement status based on linked ACH payment statuses.
 * Mirrors PaymentArrangementACHListener logic from MR !1368.
 *
 * EnumSets:
 *   PENDING  = { PENDING, SENT, ACK_RECEIVED, PICKED_TO_SEND, STATUS_UPDATE_PENDING, PENDING_TO_RERUN }
 *   SUCCESS  = { SETTLED, COMPLETED, SETTLED_IN_RERUN }
 *   isFailure = !PENDING && !SUCCESS (residual logic)
 *
 * Priority: FAILURE > PENDING > SUCCESS
 *
 * State machine:
 *   hasFailure  -> FAILED  (is_active=false, current_rating=null)
 *   hasPending  -> IN_PROGRESS (is_active=true)
 *   all SUCCESS -> SUCCESS (is_active=false)
 *     + SETTLEMENT -> account = SETTLED_IN_FULL
 *     + NORMAL -> account unchanged
 */
async recalculateAchArrangementStatus(arrangementPk: string): Promise<string> {
  const PENDING_STATUSES = ['PENDING', 'SENT', 'ACK_RECEIVED', 'PICKED_TO_SEND', 'STATUS_UPDATE_PENDING', 'PENDING_TO_RERUN'];
  const SUCCESS_STATUSES = ['SETTLED', 'COMPLETED', 'SETTLED_IN_RERUN'];

  const payments = await this.getAchPaymentsByArrangement(arrangementPk);

  const hasFailure = payments.some(p => {
    const status = String(p.status);
    return !PENDING_STATUSES.includes(status) && !SUCCESS_STATUSES.includes(status);
  });
  const hasPending = payments.some(p => PENDING_STATUSES.includes(String(p.status)));

  let newStatus: string;
  let isActive: boolean;

  if (hasFailure) {
    newStatus = 'FAILED';
    isActive = false;
  } else if (hasPending) {
    newStatus = 'IN_PROGRESS';
    isActive = true;
  } else {
    newStatus = 'SUCCESS';
    isActive = false;
  }

  await this.executeUpdate(
    `UPDATE uown_sv_payment_arrangement SET status = $1, is_active = $2 WHERE pk = $3`,
    [newStatus, isActive, arrangementPk],
  );

  // SETTLEMENT + SUCCESS -> account transitions to SETTLED_IN_FULL
  if (newStatus === 'SUCCESS') {
    const arrangement = await this.getPaymentArrangementByPk(arrangementPk);
    if (arrangement && String(arrangement.arrangement_type) === 'SETTLEMENT') {
      await this.executeUpdate(
        `UPDATE uown_sv_account SET account_status = 'SETTLED_IN_FULL' WHERE pk = $1`,
        [arrangement.account_pk],
      );
    }
  }

  // FAILED -> clear current_rating (matches Java: arrangement.setCurrentRating(null))
  if (newStatus === 'FAILED') {
    await this.executeUpdate(
      `UPDATE uown_sv_payment_arrangement SET current_rating = NULL WHERE pk = $1`,
      [arrangementPk],
    );
  }

  return newStatus;
}
```

## Edge Cases

### 1. Account reuse across serial CTs
Each CT needs its own dedicated account. If accounts are reused, a previous arrangement (is_active=true) would block the new one. The `beforeAll` must allocate separate accounts to each CT slot (CT-01 through CT-18 = 18 unique accounts for DB CTs, but CT-06 doesn't modify the account destructively and CT-UI-01..04 reuse accounts from earlier CTs).

**Mitigation**: Allocate 15 accounts in beforeAll. CT-05 uses a separate account from the NORMAL CTs. CT-UI-01..04 reuse CT-01, CT-14, CT-08, CT-05 accounts (read-only UI verification).

### 2. CT-05 is DESTRUCTIVE
Creating a SETTLEMENT arrangement and transitioning it to SUCCESS changes the account to SETTLED_IN_FULL. This account cannot be reused for any future test.

**Mitigation**: CT-05 is placed last in Group 1. Its account is reused only by CT-UI-04 (which reads SETTLED_IN_FULL status).

### 3. CT-06 timing sensitivity
CT-06 verifies that the arrangement remains NOT_STARTED without any sweep. There's a theoretical race condition if a background sweep processes the account between creation and verification.

**Mitigation**: Use a 5-second re-verification window. If the arrangement status changes during this window, it means a background sweep is active in qa2 — flag as environment issue, not test failure.

### 4. ACH installment amounts
The test uses small installment amounts ($100) that may be below minimum thresholds for some account configurations. However, since we're testing the listener/recalculate logic (not actual payment processing), the amount doesn't affect the test outcome — we UPDATE statuses directly in DB.

### 5. `recalculateAchArrangementStatus` vs real listener
The helper simulates the listener by applying the same logic. However, the real listener fires as a `@TransactionalEventListener(phase = AFTER_COMMIT)` triggered by `getStatusDatePaymentsListSweep`. The helper applies the logic synchronously. Any discrepancy between the helper and the real listener would be a test infrastructure issue, not a product bug.

**Mitigation**: Document that DB simulation tests validate the *rules* (PENDING/SUCCESS/FAILURE classification), not the *trigger mechanism* (sweep -> event -> listener). The trigger mechanism is implicitly validated if the MR is deployed and the EnumSets are correct.

### 6. UI session expiry in stg/qa2
Long serial test suites (22 CTs) may cause Servicing Portal session expiry before UI CTs execute. The `PaymentArrangementPage.navigateDirectly` method has built-in session recovery.

**Mitigation**: UI CTs use `navigateDirectly(testEnv.servicingUrl, accountPk)` which handles re-authentication automatically.

### 7. `is_active` not read after arrangement API creation
The `getPaymentArrangement` helper returns the most recent arrangement for an account. If an old inactive arrangement exists, the query might return the wrong one.

**Mitigation**: Use `ORDER BY pk DESC LIMIT 1` in the query (already implemented). Also verify `is_active = true` immediately after creation.
