## Descrição

O `PaymentArrangementACHListener` foi refatorado no MR !1368 para resolver classificação incorreta de statuses ACH intermediários. A lógica anterior de `isPending` e `isFailure` deixava alguns statuses em "zona cinza" (ACK_RECEIVED, PICKED_TO_SEND, PENDING_TO_RERUN, CANCELLED, BLOCKED_ACCOUNT), levando a resultados imprevisíveis na avaliação do arrangement.

A refatoração introduziu dois `EnumSet`s explícitos:

- **PENDING** = { PENDING, SENT, ACK_RECEIVED, PICKED_TO_SEND, STATUS_UPDATE_PENDING, PENDING_TO_RERUN }
- **SUCCESS** = { SETTLED, COMPLETED, SETTLED_IN_RERUN }
- **isFailure** = tudo que NÃO está em PENDING nem em SUCCESS (lógica residual)

Prioridade de avaliação: **FAILURE > PENDING > SUCCESS**

Mudanças comportamentais chave:
- ACK_RECEIVED, PICKED_TO_SEND, PENDING_TO_RERUN: antes "zona cinza" (poderia ser FAILED) → agora IN_PROGRESS
- CANCELLED, BLOCKED_ACCOUNT: antes "zona cinza" → agora FAILED (lógica residual)
- Arrangement só avança para SUCCESS quando **todos** os ACH payments estão no set SUCCESS

## Decisões Técnicas

- **Estratégia de simulação DB em stg:** Mesma da execução em qa2 — UPDATE direto em `uown_sv_achpayment.status` + helper `recalculateAchArrangementStatus`, que espelha a lógica Java do MR !1368 com os EnumSets PENDING e SUCCESS. Justificativa: stg também não dispara Profituity controlável a partir do framework de testes; o helper local replica a lógica do listener.
- **Contas ACTIVE pré-existentes em stg:** 20 contas elegíveis encontradas (≥ 18 exigidas). Faixa: 589041..589074. Nenhuma criação de conta foi necessária.
- **CT-05 destrutivo em stg:** A conta `589056` foi permanentemente transicionada para `SETTLED_IN_FULL`. Não usar essa conta em testes futuros que exijam ACTIVE. (Em 2026-04-27, o destrutivo havia sido em `589067` — `589056` era CT-12 naquele run, agora é CT-05 porque a janela `ORDER BY pk DESC LIMIT 20` deslocou após a baixa de `589067`.)
- **Re-auth de sessão em todos os CT-UI:** Cada um dos 4 CT-UI logou `[PA.navigateDirectly] Session expired in stg — re-authenticating` e completou normalmente.
