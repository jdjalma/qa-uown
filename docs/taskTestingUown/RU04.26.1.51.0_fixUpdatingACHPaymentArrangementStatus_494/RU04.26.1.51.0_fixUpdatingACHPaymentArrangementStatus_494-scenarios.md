## Contexto da Mudança (MR !1368)

O `PaymentArrangementACHListener` foi refatorado. A lógica de `isPending` e `isFailure` agora usa dois `EnumSet`s fixos:

```
PENDING = { PENDING, SENT, ACK_RECEIVED, PICKED_TO_SEND, STATUS_UPDATE_PENDING, PENDING_TO_RERUN }
SUCCESS = { SETTLED, COMPLETED, SETTLED_IN_RERUN }
isFailure = tudo que NÃO está em PENDING nem em SUCCESS
```

O arrangement só avança para `SUCCESS` quando **todos** os ACH payments estão em `SUCCESS`.

## Mecanismo de Sweep

### Como o listener é disparado

O `PaymentArrangementACHListener` é um `@TransactionalEventListener(phase = AFTER_COMMIT)`.  
**Ele SÓ é disparado pelo `getStatusDatePaymentsListSweep`**, via o seguinte fluxo:

```
POST /uown/svc/sendACHPaymentsSweep
    → PENDING → PICKED_TO_SEND (enviado para Profituity)

POST /uown/svc/getSendACHPaymentsStatusSweep  (opcional, busca ACK)
    → PICKED_TO_SEND → ACK_RECEIVED

POST /uown/svc/getStatusDatePaymentsListSweep   ← TRIGGER DO LISTENER
    → Profituity retorna status final
    → ProfituityPaymentStatusParser atualiza uown_sv_achpayment.status
    → Publica ACHPostUpdateEvent(achPaymentPk)  [somente se payment_arrangement_pk != null]
    → PaymentArrangementACHListener.handle() dispara AFTER_COMMIT
    → Recalcula status do arrangement
```

### Sweeps por cenário

| Cenário | Sweeps necessários | Fallback (Profituity inativo) |
|---------|-------------------|-------------------------------|
| SUCCESS: SETTLED | `sendACH` → `getStatusDate` | DB: UPDATE status=SETTLED → `recalculateAchArrangement` |
| SUCCESS: COMPLETED | `sendACH` → `getStatusDate` | DB: UPDATE status=COMPLETED → `recalculateAchArrangement` |
| SUCCESS: SETTLED_IN_RERUN | `rerunACH` → `getStatusDate` | DB: UPDATE status=SETTLED_IN_RERUN → `recalculateAchArrangement` |
| PENDING: PENDING | nenhum (estado inicial) | — |
| PENDING: SENT | `sendACH` | DB: UPDATE status=SENT (arrangement não avança) |
| PENDING: ACK_RECEIVED | `sendACH` → `getSendStatus` | DB: UPDATE status=ACK_RECEIVED |
| PENDING: PICKED_TO_SEND | `sendACH` (parcial) | DB: UPDATE status=PICKED_TO_SEND |
| PENDING: STATUS_UPDATE_PENDING | `getStatusDate` (parcial) | DB: UPDATE status=STATUS_UPDATE_PENDING |
| PENDING: PENDING_TO_RERUN | `getStatusDate` (detecta rerun) | DB: UPDATE status=PENDING_TO_RERUN |
| FAILED: RETURNED | `sendACH` → `getStatusDate` | DB: UPDATE status=RETURNED → `recalculateAchArrangement` |
| FAILED: ERROR | `sendACH` → `getStatusDate` | DB: UPDATE status=ERROR → `recalculateAchArrangement` |
| FAILED: CANCELLED | `disableACHPayment` API | DB: UPDATE status=CANCELLED → `recalculateAchArrangement` |
| FAILED: BLOCKED_ACCOUNT | `sendACH` → `getStatusDate` | DB: UPDATE status=BLOCKED_ACCOUNT → `recalculateAchArrangement` |

> **Nota qa2:** Como não podemos controlar o que o Profituity retorna para cada status específico,
> todos os cenários usarão **fallback DB** com o helper `recalculateAchArrangementStatus` (a ser criado).
> Se Profituity estiver ativo, o CT-SM-ACH-01 usará o fluxo real como smoke test.

## User Stories

### US-ACH-01: Listener reconhece statuses de conclusão como SUCCESS

**Como** sistema de cobrança,  
**Quero** que o arrangement seja marcado como `SUCCESS` quando todos os ACH payments tenham status `SETTLED`, `COMPLETED` ou `SETTLED_IN_RERUN`,  
**Para** garantir que apenas pagamentos efetivamente liquidados concluam o acordo.

**Critérios de Aceite:**
- [ ] Todos SETTLED → arrangement SUCCESS
- [ ] Todos COMPLETED → arrangement SUCCESS
- [ ] Todos SETTLED_IN_RERUN → arrangement SUCCESS
- [ ] Mix dos três → arrangement SUCCESS
- [ ] Arrangement tipo SETTLEMENT → account = SETTLED_IN_FULL
- [ ] Visível no UI do Servicing Portal

---

### US-ACH-02: Listener reconhece statuses de andamento como PENDING

**Como** sistema de cobrança,  
**Quero** que o arrangement permaneça em `IN_PROGRESS` enquanto qualquer ACH payment estiver em status de andamento,  
**Para** não concluir prematuramente um acordo com pagamentos ainda em processamento.

**Critérios de Aceite:**
- [ ] PENDING → arrangement NOT_STARTED (nenhum listener disparado ainda)
- [ ] SENT → arrangement IN_PROGRESS
- [ ] ACK_RECEIVED → arrangement IN_PROGRESS ← **novo no MR**
- [ ] PICKED_TO_SEND → arrangement IN_PROGRESS ← **novo no MR**
- [ ] STATUS_UPDATE_PENDING → arrangement IN_PROGRESS
- [ ] PENDING_TO_RERUN → arrangement IN_PROGRESS ← **novo no MR**

---

### US-ACH-03: Listener reconhece statuses desconhecidos como FAILURE

**Como** sistema de cobrança,  
**Quero** que qualquer status fora dos sets PENDING e SUCCESS seja tratado como falha,  
**Para** garantir que pagamentos cancelados, devolvidos ou com erro não concluam o acordo.

**Critérios de Aceite:**
- [ ] RETURNED → arrangement FAILED
- [ ] ERROR → arrangement FAILED
- [ ] CANCELLED → arrangement FAILED ← **novo comportamento MR**
- [ ] BLOCKED_ACCOUNT → arrangement FAILED ← **novo comportamento MR**
- [ ] Account NÃO muda de status em caso de FAILED

---

### US-ACH-04: Mix de statuses respeita a prioridade de avaliação

**Como** sistema de cobrança,  
**Quero** que a avaliação respeite a prioridade: FAILED > PENDING > SUCCESS,  
**Para** que um único pagamento falhado ou pendente bloqueie o avanço para SUCCESS.

**Critérios de Aceite:**
- [ ] SUCCESS + PENDING → IN_PROGRESS (não avança)
- [ ] SUCCESS + FAILED → FAILED
- [ ] PENDING + FAILED → FAILED

| Comportamento novo | CT(s) que cobrem |
|-------------------|-----------------|

| PICKED_TO_SEND agora é PENDING | CT-09 |

| CANCELLED agora é FAILURE (lógica residual) | CT-14, CT-18 |

| isFailure = residual (tudo fora de PENDING e SUCCESS) | CT-14, CT-15 |
| SUCCESS só quando TODOS em { SETTLED, COMPLETED, SETTLED_IN_RERUN } | CT-01..CT-05 |
