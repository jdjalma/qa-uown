The hybrid E2E+DB pattern (DB query as setup, UI assertion as validation) is the correct approach here.

## DB Queries for Setup

### Find a lead WITH an invoice (CT-02 + CT-05)
```sql
SELECT
  lead.pk AS lead_pk,
  invoice.merchant_invoice_number AS invoice_number
FROM uown_los_lead lead
JOIN uown_los_invoice invoice ON invoice.lead_pk = lead.pk
WHERE invoice.merchant_invoice_number IS NOT NULL
  AND invoice.merchant_invoice_number != ''
ORDER BY lead.row_created_timestamp DESC
LIMIT 1;
```

### Find a lead WITHOUT an invoice (CT-03)
```sql
SELECT lead.pk AS lead_pk
FROM uown_los_lead lead
LEFT JOIN uown_los_invoice invoice ON invoice.lead_pk = lead.pk
WHERE invoice.pk IS NULL
ORDER BY lead.row_created_timestamp DESC
LIMIT 1;
```

## Edge Cases

1. **Lead with multiple invoices**: The SQL uses `LEFT JOIN uown_los_invoice invoice ON invoice.lead_pk = lead.pk` which could produce duplicate rows if a lead has multiple invoices. The backend likely handles this via DISTINCT or CTE deduplication. If duplicates appear in the UI, the test should log a warning but not fail (backend concern, not frontend).

2. **Case-insensitive filter matching**: The backend SQL uses `LOWER(invoice.merchant_invoice_number) LIKE LOWER(:invoiceNumber)`. CT-05 should use the exact case from DB to avoid false negatives. Case-insensitive matching is a backend feature, not explicitly tested in this SPEC (out of scope for the column+filter task).

3. **Partial match via LIKE**: The SQL uses `LIKE` which supports wildcards (`%`). The frontend trims whitespace but does not add wildcards. The backend may or may not wrap the value with `%`. CT-05 uses an exact invoice number value to avoid ambiguity.

4. **Sort indicators in headers**: The Leads table includes sort indicator characters in headers. `getCleanHeaders()` already strips `[triangles/arrows]` -- no additional handling needed.

5. **Origination auth not pre-stored**: The `tests/auth.setup.ts` has origination auth COMMENTED OUT. The test must login inline via `LoginPage`. The `task-testing` project depends on `auth-servicing` (not origination).

6. **Empty table on initial load**: The Leads page loads with today's date range by default. If no leads were created today, the table may be empty. The test uses Lead PK filter (CT-02, CT-03) and Invoice Number filter (CT-05) to search specific leads, so this is not an issue.
