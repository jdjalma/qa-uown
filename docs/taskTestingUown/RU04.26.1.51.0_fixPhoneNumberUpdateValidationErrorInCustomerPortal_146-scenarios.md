# Cenários de Teste — Fix Phone Number Update Validation Error in Customer Portal

**Task:** #146 — UOWN | Customer Portal | Fix phone number update validation error  
**Milestone:** RU04.26.1.51.0  
**Portal:** Website (Customer Portal)  
**Test Name:** `RU04.26.1.51.0_fixPhoneNumberUpdateValidationErrorInCustomerPortal_146`

---

## Contexto do Bug

Ao navegar para **Account Settings > Update Contact Info** e alterar o número de telefone, o frontend
chamava `POST /uown/svc/sendVerificationCode/{newPhone}` ANTES de salvar o telefone no banco.
Como o novo número ainda não estava vinculado a nenhuma conta, o backend retornava HTTP 500:
> "We could not find an active account for customer with phone number [number] in our system."

**Fix (frontend):** o fluxo foi corrigido para salvar o telefone via
`createOrUpdatePrimaryCustomerContactInfo` antes de chamar `sendVerificationCode`.

## User Stories

### US-CP-01: Atualizar número de telefone no Customer Portal

**Como** cliente autenticado no Customer Portal,  
**Quero** atualizar meu número de telefone em Account Settings > Update Contact Info,  
**Para** que meus dados de contato fiquem corretos no sistema.

#### Critérios de Aceite
- [ ] Usuário consegue alterar o número de telefone sem receber HTTP 500
- [ ] Sistema salva o novo telefone via `createOrUpdatePrimaryCustomerContactInfo`
- [ ] Se verificação via SMS é exigida, código é enviado com sucesso ao novo número
- [ ] DB reflete o novo telefone após salvar
- [ ] Mensagem de sucesso é exibida na UI

### US-CP-02: Não bloquear atualização de telefone com erro de validação indevido

**Como** cliente autenticado,  
**Quero** que o sistema não me exiba erro "could not find an active account" ao atualizar telefone,  
**Para** que a atualização de contato seja fluída e sem falsos erros.

#### Critérios de Aceite
- [ ] Nenhum popup de erro HTTP 500 é exibido ao salvar novo número de telefone
- [ ] Request para `sendVerificationCode` (se chamada) retorna 200 — não 500
- [ ] Fluxo completa sem erros inesperados

## Cenários de Teste

### CT-01: Happy path — atualizar telefone via UI, verificar payload/response e persistência no DB (Hybrid)

**Tipo:** Hybrid (E2E + Network Intercept + DB)
**Portal:** Website (Customer Portal)
**Projeto Playwright:** `website-ui`
**Pré-condição:** Conta ativa no Customer Portal com email e telefone cadastrado

**Passos:**
1. Navegar para `WEBSITE_URL`
2. Fazer login com email do cliente (OTP via email)
3. Ativar interceptação de network para `**/createOrUpdatePrimaryCustomerContactInfo` e `**/sendVerificationCode/**`
4. Navegar para Account Settings > Update Contact Info
5. Alterar o número de telefone para um novo número válido (ex: `areaCode: "555"`, `phone: "1230001"`)
6. Clicar em Save Changes
7. **Verificar payload** interceptado de `createOrUpdatePrimaryCustomerContactInfo`:
   - `phoneList[0].areaCode` = área correta
   - `phoneList[0].phoneNumber` = número novo correto
   - `accountPk` = accountPk da conta autenticada
8. **Verificar response** de `createOrUpdatePrimaryCustomerContactInfo`: HTTP 200, `phoneList` contém novo número
9. **Verificar que nenhuma chamada a `sendVerificationCode` retornou HTTP 500**
10. **Verificar DB**: `SELECT area_code, phone_number FROM uown_sv_phone WHERE account_pk = {accountPk}` contém novo número
11. **Verificar UI**: nenhum popup de erro "could not find an active account"; mensagem de sucesso visível

**Resultado esperado:**
- `createOrUpdatePrimaryCustomerContactInfo` → HTTP 200, body com novo telefone
- `sendVerificationCode` (se chamada) → HTTP 200 — nunca 500
- DB: `uown_sv_phone.area_code || phone_number` = novo número para `account_pk`
- UI: sem popup de erro HTTP 500; fluxo completa com sucesso

**Tags:** `@regression @website @account-settings @phone-update @hybrid`

### CT-02: Regressão API — `createOrUpdatePrimaryCustomerContactInfo` aceita novo telefone

**Tipo:** API
**Portal:** SVC
**Pré-condição:** Conta ativa com `accountPk` conhecido e token SVC válido

**Passos:**
1. `GET /uown/svc/getPrimaryCustomerContactInfo/{accountPk}` → obter telefone atual e `phonePk`
2. `POST /uown/svc/createOrUpdatePrimaryCustomerContactInfo` com `phoneList` contendo novo número
3. `GET /uown/svc/getPrimaryCustomerContactInfo/{accountPk}` → verificar que telefone foi atualizado

**Resultado esperado:**
- Step 2: HTTP 200, response contém novo número de telefone
- Step 3: novo `areaCode` + `phoneNumber` presentes na lista de telefones da conta

**Tags:** `@regression @api @phone-update`

### CT-03: Regressão backend — `sendVerificationCode` retorna 200 após telefone ser salvo

**Tipo:** API
**Portal:** SVC / Customer Portal
**Pré-condição:** Conta ativa com `accountPk` conhecido

**Passos:**
1. `POST /uown/svc/createOrUpdatePrimaryCustomerContactInfo` com novo telefone (areaCode + phoneNumber) → HTTP 200
2. Formatar como `{areaCode}{phoneNumber}` (mesmo número recém-salvo)
3. `POST /uown/svc/sendVerificationCode/{formattedPhone}?company=UOWN`

**Resultado esperado:**
- Step 1: HTTP 200, response contém novo telefone
- Step 3: HTTP 200 (void) — número agora está no DB associado à conta → sem 500

**Tags:** `@regression @api @phone-verification`
