### Failure root-cause analysis (6th pass)

**F-003 (S3) — `[class*="customOptionStyles"]` returns 0 visible elements at runtime despite 2 matches in pre-run MCP validation.**

The evidence chain:
- MCP at 13:13 UTC: selector returned 2 elements, filter regex worked, click flipped state.
- Test run at ~14:18 UTC: `row.waitFor({ state: 'visible', timeout: 5_000 })` timed out → 0 matches.
- `error-context.md` snapshot at failure: combobox open, options present in a11y tree, but no DOM class confirmation available from the snapshot alone.

The discrepancy is real and reproducible (this is the failure mode, not a flake — the test failed deterministically). Three explanations are possible, listed by likelihood:

1. **FE build drift between MCP validation and test run.** The CSS-Module hash AND the prefix can change when webpack output module-IDs change. If the prefix is no longer `customOptionStyles` (e.g. shortened to `customOption` or the module was renamed), the selector returns 0. Verification requires reading `document.querySelectorAll('[id^="react-select-"][id*="-option-"]').forEach(el => el.className)` in the live page during a fresh test run.

2. **Portal-mounted options counted as non-visible by Playwright.** React-select sometimes renders options inside a `react-portal` outside the main DOM tree (especially with `menuPortalTarget={document.body}`). Playwright's `state: 'visible'` requires the element to be in viewport AND not have `visibility: hidden` / `display: none` / `opacity: 0` on any ancestor. A portaled option above the viewport top, or hidden by CSS during transitions, would fail visibility despite being in DOM.

3. **MCP validation captured a stale state.** Less likely but possible: the MCP browser session at 13:13 UTC may have been on a different qa1 deploy than the test runner used at 14:18 UTC if any FE redeploy hit qa1 in between.

### Path forward (NOT executed — pipeline closes per orchestrator brief)

Two strategies that bypass the volatile CSS-Module class entirely:

1. **Keyboard contract (recommended):** the widget self-documents its keyboard contract in the a11y live region (snapshot e81). Code:
   ```ts
   await this.openFilters();
   await this.activeCombobox.click();
   await this.activeComboboxOpen.waitFor({ state: 'visible', timeout: 5_000 });
   if (state === 'Inactive') {
     await this.page.keyboard.press('ArrowDown');
   }
   await this.page.keyboard.press('Enter');
   ```
   Zero selectors against option DOM. Works regardless of CSS-Module hash, portal mount, or a11y name.

2. **React-select option ID:** the snapshot shows option rows mount with `id="react-select-N-option-{0|1}"` (line 99 of the page-object docstring confirms this from earlier MCP capture — `react-select-2-option-0` clicked manually flipped state). Code:
   ```ts
   const idx = state === 'Active' ? 0 : 1;
   await this.page.locator(`[id^="react-select-"][id$="-option-${idx}"]`).click();
   ```
   Framework-generated IDs; the N prefix varies but the suffix shape is stable.

Either strategy would close F-003 in a single iteration. Neither was attempted in passes 1–6 because each iteration committed to a different DOM hypothesis without falling back to keyboard or ID selectors.
