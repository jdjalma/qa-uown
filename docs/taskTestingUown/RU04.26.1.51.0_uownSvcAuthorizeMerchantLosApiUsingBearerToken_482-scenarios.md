## Context

The `RequestFilter` in SVC now enforces format rules per `ApiUserType`:

| `apiUserType` | Required `Authorization` format | 401 if sent wrong format? |
|---|---|---|
| `MERCHANT` | `Bearer <key>` | Yes — raw key → 401 |
| `VENDOR` | Raw key (no Bearer) | Yes — Bearer → 401 |
| `null` (legacy) | Both accepted | No |

Affected path: `/uown/los/merchant/applications` (and all sub-paths).  
Auth endpoint: `POST /uown/auth/authorize` → `{ key, expires, error }`.  
`DefaultLosApiVersionRequestWrapper` injects `X-API-Version: 2` when the header is absent.

---

## Prerequisites (DB)

A test user must exist in `uown_api_user` with:
- `api_user_type = 'MERCHANT'`
- Valid `username` and `password`
- A valid (non-expired) API key in `uown_api_key`

Credentials stored in `.env` as:
```
LOS_PARTNER_USERNAME=<merchant_partner_user>
LOS_PARTNER_PASSWORD=<merchant_partner_password>
```

## User Stories

### US-482-01: Merchant partner authenticates with Bearer token

**Como** merchant parceiro (sistema externo),  
**Quero** autenticar nas APIs LOS usando `Authorization: Bearer <token>`,  
**Para** criar e consultar aplicações de financiamento de forma segura.

#### Critérios de Aceite
- [ ] Posso obter um API key via `POST /uown/auth/authorize`
- [ ] Com `Authorization: Bearer <key>` e `X-API-Version: 2`, o `RequestFilter` permite acesso ao endpoint (não retorna 401)
- [ ] Sem o header `Authorization`, recebo 401 "Invalid API KEY"
- [ ] Enviando a raw key sem prefixo `Bearer`, recebo 401 (regra MERCHANT)

---

### US-482-02: API version header é opcional para endpoints do parceiro

**Como** merchant parceiro,  
**Quero** chamar os endpoints LOS sem obrigatoriamente enviar `X-API-Version`,  
**Para** simplificar a integração sem precisar gerenciar versões manualmente.

#### Critérios de Aceite
- [ ] Chamada sem `X-API-Version` é tratada como versão 2 (padrão via `DefaultLosApiVersionRequestWrapper`)
- [ ] Chamada com `X-API-Version: 1` retorna 404 (sem handler v1)

---

### US-482-03: Regressão — outros paths não afetados

**Como** sistema interno (SVC/TMS) ou outro cliente,  
**Quero** que minha autenticação existente continue funcionando,  
**Para** não quebrar integrações já em produção.

#### Critérios de Aceite
- [ ] `POST /uown/svc/*` com API key existente (`svcApiKey`) ainda retorna 200/2xx
- [ ] Paths fora de `/uown/los/merchant/applications` não são afetados pela nova regra Bearer

## Test Scenarios

### CT-01: [Happy Path] Authorize → Bearer token → create application (auth passes)

**Tipo:** API  
**Pré-condição:** User com `api_user_type = MERCHANT` existe no DB com credenciais em `.env`

**Passos:**
1. `POST /uown/auth/authorize` com `{ username, password }` do LOS partner
2. Verificar resposta: `200`, `body.key` não vazio, `body.error` null
3. `POST /uown/los/merchant/applications` com `Authorization: Bearer <key>`, `X-API-Version: 2`, body JSON mínimo
4. Verificar que a resposta **não é 401** (pode ser 400/422 por validação de negócio — o que importa é que a autenticação passou)

**Resultado esperado:**
- Step 2: HTTP 200, `key` presente, `error` null
- Step 4: HTTP ≠ 401 (auth passou; status business esperado: 400 ou 422)

**Tags:** `@regression @critical @smoke`

---

### CT-02: [Happy Path] Bearer token funciona em todos os sub-endpoints (smoke)

**Tipo:** API  
**Pré-condição:** API key válida obtida via CT-01 (ou setup compartilhado)

**Passos (parallel steps usando o mesmo Bearer token):**
1. `POST /uown/los/merchant/applications/search` com body mínimo → não 401
2. `POST /uown/los/merchant/applications/{id}/invoices` com body mínimo → não 401
3. `POST /uown/los/merchant/applications/{id}/settlements` com body mínimo → não 401
4. `POST /uown/los/merchant/applications/{id}/leases` com body mínimo → não 401

**Resultado esperado:** Todos retornam status ≠ 401 (400/422 OK — auth passou)

**Tags:** `@regression @smoke`
