## Descrição

Novo endpoint `PUT /uown/svc/payments/credit-cards/{ccTransactionPk}` para editar ou cancelar transações de cartão de crédito com status PENDING no portal Servicing. O frontend adiciona uma coluna "Edit" na tabela CC Transactions com ícone de lápis para transações PENDING, abrindo um modal "Edit Pending Credit Card Payment" com campos de Posting Date, Amount e Comment, além de botões SAVE (editar) e REMOVE (cancelar).

**Escopo da tarefa:**
- Endpoint PUT para atualizar `amount`, `postingDate`, `comment` e `status` (PENDING ou CANCELLED)
- Modal de edição com campos Posting Date, Amount, Comment
- Botão SAVE (persiste alterações com status=PENDING) e botão REMOVE (persiste com status=CANCELLED)
- Ícone de edição visível apenas para transações PENDING
- Transações CANCELLED exibidas com strikethrough visual
- Activity log registrando cada alteração
- Recálculo de rating ao cancelar todas as transações PENDING

### CT-NEG-04

**Objetivo:** Documentar que amount zero, negativo e null são aceitos pelo backend (bug de validação).

| null | 400 | 200 (aceito) |

> Este comportamento é causado pelo BUG-01 documentado abaixo.

#### Como verificar manualmente

3. PUT com `amount: null` → esperado 400, recebido 200

## Bugs de Aplicação Encontrados

### BUG-01 — Missing @Valid no controller impede validação Bean Validation do DTO

**Status:** OPEN
**Severidade:** High

**Descrição:** O DTO `UpdateCreditCardTransactionDto` possui anotações Bean Validation (`@DecimalMin("0.0", inclusive=false)`, `@NotNull`, `@FutureOrPresent`, `@Size(max=500)`) mas o controller (`CreditCardTransactionController`) não possui a anotação `@Valid` no parâmetro `@RequestBody`. Como resultado, nenhuma validação de campo é aplicada e valores inválidos (amount=0, amount=-5, amount=null, postingDate no passado) são aceitos com HTTP 200.

**Como Reproduzir:**
1. Enviar PUT com `amount: 0` ou `amount: -5` ou `postingDate: '2025-01-01'`
2. Observar resposta HTTP 200 (esperado HTTP 400)
3. Verificar no DB que os valores inválidos foram persistidos

**Evidência:** CT-NEG-04 logs: `Amount=0 → status=200, ok=true`; CT-NEG-05 logs: `Past date response: status=200, ok=true`

**Cenário que detectou:** CT-NEG-04, CT-NEG-05

**Causa provável:** O desenvolvedor adicionou as anotações no DTO mas esqueceu de adicionar `@Valid` no parâmetro do controller. Correção: adicionar `@Valid` antes de `@RequestBody UpdateCreditCardTransactionDto dto` no método do controller.

---

### BUG-02 — Backend retorna 500 em vez de 400 para erros de validação

**Status:** OPEN
**Severidade:** Medium

**Descrição:** Quando campos obrigatórios são null (postingDate=null, status=null) ou violam restrições (comment >500 chars), o backend retorna HTTP 500 com `SvcException` genérica em vez de HTTP 400 com mensagens de validação específicas. Isso ocorre porque sem `@Valid`, os valores null chegam ao service layer e causam `NullPointerException`, e strings longas causam erro de truncação no banco.

**Como Reproduzir:**
1. PUT com `postingDate: null` → retorna HTTP 500
2. PUT com `status: null` → retorna HTTP 500
3. PUT com comment de 501 caracteres → retorna HTTP 500

**Evidência:** CT-NEG-06 logs: `null postingDate → status=500`; CT-NEG-08 logs: `Comment 501 chars → status=500`

**Cenário que detectou:** CT-NEG-06, CT-NEG-08

**Causa provável:** Consequência direta do BUG-01 (sem `@Valid`). Com `@Valid` habilitado, Spring retornaria 400 automaticamente via `MethodArgumentNotValidException` handler. Adicionalmente, o `@ControllerAdvice` pode precisar de handler para mapear essas exceções a 400.

---

### BUG-03 — Re-cancel de transação CANCELLED retorna 500 em vez de 400

**Status:** OPEN
**Severidade:** Low

**Descrição:** Ao tentar cancelar uma transação que já está CANCELLED, o backend rejeita corretamente a operação, porém retorna HTTP 500 em vez de HTTP 400 com mensagem explicativa (ex: "Transaction already cancelled").

**Como Reproduzir:**
1. Cancelar transação PENDING via PUT com `status: 'CANCELLED'` → 200 OK
2. Repetir o mesmo PUT na transação agora CANCELLED
3. Observar resposta HTTP 500 (esperado HTTP 400)

**Evidência:** CT-NEG-09 logs: `Re-cancel response: status=500, ok=false`

**Cenário que detectou:** CT-NEG-09

**Causa provável:** O service verifica que a transação não é PENDING e lança `SvcException`, mas o tratamento de exceções converte para 500 em vez de 400. O método deveria verificar o status atual e retornar `ResponseEntity.badRequest()` com mensagem clara.
