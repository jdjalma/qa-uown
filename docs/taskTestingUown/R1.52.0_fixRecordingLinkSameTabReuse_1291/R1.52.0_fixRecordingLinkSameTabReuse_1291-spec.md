## Context — bug raiz e fix técnico (input do orquestrador)

Origination usa **Sentry Session Replay** durante o preenchimento da aplicação pelo cliente. Ao chegar no fluxo de complete, o front envia `replayId` ao backend (linkado ao `leadPk`) e o salva em `sessionStorage` como `sentUuid` para evitar reenvio.

**Bug original:** ao abrir uma 2ª aplicação na MESMA aba após uma 1ª aplicação incompleta, `sessionStorage.sentUuid` ainda existia → o código fazia `idSent === replayId` e pulava o envio do recording → 2ª lead ficava **sem recording link** na lead page do Origination.

**Fix em MR !1448:**
- `domain/stores/customer.tsx` — `storeRecordingInfo()` agora persiste `shortCode` junto de `sentUuid`.
- `pages/[shortCode]/complete/index.tsx` — condição de envio passou a ser `!isSameId || storedShortCode !== shortCode`. Mudança de `shortCode` força reenvio.
- `pages/appComplete/index.tsx` — cleanup remove `shortCode` e `leadPk` do `sessionStorage` no fim do fluxo.

**Implicação para o teste:** `sessionStorage` é **per-tab**. Reproduzir o bug exige reuso do `BrowserContext` + `page` (mesma aba). Novo contexto ou nova aba não reproduz. Esta é a precondição arquitetural mais importante deste SPEC.

### Assertion strategy — recording link via DB

**Atualizado 2026-05-22:** investigação MCP confirmou que **não existe UI element** para o recording link na lead page do Origination. A asserção primária migra de UI para DB.

**Fonte de verdade:** tabela `uown_lead_recording` em `svc` DB (porta 5445, user `svc_user`).

```sql
SELECT uuid
  FROM uown_lead_recording
 WHERE lead_pk = $1
 ORDER BY pk DESC
 LIMIT 1
```

**Helper sugerido (a criar pelo implementer):**

```ts
// src/helpers/recording.helpers.ts
import type { DbClient } from '@helpers/database.helpers.js';

export async function assertRecordingLinkInDb(
  db: DbClient,
  leadPk: number,
): Promise<string> {
  const row = await db.waitForRecord<{ uuid: string }>(
    'SELECT uuid FROM uown_lead_recording WHERE lead_pk = $1',
    [leadPk],
    { timeoutMs: 10_000, label: `recording for lead ${leadPk}` },
  );
  if (!row?.uuid) {
    throw new Error(`Recording link missing for lead ${leadPk} — uown_lead_recording row absent`);
  }
  return row.uuid;
}
```

**Justificativa de regra #14 mantida:** caminho do bug é UI puro (preenchimento da app, navegação same-tab, sessionStorage). DB é só para post-condition validation porque não há UI rendering. Regra #15 (DOM-first) satisfeita — investigação MCP feita e conclusão é "no DOM".

**Asserção secundária (corrigida 2026-05-22 pós-run):**
- ~~`lead1.uuid !== lead2.uuid` em CT-01/CT-02~~ — INCORRETO. Sentry Session Replay é per-tab; same-tab navigation produz o **mesmo** `replayId` para ambas as leads. Unicidade de uuid NÃO confirma que recordings são distintos — presença de row em `uown_lead_recording` para cada `leadPk` é a asserção correta. Log informativo do uuid pode ser mantido (sem `expect`).
- `lead1.row_created_timestamp < lead2.row_created_timestamp` — order de execução correto (ainda válido).
