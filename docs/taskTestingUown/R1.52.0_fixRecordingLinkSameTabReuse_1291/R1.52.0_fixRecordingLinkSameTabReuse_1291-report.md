### F-001 — análise detalhada (assertion espúria)

**Observado:**
- CT-01: `lead 11702.uuid = 56c0e97ff0494868b2cf3a57cfbdeb3c`; `lead 11703.uuid = 56c0e97ff0494868b2cf3a57cfbdeb3c` (idêntico).
- CT-02: `lead 11704.uuid = 86eb38d772c64b87886a29a3fed54489`; `lead 11705.uuid = 86eb38d772c64b87886a29a3fed54489` (idêntico).
- `sessionStorage` capturado em CT-01 entre apps: `{"sentUuid":"56c0e97ff0494868b2cf3a57cfbdeb3c","shortCode":"6zGOhqH7","leadPk":null}` — i.e. exatamente o estado pré-fix-correto (sentUuid carregado de app1).

**Interpretação:**
- Sentry Session Replay gera um `replayId` por **sessão de browser/aba**, não por aplicação. Quando o mesmo tab navega para `/{shortCode}/complete` de app2, o Sentry SDK continua na **mesma replay session** → mesmo `replayId`.
- O bug pré-fix (MR !1448) era que o FE comparava `sentUuid === replayId` e PULAVA o POST do replayId — então lead2 ficava **sem row** em `uown_lead_recording`.
- O fix mudou a condição para `!isSameId || storedShortCode !== shortCode` — força reenvio quando `shortCode` muda → backend cria a row para lead2 com o mesmo replayId (Sentry session é a mesma).
- **Comportamento esperado pós-fix:** lead1 e lead2 same-tab têm o mesmo `uuid` (mesma replay session apontando para duas leads); o que importa para o AC do Marcos é que **ambas as leads tenham row** em `uown_lead_recording`.

**Conclusão:** AC-4 do Marcos está satisfeito — "link for the recording should be present" significa "row existe", não "row tem uuid distinto". A asserção `expect(uuid1).not.toBe(uuid2)` na linha 268 (CT-01) e linha 350 (CT-02) foi um overshoot interpretativo do implementer no SPEC § "Asserção secundária (opcional, anti-cross-pollination)". O comentário do SPEC chamava isso de "asserção secundária opcional anti-cross-pollination", mas o teste a marcou como `expect` bloqueante.

**Classificação:** `[INCORRETO_NO_TESTE]` — NÃO é bug do produto. Test deve relaxar essa asserção; pode manter um log informativo do diff sem `expect`.

Source: (i) DB query qa1 svc — direta; (ii) sessionStorage snapshot capturado pelo próprio test (`console.log` em stdout, linha 28 do output); (iii) MR !1448 diff descrito no SPEC § "Context"; (iv) Sentry SDK behavior — public docs sobre Session Replay (replay scope = browser session).

### F-002 — env-gap soft (não-bloqueante)

**Observado:** o pre-flight `SELECT EXISTS (SELECT 1 FROM uown_gowsign_template WHERE state = $1 AND (is_active IS NULL OR is_active = true))` em CT-02 e CT-04 falha porque a tabela não existe em qa1 svc DB. O helper engole o erro com `.catch(() => null)` e emite `[OBSERVAÇÃO]` "qa env has no GoSign template for CA".

**Realidade:** a tabela canônica de templates GowSign em qa1 svc DB tem outro nome (provavelmente `uown_signing_template` ou similar — não confirmado nesta execução; `information_schema.tables WHERE table_name ILIKE '%gowsign%'` → 0 rows; `'%esign%'` → `uown_dms_esign_document, uown_esign_document, uown_esign_document_history, uown_esign_event_trigger_log`).

**Implicação:** a "GoSign coverage" claim em CT-02/CT-04 é **não-verificável** com o pre-flight atual. As leads de CA (`state=CA + merchant=TireAgent ONLINE`) provavelmente roteiam para GOWSIGN conforme regra de routing documentada (`testing.md` § "E-sign Provider Routing"), mas isso não foi confirmado no run.

**Classificação:** `[OBSERVAÇÃO]` — env-gap soft do test infrastructure (nome de tabela errado no helper). Não-bloqueante para a regressão do recording link (vendor-agnóstico). Capturar para improvement.

Source: (i) DB introspection `information_schema.tables` qa1 svc; (ii) test stdout (annotation `[OBSERVAÇÃO]` emitida); (iii) `tests/e2e/origination/R1.52.0_fixRecordingLinkSameTabReuse_1291.spec.ts:296-303`.

### F-003 — test gap em CT-05 (`/{shortCode}/complete` tem program selector antes do CC form)

**Observado:** Quando o test navega para `/{shortCode}/complete` e chama `contract.fillCreditCardInfo({...})`, o seletor `#ccFirstName, input[placeholder*="First Name"]` dá timeout 15s. Snapshot anexado (`error-context.md`) mostra que a página atual é "Choose the payment program that works best for you" com cards `Weekly` / `Bi-Weekly` / botões "Choose Payment Program".

**Conclusão:** test gap claro. Não é bug do produto — a página de program selection é parte normal do fluxo customer-facing. Fix do test: adicionar `await contract.selectPlanFromCompletePage('Bi-Weekly')` antes do `fillCreditCardInfo`.

**Classificação:** `[CONFIRMADO]` test-gap. Re-implementação para CT-05 necessária.
