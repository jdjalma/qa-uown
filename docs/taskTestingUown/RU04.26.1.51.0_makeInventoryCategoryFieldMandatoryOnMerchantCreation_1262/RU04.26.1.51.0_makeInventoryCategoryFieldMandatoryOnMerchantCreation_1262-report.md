## Descrição

A tarefa #1262 torna o campo **Inventory Category** obrigatório no formulário de criação de Merchant no portal Origination. Os critérios de aceite são:

1. O campo Inventory Category deve ser visualmente obrigatório no formulário de criação (indicador de asterisco).
2. Deve haver validação de UI — o campo deve estar visualmente marcado como obrigatório e validado antes de salvar.
3. A ação de salvar deve ser bloqueada quando Inventory Category estiver vazio; o campo deve ser destacado.
4. Remover o toast genérico de erro `"Unable to create a new inventory category"` (confuso para o usuário) e substituir por mensagem inline `"Inventory Category is Required"`.

## Bugs de Aplicação Encontrados

### BUG-03 — Clear visual do Inventory Category não bloqueia save; Yup required ignora valor pré-existente em Formik state

**Status:** OPEN
**Severidade:** High

**Descrição:** Quando o usuário limpa o campo Inventory Category em um merchant (via clone ou edição de existente), o UI mostra o campo vazio, mas o estado Formik retém o valor anterior. Ao clicar SAVE:
- NENHUMA mensagem inline `"Inventory Category is Required"` aparece.
- O `POST /uown/createOrUpdateMerchant` é disparado com `inventoryCategory: "<valor anterior>"` (ex.: `"OTHER"`).
- O backend aceita e persiste o valor no DB (merchant criado/atualizado com a categoria que o usuário achou ter limpado).
- A `Yup.mixed().nullable().required('Inventory Category is Required')` NÃO dispara porque `formik.values.inventoryCategory` nunca é null — o clear afeta apenas a apresentação visual do react-select.

**Como Reproduzir (confirmado manualmente por QA em qa2):**
1. Acessar `https://origination-qa2.uownleasing.com/merchant/new`.
2. Clonar o merchant `OL90294-0001` (Progress Mobility Acquisition LLC).
3. Ajustar `refMerchantCode` para um valor único (ex.: `OL90294-0001_clone_clone`).
4. Abrir o dropdown Inventory Category e limpar o valor visualmente.
5. Clicar SAVE.
6. **Observar:** sem inline error, merchant é criado com sucesso e o DB reflete `inventoryCategory = 'OTHER'` (valor residual do clone).

O mesmo comportamento foi confirmado na edição (merchant pk=1564, `refMerchantCode = OL90294-0001_clone_clone`): limpar o Inventory Category e salvar atualiza `rowUpdatedTimestamp` sem inline error, e o campo permanece com o valor anterior no DB.

**Evidência:**
- Request create: `POST /uown/createOrUpdateMerchant` com payload `"inventoryCategory": "OTHER"` → response 200, `merchantPK: 1564`, `isNew: true`, `rowCreatedTimestamp: 2026-04-20T21:10:02.601125483`.
- Request edit: `POST /uown/createOrUpdateMerchant` com payload `"inventoryCategory": "OTHER"` (mesmo após clear visual) → response 200, `merchantPK: 1564`, `isNew: false`, `rowUpdatedTimestamp: 2026-04-20T21:14:01.564244029`.
- Testes automatizados replicam o cenário e estão marcados como `test.fixme`: CT-04 (`@clone @bug`) e CT-06 (`@edit @bug`) — ambos asserting o comportamento CORRETO esperado (save bloqueado com inline error, sem POST, sem mutação no DB). Quando a task #1262 for considerada cumprida, esses testes devem ser removidos de `test.fixme` e passarão a rodar normalmente.

**Cenários que detectaram:** CT-04 (clone), CT-06 (edit).

**Causa provável:** o react-select do Inventory Category é renderizado sem `isClearable` (não há clear indicator DOM) e o custom `handleKeyDown` em `merchant-helper.tsx` intercepta `Backspace`/`Delete` para lógica de criação de categoria — portanto nenhum caminho de UI dispara `onChange(null)` para atualizar o Formik state. O Yup `required` sempre vê o valor anterior.

**Impacto:** o critério de aceite #1262 "save bloqueado quando Inventory Category vazio" está quebrado para os fluxos de clone e edit — um usuário pode salvar um merchant/alteração acreditando que o campo está vazio, mas o sistema persiste o valor residual, mascarando possíveis problemas de categorização e violando a regra de auditoria (o usuário crê que modificou o campo mas nada mudou).

### BUG-01 — React-select Inventory Category sem `isClearable` (causa raiz de BUG-03)

**Status:** OPEN
**Severidade:** Low

**Descrição:** O react-select do campo Inventory Category é renderizado sem a propriedade `isClearable` (o DOM não expõe ícone "X" de clear) e o `handleKeyDown` customizado em `merchant-helper.tsx` intercepta as teclas `Backspace`/`Delete` para a lógica de creatable options. Como consequência, não existe caminho de UI que dispare `onChange(null)` para o Formik — tanto no fluxo de create (clone) quanto em edit, o clear executado pelo usuário é apenas visual e não atualiza o estado do formulário. Este bug é classificado como Low isoladamente (UX quirk), mas atua como **causa raiz do BUG-03**: ao combinar o clear puramente visual com o Yup `required` observando apenas o Formik state, o sistema aceita e persiste valores residuais que o usuário acredita ter removido.

**Como Reproduzir:**
1. Autenticar em `https://origination-qa2.uownleasing.com` e abrir o formulário de create (`/merchant/new`) ou a edição de qualquer merchant com Inventory Category pré-populado.
2. Tentar limpar o valor pelo botão X do react-select (inexistente) e pelas teclas Backspace/Delete (capturadas pelo `handleKeyDown` customizado).
3. Observar no DOM/DevTools que o estado do react-select e do Formik permanece com o valor anterior apesar da aparência visual.

**Evidência:** Inspeção do componente react-select em `merchant-helper.tsx` (sem `isClearable`), `handleKeyDown` customizado interceptando Backspace/Delete, e DevTools mostrando `formik.values.inventoryCategory` inalterado após o clear visual.

**Cenário que detectou:** CT-04, CT-06 (consequência observada via BUG-03).

**Causa provável:** decisão de implementação para suportar creatable options bloqueia o fallback nativo do react-select de limpar valor com Backspace, e a ausência de `isClearable` remove o único outro caminho de UI para nulificar o campo.

**Relação:** isoladamente é um UX quirk (Low); em conjunto com o Yup `required` observando apenas o Formik state, viabiliza o BUG-03 (High).
