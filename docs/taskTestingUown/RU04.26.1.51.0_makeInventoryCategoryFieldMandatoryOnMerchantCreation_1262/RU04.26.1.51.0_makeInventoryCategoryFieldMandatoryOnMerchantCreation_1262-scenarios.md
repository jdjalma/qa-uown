## User Stories

### US-1262-01: Campo Inventory Category obrigatório na criação de Merchant

**Como** Admin da Origination,
**Quero** que o campo Inventory Category seja visualmente obrigatório e validado antes do save,
**Para** evitar submeter o formulário sem esse valor e receber um erro genérico ("Unable to create a new inventory category") do backend.

#### Critérios de Aceite
- [ ] O campo "Inventory Category" exibe indicador visual de obrigatório (asterisco) na tela de criação.
- [ ] Ao tentar salvar com o campo vazio, o save é bloqueado no client-side.
- [ ] Mensagem inline `"Inventory Category is Required"` aparece abaixo do campo.
- [ ] A mensagem genérica `"Unable to create a new inventory category"` NÃO é exibida.

### US-1262-02: Inventory Category preservado ao Clonar merchant

**Como** Admin da Origination,
**Quero** que ao clonar um merchant o campo Inventory Category do merchant origem seja copiado para o novo merchant,
**Para** não precisar redigitar esse valor e evitar o bloqueio de save.

#### Critérios de Aceite
- [ ] Após selecionar um merchant na lista de Clone, o campo Inventory Category da nova cópia é pré-populado com o valor do merchant origem.
- [ ] Save da cópia (sem alterar o Inventory Category) conclui com sucesso e cria um novo merchant.
- [ ] Se o usuário limpar o Inventory Category após clonar, a mesma regra de US-1262-01 se aplica (save bloqueado, mensagem inline).

### US-1262-03: Campo Inventory Category obrigatório na edição de Merchant existente

**Como** Admin da Origination,
**Quero** que na edição de um merchant existente o Inventory Category também seja tratado como obrigatório,
**Para** manter a consistência da regra entre criação e edição.

#### Critérios de Aceite
- [ ] Ao abrir `/merchant/{refMerchantCode}`, o campo Inventory Category está visualmente marcado como obrigatório (asterisco).
- [ ] Se o usuário limpar o valor e clicar SAVE, o save é bloqueado com `"Inventory Category is Required"` inline.
- [ ] Se o usuário definir um valor válido e clicar SAVE, o update conclui com sucesso (toast `"Merchant updated successfully"`).

## Estratégia de validação (UI + API + DB + Log via UI)

Cada CT segue o padrão triplo do framework:
- **UI**: toasts, mensagens inline, estado do formulário, navegação.
- **API**: interceptar `POST /uown/createMerchant` / `POST /uown/updateMerchant` — em cenários de sucesso valida payload e status 200; em cenários de bloqueio confirma que o request NÃO é disparado.
- **DB**: `SELECT` em `merchant` — em sucesso valida que `inventory_category`, `updated_at`/`created_at` e `updated_by`/`created_by` refletem a mutação; em bloqueio valida que baseline permanece inalterada.
- **Log (via UI)**: navegar para `/merchantModificationHistory` e filtrar pelo merchant — confirmar presença (sucesso) ou ausência (bloqueio) de linha correspondente à tentativa.
