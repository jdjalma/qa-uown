## 1. Business rule under test

When an application is approved, the customer receives an approval email. The template used depends on the **merchant**, not just the brand:

| Merchant group | Template file | Resolution |
|----------------|---------------|------------|
| **EZPawn locations** (e.g., `TF10078-0001`) | `correspondence/templates/approval-email-ezpawn.html` (539 lines — V2 marketing design) | `templateService.getCurrentTemplateByNameAndMerchant("ApprovalEmail", refMerchantCode)` returns the EZPawn-bound row |
| **All other UOWN merchants** | `correspondence/templates/approval-email-general.html` (153 lines) | Same call returns the generic row (no merchant binding) |
| **All other Kornerstone merchants** | `correspondence/templates/kornerstone/approval-email-general.html` | Same call with `templateName = "KORNERSTONE_ApprovalEmail"` |

**Backend implementation (svc):**
1. `ApprovalEmailService.sendApprovalEmail(lead)` sets `templateName = "ApprovalEmail"`.
2. `CorrespondenceService.createCorrespondence(...)`:
   - Prepends `KORNERSTONE_` when the account/merchant is Kornerstone brand.
   - Calls `templateService.getCurrentTemplateByNameAndMerchant(templateName, merchant.getMerchantInfo().getRefMerchantCode())`.
3. DB column `uown_template.merchant_ids` maps merchant codes to template rows (migration on feature branch `R1.51.0_ezpawn-approval-email`; not yet merged into `R1.51.0` as of 2026-04-23).

**Deployment status caveat:** feature branch `R1.51.0_ezpawn-approval-email` is NOT yet merged into `R1.51.0` — the `getCurrentTemplateByNameAndMerchant` path + `merchant_ids` schema are pending. The sandbox build may still route all approval emails through the generic template. If the positive scenario (CT-01) observes the generic template and not the EZPawn one, classify as `[ENV-GAP]` (feature not deployed) — not a bug. Re-run once sandbox picks up the merged branch.

## 2. Template differentiators (to assert in the email body)

| Marker (case-insensitive, text-only) | Only in EZPawn | Only in General | Shared |
|----|:---:|:---:|:---:|
| `You've got this` | ✅ | ❌ | — |
| `don't miss` | ❌ | ✅ | — |
| `approved lease amount` | — | — | ✅ |
| `lease-to-own` / `spending limit` / `Start Shopping` | — | — | ✅ |

- Body size is also indicative (`ezpawn` ≈ 539 lines / `general` ≈ 153 lines) but we prefer text markers since rendering/IMAP may re-wrap HTML.
- Subject line is identical across templates (`You're Approved — Start Shopping with Uown!`), so subject alone CANNOT distinguish templates.

## 3. User Stories

### US-TMPL-01 — As a customer of an EZPawn merchant
**Como** cliente que envia uma aplicação em um merchant EZPawn (`TF10078-0001`),
**Quero** receber um email de aprovação com o template dedicado EZPawn (`approval-email-ezpawn.html`),
**Para** que a comunicação reflita o branding e copy específicos definidos pelo marketing para EZPawn.

#### Critérios de Aceite
- [ ] A aplicação é aprovada (estado `UW_APPROVED`).
- [ ] O email recebido na caixa do `applicant.email` carrega o template EZPawn (contém `You've got this`; NÃO contém `don't miss`).
- [ ] Todas as imagens `src=...storage.googleapis.com/uown/...` retornam HTTP 200.
- [ ] Subject line contém `approval`/`approved` (assertion case-insensitive).

### US-TMPL-02 — As a customer of any non-EZPawn UOWN merchant
**Como** cliente que envia uma aplicação em um merchant UOWN não-EZPawn (ex.: TireAgent, TerraceFinance `OL90202-0001`),
**Quero** continuar recebendo o email de aprovação padrão,
**Para** que a introdução do template EZPawn não quebre o fluxo existente para outros merchants.

#### Critérios de Aceite
- [ ] Email recebido contém os marcadores do `approval-email-general.html` (contém `don't miss`; NÃO contém `You've got this`).
- [ ] Brand footer/portal URL continua UOWN (não Kornerstone).

### US-TMPL-03 — As a customer of a Kornerstone merchant (regression)
**Como** cliente de um merchant Kornerstone não-EZPawn,
**Quero** continuar recebendo o email Kornerstone padrão,
**Para** garantir que o roteamento de template por merchant não regrediu o branding Kornerstone.

#### Critérios de Aceite
- [ ] Email recebido carrega assinatura Kornerstone (`CustomerService@kornerstoneliving.com` ou `CS@kornerstoneliving.com`, footer Kornerstone).
- [ ] Body NÃO contém `You've got this` (não é ezpawn).

> **Justificativa de modalidade (inviolável — `.claude/context/shared/ssn-test-catalog.md`):** a feature depende do **merchant selection**, não de modalidade de programa. A emissão do approval email ocorre identicamente para qualquer aplicação `UW_APPROVED` (13m ou 13m+16m). Cobriremos apenas Modalidade A (13m) usando `generateTestSSN(true)` para minimizar tempo de execução. Modalidade B (13m+16m Kornerstone) é exercitada implicitamente em CT-03 (Kornerstone regression). Modalidade C (Second Look) e D (Denied) não disparam approval email e ficam fora do escopo.
