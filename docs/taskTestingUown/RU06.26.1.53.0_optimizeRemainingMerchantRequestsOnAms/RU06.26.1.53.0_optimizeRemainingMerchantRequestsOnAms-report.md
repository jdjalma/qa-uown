# Optimize Remaining Merchant Requests on AMS — stg execution report

## 1. Scope

This work item (frontend MR `!171` + backend `svc!1446`, both **merged**) removes the remaining `getAllMerchants` calls from the AMS **user list** and **user details** pages, replacing merchant selection with a **server-paginated component** and turning the user-details merchant card into a **count + inline (non-modal) editor**.

Re-validated in **stg** the 10 test cases that QA-passed in **qa2** (TC-01…TC-04, TC-07…TC-12; TC-05/TC-06 not in scope).

## 2. Method (rules #15 / #18 — UI-first)

- DevTools network capture per page load / interaction (`browser_network_requests`).
- Raw API response + request body inspection (`browser_network_request`) and JWT decode for the 403 diagnosis.
- Activity-log validation for every write (rule #13).

## 3. Core ticket goal — VERIFIED

`getAllMerchants` / `getAllAvailableMerchants` is **never called** on the user-list load, the create-user modal open, or the user-details load. Source-tagged from live network capture. Endpoints used instead:

| Context | Request observed |
|---------|------------------|
| Users list load | `GET /user?search=&page=0&size=10` (server-paginated, no merchant prefetch) |
| Create-user modal open | only `/renew` + `/role` — **no merchant call** |
| User-details load | `/user/{u}`, `/user/{u}/group`, `/role`, `/permission`, `/user/getLogs/...` — **no merchant call** |
| Merchant inline editor | `GET /uown/merchants?page=0&size=10&search=&isActive=true&merchantRefCodes=<assigned>` (paginated) |
| Users-list merchant filter | `GET /user?...&merchantCodes=%<code>%` |

## 5. Findings & Observations (conservative — rule #10)

### 5.1 [DIVERGENCE — needs team confirmation] Create-user modal has no merchant selector in stg
stg's create-user modal is identity-only (no merchant **and** no role selector), while qa2's TC-01 exercised a paginated merchant selector **inside** that modal. Yet the rest of the task **is** deployed on stg (count-only card, inline editor, no `getAllMerchants`). Most likely the stg AMS frontend build lags qa2 on the create-modal portion of MR `!171`, or the create flow there assigns merchants post-creation only. **Not classified as a defect** — recommend the team confirm whether the in-modal merchant selector is expected on stg or stg is behind qa2.

### 5.2 [ENV BLOCKER on writes — root-caused, not a task defect] Token renewal yields empty-permissions token → 403 + force-logout
The first write (`PUT`/`POST`) issued after a mid-session `/renew` returned **403 `{"unauthorized":true}`** and the app force-logged-out. Root cause (JWT decode): the **renewed** token carried `data.permissions = {}` (empty) with blank `authorization`, whereas a **fresh login** token carries full permissions (`restricted.modify.edit_merchant_panel:true`, PUT/POST/DELETE `true`). **Workaround used:** re-login immediately before each write — every write then returned **200**. This is an AMS auth/session issue, **independent of this task**, but it intermittently blocks merchant saves and user creation. **Recommend flagging to the AMS frontend/auth team** — long-lived real sessions could hit the same logout on save. (Implicit-requirement note per rule #11.)

### 5.3 [OBSERVATION] Multi-code filter separator is comma, not colon
The test instruction says multiple codes "should still work using colon". Empirically **comma works** (each code wrapped `%code%`, union returned) and **colon does not** (whole string wrapped as one literal pattern → 0 results). Confirm the intended delimiter / instruction wording.

### 5.4 [OBSERVATION] Search matches non-displayed fields
`search=Mehran` returned `Versailles Furniture / KS9199` (raw API confirmed), whose visible columns contain no "Mehran" — likely an indexed field not shown (owner/contact). Pre-existing search-relevance behavior, **unrelated to this task**.

### 5.5 [OBSERVATION — security, pre-existing] Password in URL query on user creation
`POST /user?password=<plaintext>` sends the new user's password as a **URL query parameter** (risk of exposure in access logs). Pre-existing AMS behavior, **unrelated to this task** — worth a separate ticket.

### 5.6 [OBSERVATION] Persistent console error
Every AMS page logs one `[mobx] uncaught error in 'Reaction'`. Present on initial load before any action — pre-existing baseline, **not** triggered by the task flows.
