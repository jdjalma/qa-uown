## Test Strategy

- **Approach:** **Hybrid, UI-first (rule #14).** Setup via API (`createPreQualifiedApplication` → `sendInvoice` → `getMissingFields` → `submitApplication`); exercise + primary assertion via the **rendered GowSign iframe / signed PDF** in the browser; DB assertions as cross-cutting complement (routing selection, activity log, dispatch variables).
- **Why not API-only:** the deliverable is rendered contract *content* (clauses, tokens, page order). The Daniel's Jewelers CA bug (empty PDF placeholders, 2026-05-06) is the canonical proof that reading backend logs does NOT detect a rendering regression. Rule #14 is mandatory here.
- **Level split per scenario:**
  - Setup: API (fast, deterministic). Merchant preflight (rule #12) runs for the *created* lead's merchant — do NOT auto-heal an out-of-scope merchant.
  - Exercise: browser (drive `/complete` → Terms → GowSign iframe).
  - Assertion: iframe/PDF text (MUST/MUST-NOT) + DB (`uown_esign_document`, `uown_los_lead_notes`, dispatch variables).
- **Environments:** primary **qa2** (more stable; Kornerstone-16m route confirmed there); **stg** for DoD (the milestone `RU07.26.1.54.0` deploy target — confirm MA templates are deployed there). Avoid qa1 for `sendApplication` if the DV360 outage is active. **First action before implementation: `getGowSignTemplatesForState(db,'MA')` in the target env** to confirm the templates were deployed.
- **Suites to activate (per `gowsign-knowledge` / `regression-suites-map`):** `gowsign-contract-content*` + `gowsign-signature-fields` for the new template; **SignWell regression is MANDATORY** (CT-09) — validate the fallback + MA clause parity, do not validate GoSign alone.
- **Test data:** fresh per run (rule #9) — fresh SSN, realistic applicant (`buildTestData({ realistic: true })`), fresh MA customer. No DB mutation to force state.
- **Reuse existing artifacts (rule #2 — do NOT recreate):** `createPreQualifiedApplication`, `sendInvoice`, `getMissingFields`, `submitApplication`, `MissingDataFormPage`, `ContractCompletePage`, `TermsOfAgreementPage`, `AlternativeContractModalPage.getGowSignFrame`, `signGowSignInFrame` (`src/helpers/gowsign-signing.helper.ts`), `getGowSignTemplatesForState` + `assertSelectedTemplateForLead` (`src/helpers/gowsign-template-db.helpers.ts` / `esign-db.helpers.ts`), `waitForLeadStatus`. Iframe detection by class `alternative-contract-vendor_iframe__nSb3A` or substring `gowsign` — never `gowsign.com` (host varies by env).

### ⚠️ Automation feasibility gate (S1 — escalate, mirrors IL #576)

The IL/AL siblings proved that the only route to a **16-month** lease is a **Kornerstone ONLINE** merchant whose `/complete` page is behind the **NeuroID** anti-bot gate. Both automated entry paths are blocked: API `submitApplication` returns `"Failed to verify identification."` (no esign doc, lead stuck UW_APPROVED) and browser auto-advance fails. The only proven 16m render path is **MCP-manual** (fill CC → reload → click "Sign Contract"). **If MA's only 16m route is Kornerstone**, then CT-02..CT-06/CT-08 for 16m are validatable **only by MCP-manual render**, not as CI specs — an architecture blocker to escalate to the user, exactly as done for #576. What IS automatable regardless: **CT-01's 13m negative control** and **CT-07's sanity reference** via a UOWN-gateway MA 13m lease (`MA_2025_SAC`, no NeuroID), and **CT-09's SignWell fallback**. **Discovery must resolve whether a UOWN-gateway route to a MA 16m lease exists (Route B: a merchant with a 16m-only program serving MA)** before committing to a CI implementation for the 16m content.

## Test Design Techniques applied

- **Decision table (routing) — CT-01:** state MA × term {13, 16} → {`MA_2025_SAC`, `MA_2025_SAC_16_MONTHS`}, both GOWSIGN. Two columns, two distinct outcomes → both are test cases (16m = positive under test, 13m = negative control that also proves term-specific selection).
- **Equivalence partitioning + Error Guessing (content) — CT-02..CT-06:** each clause is a MUST partition (correct wording present) + a MUST-NOT partition (donor-state wording absent). Error-guessing seeds from siblings: `nextPaymentDueAmount` blank (AL), the "16-\nmonth" line break in the appendix (OH), header-token vs literal state name (OH: "AGREEMENT-OH" present, literal "Ohio" absent → for MA anchor on the header token, not the word "Massachusetts"), float repr in money tokens (`toBeCloseTo`).
- **State transition — CT-08:** UW_APPROVED → CC_AUTH_PASSED → CONTRACT_CREATED → SIGNED; esign doc SENT_TO_CUSTOMER → SIGNED → STORED; each transition emits an activity-log row (rule #13).
- **Comparison/diff — CT-07, CT-09:** 16m template vs standard MA SAC (state-law clauses), and MA GowSign vs prior MA SignWell (migration parity).

- **Steps:**

  - DB: `assertSelectedTemplateForLead(db, leadPk13, 'MA_2025_SAC')`, client=GOWSIGN.
  - Oracle: CT-01.
- **Edge cases covered:** term-specific selection (16m must NOT pick the 13m template and vice versa).
- **Pitfalls considered:** esign doc only exists at CONTRACT_CREATED (not UW_APPROVED); matrix row MA still says SIGNWELL (stale — do not assert against the matrix, assert against live DB template availability).

### Scenario 2 — Item 4 / Item 4a / token render of the MA 16m contract (P0)
- **Technique:** Equivalence partitioning (MUST/MUST-NOT) + Error Guessing.
- **Persona:** Customer reading the contract in the GowSign iframe on the signing page; agent auditing the signed PDF.
- **Setup:** the MA 16m lead from Scenario 1 rendered in the GowSign iframe (MCP-manual if NeuroID-gated).
- **Steps:**
  1. Capture the iframe body / PDF flatten text; normalize whitespace.
- **Validations (UI-first, rendered content):**
  - CT-02 MUST: heading exactly "Promotional-Payoff Option:"; exercise-date daily-accrual phrase ($ costPriceWithFeeNoTax + tax − on-time payments + accrued daily lease fees through the exercise date); late payment voids.
  - CT-02 MUST-NOT: no "During the first … days", no "This option expires on", no "Cash Price, [epoFeeText]", no "You have other purchase options described below", no `{{epoDays}}`/`{{epoExpiryDate}}` (populated or blank).
  - CT-03 MUST: Item 4a exercise-date EPO; tail `[totalNumberOfPayments]` payments of `$ [nextPaymentDueAmount]` (plus tax); Total `$ [contractAmount]` (including tax)(including the application fee); "Taxes are subject to changes in the applicable tax rate."; contains "Total Cost," + "you will own the Property"; `nextPaymentDueAmount` **non-empty**.
  - CT-03 MUST-NOT: no `[payOffDiscountPercent]%` discount (the standard MA SAC formula).
  - CT-04: every monetary token non-empty and matching `uown_esign_document.request.document.variables` (`toBeCloseTo`); zero `{{ }}`; logo present; no `[DocumentDispatchService][GowSign] … variables map missing …` note for this lead.
- **Edge cases covered:** blank `nextPaymentDueAmount` (AL regression); float repr; hyphenated-heading-with-exercise-date combination (MA-specific — freeze exact string from real render).
- **Pitfalls considered:** iframe host varies by env (detect by class/substring, not `gowsign.com`); post-signature `request` field becomes the string `getDocumentStatus` (query dispatch variables pre-signature or by lead+client).

### Scenario 3 — Lease-purchase-plan footnote + EARLY PURCHASE OPTION appendix + page order (P0)
- **Technique:** Equivalence partitioning (MUST/MUST-NOT) + page-order check.
- **Persona:** Customer reading the plan footnote and the appendix page.
- **Setup:** same rendered MA 16m contract.
- **Validations:**
  - CT-05 MUST: "Traditional [N] Month Lease Purchase Plan*", "*with a Promotional Payoff option", "Early Purchase Option Available Off Unpaid Balance", populated `[leasePurchasePlan]` grid, footnotes "* These amounts include sales tax…" and "** Total Cost includes sales tax, application fee and delivery fee.".
  - CT-05 MUST-NOT: no EPO chart `[table|earlyPurchaseOption]`, no numbered Options 1 & 2.
  - CT-06 MUST: singular "EARLY PURCHASE OPTION"; "This Agreement has a 16-month term for ownership."; "you may purchase the Property at any time"; "For a 16-\n?month lease, the EPO price is calculated as:" (watch the line break); the 5 bullets (cost of leased goods; plus taxes; plus applicable fees; plus accrued daily lease fees through exercise date; minus payments made excluding taxes/fees); "To exercise this option, you must call us at [phone]".
  - CT-06 MUST-NOT: no plural "OPTIONS", no Option 1/2 structure, no "Purchase the Property by [epoExpiryDate]", no sub-bullets a–e, no customer-initials line on this page.
  - CT-06 page order: **Item 14 (ACH Payments) follows immediately** after the appendix, page not duplicated.
- **Edge cases covered:** "16-\nmonth" line break defeats an exact "16-month" match (OH precedent); duplicated appendix page.
- **Pitfalls considered:** singular vs plural is the migration discriminator (Options page is the OLD standard-SAC structure).

### Scenario 4 — State-law sanity: §1–§14 vs standard MA SAC (P1)
- **Technique:** Comparison/diff (16m derivative vs standard MA SAC).
- **Persona:** Compliance/agent auditing state-specific clauses.
- **Setup:** rendered MA 16m contract + a standard MA SAC reference (MA 13m render).
- **Validations (CT-07):** §1 "(*) Total delivery fee"; §3 "Regular lease rate **is** $ …" (MA keeps "is"); §6(a–c,g) $15 declined / $5 late per frequency / $10 in-home / $5 ACH changes; §7 reinstatement 7 days after renewal / 6 months after return (MA-specific — **NOT** SC's 60 days); §8 third-party warranty + "You are **leasing**…"; §9(c) end-of-term liability = None; §13 "Damage to Other Property"; pre-auth vendor-name data-gathering disclosure; ACH uses `nextPaymentDueAmountWithTax`, returns = $15.00; §14 CANCELLATION "last payment **due date**". MUST-NOT (16m-exclusive leakage): standard §4 promo, standard §4a %-discount, Options 1/2 appendix with initials. Any divergence = **[finding]** (does not pass silently).
- **Edge cases covered:** wrong reinstatement window dragged from a donor state (SC/TN); §3 "is" omission.
- **Pitfalls considered:** MA uses a %-discount EPO on the *standard* SAC but the *16m* template must use daily-accrual — the discount formula appearing in 16m is the primary migration bug.

### Scenario 5 — Signature completion + activity log (P1)
- **Technique:** State transition + activity-log validation (rule #13).
- **Persona:** Customer completing the GowSign ceremony; agent seeing SIGNED.
- **Setup:** the MA 16m lead at the rendered contract (MCP-manual if NeuroID-gated).
- **Steps:**

- **Validations (CT-08):**

  - DB: `uown_esign_document.status` progresses SENT_TO_CUSTOMER → SIGNED, `doc_signed_time_stamp` non-null (query by lead+client; `request` may become `getDocumentStatus`).
  - Activity log (rule #13): `uown_los_lead_notes` contains "[ContractService][isLeaseOrLeaseModSigned]…SIGNED", "[EsignRedirectService][updateSignStatus]…to SIGNED", "[ESIGNSERVICE][parseCCPeekConsent]".
- **Pitfalls considered:** Buddy widget can swallow the "Proceed" click; NeuroID reload trick required on Kornerstone; log may be async (use `waitForRecord` with backoff).
