## Descrição

Quando o cliente retorna ao portal de signing após um primeiro submit (CC ou bank já salvos),
um retry/continue NÃO deve disparar uma NOVA chamada de verificação NeuroID no backend. A primeira
tentativa chama NeuroID; retornos subsequentes reutilizam a verificação anterior. Um resultado
`NOT_ENOUGH_INTERACTION_DATA` não pode negar o cliente (AC-03). A primeira tentativa legítima deve
continuar chamando NeuroID (AC-04, regression guard).

## Observações — Ciclo 3

### O-06 — [OBSERVAÇÃO] `NOT_ENOUGH_INTERACTION_DATA` é o status padrão do ambiente qa2 (depth355)
- O site-id `depth355` do ambiente qa2 retorna consistentemente `NOT_ENOUGH_INTERACTION_DATA` quando não há
  interação suficiente com o SDK NeuroID no browser. HTTP response da API é sempre 200; o status é
  interpretado internamente pelo backend.
- Confirmado via `uown_sv_outbound_api_log` pk=8719: GET → 200 com `{"message":"success",...}`.
- O backend mapeia este status para `APPROVE` — AC-03 é exercitado em todo SUBMIT_APP no qa2.
- `[db-observation:uown_sv_outbound_api_log,pk=8719,http=200]` `[manual-validation:qa2,lead=16714]`

### O-07 — [OBSERVAÇÃO] Falha de automação CT-05: `returnToSignNoInteraction` não clicar "Sign Contract"
- O consumer portal exibe "Sign Contract" button + welcome message no retorno ao `contractUrl` (quando
  payment data já existe). O T&C só aparece após clicar o botão.
- `returnToSignNoInteraction` verificava T&C imediatamente sem clicar "Sign Contract" — timeout 15 s.
- Fix pendente na função: adicionar clique em "Sign Contract" se visível antes de checar T&C.
- NÃO impacta a feature: a evidência manual (lead 16714) prova que o signing é acessível.
- `[svc-source:spec:249-263]`

### O-08 — [OBSERVAÇÃO] Falha de automação CT-06: esign `STORED` (sweep lag)
- O signing foi completado (lead 16717 `SIGNED`). O `uown_esign_document.status=STORED` indica
  que o documento foi armazenado mas o webhook/sweep do Signwell ainda não processou a conclusão.
- A asserção `toBe('COMPLETED')` falhou por lag assíncrono, não por falha de feature.
- `[db-observation:uown_esign_document,pk=13910,status=STORED,lead=16717]`

### O-09 — [OBSERVAÇÃO] Raiz técnica do env-blocker ciclos 1 e 2 identificada
- O `sendApplication` 500 com `nested 401` origina da chamada GDS UW via `LosOutboundCall.makeRestCall`
  dentro de `UnderwritingService.routeUwRequest()`. Token Cognito expirado ou endpoint GDS instável no qa2.
- O log aparece vazio porque `AspectOutboundApiLog` faz `saveAndFlush` na mesma `@Transactional` do
  `SendApplicationService`; o rollback apaga o log junto com o lead.
- Não é bug de código de teste — infraestrutura qa2.
