- **Merchant Modification History** (`/merchantModificationHistory`)

> **Nenhum bug de aplicação foi encontrado.** Os dois achados abaixo são defeitos de **framework/spec** corrigidos durante esta validação, classificados conservadoramente. A seção `## Bugs de Aplicação Encontrados` é **omitida** porque o produto se comportou conforme esperado em todos os CTs.

### [CONFIRMADO] F-001 — `controlByLabel` capturava o controle errado no MMH

- **Tipo:** Defeito de framework (page object). **Severity:** S2 (bloqueava CT-01/CT-02). **Priority:** P1 (blocker do run, escopo de teste).
- **Sintoma:** `getMerchantSelectedCount()` retornava sempre 0 no MMH; o filtro lido era "Log Type" em vez de "Merchant".
- **Causa raiz:** O XPath `starts-with(normalize-space(.),'Merchant')` casava primeiro com a label "Merchant Ref Code" (text-input que aparece ANTES de "Merchant" na ordem do documento no MMH). Essa label não tem `filter__control` próprio, então o ancestor-walk (`ancestor-or-self::*[.//*[contains(@class,'filter__control')]][1]`) subia até o wrapper da linha de filtros inteira e devolvia o primeiro controle ("Log Type"). `[svc-source:src/pages/origination/merchant-location-filter.po.ts:188-207]`
- **Fix aplicado:** trocar o ancestor-walk por `..` (parent direto da label). O parent direto de cada label de filtro envolve exatamente o controle daquele filtro, então o escopo fica inequívoco mesmo quando múltiplas labels compartilham o mesmo prefixo `starts-with`. `[svc-source:src/pages/origination/merchant-location-filter.po.ts:203-206]`
- **Fresh repro:** sim — reproduzido nesta execução em qa2 antes do fix (CT-01/CT-02 falhavam por count=0); pós-fix os dois passaram. `[test-execution:2026-06-19]`
- **Cenário que detectou:** CT-01, CT-02.

### [CONFIRMADO] F-002 — CT-11 usava `MerchantLocationFilterPO` incompatível no Funding Queue

- **Tipo:** Defeito de spec (uso de PO errado). **Severity:** S3 (afetava só CT-11). **Priority:** P2.
- **Sintoma:** `listAvailableOptions('Location')` no Funding Queue não encontrava opções → `scrollIntoViewIfNeeded` timeout.
- **Causa raiz:** O Funding Queue renderiza as labels como `<div>` (não `<label>`) e usa IDs estáveis (`#merchantLocation`). O `MerchantLocationFilterPO` ancora por XPath de `<label>`, então não casava nada no Funding Queue. `[dom-snapshot:2026-06-19,1440x900]`
- **Fix aplicado:** adicionado `listAvailableLocations()` público em `FundingPage` (usa `fqListOptions('merchantLocation')` com o ID estável) e CT-11 atualizado para usar `funding.listAvailableLocations()`. `[svc-source:src/pages/origination/funding.page.ts]`
- **Fresh repro:** sim — CT-11 falhava antes do fix nesta execução, passou após. `[test-execution:2026-06-19]`
- **Cenário que detectou:** CT-11.

### [CONFIRMADO] F-003 — `isEmailCsvEnabled()` lia o atributo HTML `disabled` em vez da classe CSS

- **Tipo:** Defeito de framework (page object / CSV controls). **Severity:** S3 (afetava só CT-13). **Priority:** P2.
- **Sintoma:** `isEmailCsvEnabled()` usava o `isEnabled()` do Playwright (que checa o atributo HTML `disabled`). O botão "Email CSV" da app NÃO usa o atributo `disabled` — ele desabilita via classe CSS `disabledButton` + `pointer-events: none`. Logo `isEnabled()` retornava sempre `true`, mascarando o estado desabilitado e fazendo o guard data-dependent do CT-13 nunca disparar.
- **Causa raiz:** divergência entre o mecanismo de disable da app (CSS class) e a checagem do PO (atributo HTML). `[svc-source:src/pages/origination/filtered-csv-download.controls.ts:54-66]`
- **Fix aplicado:** `isEmailCsvEnabled()` passou a checar visibilidade de `SELECTORS.csvEmailButtonDisabled` (`button[class*='disabledButton']:has-text('Email CSV')`) e retornar `!cssDisabled`. `[svc-source:src/selectors/common.selectors.ts:686]` `[svc-source:src/pages/origination/filtered-csv-download.controls.ts:60-66]`
- **Fresh repro:** sim — corrigido e revalidado nesta execução; CT-13 passou (botão habilitado em qa2 no momento do run, fluxo de envio exercido até o fechamento do modal). `[test-execution:2026-06-19]`
- **Cenário que detectou:** CT-13.

## Pitfalls descobertos (devem ir para skill/catálogo via qa-doc-keeper)

> Regra #11 — requisito implícito descoberto em debug vira regra antes de fechar o pipeline. Estes pitfalls são **input obrigatório** para o `qa-doc-keeper`. Pattern source = código/skill, NÃO este report (regra #16).

1. **`controlByLabel` + `starts-with` + labels que compartilham prefixo (MMH "Merchant Ref Code" vs "Merchant").** Quando uma página de filtro tem labels cujo texto começa com o mesmo prefixo de outra label-alvo, o ancestor-walk pode subir além do controle pretendido. Usar `..` (parent direto da label) em vez de `ancestor-or-self::*[...]` mantém o escopo no controle daquela label. Já documentado em comentário no PO (`merchant-location-filter.po.ts:194-206`); candidato a entrada de pitfall em [[selector-hardening]] / [[page-object-pattern]]. `[svc-source:src/pages/origination/merchant-location-filter.po.ts:194-206]`
2. **Funding Queue usa `<div>` como label e IDs estáveis (`#merchantLocation`), divergindo do componente compartilhado `MerchantLocationFilters` (que usa `<label>`).** O `MerchantLocationFilterPO` (XPath por `<label>`) é **incompatível** no Funding Queue — usar os readers próprios da `FundingPage` (`fqListOptions`, `listAvailableLocations`). 4. **Botões "Email CSV"/"Download CSV" desabilitam via classe CSS (`disabledButton` + `pointer-events:none`), NÃO via atributo HTML `disabled`.** `isEnabled()` do Playwright retorna sempre `true` nesses botões — checar enabled/disabled exige testar a presença da classe CSS (`SELECTORS.csvEmailButtonDisabled`).
