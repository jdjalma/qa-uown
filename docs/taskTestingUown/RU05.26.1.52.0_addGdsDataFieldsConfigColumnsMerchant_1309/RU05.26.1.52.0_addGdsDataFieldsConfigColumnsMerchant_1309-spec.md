## Test Strategy

- **Approach:** **E2E, UI-first (rule #14 — MANDATORY).** The feature is a customer/agent-facing render: Config Columns panel, table columns, and a client-side-generated CSV. Per rule #14, visual render (table cells, CSV content the user downloads) CANNOT be replaced by reading the DB or API response — the Daniel's Jewelers precedent (empty placeholders only visible on render) applies directly to the NULL-cell scenario (CT-03).
  - **Setup:** API/UI-light — ensure target merchants exist with known GDS values (see Test Data Strategy). No application lifecycle needed (this feature does not touch leads).
  - **Exercise:** browser — open Config Columns, toggle checkboxes, set Active/Inactive filter, click Download CSV.
  - **Assertion (oracle):** UI table cell text + downloaded CSV content, **cross-checked against DB** (`uown_merchant.uw_pipeline / fraud_threshold / max_approval_amount`) as the value source of truth.
- **API-only justification:** N/A — feature has full UI affordance. API (`POST /uown/getMerchantsByCriteria`) is used only as a DB-equivalent cross-check oracle, never as the exercise path. DB SELECT used for value oracle (read-only, rule #9 compliant).
- **Environment:** **qa1** (feature target — must be deployed there; tests are written for and assert against qa1).
  - **CONNECTIVITY CAVEAT:** at SPEC time the qa1 DB tunnel (`127.0.0.1:5445`) returned `password authentication failed for user "svc_user"`, while qa2 on the same host:port connected fine. Column discovery below was therefore done against **qa2** (structurally representative — same app family). `qa-implementer` MUST re-verify qa1 DB connectivity (`UOWN_DB_PASS_QA1`) before the run, and the value-oracle SELECTs run against qa1. If qa1 DB stays unreachable, fall back to UI/CSV-only assertion + API `getMerchantsByCriteria` response as oracle, and flag the DB gap in the report. Also check `project_dv360_uat_qa1_outage_2026_05_18` for any active qa1 outage.
- **Suites to activate:** none beyond this task suite. No signing/dual-brand/regression-suite expansion — feature is isolated to the Merchants table view. Tag `@regression` + `@qa1`.
- **Smoke vs full:** CT-01 (panel presence) + CT-05 (Active auto-default) are the **smoke** set (P0, fast, the two highest-risk behaviors). All 9 CTs are the **full** set. DoD requires stg validation; since feature deploys to qa1 first, re-run full set on stg once promoted.
- **Parallelization:** **serialize within the file.** All CTs operate on the same `/merchant` table and toggle shared session Config Columns state (BR-01: selection is immediate, session-scoped, no Apply). Running in parallel would cross-contaminate checked-column state. Use `test.describe.serial`. Each CT must reset Config Columns to a known state at start (re-open panel, assert/define checked set) rather than assuming prior state.

## Test Data Strategy

> **Test Data Hierarchy (rule #9):** fresh data via automation is DEFAULT. Discovery confirms the env cannot be relied on for pre-populated GDS values.

**Findings (QA2, 2026-06-15 — representative; re-verify on qa1):**
- Only **1** active merchant has all three GDS fields populated: `Saslow's Jewelers` (`OW90337-0001`) → `uw_pipeline='test@!123'`, `fraud_threshold=20`, `max_approval_amount=6280.0000`.
- `terraceFinance` (named in Gherkin, `OT90202-0001_clone`) has all three **NULL** — Gherkin's named merchant is NOT valid as-is.
- Among active merchants: `uw_pipeline` set on 1, `fraud_threshold` on 2, `max_approval_amount` on 44 (but 42 of those are `0`).

**Implication & strategy:**
1. **Populated-merchant CT (CT-02, CT-04, CT-05):** do NOT hardcode `terraceFinance`. Either (a) pick a fresh merchant and SET known GDS values via `MerchantSettingPage` (`fillUwPipeline` / `fillFraudThreshold` / `fillMaxApprovalAmount` on `/merchantSetting`, or the GDS Data section on Merchant Edit) as a UI setup step, then assert those exact values downstream; OR (b) resolve the actual target merchant dynamically by DB SELECT for `uw_pipeline IS NOT NULL AND fraud_threshold IS NOT NULL AND max_approval_amount IS NOT NULL AND is_active=true` and use whatever the env returns as the oracle. **Prefer (a)** — deterministic, self-documenting values; aligns with fresh-data-via-automation default. Setting GDS values is UI-mutating but in-scope-adjacent and reversible.
   - GDS write is a DB mutation via UI form → allowed (no direct SQL). Do NOT direct-UPDATE `uown_merchant` (rule #9 / Exception 3).
2. **Empty/NULL-merchant CT (CT-03):** pick an active merchant with all three NULL (abundant — e.g. resolve dynamically by DB SELECT `... IS NULL ...`). Do not mutate it; assert empty cells.
3. **Active-set CT (CT-05/06):** the Active filter is the default page state (BR-03); the export covers all active rows. The populated merchant from (1) must appear in the Active export with correct values; the NULL merchant must appear with empty cells.

**Auto-default isolation (AC-03 — how to guarantee fields were NOT manually selected):**
- Critical for CT-05/CT-06 validity: the auto-default must be proven to come from the `isActive: true` state, NOT from a leftover manual check.
- Procedure: at CT start, **open Config Columns and assert the 3 GDS checkboxes are UNCHECKED** (read `input[name="UW Pipeline"]` etc. `value`/`checked` state). If any is checked from a prior test (serial run), uncheck it. Close panel WITHOUT touching them. THEN trigger Active export. If the CSV still contains the 3 columns → auto-default proven. For CT-06 (Inactive): same unchecked precondition, switch filter to Inactive, export, assert columns ABSENT.
- This unchecked-precondition assertion is the linchpin of the AC-03 test — it is what distinguishes "auto-defaulted by Active" from "manually left on".

**CSV validation approach:**
- Use Playwright download interception: `page.waitForEvent('download')` around the Download CSV click, then `download.path()` and read the file. Existing helper `downloads.helpers.ts` (`File download verification`) — check `references/methods.md` before adding new; extend only if a CSV-parse/column-extract helper is missing.
- Parse header row → assert presence/absence of `"UW Pipeline"`, `"Fraud Threshold"`, `"Max Approval Amount"`.
- For value assertions, locate the target merchant's data row in the CSV by `Merchant Code` and assert the 3 cell values equal the GDS oracle (DB value, format-normalized per Q2).
- File name expected `merchant-report.csv` (discovery `[confirmed]`).

### CT-02 — [P1] Table shows correct GDS values for a populated merchant
- **AC:** AC-02 · **Technique:** EP (valid/populated class).
- **Persona:** Operations agent reviewing underwriting config inline.
- **Setup:** ensure a known active merchant has GDS values (set via `MerchantSettingPage` with deterministic values, e.g. `uw_pipeline='QA-1309'`, `fraud_threshold=15`, `max_approval_amount=5000`) — fresh-data default.

- **Edge cases:** `max_approval_amount` formatting (raw vs currency) — resolve Q2 before locking exact-string assertion.
- **Pitfalls:** value-format divergence between DOM and DB; don't assert against DB raw if UI formats.

### CT-03 — [P0] New columns render EMPTY for a merchant with no GDS Data (NULL)
- **AC:** AC-02 (negative) · **Technique:** BVA (NULL boundary).
- **Persona:** same agent viewing a not-yet-configured merchant.
- **Setup:** resolve (dynamically, DB SELECT) an active merchant with all 3 GDS columns NULL; do not mutate.
- **Steps:**
  1. With the 3 GDS columns checked (from CT-02 state or re-check), view the table.
  2. `getCellForMerchant(nullMerchantCode, ...)` for each of the 3 columns.
- **Validations:** UI — cells render empty (per `[ASSUNÇÃO]`: empty string; confirm Q-G2 post-deploy — could be `—`/`0`). No `null`/`undefined`/`NaN` text leaked. CSV serialization (cross with CT-05) must not break on NULL.
- **Edge cases:** the empty-render representation is the open question — assertion is parametrized on Q-G2 resolution.
- **Pitfalls:** rule #14 / Daniel's Jewelers — this bug class is only visible on render; reading API/DB NULL would hide a UI `"null"` leak. UI assertion is mandatory.

### CT-04 — [P1] Manual selection exports correct GDS values in CSV
- **AC:** AC-02 · **Technique:** EP (export path).
- **Setup:** same populated merchant as CT-02.
- **Steps:**
  1. `openConfigColumns()`; check the 3 GDS columns; close.
  2. `downloadCsv()` → parse.
- **Validations:** CSV header includes the 3 column names · the populated merchant's row holds the correct 3 values (== DB oracle, format-normalized) · file name `merchant-report.csv`.

### CT-05 — [P0/smoke] Active export auto-includes the 3 GDS fields by default
- **AC:** AC-03 (positive) · **Technique:** Decision Table (isActive=true × manual=unchecked).
