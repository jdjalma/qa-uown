## Source Code Analysis

### RequestFilter.java (auth enforcement)

The `RequestFilter.doFilter()` at `/home/jose/projects/svc/src/main/java/com/uownleasing/svc/logging/RequestFilter.java` routes requests based on path:

1. **`/uown/los/merchant/applications*`** -- delegated to `LosMerchantApiAuthPaths.requiresLosMerchantApiAuth()` which calls `apiKeyService.validateLosMerchantAuthorization(authorizationHeader)`
2. **`/uown/svc*`** -- checks raw key against hardcoded `svcApiKey`
3. **`/uown/tms*`** -- checks raw key against hardcoded `tmsApiKey`, with fallback to `apiKeyService.isValid()`
4. **Other paths** (including `/uown/auth/*`) -- no auth check (pass-through)

### ApiKeyService.validateLosMerchantAuthorization()

At `/home/jose/projects/svc/src/main/java/com/uownleasing/svc/service/ApiKeyService.java`:

1. Blank header -> `false` (401)
2. Parses header via `ParsedAuthorization.fromHeader()`:
   - Detects `Bearer ` prefix (case-insensitive via `regionMatches`)
   - Extracts key (with or without Bearer prefix)
3. Looks up key in `uown_api_key` table via `getByKey(parsed.key())`
4. Checks expiration
5. Loads `ApiUser` via FK `api_user_pk`
6. Calls `authorizationFormatAllowed(apiUserType, bearerFormat)`:
   - `MERCHANT` -> `bearerFormat` must be `true` (Bearer required)
   - `VENDOR` -> `bearerFormat` must be `false` (raw key required)
   - `null` (legacy) -> always `true` (accepts both)

### DefaultLosApiVersionRequestWrapper

At `/home/jose/projects/svc/src/main/java/com/uownleasing/svc/logging/DefaultLosApiVersionRequestWrapper.java`:

Wraps the request to inject `X-API-Version: 2` when the header is absent. The `LosExternalMerchantController` has `headers = "X-API-Version=2"` in `@RequestMapping`, so requests without this header would get 404 without the wrapper.

### AuthController

At `/home/jose/projects/svc/src/main/java/com/uownleasing/svc/rest/AuthController.java`:

- `POST /uown/auth/authorize` -> `AuthService.authorize(request)` -> returns `GetApiKeyResponse { key, expires, error }`
- Path does NOT start with `/uown/svc` or `/uown/tms` or `/uown/los/merchant/applications`, so it goes through the filter with NO auth check (pass-through)
- `GetApiKeyRequest { username, password }` -> authenticates via `ApiUserRepo.getByInfo_UsernameAndInfo_Password()`
- On success: generates new key with 1-day expiration
- On failure: returns response with `error = "Authentication Failed. Incorrect username and/or password"`, key=null

## Edge Cases

1. **Bearer prefix case sensitivity**: `ApiKeyService.isBearerAuthorization()` uses `regionMatches(true, ...)` which is case-insensitive. `"bearer <key>"`, `"BEARER <key>"`, and `"Bearer <key>"` should all work. However, the test focuses on the standard `Bearer` format per RFC 6750. The implementation detail covers edge cases automatically -- no explicit CT needed.

2. **Expired key**: The authorize endpoint generates a key with 1-day expiration (`DateUtils.getDefaultLocalDateTimeNow().plus(1, ChronoUnit.DAYS)`). Since we use a fresh key per test run, expiration is not a concern. No CT for expired key (would require DB manipulation to set past expiration, which is out of scope).

3. **401 body format**: The `RequestFilter` returns the raw string `"Invalid API KEY"` (not JSON). The response `Content-Type` is not explicitly set in the filter, so Playwright's `parseResponse()` may return it as a string in `body` rather than JSON. The `LosPartnerApplicationClient` should handle non-JSON 401 responses gracefully.

4. **Empty Bearer token**: `Authorization: Bearer ` (with empty token after prefix) -- `ParsedAuthorization.fromHeader()` extracts empty string, `keyBlank()` returns true, validation fails. Covered implicitly by CT-03/CT-04 behavior, not a separate CT.

5. **SVC endpoint for CT-07**: The test needs a safe `/uown/svc/*` endpoint that accepts a POST with the SVC API key and does NOT have side effects. Options:
   - `POST /uown/svc/triggerScheduledTask/testConnectivity` (if it exists)
   - Or use any idempotent SVC GET endpoint
   - The implementation agent should verify which endpoint is safe to use.

6. **VENDOR user type**: The task only describes MERCHANT enforcement. Testing VENDOR behavior (raw key accepted, Bearer rejected) is out of scope since the task doesn't mention VENDOR. If needed in future, a separate task should cover it.

7. **Sub-endpoint path parameter `{id}`**: For CT-02b/c/d, the `{id}` path parameter uses a dummy value `"test"`. The auth check happens BEFORE controller dispatch, so the value doesn't matter for auth validation. Any 400/404/500 after auth passes is acceptable.
