# QA Onboarding — UOwn Leasing

---

## 1. The Product

UOwn is a **Lease-to-Own** fintech platform (rental with a purchase option). The system operates through **4 portals**:

| Portal | Function |
|--------|--------|
| **Origination** | Application pipeline — from lead to funding (17 stages), merchant pages, programs, states |
| **Servicing** | Active account management — payments, modifications, cancellations, refunds |
| **Website** | Customer portal — login, payments, viewing, updating data |
| **AMS** | Administration — user and merchant management |

The business-rules documentation covers **69+ sections** in [docs/business-rules/](business-rules/), including: application pipeline, underwriting, contracts and e-sign, financial calculations, payments (CC/ACH), funding, delinquency, and more.

## 2. The QA Work Process

The workflow follows these steps:

### a) Refinement

- We join the refinement calls to understand the scope of the task
- I write down everything that is said during the call
- If any doubt remains, I ask during the call itself. If it still remains, I send it to the people responsible (the India team) afterward

### b) Receiving the task

- The devs deliver the task marked as ready, containing the **testing steps**

### c) Manual analysis

- I run through the process manually following the testing steps
- I validate **persistence in the database** — this lets me discover new tables and assess the impact of the change

### d) Scenario creation with AI

- I take the task content + the information from my analysis and send it to the AI
- I ask it to generate test scenarios considering:
  - The task content
  - The documented business rules ([docs/business-rules/](business-rules/))
  - The backend project (svc)
  - The project the task covers (origination, servicing, etc.)

### e) Scenario review

- I re-read all the scenarios generated by the AI
- I validate whether they cover the task requirements and the findings from the manual analysis

### f) Parallel execution

- When the scenarios cover everything, I ask the AI to **create and run the automated tests**
- At the same time, I run the **manual tests** using the scenarios document as a guide

### g) Results analysis

- I analyze the results of my manual test
- I read the test report generated by the AI

### h) Report on the task

- I process the report and add it to the GitLab task
- Format: the project's own standard (defined by the [[test-report-standard]] skill)
- **We do not use Gherkin/Cucumber** — we migrated from Selenium+Cucumber to Playwright and did not add the Cucumber layer because it is not used by the development team

## 3. The Automated Testing Project

### Stack

- **Playwright** `^1.50.0` — browser automation + API tests
- **TypeScript** `^5.6.0` (strict mode) — strong typing
- **PostgreSQL** via `pg` — persistence validation
- **IMAP** via `imapflow` — OTP extraction from emails
- **Allure** — extended reports
- **GitLab CI** — continuous integration

### Structure

```
automation/
├── src/
│   ├── api/          → 19 typed REST clients + bodies + responses
│   ├── pages/        → Page Objects per portal (hierarchical inheritance)
│   ├── support/      → Fixtures, hooks, custom reporter
│   ├── config/       → Environment configuration, constants
│   ├── helpers/      → Database, email, dates, tables, navigation
│   ├── selectors/    → Centralized CSS/XPath selectors
│   ├── data/         → Test data (merchants, cards, states)
│   └── types/        → Enums and interfaces
├── tests/
│   ├── e2e/          → Browser tests (per portal)
│   ├── api/          → API-only tests (no browser)
│   ├── ci/           → Critical tests for CI/CD
│   ├── smoke/        → Smoke tests
│   └── taskTestingUown/  → Tests tied to GitLab tasks (300+)
├── docs/
│   ├── business-rules/   → Business rules (12 chapters + 6 appendices)
│   └── taskTestingUown/  → Persistent reports per task
└── .claude/
    ├── agents/       → 13 specialized AI subagents
    ├── context/      → Reference documentation for agents
    ├── commands/     → Slash commands (/qa-flow)
    └── rules/        → Per-domain rules
```

### How each test works

Every test automatically receives, via fixtures (`src/support/base-test.ts`):

| Fixture | What it provides |
|---------|---------------|
| `testEnv` | URLs, credentials, database config for the environment |
| `api` | 19 typed API clients ready to use |
| `db` | PostgreSQL pool with polling (`waitForRecord()`) |
| `email` | OTP extraction via IMAP |
| `ctx` | Shared state between steps (leadPk, accountNumber, etc.) |
| `page` | Playwright page with automatic screenshot on failure |

Validation across **3 layers**:

1. **Payload/Response** — HTTP status + body fields
2. **Database** — query the table, verify created/changed records
3. **UI** — navigate the page, verify rendering

### Available environments

`sandbox` (default), `qa1`, `qa2`, `stg`, `dev1`, `dev2`, `dev3`

### Tests per task (Task Testing)

Each GitLab task generates a directory in `docs/taskTestingUown/` following the convention:

```
{milestone}_{camelCaseTitle}_{issueNumber}
Example: R1.49.1_separateShortCodeInANewEntity_469
```

Containing:

- `{testName}.spec.ts` — the test file
- `{testName}-report.md` — persistent execution report

### AI Agents

The project uses **5 specialized agents** orchestrated by `CLAUDE.md`. Each one loads skills (in `.claude/skills/`) on demand — no slash commands; the agents decide.

| Agent | Role |
|--------|-------|
| `qa-planner` | QA Strategist — analyzes scope, validates AC, prioritizes by risk, chooses strategy, produces a SPEC |
| `qa-implementer` | QA Engineer — implements tests, page objects, API clients, helpers, DB validation |
| `qa-debugger` | QA Investigator — DOM-first, root-cause, conservative classification, feeds the pitfall catalog |
| `qa-validator` | QA Reviewer — runs tests, validates coverage vs risk, produces report |
| `qa-doc-keeper` | Knowledge Curator — updates catalogs and pitfalls (ALWAYS last) |

Standard flow per task type:

| Task signal | Sequence |
|------------------|-----------|
| New feature / new test | `qa-planner` → `qa-implementer` → `qa-validator` → `qa-doc-keeper` |
| Failing / flaky test | `qa-debugger` → (`qa-validator` if a report exists) → `qa-doc-keeper` |
| Update docs / catalog | `qa-doc-keeper` |

> Full details in [`CLAUDE.md`](../CLAUDE.md) (orchestrator) and in each `.claude/agents/{name}.md`.

## 4. End-to-end flow summary

```
Refinement → Task ready (devs) → Manual analysis (DB + impact)
    → AI generates scenarios (task + business rules + backend)
    → Scenario review
    → Parallel execution (AI = automated | QA = manual)
    → Results analysis (manual + AI report)
    → Report added to the task
```
