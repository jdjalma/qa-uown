# E2E Test Report Standard

> Standard format for test execution reports generated by `validate-results` and `qa-flow`.
> **ALL reports MUST be written in Portuguese (PT-BR).** Only technical values (code, URLs, enums) remain in original form.
> Full template and examples: [references/template.md](references/template.md)

## Reports = history, not pattern source

Reports em `docs/taskTestingUown/` sao **registros do que aconteceu em uma execucao especifica**. NAO sao fonte de patterns (regra #16).

- **Fonte de pattern de codigo** -> skills (`.claude/skills/`) + codigo (`src/`, `tests/`)
- **Fonte de pattern de dominio** -> skills ([[qa-domain-reflexes]], [[merchant-preflight]], etc.)
- **Licoes aprendidas** -> pitfalls em [[application-lifecycle]] ou skill domain-especifica
- **Categorias volatile** -> [[volatile-knowledge-registry]]

Re-leitura de report por agente permitida apenas para: (a) `qa-validator` atualizando apos nova execucao, (b) reproducao manual via leadPk/accountPk, (c) auditoria solicitada pelo user. **Inferir pattern a partir de report antigo = bug do agente.**

Todo report gerado DEVE incluir o disclaimer no topo, logo apos o `# Relatorio de Teste:` header (ver template).

## Report Structure (sections)

| Section | Required | Content |
|---------|:--------:|---------|
| Informacoes da Tarefa | Yes | Task title, URL, milestone, labels, pipeline |
| Descricao | Yes | Task description in markdown |
| Execucao do Teste | Yes | File, environment, project, date, duration, result, video/trace |
| Evidencias (Dados Utilizados/Criados) | Yes | Lead/Account PKs with role + created/existing flag |
| Capturas de Tela | Yes | Screenshot table (N/A for API-only) |
| Cenarios | Yes | CT-XX blocks (see format below) |
| Cobertura dos Requisitos | Conditional | Only when task has explicit AC/requirements |
| Bugs de Aplicacao Encontrados | Conditional | Only when bugs found |
| Resumo da Validacao | Yes | Validation checklist table |

> Full template with all sections: [references/template.md](references/template.md)

## Artifact Rules

- Create/update after EVERY execution — never skip
- Never leave PENDING values after successful execution
- If `.md` exists: preserve Informacoes da Tarefa + Descricao, update everything else
- If task metadata missing: omit rows (no placeholders)
- Sections Cobertura dos Requisitos and Bugs de Aplicacao Encontrados are conditional

## Scenario Format (CT-XX)

Each scenario MUST have:
- Header: `### CT-XX` — only prefix CT and scenario code
- `**Objetivo:**` — one sentence, what the scenario validates
- `**O que e verificado:**` — system behavior in business language, NOT test code or assertions
- `#### Como verificar manualmente` — numbered steps, specific URLs, expected values, SQL/curl/UI navigation
- Status: `**PASSOU**` or `**FALHOU**` (with `> Falha: {error}`) or `**SKIPPED**` (with `> Motivo:`)
- `---` separator after every scenario (including last)
- Values from REAL execution output — never placeholders

> Full format with examples: [references/template.md](references/template.md)

## Bug Format

Section name: always `## Bugs de Aplicacao Encontrados`. Each bug: `### BUG-{N}` with Status, Severidade, Descricao, Como Reproduzir, Evidencia, Cenario que detectou, Causa provavel. Sequential numbering (BUG-01, BUG-02). Omit section entirely if no bugs.

> Full format: [references/template.md](references/template.md)

## Screenshots

Path: `docs/taskTestingUown/{testName}/{testName}-{NN}-{desc}.png` (inside task folder, NOT `reports/`).
At least 1 per CT, taken immediately after the key assertion. API-only: no screenshot needed.

## Report Location

All reports live in `docs/taskTestingUown/` (gitignored — local only).

| File | Created by |
|------|-----------|
| `{testName}-report.md` | `qa-validator` |
| `{testName}-scenarios.md` | `qa-flow` Fase 2 |

No separate bugs file — bugs are documented inside `-report.md`.

## Source-Tagging — Evidence Provenance

Toda afirmacao tecnica num report deve carregar **tag de proveniencia**. Operacionaliza regra #10 (conservadora) e regra #16 (reports = history).

### Tag Taxonomy

| Tag | When to use |
|-----|-------------|
| `[svc-source:path:line]` | Citing backend source code |
| `[db-observation:filter]` | Citing DB row(s) read via SELECT |
| `[dom-snapshot:date,viewport]` | Citing DOM state via MCP Playwright |
| `[api-response:endpoint,status]` | Citing real API response during execution |
| `[memory:slug]` | Citing memory entry (ALWAYS add "verify against current state") |
| `[skill:name]` | Citing pitfall or knowledge from a skill |
| `[user-provided:name,date]` | Information provided by user in conversation |
| `[test-execution:run-id]` | Citing test execution result/log |
| `[doc:path]` | Citing repo documentation |
| `[external-doc:url]` | Citing external doc (Confluence, GitLab issue) |

### Where to apply (MANDATORY)

- Every `**Evidencia:**` and `**Causa provavel:**` line in bug reports
- Every `[CONFIRMADO]` classification — **without primary source tag, degrades to `[HIPOTESE]`**
- System behavior assertions in `**O que e verificado:**` (when beyond obvious)
- Every leadPk/accountPk in Evidencias table

### Degradation Rule

| Original | Only [memory] or [skill] tag | Final |
|----------|------------------------------|-------|
| `[CONFIRMADO]` | yes | `[HIPOTESE]` |
| `[HIPOTESE]` | yes | `[OBSERVACAO]` |
| `[OBSERVACAO]` | yes | (keeps) |

Memory is a dated record — may be stale. Drift-prone categories in [[volatile-knowledge-registry]] always require primary source tag.

> Full examples: [references/template.md](references/template.md)

### Cross-references

- Rule #10 (conservative bug classification): tags = foundation for `[CONFIRMADO]`
- Rule #16 (reports = history): `[memory:]` and `[skill:]` tags reinforce pattern source is outside report
- Memory `feedback_consult_svc_when_unsure`: `[svc-source:]` is the operationalization
- Skill [[volatile-knowledge-registry]]: drift-prone categories require primary source + date tag
