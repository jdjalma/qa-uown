| 146 | **A test asserting `reverse_date`/`status=REVERSED` immediately after an ACH Fully Refund fails or times out, while the equivalent CC refund passes instantly.** | CC Fully Refund is **synchronous** — completes within the same HTTP request (3 activity-log lines land in the same second: CREDIT transaction, SALE→REFUNDED transition, "Refund CC Payment complete"). ACH Fully Refund is **asynchronous** — `refundPayment` creates a **new `ACHPayment`** with `status=PENDING, achProcessType=REFUND`; the ORIGINAL payment's `reverse_date` stays `NULL` until that credit ACH settles via sweep/vendor (Profituity) — same async pattern as ACH payment creation (see [[payment-flows]] "ACH sweep chain"). Pre-existing behavior (not introduced by servicing#519/MR !700); first documented during the Sticky reverse/refund regression (CT-06) because contrasting CC vs ACH side-by-side made the asymmetry visible. | For ACH refund tests: assert the NEW `ACHPayment` row (`status=PENDING, achProcessType=REFUND`) as the immediate evidence; do NOT assert the original payment's `reverse_date` without waiting for the ACH sweep/settlement. For CC refund tests: assert immediately, no wait needed. Confirmed live sandbox 2026-06-30 (account 6042 CC $60.08 synchronous vs account 17300 ACH $164.59 asynchronous). See `docs/business-rules/05-pagamentos.md §53` "CC Refund (Synchronous) vs ACH Refund (Asynchronous)" and `docs/knowledge-base/sticky-payment-refund.md`. |
