| 144 | **`setProratedDate` (or any method targeting `input#proratedDate`) hangs on `waitForProratedResult` — result field stays `-`, no `getProrateAmount` API call fires.** **Symptom:** date is visually set in the field (screenshot confirms value), but `waitForFunction` times out after 30 s with result field showing `-`; `uown_sv_inbound_api_log` shows no `getProrateAmount` entry. **Root cause:** `input#proratedDate` is a React DatePicker (RDP) field (`type="search"`, `maxlength=10`, MM/DD/YYYY format). RDP fires Formik `onChange` when a day is selected in the calendar popup — plain `fill()` and `pressSequentially` + generic blur do NOT trigger `onChange`. **Fix (confirmed 2026-06-26, CT-02 `$705.39` / CT-03 `$1,410.77`, zero-delta vs API):** use the **native HTMLInputElement value setter + dispatchEvent** pattern — same as `fillArrangementSchedule` for arrangement date fields: `Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value')?.set` → call with target value → `dispatchEvent(new Event('input', { bubbles: true }))` → `dispatchEvent(new Event('change', { bubbles: true }))` → `el.blur()`. **Detection:** `input[type="search"][maxlength="10"]` with MM/DD/YYYY format inside a Servicing Formik modal = RDP pattern; do NOT use `fill()` or `pressSequentially`. See `src/pages/servicing/servicing-base.page.ts` → `setProratedDate` + `docs/knowledge-base/scheduled-payments.md` gap #6. |
| 145 | **`authorizeCreditCard` returns `{ "error": "Credit Card is invalid.", "preAuthStatus": "NOT_RUN" }` even with a valid card BIN and correct CVC.** **Symptom:** `preAuthStatus: "NOT_RUN"` with message "Credit Card is invalid." — initially confused with Kount IP-unavailable (also returns `NOT_RUN`), but Kount pre-auth is not even attempted. Backend log: `[UownClient][checkCCLastNameMatch] Invalid CC info(no matched the two lastnames) - card : <cardLastName>, customer : <leadLastName>`. **Root cause:** `ApplicationService.java` calls `ccLastNameMatchService.checkCCLastNameMatch(ccInfo, lead)` before invoking Kount. If the `ccLastName` submitted to the CC form (`#ccLastName`) does NOT match the lead's `uown_los_customer.last_name`, the backend calls `declineCreditCard()` immediately and skips Kount entirely. This fires on the consumer portal CC page (`/{shortCode}/complete`) and on direct `POST /uown/los/authorizeCreditCard` calls. **Most common test cause:** using `buildTestData({ realistic: true })` generates a random `lastName` per call; if the CC body is built with a different (e.g. hardcoded "Bell" or a separate `randomApplicant()` call) lastName, the match fails. **Fix:** (a) use `buildTestData({ realistic: false })` → deterministic `lastName = "TestLNufl"` — then reuse the same value in the CC body; or (b) capture `applicant.lastName` from the `sendApplication` step and pass it explicitly to both the CC form and the API body. **Source:** `svc/src/main/java/com/uownleasing/svc/service/ApplicationService.java` lines 173-175 + `ccLastNameMatchService.checkCCLastNameMatch`. Confirmed live stg leads 7218254/7218270 + qa2 leads 16649/16651 (AL/NY GowSign investigations). |
