## How to use this skill

1. Before asserting anything about a category listed below, **consult the indicated primary source**
2. Cite the source with a provenance tag (see [[test-report-standard]] section 9)
3. If memory conflicts with the current primary source, **update the memory** (do not trust stale memory)
4. If the primary source is inaccessible, classify the finding as `[HIPÓTESE]` at most (never `[CONFIRMADO]`)
5. To check the **current volatility** of a specific file, read `docs/business-rules/_index.md` or `docs/knowledge-base/_index.md` (the `Volatilidade` column, regenerated by `node scripts/docs-tooling.mjs index`) — do not trust this skill's static list as the source of current volatility

### 2. Sweep SQL (uown_scheduled_task)

**Why volatile:** SQL is editable via an admin endpoint; drift across sandbox/qa1/qa2/stg is frequent; the `sticky.io` lib and svc MRs diverge.

**Primary source:**
- Current SQL: query `SELECT scheduled_task_name, sql_to_pick_accounts, cron_trigger FROM uown_scheduled_task WHERE scheduled_task_name='{X}'`
- Expected spec: read source code in `svc` (`@ScheduledTask` annotations)
- Updates via: `POST /uown/createOrUpdateSqlConfig` (see memory `reference_sqlconfig_admin_endpoint`)

**Real schema (confirmed qa1 2026-06-10, S2):**
- `uown_scheduled_task`: name column = **`scheduled_task_name`** (NOT `name`); cron = **`cron_trigger`** (NOT `cron`); **`last_trigger_time` reflects only MANUAL triggers, NOT Quartz executions** — the real-execution oracle = `qrtz_triggers.prev_fire_time` + `uown_sweep_logs`.
- `uown_sweep_logs`: real columns = **`sweep_name`**, **`number_of_records_processed`**, **`error`**, **`source`**, **`start_time`** / **`end_time`** (NOT `task_name`/`processed`/`created_timestamp`). `number_of_records_processed` is written AFTER processing (immediate read = 0; app-lifecycle pitfall #87).
- See app-lifecycle pitfalls #94 (`uown_scheduled_task`) and #95 (`uown_sweep_logs`).

**Known drift-prone cases:**
- `StickyRecoverCancelSweep` (sandbox pk81, cron `0 0 17 * * ?`) — `sql_to_pick_accounts` = `sticky_transaction_id NOT NULL AND account_status <> 'ACTIVE' AND recovery_status NOT IN ('RECOVERED','FAILED','CANCELED')`. **`ScheduledTaskClient.createOrUpdateScheduledTask` mutates this SQL** (narrow-to-one-row in tests) → ALWAYS restore via snapshot (`try/finally`). See `docs/knowledge-base/sticky-recover-cancel-sweep.md`.

**Tag to use:** `[db-observation:uown_scheduled_task WHERE scheduled_task_name=...]` + `[svc-source:{TaskClass}.java]`

**Related memories:** `project_sticky_io_485_drift`, `project_svc_485_sticky_recover`, `reference_get_scheduled_task_by_name`, `reference_sql_config_uppercase`

### 11. Sentry Session Replay — semantics per-tab (not per-application)

**Why volatile:** the `replayId` returned by `Sentry.getReplay?.getReplayId` is per-browser-session/tab, not per-application. Tests that assert UUID uniqueness between two apps in the same tab are incorrect. Behavior confirmed in qa1 2026-05-22 — two distinct leads in the same tab shared the same `uuid` in `uown_lead_recording`. Easy to get wrong when specifying ("each app's recording must be distinct").

**Primary source:**
- Sentry public docs about Session Replay (replay scope = browser session)
- DB `uown_lead_recording`: column `uuid` = Sentry replay ID; presence of a row = recording linked; `uuid` uniqueness across rows of distinct tabs (NOT the same tab)
- Table: `SELECT pk, lead_pk, uuid FROM uown_lead_recording WHERE lead_pk IN (...)` — same tab → same uuid; separate tabs → distinct uuid (the Playwright fixture creates a new context per test)

**Assertion rule:**
- AC "recording link present" = row EXISTS in `uown_lead_recording` for the `leadPk` → `expect(uuid).toBeTruthy`
- Do NOT assert `uuid1 !== uuid2` when both leads were created in the same Playwright tab
- For same-tab tests (CT-01/CT-02 of #1291): UUID equality is EXPECTED and confirms the Sentry SDK is running normally

**Tag to use:** `[db-observation:uown_lead_recording WHERE lead_pk=...]` + `[sentry-docs:session-replay-scope]`

**Related memories:** `reference_sentry_replay_per_tab` (2026-05-22)
