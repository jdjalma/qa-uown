# Test Report (readable by all areas)

Transforms the test scenarios and coverage of a demand into an **executive report in plain language, free of technical jargon**, that any team can understand. Takes the output of `/test-scenarios` (Gherkin scenarios + AC→scenario matrix in `docs/scenarios/`) and the **reported results** of execution, and produces a document that answers, in 30 seconds: *what was tested, what works, and whether it can be released.*

> **This skill does not execute tests.** The results of each scenario (passed / passed with caveat / failed / not tested) are **reported** by whoever ran the tests. The skill focuses on **consolidating and communicating** — never on determining results independently, and **never on inventing results**.

## Process

1. **Understand the input.** Locate and read the scenarios file (`docs/scenarios/<demand>.md`, output of `/test-scenarios`) — if only the demand name is provided, search with Glob for `docs/scenarios/**/*.md`. Extract: the **demand**, the **Acceptance Criteria (AC)** with IDs, the **scenarios**, and the **AC→scenario matrix**.
2. **Obtain the results.** For each scenario, record the **reported** result: ✅ meets, ⚠️ meets with caveat, ❌ does not meet, or — not tested; when ⚠️/❌, note the **observed problem**. Result missing? Apply the [missing-result protocol](#missing-result-protocol). **Never invent.**
3. **Consolidate status per AC** from the scenarios that cover it (see [consolidation rules](#ac-status-consolidation)) and calculate the **% of ACs met**.
4. **Define the overall verdict** (🟢/🟡/🔴) according to the [verdict rules](#overall-verdict).
5. **Translate to business language.** Each scenario becomes a **behavior sentence**; each failure becomes a **user/business impact** (see [language rules](#plain-language-rules) and [references/translation-guide.md](references/translation-guide.md)).
6. **Build the report** by copying [assets/report-template.md](assets/report-template.md) and filling it from the top (verdict) down (technical appendix).
7. **Self-validate** against the language rules and the [checklist](#self-validation-before-saving). Found jargon, or a status painted green without basis? Fix it and re-validate. **Only proceed when it passes.**
8. **Save the `.md`** to `docs/reports/<demand>.md` and report the path.

```
[ ] Scenarios file read (demand, ACs, scenarios, matrix)
[ ] Result of each scenario obtained (without inventing; absent = Not tested)
[ ] Status per AC consolidated + % of ACs met calculated
[ ] Overall verdict defined (🟢/🟡/🔴) with explicit criterion
[ ] Scenarios translated to behavior; failures translated to business impact
[ ] No technical jargon (language self-validation)
[ ] Honesty: what was not tested is declared; no "watermelon reports"
[ ] .md saved in docs/reports/ and path reported
```

## Plain language rules

The most important section. The report serves readers who are **not** technical.

1. **Non-technical audience.** The reader understands without any knowledge of software testing. Self-check per sentence: *would a business person understand this without asking?*
2. **Jargon is forbidden.** No "Gherkin", "BDD", "Given/When/Then", "test case", "regression", "smoke", "mock", "flaky", "endpoint", "error code", "stack trace", module/class names. Unavoidable concept? Translate it ([references/translation-guide.md](references/translation-guide.md)).
3. **Scenario becomes behavior.** Each scenario is a user-observable behavior sentence (*"the customer is able to register with valid data"*), never the technical steps.
4. **Failure becomes impact.** Describe the consequence for the user/business (*"the customer loses their cart items when refreshing the page"*), not the technical error.
5. **Status in business language** — use the [status translation table](#status-translation).
6. **Anchor in the AC text.** Repeat the Acceptance Criterion text so the business recognizes what was agreed upon.
7. **Verdict first, detail later.** Conclusion at the top; technical detail only in the appendix.
8. **Concise.** Short sentences, one idea per sentence.
9. **Honesty above all (anti-"watermelon report").** Never mark ✅/🟢 something with a relevant caveat or that was not tested. Always declare what was left out. Green on the outside with a problem on the inside destroys trust in the report.

## Traffic light (fixed legend)

**Always** use these four signals, with labels (not color alone, for accessibility):

- ✅ **Meets** — works as expected.
- ⚠️ **Meets with caveat** — works, but with a limitation that needs attention.
- ❌ **Does not meet** — does not work as expected.
- — **Not tested** — was not possible to verify.

### Status translation

| Execution result (technical) | In the report (business) | Signal |
|---|---|---|
| Passed | Works as expected · Meets | ✅ |
| Passed with observation | Works with caveat | ⚠️ |
| Failed | Does not work as expected · Does not meet | ❌ |
| Blocked / not executed | Was not possible to test · Not tested | — |

## AC status consolidation

The status of each Acceptance Criterion comes from the scenarios that cover it, **in this priority order**:

1. ❌ **Does not meet** — if **any** scenario for the AC failed.
2. ⚠️ **Meets with caveat** — if none failed, but there is a scenario with a caveat **or** part of the AC's scenarios were not tested (partial coverage).
3. — **Not tested** — if **no** scenario for the AC was tested, **or** the AC has no linked scenario (gap → record under "What was not verified").
4. ✅ **Meets** — if **all** scenarios for the AC meet, with no caveat.

**% of ACs met** = number of ✅ ACs ÷ total ACs. Also report how many are ⚠️, ❌, and —.

## Overall verdict

- 🟢 **Ready to release** — all ACs ✅; nothing pending testing or open issues.
- 🟡 **Ready with caveats** — ACs met with ⚠️, or untested items of low impact; no ❌ that prevents use. **List the caveats.**
- 🔴 **Not ready** — at least one AC ❌ that prevents use, or insufficient coverage to decide safely.

> The **form** is rigid (`.md` structure, traffic light, no jargon); the **judgment** of severity (what counts as "low impact") is open — but **make the criterion explicit** in the report.

## Missing result protocol

Do not invent results. When a scenario's result has not been reported:

- **Someone can provide it** → **ask the user** for the results of the pending scenarios (and the observed problem, when ⚠️/❌).
- **No way to obtain it** → mark **— Not tested** and declare it in the "What was not verified" section. Never assume it passed.
- **Orphan scenario** (without AC) or **AC without scenario** → record as an explicit gap; do not hide it.

## Self-validation (before saving)

- [ ] No technical term leaked (scan for: Gherkin, Given/When/Then, regression, endpoint, error NNN, module name).
- [ ] Each scenario is expressed as a **behavior sentence**, not as steps.
- [ ] Each failure is described as **impact** on the user/business.
- [ ] Verdict, % coverage, and number totals match the AC×result table.
- [ ] Nothing incorrectly marked ✅/🟢; what was not tested is declared.
- [ ] Traffic light legend present before the first use of signals.

## Output

**Copy [assets/report-template.md](assets/report-template.md) and fill it in.** Save to `docs/reports/<demand>.md` (kebab-case), or follow the project's convention if one exists — check with Glob for `docs/reports/**/*.md` first. If a report already exists for the same demand and the team wants to keep a version history, include the date in the name (`<demand>-yyyy-mm-dd.md`). After saving, report the path. If the user only wants to see it in the chat, present it without saving.

## Integration with the scenarios skill (`/test-scenarios`)

This skill **consumes** what `/test-scenarios` produces (`docs/scenarios/<demand>.md`: scenarios + AC→scenario matrix). Natural flow: `/test-scenarios` writes the scenarios → the team executes and records results → `/test-report` transforms everything into a document readable by all areas.

## Distinction from the project's technical reports

This report serves the **management/product/business audience** — no technical jargon. It is distinct from the two technical artifacts in the automation pipeline:

| Artifact | Saved to | Audience | Generated by |
|---|---|---|---|
| This report (`/test-report`) | `docs/reports/<demand>.md` | Management / product / stakeholders | This skill |
| Evidence (`task-evidence-report`) | `docs/taskTestingUown/{task}/{task}-evidence.md` | GitLab/Jira — QA validation proof | `qa-validator` when closing the pipeline |
| Technical report (`test-report-standard`) | `docs/taskTestingUown/{task}/{task}-report.md` | Internal QA — execution history | `qa-validator` after each execution |

All three can coexist for the same demand — each one serves a different reader.
